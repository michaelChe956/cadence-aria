# 项目管理工作台与 Issue 全生命周期设计

> 版本: v1.1 | 日期: 2026-05-14

## 1. 概述

将 Cadence Aria 从单一任务执行工具升级为项目管理工作台，引入项目、代码库（原 Workspace）、Issue 全生命周期管理，重新定义需求澄清和代码开发两大流程。

### 1.1 核心变更

- 新增项目管理概念（Project → Repository → Issue 层级）
- Workspace 概念重命名为代码库（Repository）
- 首页重构为项目管理工作台
- Issue 全生命周期：需求澄清 → 代码开发 → 测试验收（测试验收为占位阶段，本版本不进入执行链）
- 需求澄清流程产出 Story Spec + Design Spec
- 代码开发流程产出 Work Item + Plan + 代码变更
- Provider 执行过程流式透明展示

### 1.2 v1.1 修订边界

本版本在 v1.0 基础上补齐评审中发现的阻塞问题，采用“产品索引与执行真相源分层”的兼容方案：

- `~/.aria/` 只保存项目、代码库、Issue、最近打开项目、用户偏好等产品级索引。
- `{repo}/.aria/runtime/` 继续作为 task、checkpoint、provider runs、artifacts、projection 的执行真相源。
- Issue 通过 `IssueRuntimeBinding` 指向现有 runtime task，不直接替代 runtime task。
- 不要求一次性迁移已有 `.aria/runtime` 数据；首次打开时可从现有 runtime 重建产品索引。
- 测试验收阶段在 UI 中可灰显展示，但本版本状态机不自动进入 `Acceptance`。

### 1.3 设计决策记录

| 决策项 | 结论 |
|--------|------|
| 项目/代码库/Issue 关系 | 项目 1:N 代码库，Issue 属于项目但绑定单个代码库 |
| Agent 协作模式 | Claude Code 生成 + Codex CLI review，复用现有 provider 机制 |
| Design spec 前后端 | 动态决定，后端先行，前端基于后端设计 |
| 数据存储位置 | `~/.aria/` 保存产品索引，`{repo}/.aria/runtime/` 继续保存执行数据；不做破坏性迁移 |
| Worktree 时机 | 只有代码开发阶段才开 worktree |
| 首页交互 | 有最近项目直接进入，没有则引导创建 |
| 架构复用 | 在现有 runtime_units 架构上扩展 |
| Work item 粒度 | 细粒度（一个 agent session 可完成） |
| Dispatch | 去掉，保留 plan，plan 需人工确认 |
| Work item 执行 | 依赖 DAG + 写范围冲突检测共同决定可并行集合 |
| Provider 选择 | Per-node 级别，用户可从前端选择；静态节点契约给出推荐与允许范围 |
| 回退机制 | 按执行记录回退，soft-delete + 状态恢复；需求阶段恢复 spec 版本，开发阶段恢复 runtime checkpoint 并 reset 对应 worktree |

### 2.1 核心实体

```rust
// 代码库（原 Workspace）
struct Repository {
    id: Uuid,
    project_id: Uuid,
    name: String,
    path: PathBuf,                  // 本地 git 仓库路径
    repo_hash: String,              // canonical path hash，用于 worktree / runtime 定位
    runtime_root: PathBuf,          // 派生自 path/.aria/runtime
    default_policy_preset: String,
    default_provider_mode: String,
    created_at: DateTime,
    updated_at: DateTime,
}

// Issue（议题）
struct Issue {
    id: Uuid,
    project_id: Uuid,
    repo_id: Uuid,                  // 绑定的代码库
    title: String,
    description: Option<String>,
    change_id: String,             // OpenSpec / runtime change 标识
    phase: Phase,                  // 当前产品阶段
    status: IssueStatus,
    active_binding_id: Option<Uuid>,
    created_at: DateTime,
    updated_at: DateTime,
}

// Issue 与现有 runtime task 的绑定关系
struct IssueRuntimeBinding {
    id: Uuid,
    issue_id: Uuid,
    repo_id: Uuid,
    change_id: String,
    task_id: Option<String>,
    session_id: Option<String>,
    runtime_root: PathBuf,          // {repo}/.aria/runtime
    task_root: Option<PathBuf>,     // {repo}/.aria/runtime/tasks/{task_id}
    status: RuntimeBindingStatus,
    created_at: DateTime,
    updated_at: DateTime,
}

// 每次 provider 或人工 gate 操作的产品级执行记录
struct ExecutionRecord {
    id: Uuid,
    issue_id: Uuid,
    binding_id: Uuid,
    node_id: String,
    executor_config: Option<ExecutorConfig>,
    gate_id: Option<Uuid>,
    status: ExecutionStatus,
    before_state: StateSnapshot,
    after_state: Option<StateSnapshot>,
    runtime_checkpoint_id: Option<String>,
    output_ref: Option<String>,
    dropped: bool,
    created_at: DateTime,
    updated_at: DateTime,
}
```

`Issue` 是产品对象，不能直接替代现有 runtime task。所有 task、checkpoint、provider run、artifact、projection 的执行真相仍通过 `IssueRuntimeBinding` 指向仓库内 `{repo}/.aria/runtime/`。

#### 2.4.2 执行真相源：`{repo}/.aria/runtime/`

```
{repo}/.aria/runtime/
├── tasks/
│   └── {task_id}/
│       ├── state.json
│       ├── projection.json
│       ├── checkpoints/
│       ├── provider-runs/
│       ├── artifacts/
│       └── reports/
└── web/
    ├── workspaces.json   # 兼容现有版本，后续由产品索引替代
    └── issues.json       # 兼容现有版本，后续由产品索引替代
```

#### 2.4.3 兼容策略

- 新工作台读取 `~/.aria` 产品索引；若索引缺失，可从已登记的 repo `{repo}/.aria/runtime/tasks/` 重建 Issue-runtime 绑定。
- 现有 runtime 文件不移动、不删除、不改写路径语义。
- 产品索引中的 `ExecutionRecord` 只保存摘要、状态、指针和回退边界；provider stdout/stderr、artifact 正文仍以 runtime task root 为准。
- 同一 Issue 允许多次 runtime binding，但同一时刻只能有一个 `active_binding_id`。

## 3. 流程编排与节点设计

### 3.1 节点定义

**需求澄清阶段：**

| 节点 | 作用 | Provider | 硬 Gate |
|------|------|----------|---------|
| N04 | 用户需求初步澄清 | Claude Code | 否 |
| N05 | 生成 story spec | Claude Code | 否 |
| N06 | Review story spec（最多 2 次 review） | Codex | 否 |
| N06.5 | Story spec 人工确认 | 无 | **是** |
| N07 | 生成后端 design spec | Claude Code | 否 |
| N08 | Review 后端 design spec（最多 2 次 review） | Codex | 否 |
| N07b | 生成前端 design spec（可选） | Claude Code | 否 |
| N08b | Review 前端 design spec（最多 2 次 review） | Codex | 否 |
| N09 | Design spec 人工确认 | 无 | **是** |

**代码开发阶段：**

| 节点 | 作用 | Provider | 硬 Gate |
|------|------|----------|---------|
| N10 | 生成 work items | Claude Code | 否 |
| N10.5 | Work item 人工确认（标注执行方式/依赖） | 无 | **是** |
| N11 | 为当前 work item 生成 plan | Claude Code | 否 |
| N11.5 | Plan 人工确认 | 无 | **是** |
| N16 | 编码执行 | 按节点执行配置 | 否 |
| N17 | 测试 | 按节点执行配置 | 否 |
| N18 | 代码审查 | 按节点执行配置 | 否 |
| N19 | 返工（loop back to N17） | 按节点执行配置 | 否 |

### 3.2 链路编排

```
run_clarification_chain:
  N04 → N05 → N06(review_attempts≤2) → N06.5[硬gate]
       → N07 → N08(review_attempts≤2)
       → [if frontend needed] N07b → N08b(review_attempts≤2)
       → N09[硬gate]

run_development_chain:
  N10 → N10.5[硬gate]
  Per work_item (依赖 DAG ready 且 allowed_write_scope 不冲突):
    创建/绑定外置 worktree
    → N11 → N11.5[硬gate]
    → N16 → loop { N17 → N18 → if blocking: N19 → back to N17 }

acceptance_chain:
  本版本不执行；Issue 完成代码开发后停留在 Development/Completed，
  Acceptance 仅在 UI 进度条中灰显占位。
```

### 3.3 Review 循环机制

```
review_loop(generate_node, review_node, max_review_rounds=2):
  run generate_node
  for attempt in 1..=max_review_rounds:
    run review_node
    if review_result == "pass":
      break
    if attempt == max_review_rounds:
      标记为 needs_human_review，推送到前端
      break
    run generate_node (基于 review feedback 修订)
```

`max_review_rounds=2` 表示最多执行 2 次 review，而不是最多 2 次返修。若两次 review 后仍未通过，流程暂停到前端人工处理。

### 3.4 硬 Gate 机制

与现有 `PausedForApproval` 不同，硬 gate 无条件暂停，不受 PolicyPreset 控制。硬 gate 必须持久化为 `GateRecord`，并写入对应 `ExecutionRecord`。

```rust
enum GateType {
    PolicyControlled,  // 现有行为，受 PolicyPreset 控制
    HardGate,          // 无条件暂停等人工确认
}
```

硬 gate 节点行为：
1. 将当前产物推送到前端展示
2. 暂停流程，等待用户操作
3. 用户可以：确认继续 / 要求修改（回退到 generate 节点）/ 终止

后端最小 API：

```
POST /api/issues/{issue_id}/gates/{gate_id}/confirm
POST /api/issues/{issue_id}/gates/{gate_id}/request-change
POST /api/issues/{issue_id}/gates/{gate_id}/terminate
```

SSE 事件：

```
gate.opened {
  issue_id,
  gate_id,
  node_id,
  artifact_refs
}

gate.resolved {
  issue_id,
  gate_id,
  node_id,
  decision: "confirmed" | "change_requested" | "terminated"
}
```

Gate 状态恢复规则：
- 服务重启后从 `~/.aria/projects/{project_id}/issues/{issue_id}/gates/` 恢复所有 `Open` gate。
- 同一 binding 同一节点最多允许一个 open hard gate。
- `request-change` 必须记录用户修改意见，并把流程路由回对应 generate 节点。
- `terminate` 只终止当前 Issue 绑定，不删除历史执行记录。

## 5. Provider 执行流式展示

### 5.1 执行卡片模型

每次 provider 调用展示为一个执行卡片，包含三段：

```
┌─ Provider 执行卡片 ─────────────────────────────────┐
│ [Claude Code]  N05 story_spec_generate   ● 执行中    │
├─────────────────────────────────────────────────────┤
│ ▶ 输入 (可折叠)                                      │
│   输入摘要 + 可审计 input_full_ref                  │
├─────────────────────────────────────────────────────┤
│ ▶ 实时输出 (流式，类终端)                             │
│   逐行/逐 chunk 展示 agent 输出                      │
├─────────────────────────────────────────────────────┤
│ ▶ 产物输出 (完成后，Markdown 渲染)                    │
│   最终生成的 spec / plan / code                      │
└─────────────────────────────────────────────────────┘
```

### 5.2 SSE 事件增强

```rust
// 新增事件类型
ProviderInputPrepared {
    node_id: String,
    provider: String,           // "claude_code" | "codex"
    input_summary: String,
    input_full_ref: Option<String>,  // 指向 runtime task root 下的持久化文件
    redaction_applied: bool,
}

ProviderOutputStream {
    node_id: String,
    provider_run_id: String,
    chunk: String,
    timestamp: u64,
}

ProviderOutputCompleted {
    node_id: String,
    provider_run_id: String,
    artifact_type: String,
    artifact_summary: String,
    artifact_ref: Option<String>,
}

ReviewRoundStarted {
    node_id: String,
    attempt: u32,
    reviewer: String,
}

ReviewRoundCompleted {
    node_id: String,
    attempt: u32,
    result: String,             // "pass" | "revise" | "needs_human"
    feedback: Option<String>,
}
```

事件流约束：
- SSE payload 不直接携带完整 system prompt、上下文包或大段文件内容。
- `input_full_ref` 指向 runtime task root 内文件；前端按需读取，并受权限、大小和脱敏策略限制。
- provider 输出按 `provider_run_id` 归档，支持前端刷新后按 cursor resume。
- 默认 replay 最近 512 条内存事件；历史详情从 runtime task root 中的持久化日志读取。

```
后端:
  spawn agent process → 捕获 stdout pipe
  → 逐 chunk 读取 → 写 provider-output.jsonl → SSE 推送 ProviderOutputStream
  → 进程结束 → 解析产物 → 推送 ProviderOutputCompleted

前端:
  SSE EventSource 监听（带 last_event_id/cursor）
  → ProviderOutputStream → append 到 StreamSection
  → ProviderOutputCompleted → 渲染 ArtifactSection
```

### 5.5 输入与输出安全边界

- 输入全文展示前必须做 secret redaction，至少覆盖 API key、token、私钥、cookie、Authorization header。
- 单个输入全文默认最大展示 200KB，超过后展示摘要和文件引用。
- Provider stdout/stderr 默认按流式输出展示，但历史加载从持久化日志分页读取。
- 产品索引只保存摘要和引用，不复制 provider 原始输入输出正文。

## 6. Provider 选择机制

### 6.1 设计原则

每个使用 provider 的节点，用户可以从前端选择使用哪个 agent。系统提供默认推荐，但用户可以覆盖。

### 6.2 选择粒度：Per-Node（每次执行）

参考 vibe-kanban 的 per-action 模式，executor 选择嵌入每次执行记录中。静态 `NodeExecutionContract` 负责声明推荐 executor 与允许 executor，`ExecutionRecord` 负责记录本次实际选择。

```rust
struct NodeExecutorPolicy {
    node_id: String,
    recommended_executor: AgentType,
    allowed_executors: Vec<AgentType>,
    required_capabilities: Vec<String>,
}

// ExecutionRecord 完整字段见 2.1；此处只强调 executor 选择字段。
struct ExecutionRecord {
    id: Uuid,
    issue_id: Uuid,
    binding_id: Uuid,
    node_id: String,              // "N05", "N06", "N16" 等
    executor_config: Option<ExecutorConfig>,
    status: ExecutionStatus,
    before_state: StateSnapshot,  // 执行前状态快照
    output_ref: Option<String>,
    created_at: DateTime,
}

struct ExecutorConfig {
    executor: AgentType,          // claude_code | codex | gemini | ...
    model_id: Option<String>,     // 具体模型
    variant: Option<String>,      // 预设变体
}
```

### 6.3 默认推荐

| 节点类型 | 推荐 Agent | v1.1 当前落地默认 | 说明 |
|---------|------------|------------------|------|
| spec/design/work item/plan generate（N05/N07/N10/N11） | Claude Code | Claude Code | 生成类任务 |
| review 类（N06/N08/N18） | Codex | Codex | 审查类任务 |
| coding（N16） | Claude Code | Codex | 现有契约中 N16 为 Codex；Claude Code 作为允许 executor 需通过能力矩阵后启用 |
| testing（N17） | Claude Code / Codex | Codex | 现有契约中 N17 为 Codex；后续允许用户覆盖 |
| rework（N19） | Claude Code | Codex | 现有契约中 N19 为 Codex；Claude Code 作为目标推荐 |

用户可以在节点执行前通过前端 Agent Picker 覆盖默认选择，但只能选择 `allowed_executors` 中且可用性检查通过的 executor。

### 6.4 前端交互

```
┌─ 节点执行前 ─────────────────────────────────┐
│ N05 Story Spec 生成                           │
│                                              │
│ Agent: [Claude Code ▼]  Model: [sonnet ▼]    │
│                                              │
│ [▶ 开始执行]                                  │
└──────────────────────────────────────────────┘
```

- 硬 gate 节点（N06.5/N09/N10.5/N11.5）不需要 agent 选择（纯人工确认）
- 自动执行的节点（PolicyPreset 允许自动运行时）使用节点落地默认 agent，但用户可以在项目/代码库设置中修改默认值
- 若用户选择的 executor 与节点能力矩阵不兼容，开始执行按钮置灰并展示原因

### 6.5 可用性检查

启动时检查已安装的 agent CLI 工具：
- Claude Code: 检查 `claude` 命令是否可用
- Codex: 检查 `codex` 命令是否可用
- 其他: 按需扩展

不可用的 agent 在选择器中灰显，提示安装方式。

可用性检查结果必须同时包含：
- CLI 是否存在
- 是否已登录或授权
- 当前节点所需能力是否支持（读、写、测试、review、流式输出）
- 当前 sandbox / worktree / allowed_write_scope 是否满足 executor 要求

## 7. 执行回退机制

### 7.1 设计原则

参考 vibe-kanban 的 ExecutionProcess soft-delete + state restore 模式。每次 provider 调用或硬 gate 决策记录为一个执行记录，包含执行前的产品状态快照和可选 runtime checkpoint。回退时先 soft-delete 产品索引中的目标及之后记录，再按阶段恢复 spec 版本或 runtime checkpoint。

### 7.2 执行记录与状态快照

```rust
struct ExecutionRecord {
    id: Uuid,
    issue_id: Uuid,
    binding_id: Uuid,
    node_id: String,
    executor_config: Option<ExecutorConfig>,
    gate_id: Option<Uuid>,
    status: ExecutionStatus,       // running | completed | failed | dropped
    before_state: StateSnapshot,   // 执行前快照
    after_state: Option<StateSnapshot>, // 执行后快照
    runtime_checkpoint_id: Option<String>,
    output_ref: Option<String>,
    created_at: DateTime,
    updated_at: DateTime,
    dropped: bool,                 // soft-delete 标记
}

// 状态快照内容因阶段不同
enum StateSnapshot {
    // 需求澄清阶段：记录 spec 产物版本
    ClarificationState {
        story_spec_version: Option<u32>,
        design_spec_backend_version: Option<u32>,
        design_spec_frontend_version: Option<u32>,
    },
    // 代码开发阶段：记录 git commit SHA
    DevelopmentState {
        work_item_id: Uuid,
        worktree_path: PathBuf,
        before_head_commit: String,  // worktree 的 HEAD SHA
        runtime_checkpoint_id: Option<String>,
    },
}
```

### 7.3 回退流程

**需求澄清阶段回退：**

```
用户在前端点击某个执行记录的"回退到此步骤"
  → soft-delete 该记录及之后的所有记录（dropped = true）
  → 恢复 spec 产物到该记录的 before_state 版本
  → 恢复 issue.phase / status / gate 状态
  → 流程状态回退到该节点，等待重新执行
```

不涉及 git 操作，只是 spec JSON 文件的版本回退。

**代码开发阶段回退：**

```
用户在前端点击某个执行记录的"回退到此步骤"
  → soft-delete 该记录及之后的所有记录
  → 若存在 runtime_checkpoint_id，先恢复 runtime state/projection
  → git reset 对应 worktree 到 before_head_commit
  → 流程状态回退到该节点
```

涉及 git reset，需要用户确认（参考 vibe-kanban 的 RestoreLogsDialog）。开发阶段禁止在 repo root 执行 reset，只允许在该 work item 绑定的 worktree 路径内执行。

### 7.4 前端交互

每个执行卡片上显示回退按钮：

```
┌─ 执行记录 #3 ─────────────────────────────────┐
│ [Claude Code] N07 design_spec_backend  ✓ 完成  │
│ ...                                           │
│                              [↩ 回退到此步骤]  │
└───────────────────────────────────────────────┘
┌─ 执行记录 #4 ─────────────────────────────────┐
│ [Codex] N08 design_spec_backend_review ✓ 完成  │
│ ...                                           │
│                              [↩ 回退到此步骤]  │
└───────────────────────────────────────────────┘
```

点击回退后弹出确认对话框：
- 显示将被回退的执行记录列表
- 代码开发阶段额外显示 git reset 信息
- 如果 worktree 有未提交变更，需要用户确认是否强制回退
- 显示将恢复的 `runtime_checkpoint_id` 与 worktree path

### 7.5 回退后的状态

回退后：
- 被 soft-delete 的记录在前端以灰色/删除线展示（可选隐藏）
- 流程从回退点重新开始，用户可以选择不同的 agent 重新执行
- 回退不删除数据，保留完整历史（审计追溯）
- 回退后的新执行记录继续追加，不复用已 dropped 的记录 ID

## 8. Worktree 管理

### 8.1 目录结构

Git worktree 不放在 repo root 内，避免污染主工作区：

```
<repo_parent>/.aria-worktrees/
└── <repo_name>-<repo_hash>/
    └── <issue_short_id>/
        ├── <work_item_short_id>/
        └── integration/          # 后续集成阶段预留，本版本不自动合并
```

`repo_hash` 使用代码库 canonical path hash，与 `Repository.repo_hash` 一致。

### 8.2 分支命名

```
aria/{issue_short_id}/{work_item_short_id}
例: aria/001/wi-001
```

### 8.3 生命周期

```
Work item 确认为 agent 可执行
  → 依赖 DAG ready
  → allowed_write_scope 与活跃 work item 不冲突
  → 创建外置 worktree（基于默认分支）
  → Plan 生成 → 人工确认
  → Coding → Testing → Review → Rework loop
  → 完成：保留 worktree 等待用户合并/清理
```

### 8.4 并行执行

```
Work items: [WI-001, WI-002, WI-003]
依赖: WI-002 depends_on WI-001, WI-003 独立

执行:
  并行组1: WI-001 + WI-003 (各自 worktree，同时执行)
  并行组2: WI-002 (等 WI-001 完成后启动)
```

并行调度必须同时满足：
- `depends_on` 已完成或不需要执行。
- `allowed_write_scope` 与当前 active work item 不重叠。
- 若写范围无法判断，默认按冲突处理，进入等待队列。
- 等待队列按 FIFO 重试；释放 active work item 后重新计算可执行集合。

写范围冲突检测复用现有 worktree lease 思路，不能只依赖用户手动选择串行/并行。

### 8.5 Coding 阶段 Provider 输入

| 节点 | Provider | 输入 |
|------|----------|------|
| Plan 生成 | Claude Code | work_item + story_spec + design_spec + 代码库上下文 |
| Coding | 按节点执行配置 | plan + work_item + spec（constraint context 注入），working_dir = worktree |
| Testing | 按节点执行配置 | coding 产物 + acceptance_scenarios + verification_commands |
| Code Review | 按节点执行配置 | git diff + acceptance_scenarios |
| Rework | 按节点执行配置 | review feedback + plan + 当前 worktree 状态 |

## 9. 分阶段实施计划

### Phase 1：基础设施

项目管理 + 代码库管理 + `~/.aria/` 产品索引层 + runtime 兼容绑定 + 首页工作台骨架。

- Project / Repository 数据模型和持久化
- IssueRuntimeBinding 数据模型和持久化
- 从 `{repo}/.aria/runtime` 重建产品索引的兼容读取能力
- Workspace → Repository 重命名（后端 + 前端）
- 项目切换、最近项目记忆
- 首页重构（Issue 列表 + 代码库管理弹窗）
- Issue 增加 phase 字段
- ExecutionRecord 数据模型和持久化
- Provider 可用性检查机制

### Phase 2：需求澄清流程

Story spec + Design spec 完整生成/review/确认链路。

- 新增 runtime units：story_spec_generate/review、design_spec_generate/review
- 硬 gate 机制（GateRecord + gate API + gate SSE）
- Review 循环逻辑（最多 2 次 review）
- Codex CLI provider 接入
- 前端 spec 产物展示 + gate 确认
- 前端 Agent Picker（节点执行前选择 provider）
- 需求澄清阶段回退机制（spec 版本回退）

### Phase 3：代码开发流程

Work item + Plan + Coding 完整链路。

- 新增 runtime units：work_item_generate/confirm、plan_generate/confirm
- 外置 Worktree 管理（创建、状态追踪、清理）
- Work item 依赖 DAG 排序、allowed_write_scope 冲突检测和等待队列
- 复用 coding/testing/code_review/rework 节点
- 前端 work item 列表 + 执行状态
- 代码开发阶段回退机制（runtime checkpoint + worktree git reset + soft-delete）

### Phase 4：Provider 流式展示增强

全流程 provider 执行过程透明化。

- SSE 事件增强
- Provider input 摘要/引用分离与 secret redaction
- SSE cursor resume 与 provider-output 持久化日志
- ProviderExecutionPanel 重构
- Review 轮次指示器
- 多 work item 并行 tab 切换
- 回退确认对话框（含 git reset 信息展示）

### Phase 依赖

```
Phase 1 → Phase 2 → Phase 3
Phase 1 → Phase 4 (可与 Phase 2/3 并行)
```
