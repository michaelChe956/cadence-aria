## 1. 评审目标

1. 研发人员是否能完整理解当前技术方案和计划文档。
2. 当前文档是否足够研发直接实施落地。

## 2. 总体结论

当前文档体系的方向是正确的：MVP 文档定义了一期边界，总契约定义了对象、协议、Provider、OpenSpec 和收口规则，IO 契约补齐了节点输入输出与 Provider 协作，P1-P4 将实施拆成了可分工的阶段。

但按当前清单直接交给研发仍有风险。核心问题不是主流程缺失，而是：

1. 代码级补齐规格未包含在交付清单中。
2. 少数字段和路由在文档间存在不一致。
3. 若干 4-22 上游协议和 MVP 强制项没有在 P1-P4 中被明确锁成测试项。

因此本次评审结论为：

**当前方案可以让研发理解总体架构和实施路线，但还不能作为完全可实施的冻结版。必须先修复本评审列出的 P0/P1 问题，并将 `2026-04-24_技术方案_Aria一期评审后实施规格补齐_v1.1.md` 纳入正式技术方案包。**

## 3. P0 阻断问题

### P0-1. 正式技术方案清单缺少代码级实施规格

**问题**

P1-P4 和导读文档均明确依赖：

`cadence/designs/2026-04-24_技术方案_Aria一期评审后实施规格补齐_v1.1.md`

该文件补齐 Rust 类型、Projection/OpenSpec 编译规则、daemon 发现、artifact 路径、prompt 模板、fixture、worktree、N23 集成、approval gate 等实现细节。若研发只收到 4-23 技术方案和 P1-P4 计划，将无法独立完成代码落地。

**影响**

- P1 缺 daemon 发现协议、runtime 路径和 Rust 基础类型的最终裁定。
- P2 缺 Projection/OpenSpec compiler 的字段、parser、fixture、golden JSON 裁定。
- P3 缺 prompt 模板、provider structured output、ProviderRunRecord 审计字段。
- P4 缺 worktree lease、candidate commit、integration branch、rollback 和 N26 gate 规则。

**修订要求**

将 `2026-04-24_技术方案_Aria一期评审后实施规格补齐_v1.1.md` 纳入正式技术方案清单，并在计划总览中标明：

- `v1.2 + 实现总契约 + IO 契约` 是 phase1 implementation profile。
- `评审后实施规格补齐_v1.1` 是代码级实施规格。
- P1-P4 启动前必须满足该文档对应阶段准入门槛。

### P0-2. Runtime Snapshot 字段存在内容丢失风险

**问题**

4-22 全局协议和 MVP 都要求 `runtime_snapshot` 保留 canonical 字段，包括：

- `snapshotId`
- `sessionId`
- `taskId`
- `nodeId`
- `phase`
- `timestamp`
- `effectivePolicy`
- `artifactRefs`
- `providerRunRefs`
- `worktreeRef`
- `reworkCounter`
- `riskRegistry`
- `loopCounters`
- `supersededArtifactRefs`
- `nodeSpecificFields`

但 `评审后实施规格补齐_v1.1` 中的 Rust `RuntimeSnapshot` 使用了不同字段集合，缺少部分上游字段，并把时间字段写成 `createdAt`。P1 计划中的最小 runtime snapshot 字段也偏少。

**影响**

这会违反“不得丢失 4-22/MVP 内容”的要求，并影响：

- provider run 审计回放
- worktree 关联
- Risk Registry 恢复
- rework counter 和 loop counter 恢复
- 回流 superseded 产物判定
- 与上游 `runtime_snapshot` canonical schema 的一致性

**修订要求**

将 `RuntimeSnapshot` Rust 类型与 P1 计划最小字段统一回上游 canonical 字段：

- 保留上游字段名，尤其 `timestamp` 不应被 `createdAt` 替代。
- `providerRunRefs`、`worktreeRef`、`reworkCounter`、`riskRegistry` 必须进入 Rust 类型或明确作为兼容字段。
- 若补齐规格需要增加 `projectionRefs`、`constraintBundleRefs`、`createdAt`，只能作为 additive fields，不能替代 canonical 字段。
- P1 的 `runtime_snapshot` 测试必须覆盖全部 canonical 字段的序列化/反序列化。

### P0-3. `new_task` 成功响应字段不一致

**问题**

总契约命令表中 `new_task` 成功响应只包含：

- `task_id`
- `phase`
- `intake_ref`

但 P1 计划要求成功响应包含：

- `task_id`
- `phase`
- `intake_ref`
- `change_id`

补齐规格中的 fixture 也仍偏向前者。

**影响**

`changeId` 是 OpenSpec 强约束的入口。若 `new_task` 响应不返回 `change_id`，REPL 和后续调试路径无法明确知道本 EpicTask 绑定的 OpenSpec change。

**修订要求**

统一为：

- `new_task` 成功响应必须返回 `task_id`、`phase`、`intake_ref`、`change_id`。
- P1 的 `repl_wire` 和 `task_init_and_intake` 测试必须锁定该字段。
- 补齐规格中的 fixture 和 API 表同步更新。

### P0-4. OpenSpec bundle 字段命名仍有歧义

**问题**

总契约 6.3.1 已裁定 `OpenSpecConstraintBundle` 顶层 JSON 字段固定使用 camelCase：

- `proposalConstraints`
- `requirementConstraints`
- `designConstraints`
- `taskConstraints`
- `traceabilityRequirements`
- `coverageModel`

P2 计划也要求 `scope_constraints`、`requirement_ids`、`task_ids`、`traceability_map` 只能作为内部 helper 输出或 payload 子字段。

但总契约 10.3 又写“编译结果至少包含” `scope_constraints`、`requirement_ids` 等 snake_case 字段，容易让研发误认为这些是跨模块契约字段。

**影响**

P2、P3、P4 可能按不同 JSON 字段消费 bundle，导致约束器无法稳定作为全流程输入。

**修订要求**

修改总契约 10.3 表述：

- 顶层序列化字段只允许使用 6.3.1 的 camelCase 字段。
- `scope_constraints`、`requirement_ids`、`task_ids`、`traceability_map` 只能作为内部 helper 或具体 payload 子字段。
- P2 `openspec_bundle_schema` golden test 必须锁定字段名。

## 4. P1 重要问题

### P1-1. 横切节点覆盖未在计划中完整显式化

**问题**

4-22 节点总目录定义了 `X01-X09`，但 P1-P4 计划中显式出现的横切节点主要是 `X01` 和 `X08`。虽然部分能力以模块方式存在，例如 `provider_run`、`artifact_validate`、`checkpoint_and_recovery`，但计划没有清晰映射到 `X03-X07/X09` 的协议语义。

**影响**

研发可能实现了模块，但没有保留横切节点事件、snapshot、checkpoint 和审计边界。

**修订要求**

在计划总览或各 P 阶段补充横切节点覆盖矩阵：

| 横切节点 | 计划落位 | 最低验收 |
|----------|----------|----------|
| `X01` approval_gate_open | P1/P4 | gate open event、gate registry、恢复 |
| `X02` approval_gate_resume | P1/P4 | approve/reject/reply 命令与恢复 |
| `X03` provider_run_dispatch | P3 | ProviderRunRecord started |
| `X04` provider_run_collect | P3 | raw output、structured output、failed/completed event |
| `X05` artifact_validate | P2/P3/P4 | canonical + phase1 profile validator |
| `X06` checkpoint_persist | P1-P4 | 每节点 checkpoint/runtime snapshot |
| `X07` runtime_recover | P1 | reconnect/recovery smoke |
| `X08` manual_intervention_hold | P1/P3/P4 | reason code 与终态记录 |
| `X09` policy_override_apply | P1 | effectivePolicy、降级记录、event/snapshot |

### P1-2. MVP 中若干强制规则没有被计划测试锁住

**问题**

MVP 中明确要求：

- 非 `conservative` 策略在一期显式降级并记录 `policy_mode_degraded`。
- 同一 WorkTask 连续 2 次集成失败后进入 `X08 manual_intervention`，reason 为 `integration_retry_limit_exceeded`。
- Risk Registry 至少支持 riskId 分配、落盘、恢复、产物引用解析。
- 回流时必须写入 `supersededArtifactRefs`，节点进入前不得引用 superseded 产物。

这些规则在 P1-P4 中没有全部被明确列为测试项。

**影响**

研发可能只跑通主流程，但遗漏协议不漂移要求。

**修订要求**

补充以下测试：

- P1：`policy_override_degrade.rs`，覆盖 `balanced/aggressive -> conservative` 降级和 `policy_mode_degraded` 记录。
- P1/P2：`runtime_snapshot_schema.rs`，覆盖 `riskRegistry`、`reworkCounter`、`supersededArtifactRefs`、`providerRunRefs` 等字段。
- P2/P3：`superseded_artifact_refs.rs`，覆盖回流后旧产物不可作为输入。
- P4：`integration_retry_limit.rs`，覆盖连续 2 次集成失败后进入 `X08`。
- P3/P4：`risk_registry_minimal.rs`，覆盖 riskId 创建、引用、落盘、恢复。

### P1-3. `N26` 一期口径需要统一

**问题**

MVP 写“一期实现单元 21 个”，并说明 `N26 patch_followup_dispatch` 不自动执行但保留协议定义。P4 计划又要求实现 `patch_followup_dispatch.rs`，且 gate 确认后允许进入 `N26`。

**影响**

研发对 `N26` 是否属于一期实现范围会产生歧义。

**修订要求**

统一表述为：

**一期不自动派生 N26，但必须实现 gated N26 handler。只有 `N25 overallDecision=followup` 且用户通过 gate 显式确认后，daemon 才允许进入 `N26 patch_followup_dispatch`。**

同时将实现单元口径改为：

- `21 个主线 runtime units`
- `1 个 gated followup handler`

或在 MVP 中明确 `N26` 不计入主线 21 个实现单元。

## 5. 覆盖判断

### 5.1 主流程节点覆盖

4-22 节点总目录定义 `N00-N28` 共 29 个主流程节点。P1-P4 对主流程覆盖基本完整：

- P1 覆盖 `N00-N03`
- P3 覆盖 `N04-N12`
- P4 覆盖 `N13-N28`

其中 `N14`、`N22` 在检索中没有以独立标题出现，但分别包含在 P4 的 `N13-N15` 执行准备链和 `N21-N24` 集成链路中。建议在 P4 的任务标题或断言中显式写出 `N14`、`N22`，避免研发误以为可省略独立协议边界。

### 5.2 产物覆盖

MVP 要求 17 类一期产物：

- 16 类业务产物
- `runtime_snapshot`

P2 计划已要求 17 类一期产物进入统一 validator，覆盖方向正确。

仍需修订的是：

- `runtime_snapshot` 字段必须完全对齐 4-22 全局协议。
- JSON artifact 的 `_aria` 扩展需被 phase1 profile validator 锁定。
- 回流 `supersededArtifactRefs`、Risk Registry、loop counters 不应只停留在设计层，必须进入测试和完成判定。

### 5.3 数据面覆盖

P2 对 Document Operation、Projection、OpenSpec bundle、Traceability 的拆解是合理的，且符合 MVP “不能直接解析 Markdown 原文做 routing/coverage”的要求。

需要补强的是：

- Projection/OpenSpec golden fixture 必须成为准入条件，而不是实现后补充项。
- OpenSpec bundle 顶层字段必须唯一裁定。
- `DocumentModel`、HeadingModel、hash/version 策略应以补齐规格为准，不能由研发自行发挥。

### 5.4 Provider 覆盖

P3 对 fake provider、真实 CLI adapter baseline、capability probe、compatibility matrix、provider error registry、ProviderRunRecord 的拆解基本足够。

需要补强的是：

- ProviderRunRecord 字段必须与补齐规格保持完全一致。
- CLI baseline 不验证模型质量是合理的，但必须验证 spawn、timeout、stdout/stderr、structured output、filesModified、missing/unauthorized/permission denied。
- fake provider 必须使用正式 sentinel 格式，避免测试路径和真实路径分叉。

## 6. 对两个核心问题的回答

### 6.1 是否能让研发人员完全理解清楚

**结论：基本能理解总体架构，但还不能保证完全理解可编码细节。**

原因：

- 导读文档已经清楚说明 `v1.2`、总契约、补齐规格、计划文档各自职责。
- P1-P4 已按工程边界拆成 REPL/daemon、数据面、provider/planning、execution/final closure。
- 但若技术方案包缺少 `评审后实施规格补齐_v1.1`，研发会缺少字段、模板、fixture、daemon 发现、worktree/N23 等代码级决策。
- 少数字段冲突会造成不同负责人按不同契约实现。

### 6.2 是否足够研发实施落地

**结论：纳入补齐规格并修复 P0/P1 问题后，足够落地；当前版本不建议直接冻结开工。**

当前已具备：

- 清晰阶段拆解
- 文件级目标结构
- TDD 步骤
- 测试命令
- fake provider 与真实 CLI baseline 分层
- OpenSpec / Projection / Traceability 数据面规划

当前仍缺：

- 技术方案包完整性
- `RuntimeSnapshot` canonical 字段对齐
- `new_task` 响应字段统一
- OpenSpec bundle 字段唯一裁定
- 横切节点与若干 MVP 强制规则的测试锁定

## 8. 最终裁定

本次评审不建议直接将当前技术方案和计划文档作为冻结版交付研发。

建议处理方式：

1. 先发布 `Aria一期技术方案与实施计划_v1.1` 修订包。
2. 将 `评审后实施规格补齐_v1.1` 正式纳入技术方案索引。
3. 修复本评审列出的 P0/P1 问题。
4. 再进入 P1 编码。

若完成上述修订，当前文档体系可以达到：

- 研发能理解完整架构和分工。
- 研发能按 P1-P4 直接实施。
- MVP 和 4-22 上游文档集的节点、横切能力、产物规范和全局协议内容不会丢失。
