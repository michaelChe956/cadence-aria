## 2. 总体结论

当前文档已经具备实施主线：P1-P4 拆分清楚，REPL/daemon、Document Operation、Projection、OpenSpec、Provider、执行集成和最终收口的边界基本明确。

但当前版本仍不能直接判定为“研发可完全无歧义落地”。主要原因是：少数关键类型缺失、canonical artifact 与 fixture/prompt 口径不一致、枚举值漂移、计划测试覆盖缺口会让研发在实现时自行补口径，存在协议漂移和内容丢失风险。

## 3. 问题清单

### P0-1. `ProviderRunRecord` 在 v1.1 中被声明补齐，但缺少 Rust 类型定义

**问题描述**

`2026-04-24_技术方案_Aria一期评审后实施规格补齐_v1.1.md` 声明 `ProviderRunRecord` 审计字段不足已补齐，位置指向第 4.7 章。但第 4.7 章实际只给出了 `NodeExecutionContract`、`WorkflowDisciplineSpec`、`NodePromptTemplateRef`、`ProviderContextPackage`，没有 `ProviderRunRecord` 的 Rust struct。

P3 计划又要求以该补齐规格实现 `ProviderRunRecord`，因此研发执行 P3 时会缺少代码级字段真相源。

**影响**

- provider 审计字段可能由不同实现人员自行定义。
- `providerCapabilityRef`、`adapterCompatibilityRef`、`adapterInputRef`、`adapterOutputRef`、`timeoutStatus`、`sandboxMode`、`approvalPolicy` 等字段可能丢失或命名漂移。
- P3 的 CLI adapter baseline 和 provider error route 测试无法稳定锁定 schema。

**建议**

在 v1.1 或后续 v1.2 中新增 `ProviderRunRecord` Rust struct，并与 `Aria一期实现总契约_v1.0` 第 6.8 章的最小字段完全对齐。

---

### P0-2. Prompt / fixture 偏向 Projection，未完整覆盖 2026-04-22 canonical artifact 最小字段

**问题描述**

上游产物规范要求：

- `spec` 必须包含 `scope`、`user_stories`、`functional_requirements`、`success_criteria`、`open_items`。
- `design` 必须包含 `architecture_summary`、`frontend_design/backend_design` 至少一项、`data_model`、`api_contracts`、`risk_list`。
- `plan` 必须包含 `work_packages`、`dependencies`、`acceptance_checks`、`parallelism_hints`。

但 v1.1 的 N05/N07 prompt 和 fixture 主要覆盖 Projection 所需章节，未完整覆盖 canonical 最小字段。例如：

- N05 模板没有要求 `scope`。
- v1.1 `artifacts/spec.md` fixture 缺少 `scope`、`open_items`、`non_functional_requirements` 示例。
- N07 模板没有要求 `architecture_summary`、`frontend_design/backend_design`。
- v1.1 `artifacts/design.md` fixture 缺少 `风险` 与 `待确认项`，也没有按 canonical 字段体现 `architecture_summary`。
- v1.1 `artifacts/plan.md` fixture 把验收写在 work package 表格列里，但没有独立体现上游 `acceptance_checks`。

**影响**

- P2/P3 golden tests 可能通过，但 canonical validator 在真实流程中失败。
- 研发会误以为 Projection heading 就等同于 canonical schema。
- 存在从 v1.0 到 v1.1 的内容丢失风险，尤其是待确认项、非功能需求、风险项。

**建议**

补齐 v1.1 fixture 和 prompt：

- `spec` fixture 至少包含 `范围`、`用户故事`、`功能需求`、`成功标准`、`待确认项`、`非功能需求`。
- `design` fixture 至少包含架构摘要、前端或后端设计、数据模型、API 契约、风险、待确认项。
- `plan` fixture 除 `工作包/依赖/并行分组` 外，应明确映射或新增 `验收检查`，避免丢失 `acceptance_checks`。
- P2 validator 测试必须同时覆盖 canonical schema 和 Projection schema。

### P0-3. `N08 design_review` 的枚举值使用错误

**问题描述**

v1.1 第 9.4 章的 N08 模板差异项写为 `decision=pass/backtrack/hold` 候选。该枚举属于 `spec_gate_decision`，不是 `design_review`。

2026-04-22 全局协议规定 `design_review.review_decision` 的取值应为：

- `pass`
- `revise`
- `conditional_pass`

**影响**

- P3 prompt registry 会诱导 provider 输出错误 schema。
- `N08 -> N09/N10` 路由可能按错误枚举实现。
- `design_review` canonical validator 与 provider structured output parser 可能冲突。

**建议**

将 N08 模板差异项改为输出 `review_decision=pass/revise/conditional_pass`，并在 P3 `context_builder` 与 `planning_chain_fake_provider` 测试中锁定。

### P1-1. `execution_mode` 取值口径漂移

**问题描述**

`Aria一期实现总契约_v1.0` 中 worktask 级执行方式明确为：

- `agent_only`
- `human_assisted`
- `human_required`

但 v1.1 Rust enum 定义为：

- `Agent`
- `Human`
- `Hybrid`

经 `serde(rename_all = "snake_case")` 后实际 JSON 为：

- `agent`
- `human`
- `hybrid`

v1.1 fixture 也使用了 `agent`。

**影响**

- `PlanProjection.work_packages[].execution_mode` 与 `dispatch_package._aria.worktask_routing[]` 之间可能不一致。
- P4 的 execution route 可能无法判断 `human_assisted` 和 `human_required` 的不同路由语义。
- Dashboard / read model 的执行方式字段会出现多套枚举。

**建议**

统一为总契约口径：`agent_only`、`human_assisted`、`human_required`。Rust enum 可命名为 `AgentOnly`、`HumanAssisted`、`HumanRequired`，并用 `serde(rename_all = "snake_case")` 输出目标值。

### P1-2. P3 对 `N09 design_revision` 的产物与路由测试覆盖不足

**问题描述**

上游节点目录规定：

- `N08 design_review` 可路由到 `N09` 或 `N10`。
- `N09 design_revision` 产出 `design_revision_record`，再回到 `N08`。

P3 计划虽然创建了 `design_revision.rs`，但 P3 阶段出口清单没有列出 `design_revision_record`，`N08-N12` 测试断言也没有明确覆盖：

- `design_review.review_decision=revise`
- `N09 design_revision_record`
- 更新后的 `design`
- `N09 -> N08` 再评审
- 回流导致中间产物 superseded 的一致性

**影响**

- 研发可能只实现 happy path：`N08 pass -> N10`。
- 设计修订闭环缺失，违反 `N08/N09` 上游节点协议。
- 回流失效规则在规划阶段无法被验证。

**建议**

P3 增补测试断言：

- `design_review.review_decision=revise` 时必须进入 `N09`。
- `N09` 必须生成 `design_revision_record` 和更新后的 `design` ref。
- `N09` 后必须回到 `N08`。
- 若修订跨越中间产物，必须按回流失效规则标记相关 artifact。

### P1-3. P2 目标文件结构未显式创建 golden projection fixture

**问题描述**

v1.1 第 15 章规定 fixture 树应包含：

- `tests/fixtures/artifacts/golden/spec_projection.json`
- `tests/fixtures/artifacts/golden/design_projection.json`
- `tests/fixtures/artifacts/golden/plan_projection.json`

P2 计划强调 golden fixture，但目标文件结构只列出了 `spec.md`、`design.md`、`plan.md` 和 OpenSpec 文件，没有显式列出 golden JSON 文件。

**影响**

- 研发可能只做 runtime assertion，不落 golden JSON。
- Projection 输出格式缺少文件级锁定，后续 P3/P4 容易按不同 payload 消费。

**建议**

P2 目标文件结构补充三个 golden JSON 文件，并在完成判定中明确 projection compiler 必须与 golden JSON 对齐。

## 4. 建议修正顺序

1. 先修 `2026-04-24_技术方案_Aria一期评审后实施规格补齐_v1.1.md`：
   - 增加 `ProviderRunRecord` Rust 类型。
   - 修正 N08 枚举。
   - 修正 `ExecutionMode` 取值。
   - 补齐 canonical fixture。
2. 再同步 P2/P3 计划：
   - P2 增加 golden JSON 文件清单和 canonical validator fixture。
   - P3 增加 `N09` 修订路径测试和 `design_revision_record` 出口。
3. 最后复核 P4：
   - 确认 execution route 消费的 `execution_mode` 已统一。
   - 确认 final coverage 只消费规范后的 traceability refs。

---

## 5. 验收建议

修正后建议增加一次文档一致性验收：

- `ProviderRunRecord` 字段与总契约第 6.8 章逐项一致。
- N04-N12 prompt manifest 输出 schema 与上游产物枚举一致。
- P2 fixture 同时通过 canonical validator、projection compiler、OpenSpec bundle compiler。
- P3 fake provider 输出不会绕过 canonical validator。
- P4 execution route 只使用统一后的 `execution_mode` 枚举。

---

## 6. 一句话总结

当前方案主线可成立，但需要先修复上述 schema、枚举、fixture 和计划测试缺口，否则研发仍会在 P2/P3 阶段自行补口径，造成协议漂移或内容丢失。
