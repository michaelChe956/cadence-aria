## 4. 阻断问题

### P0-1. P4 计划中的节点编号与 4-22 节点目录不一致

**问题**

`P4执行集成与最终收口` 当前写法把执行准备链和集成链的节点编号写错：

- P4 写 `N14 worktask_register`，但上游节点目录定义为 `N13 worktask_register`。
- P4 写 `N15 worktree_prepare`，但上游节点目录定义为 `N14 worktree_prepare`。
- P4 写 `N13 session_bootstrap_execution`，但 4-22 节点目录中不存在该节点名作为 N13。
- P4 写 `N22 integration_execute`，但上游节点目录定义为 `N22 integration_prepare`。
- P4 写 `N23 integration_verify`，但上游节点目录定义为 `N23 integration_execute`。
- P4 写 `N24 merge_or_rollback`，但上游节点目录定义为 `N24 integration_verify`。

上游 canonical 节点目录应保持：

- `N13 worktask_register`
- `N14 worktree_prepare`
- `N15 execution_route_resolve`
- `N21 integration_enqueue`
- `N22 integration_prepare`
- `N23 integration_execute`
- `N24 integration_verify`

**影响**

该问题会直接导致研发按错误 node id 编写 runtime unit、event、snapshot、checkpoint、provider route 和测试，属于冻结前必须修复的问题。

**修订要求**

- 全量修正 P4 中 `N13-N15` 和 `N21-N24` 的节点编号、标题、步骤、测试名和模块说明。
- 不得引入 `N24 merge_or_rollback` 作为 canonical 节点名；rollback 可以是 `N23/N24` 内部行为或 integration report 的结果字段，但不能替代上游节点目录。

### P0-2. 总契约丢失 `N24 advisory`，且 `N20 advisory` 关键限制被写错

**问题**

`一期实现总契约` 的评审与执行节点表当前只保留了：

- `N06 advisory`
- `N20 advisory`

但缺少 `N24 advisory`。同时当前 `N20 advisory` 的关键限制写为“不直接 rollback”，该限制语义应属于 `N24 advisory`，不应落在 `N20`。

**影响**

- `N20` 的职责是 ready/block/rework 前的执行结果综合 advisory。
- `N24` 的职责是 integration verify 后的 advisory review，不直接执行 rollback。

若缺少 `N24 advisory`，P4 集成验证后的 Provider 评审边界会丢失；若把“不直接 rollback”写到 `N20`，研发会误解 `N20` 与 integration rollback 的关系。

**修订要求**

- 恢复 `N24 advisory` 行。
- 将“不直接 rollback”放回 `N24 advisory`。
- 将 `N20 advisory` 的关键限制恢复为“最终 ready / block / rework 由 daemon 决定”或等价表达。

### P0-3. Risk Registry 补齐规格仍未完整对齐 4-22 canonical 字段

**问题**

`评审后实施规格补齐_v1.1` 当前 `RiskEntry` 只包含：

- `risk_id`
- `text`
- `status`
- `source_node`

但 4-22 全局协议定义的 Risk Registry 风险对象字段为：

- `riskId`
- `description`
- `severity`
- `status`
- `sourceArtifact`
- `sourceNode`
- `createdAt`
- `updatedAt`
- `resolution`

此外，补齐规格当前将 `RuntimeSnapshot.risk_registry` 定义为 `Option<RiskRegistrySnapshot>`，但 4-22 全局协议中 `riskRegistry` 是 runtime snapshot 必填字段。

**影响**

这仍然违反“不得丢失 4-22/MVP 内容”的要求，并会影响：

- riskId 与产物引用解析
- 风险严重级别判断
- 风险创建和更新时间恢复
- mitigated / accepted 时的 resolution 审计
- runtime recovery 时 Risk Registry 是否必定存在

**修订要求**

- 将 `RiskEntry` 字段补齐到 4-22 canonical 风险对象。
- `text` 应改为或映射为 `description`。
- `resolution` 可以在 open/escalated 状态下为空，但字段语义必须保留。
- `RuntimeSnapshot.risk_registry` 建议改为必填，空风险时使用 `risks: []`，而不是省略整个 registry。

## 5. 重要问题

### P1-1. `ProjectionPayload` 少派生 `Serialize`，代码级蓝图不可直接编译

**问题**

`ArtifactProjectionRecord` 派生了 `Serialize` / `Deserialize`，并包含字段 `payload: ProjectionPayload`。但当前 `ProjectionPayload` 只派生了 `Deserialize`。

**影响**

若研发按补齐规格中的 Rust 类型直接实现，`ArtifactProjectionRecord` 的 `Serialize` 派生会失败，P2 projection golden JSON 输出链路也无法成立。

**修订要求**

将 `ProjectionPayload` 恢复为：

```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
```

### P1-2. `N06 advisory` 的 output schema namespace 容易误导

**问题**

`一期实现总契约` 当前将 `N06 advisory` 的 `outputSchemaRef` 写为：

```text
schema://aria/artifacts/spec_gate_review/v1
```

但 `N06 advisory` 明确是非正式 advisory，正式 `spec_gate_decision` 仍由 daemon 生成。

**影响**

研发可能把 `N06 advisory` 当成正式 artifact schema 处理，或者误把 advisory review 与 daemon 产出的 formal decision 混用。

**修订要求**

- 恢复或定义独立 advisory namespace，例如 `schema://aria/advisory/spec_gate_review/v1`。
- 明确 `N06 advisory` 不能替代正式 `spec_gate_decision`。

### P1-3. 补齐规格中的 `new_task` 示例和 fixture 仍漏 `change_id`

**问题**

虽然总契约和 P1 计划已经要求 `new_task` 成功响应包含：

- `task_id`
- `phase`
- `intake_ref`
- `change_id`

但 `评审后实施规格补齐_v1.1` 的 Spec Markdown 示例、projection 片段、OpenSpec 最小 fixture、API 表仍写为：

- `task_id`
- `phase`
- `intake_ref`

**影响**

P1/P2 的 fixture、golden test 和 API 文档会继续与总契约冲突，导致研发按不同来源写出不同响应模型。

**修订要求**

- 补齐规格中所有 `new_task` 响应示例统一补充 `change_id`。
- API 表 `API-001 new_task` output 改为 `task_id,phase,intake_ref,change_id`。
- projection 片段中的 acceptance text 同步更新。

## 6. 建议处理顺序

进入研发前建议按以下顺序修订：

1. 先修 P4 节点编号，保证所有计划与 4-22 节点目录一致。

5. 修 `N06 advisory` schema namespace，明确 advisory 与 formal artifact 的区别。
6. 最后统一补齐规格中所有 `new_task` 示例和 fixture 的 `change_id`。
