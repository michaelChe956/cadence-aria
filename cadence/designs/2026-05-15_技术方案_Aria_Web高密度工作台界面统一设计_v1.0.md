# Aria Web 高密度工作台界面统一设计

## 1. 背景

当前 Aria Web 已从单一执行工作台扩展为项目管理工作台入口，并支持 Issue 启动、执行工作台跳转、Provider 输出、Gate、Evidence、Diagnostics 和 Rollback 等能力。但前端视觉仍混合了早期 playful 风格、厚重彩色阴影、装饰性背景和不一致的信息层级，导致项目管理页与执行页像两个产品。

本设计统一当前用户会看到的所有页面：项目管理工作台、执行工作台、弹窗、面板、空状态和错误态。目标是把 Aria Web 收敛为高密度、冷静、可诊断的工程工具台。

## 2. 设计目标

1. 首页仍然是可操作工作台，不做 landing page。
2. 项目管理工作台和执行工作台使用同一套视觉语言、间距、状态色、按钮层级和信息密度。
3. 关键状态必须一眼可扫：Project、Issue、Workspace、Task、Node、Provider、Policy、Git、SSE、Gate、Diagnostics。
4. Provider prompt 默认只展示摘要和引用，不在普通面板中展开完整 prompt。
5. 危险动作必须明确隔离，尤其是 rollback、terminate、dirty workspace 确认。
6. 保持现有业务行为、API 契约和可访问语义，不以重设计为理由改后端模型。

## 3. 非目标

- 不引入新的 UI 组件库。
- 不实现完整 Kanban、多人协作、远端同步、PR 状态或 Acceptance 状态机。
- 不重写执行 runtime、SSE、Provider、Gate 或 Rollback 后端逻辑。
- 不把旧的历史兼容入口作为主要重设计对象；本轮只覆盖当前用户实际会看到的页面与附属层。
- 不增加营销化 hero、插画主视觉或装饰性渐变背景。

## 4. 视觉系统

### 4.1 设计方向

采用高密度工程工作台风格：信息优先、控件稳定、状态清楚、少装饰。整体参考 Minimalism / Swiss Style 的网格、克制对比和明确层级，但针对 Aria 的执行透明化场景保留足够的日志、证据和状态密度。

### 4.2 色彩

基础色：

- 页面背景：浅灰白，建议 `#F6F8FA` 或接近值。
- 主面板：白色或近白，建议 `#FFFFFF` / `#FBFCFE`。
- 文字主色：深墨色，建议 `#17202A`。
- 次级文字：灰蓝，建议 `#52616F`。
- 边线：冷灰，建议 `#DDE5EC`。

状态色：

- 主操作 / 选中：青色系，用于打开执行、推进、确认等正向动作。
- 完成：绿色系。
- 等待 / Gate / 警告：琥珀色系。
- 错误 / 终止 / Rollback 危险确认：红色系。
- 中性 / 空状态 / disabled：灰色系。

禁止大面积使用单一紫色、橙粉渐变、背景光斑、装饰性彩色阴影。颜色必须服务于状态而不是装饰。

### 4.3 字体与排版

- 保持系统 sans-serif 为主体，避免额外网络字体依赖。
- ID、task、node、hash、path、event type 使用等宽字体。
- 顶栏、面板标题、列表项标题使用较紧凑字号，不在工作台内部使用 hero 级大字。
- 字间距保持默认，不使用负字距。

### 4.4 形状与阴影

- 卡片圆角统一为 6px 到 8px。
- 常规面板使用 1px 边线和轻阴影，避免厚重偏移阴影。
- 面板不再嵌套成多层“卡片套卡片”；页面区域用栅格和边线组织，重复项才使用卡片。
- Hover 不改变布局尺寸，不使用 scale 造成跳动。

## 5. 全局信息架构

### 5.1 Shell

所有主页面共享三层结构：

1. 顶栏：产品、当前上下文、全局动作。
2. 状态带：关键运行指标。
3. 工作区：左侧导航/列表，中间主内容，右侧 Inspector。

顶栏固定在顶部，状态带位于顶栏下方。内容滚动时上下文不丢失。

### 5.2 响应式规则

- 桌面宽屏：三栏布局。
- 中等屏：两栏布局，右侧 Inspector 可下移或折叠到主内容后。
- 移动窄屏：单栏顺序为上下文、主操作、列表或证据；固定顶栏保持紧凑，避免横向滚动。
- 固定格式元素如状态带、按钮、节点条、日志行应设置稳定尺寸，动态文本使用截断、换行或等宽容器。

## 6. 项目管理工作台

### 6.1 顶栏

内容：

- `Aria Web`
- 当前项目选择器
- Issue 数
- 刷新按钮

行为：

- 无项目时，项目选择器显示空状态并保持可理解。
- 刷新不重排页面骨架，只更新数据。

视觉：

- 顶栏白底、细边线、轻阴影。
- 项目选择器和刷新按钮高度一致。

### 6.2 左栏：Issue 队列

每个 Issue 项展示：

- 标题
- 状态
- Issue ID
- Task ID 或未启动状态

状态：

- 未选中：白底、灰边。
- 选中：左侧或顶部使用青色标识，背景轻微着色。
- blocked：使用琥珀状态标记，不让整项变成强警告色。
- 空状态：显示“暂无 Issue”和明确下一步入口，但不使用大插画。

### 6.3 中栏：Issue 详情

内容：

- Issue 标题
- 状态与 ID
- 描述
- Change ID
- Workspace
- Task
- 阶段轨道：Clarification、Development、Acceptance
- 主操作：打开执行

规则：

- `打开执行` 是中栏唯一主按钮。
- 未启动或缺少 workspace/task 时按钮 disabled，并显示原因。
- Acceptance 在本版本可灰显占位，不暗示状态机已自动进入。

### 6.4 Gate 区域

当 Issue 处于 blocked 或存在 open Gate 时，中栏详情下方出现 GateActionBar。

按钮顺序：

1. 确认继续
2. 要求修改
3. 终止

视觉：

- Gate 区域用琥珀边框和轻背景。
- 终止按钮使用红色，但不与确认按钮同等视觉重量。
- Gate ID、node ID、状态必须可见。

### 6.5 右栏：Repository 与 Provider 摘要

Repository 面板：

- 当前项目
- Project ID
- Repo path
- Repo hash
- Runtime root
- Issue count

若后端暂未提供 repo 字段，UI 先保留结构并以 `-` 或“未绑定仓库”展示，不伪造数据。

Provider 执行摘要：

- event type
- node
- input ref
- input summary

不默认展示完整 prompt。需要查看完整输入时必须通过明确动作进入受控视图，并保留脱敏规则。

## 7. 执行工作台

### 7.1 顶栏

内容：

- Aria Web
- 当前 Issue
- Workspace
- TaskSwitcher
- 返回项目管理

规则：

- 返回项目管理是次级按钮。
- TaskSwitcher 保持紧凑，不因为长 change ID 撑破顶栏。

### 7.2 状态带

指标：

- Task
- Change
- Node
- Worktask
- Policy
- Provider
- Git
- SSE
- Running state

状态带使用紧凑 metric。Git dirty、SSE offline、blocked_by_gate、running 必须有明确状态色。

### 7.3 主区域：Interaction Window

移除 hero 化展示，改为执行摘要 + 操作区 + 输出流。

组成：

1. 执行摘要：当前 task、节点数、产物数、事件数、选中节点。
2. 新建任务或继续任务入口。
3. Provider 确认区：prompt 摘要、input refs、allowed write scope、verification commands、policy override。
4. 推进 / 回退 / 停止操作区。
5. Provider stream。

规则：

- 自动运行状态不遮盖整个页面，只在操作区显示局部状态。
- 回退按钮只在有 checkpoint 时出现。
- Stop、Rollback、Terminate 与普通正向动作保持视觉隔离。

### 7.4 Provider Stream

日志展示规则：

- 容器高度稳定。
- stdout、stderr、event、provider_output 使用小标签区分。
- 日志正文等宽字体，支持换行和长词断行。
- 新事件自动滚动到底部，但保留用户可滚动查看历史的能力。

### 7.5 右栏 Inspector

顺序：

1. Workflow map
2. Node workspace
3. Evidence
4. Diagnostics

Workflow map：

- 不使用装饰性光斑或渐变轨道。
- 节点按钮显示 node ID、status、provider、attempt、artifact count。
- completed、running、blocked、failed、dropped 使用明确但克制的状态样式。

Node workspace：

- 概览、inputs、run、outputs、diff 保持 tab。
- ID/path/hash 使用等宽字体。
- 空 tab 显示简短空状态。

Evidence：

- 产物列表和预览保持清晰边界。
- Markdown/JSON/source/test/log 类型用一致标签区分。

Diagnostics：

- 可作为右栏面板或底部轻量面板。
- 不使用固定大面板遮挡主要工作区。

## 8. 弹窗与附属层

### 8.1 RollbackDialog

规则：

- 红色危险语义。
- dirty workspace 必须显式确认。
- 展示 checkpoint、git head、files may change、turn/node/provider/artifact drop count。
- 主按钮文案必须说明动作结果，不只写“确认”。

### 8.2 StartIssueDialog

规则：

- 单一任务：选择 workspace 并启动。
- policy/provider/timeout 若保留，视觉上属于高级设置，不抢主操作。
- 启动按钮使用主色；关闭使用次级样式。

### 8.3 Error 与 Loading

错误：

- 顶栏下方 alert。
- 显示错误信息和可恢复动作，如刷新或重试。
- 错误文本不泄露本地敏感路径，沿用后端错误脱敏。

Loading：

- 局部 loading，保持面板尺寸稳定。
- 不用全屏遮罩覆盖普通刷新。

## 9. 可访问性与交互

- 保留现有 aria label，不随意改测试依赖的 role/name。
- 所有按钮、select、tab、dialog 必须有可见 focus ring。
- clickable 元素必须有 `cursor-pointer`。
- disabled 状态必须同时通过视觉和 `disabled` 属性表达。
- 色彩不能作为唯一状态表达，必须有文本或图标辅助。
- 支持 `prefers-reduced-motion`，非必要动画必须可关闭或移除。

## 10. 实施边界

本设计只要求前端 UI 重构：

- 可新增轻量设计 token、layout helper 或共用 shell 组件。
- 可调整 Tailwind 配置和全局 CSS。
- 可重构现有 React 组件的 JSX 与 className。
- 不改 API 路径、DTO 字段或后端业务逻辑。
- 不引入新依赖。
- 不删除现有测试覆盖；必要时更新因文案或结构变化导致的前端测试 locator。

## 11. 验收标准

功能验收：

- 打开应用默认进入项目管理工作台。
- 项目管理工作台展示项目选择、Issue 列表、Issue 详情、Repository 面板和 Provider 摘要。
- 已启动 Issue 可进入执行工作台。
- 执行工作台展示状态带、Interaction Window、Workflow map、Node workspace、Evidence、Provider stream。
- Rollback、Gate、Error、Loading、Empty state 均使用统一视觉体系。

视觉验收：

- 页面不再出现大面积橙粉渐变、背景光斑、hero 插画式主视觉和厚重彩色阴影。
- 主页面在 1440px 桌面宽度下能稳定展示高密度工作台。
- 375px 移动宽度下无横向滚动，按钮文字不溢出。
- 颜色状态一致：青色正向、绿色完成、琥珀等待/Gate、红色危险、灰色中性。

验证命令：

- `pnpm --dir web test`
- `pnpm --dir web test:e2e`
- `pnpm --dir web build`
- 浏览器检查 `http://127.0.0.1:<vite-port>/` 的桌面和移动视口。

## 12. 建议实施顺序

1. 建立全局设计 token 和基础 CSS，移除装饰性背景与全局颜色覆盖 hack。
2. 重构项目管理工作台 shell、列表、详情、右栏和空状态。
3. 重构执行工作台 shell、状态带、Interaction Window 和 Provider stream。
4. 重构右栏 Inspector：Workflow map、Node workspace、Evidence、Diagnostics。
5. 统一 RollbackDialog、StartIssueDialog、GateActionBar、Error 和 Loading。
6. 跑完整前端单元、E2E、build，并做浏览器截图检查。

## 13. 自检记录

- 无待定字段或占位需求。
- 范围限定为当前用户可见页面，不包含历史兼容入口的全面改造。
- 设计不要求后端新增字段；repo 信息缺失时允许 UI 显示未绑定状态。
- 设计与 PRD v1.1 的 Acceptance 占位约束一致，不引入 Acceptance 自动状态机。
- 设计保留 Provider prompt 透明化边界，只展示摘要和引用，不默认展开完整 prompt。
