# 大文件拆分与模块边界重构设计方案

> 版本：v1.0 | 日期：2026-06-24 | 工作分支：feat-b-0616

## 1. 概述

### 1.1 目标

将仓库中超过 800 行的源代码文件逐步拆分到 800 行以内，在**保证程序正确、测试通过、外部 API 不变**的前提下，降低单文件认知负荷，明确模块职责边界，提升可维护性。

### 1.2 范围

| 范围 | 说明 |
|------|------|
| **第一阶段（本方案）** | `src/` 下超过 800 行的 Rust 生产代码文件 |
| **第二阶段（后续可选）** | `tests/` 下超过 800 行的集成测试文件 |
| **第三阶段（后续可选）** | `web/src/` 下超过 800 行的前端 TypeScript/React 文件 |

> 说明：测试文件和前端文件同样存在体积问题，但生产代码的 Rust 后端是核心复杂度所在，优先处理。

### 1.3 约束

- **不修改业务逻辑**：只做物理拆分，不改算法、状态机、协议语义。
- **保持外部 API 不变**：`src/product/mod.rs`、`src/web/mod.rs`、`src/cross_cutting/mod.rs` 中的模块声明不变；其他文件对该模块的 `use` 路径不变。
- **保持测试通过**：每次拆分后执行 `cargo check --locked` 和 `cargo test --locked`。
- **不引入新依赖**：只使用 Rust 原生模块系统完成拆分。
- **在 `.worktrees/feat-b-0616` 中实施**：工作区已就绪，`cargo check --locked` 通过。

### 2.2 共性模式

绝大多数大文件都是以下模式的组合：

1. **大类 + 大 `impl` 块**：`WorkspaceEngine`、`CodingWorkspaceEngine` 等方法数量超过 100 个。
2. **底层辅助函数堆积**：prompt builder、parser、DTO converter、path helper、ws event mapper 等被直接内联在大文件中。
3. **测试与生产代码同文件**：部分文件测试占比 30%–50%。
4. **数据模型跨职责域混合**：`models.rs`、`coding_models.rs`、`workspace_ws_types.rs` 尚未按业务域拆分。

## 3. 设计原则

1. **目录模块 + Facade 模式**
   - 将 `xxx.rs` 改为 `xxx/mod.rs`，内部拆分为多个子模块。
   - `mod.rs` 仅负责重新导出公共 API，保持外部调用方零改动。
2. **按职责域拆分**
   - 将 prompt、parse、DTO、state、controls、provider drive、review 等按职责拆到不同子文件。
   - 同一 `struct` 可以在多个子模块中拥有 `impl` 块，Rust 支持此模式。
3. **先拆纯函数，再拆 `impl` 块**
   - 风险最低：先把不依赖 `self` 的 helper 函数迁出。
   - 再拆类型定义。
   - 最后按领域拆分 `impl` 方法。
4. **先数据，后行为**
   - `models.rs`、`coding_models.rs`、`workspace_ws_types.rs` 是纯数据结构，拆分最容易，应优先完成。
5. **可见性最小化**
   - 对外 API 保持 `pub`。
   - 子模块间共享的实现细节使用 `pub(crate)` 或 `pub(super)`。
   - 必要时将 `struct` 字段提升为 `pub(crate)`，但优先通过 getter/setter 访问。

## 4. 方案对比

### 方案 A：目录模块 + Facade（推荐）

- 将每个大文件变为同名目录模块，内部拆分子模块，`mod.rs` 统一重新导出。
- 优点：职责清晰、外部 API 不变、可逐步实施、符合 Rust 社区习惯。
- 缺点：需要调整文件路径、部分 `impl` 块需要访问私有字段。

### 方案 B：同级辅助文件

- 在同一目录下新增 `xxx_prompts.rs`、`xxx_parsers.rs` 等文件，原文件保留主入口。
- 优点：改动最小、目录结构浅。
- 缺点：文件数量爆炸、命名容易混乱、后续扩展性差。

### 方案 C：仅拆分测试和纯函数

- 只把测试和纯 helper 函数迁出，保留大 `impl` 块不动。
- 优点：改动最小、风险最低。
- 缺点：无法解决核心大类过大问题，多数文件仍超 800 行。

**推荐方案 A**： directory module + facade 是结构性、可扩展的解决方案，能够一次性解决 800 行问题，同时保持 API 稳定。

## 6. API 兼容性策略

### 6.1 模块声明不变

`src/product/mod.rs`、`src/web/mod.rs`、`src/cross_cutting/mod.rs` 中：

```rust
pub mod workspace_engine;
pub mod coding_workspace_engine;
pub mod coding_attempt_store;
// ...
```

不需要修改。Rust 会自动寻找 `workspace_engine/mod.rs` 或 `workspace_engine.rs`。

### 6.2 Facade 重新导出

每个目录模块的 `mod.rs` 统一 `pub use`：

```rust
// src/product/workspace_engine/mod.rs
mod types;
mod lifecycle;
mod plan_outline;
// ...

pub use types::{
    AuthorDecisionOutcome, EngineEvent, PendingAuthorChoiceError, ReviewDecisionOutcome,
    WorkItemBatchDecisionOutcome, WorkItemDraftDecisionOutcome, WorkItemPlanAuthorOutcome,
    WorkItemPlanCompileRecoveryOutcome, WorkspaceEngine, WorkspaceSession, WorkspaceStage,
};
pub use prompts::build_work_item_plan_revision_input;
```

### 6.3 调用方无需修改

外部调用方如 `src/web/workspace_ws_handler.rs`：

```rust
use crate::product::workspace_engine::{
    AuthorDecisionOutcome, EngineEvent, /* ... */ WorkspaceEngine,
};
```

路径和导入项保持不变。

### 6.4 可见性调整

- 将 `WorkspaceEngine` / `CodingWorkspaceEngine` 的字段从 `private` 提升为 `pub(crate)`，或在 `types.rs` 中提供 `pub(crate)` getter/setter。
- 跨子模块调用的内部方法使用 `pub(crate)`。
- 尽量减少 `pub` 暴露面，避免破坏封装。

## 9. 风险与回滚

### 9.1 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 子模块间循环依赖 | 编译失败 | 严格按依赖方向拆分，底层 parsers/prompts 不依赖上层 engine |
| 私有字段访问问题 | 编译失败 | 统一提升为 `pub(crate)` 或提供 `pub(crate)` getter |
| 测试依赖私有 API | 测试编译失败 | 将测试保留在被测子模块内，或临时暴露 `#[cfg(test)] pub use` |
| 方法互相调用复杂 | 拆分困难 | 优先抽出纯函数，再逐步拆分 `impl` 块 |
| 一次性改动过大 | 难以 review | 按阶段分多次提交/PR，每次一个模块 |

### 9.2 回滚策略

- 每个阶段开始前创建 checkpoint commit。
- 每个阶段独立在一个 commit/PR 中。
- 若某阶段测试失败且短时间内无法修复，直接回滚该阶段 commit，不影响其他阶段。

1. **单文件认知负荷降低**：每个文件聚焦一个职责域，便于理解和修改。
2. **编译并行度提升**：更多独立翻译单元，可更好地利用 `cargo` 并行编译。
3. **代码审查更容易**：PR 范围更小、职责更单一。
4. **测试定位更精确**：可按子模块运行定向测试，减少反馈时间。
5. **新功能接入更顺畅**：新增职责时可直接新增子模块，而不是往大文件里追加。
