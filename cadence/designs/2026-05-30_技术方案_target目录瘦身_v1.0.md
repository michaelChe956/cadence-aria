# 技术方案：target 目录瘦身（结构性根治）

- **日期**：2026-05-30
- **版本**：v1.0
- **分支**：product-workbench-issue-lifecycle
- **目标**：结构性根治 target 目录膨胀，让未来构建产物可控（不纠结当前 60G，随时可 `cargo clean`）

---

## 1. 背景与根因分析

worktree `product-workbench-issue-lifecycle` 的 `target` 目录达 **60G**（主仓库 target 仅 7.3G）。实测大小构成：

| 目录 | 大小 | 占比 |
|------|------|------|
| `target/debug/deps` | 47G | 78% |
| `target/debug/incremental` | 13G | 22% |
| `target/debug/build` | 445M | <1% |

### 根因（按贡献排序）

**根因 1（主因，属"用得不规范"）：88 个集成测试文件 → 88 个独立二进制**

`tests/` 目录下有 88 个 `.rs` 文件。Rust 中 `tests/` 下每个文件都被编译为一个独立可执行二进制，且静态链接整个 crate（`libcadence_aria`）+ 全部依赖。实测每个测试二进制 120–146M：

```
146M  workspace_ws_integration
136M  web_coding_ws_handler
131M  web_cli
127M  web_provider_output_events
125M  （web_* / issue_* 等十余个均为 125M）
```

当前一批 88 个二进制 ≈ 11G，加历史累积构成 deps 47G 的主力。这是可从根上改造的部分。

**根因 2：重量级 git 依赖**

```
336M  libcodex_app_server_protocol  （来自 openai/codex git 依赖）
204M  libstarlark                   （间接依赖）
```

两者共 540M，且会被链入每个测试二进制，放大根因 1。

**根因 3：陈旧产物从不回收（deps 含 4543 个文件）**

`libcadence_aria` 在 deps 中存在大量不同 hash 版本（`93fcb0f`、`5a4d7e7`、`951eaa`、`9fe0d25`…）。Cargo 每次代码变更生成新 hash 产物，旧产物不会自动删除。

**根因 4：增量编译缓存 13G**

`incremental/` 下 10+ 个 `cadence_aria-*` 目录，每个 350–618M，累积型缓存。

### 结论

根因 1 是真正的"Rust 用得不够规范"，可从结构上根治；根因 2/3/4 属于 Rust debug 构建 + 多依赖 + 缓存累积的固有特性，靠 profile 调优 + 缓存 + 定期清理缓解。

---

## 2. 方案总览

三条主线：

1. **集成测试分组合并**（主线，治本）
2. **profile 调优**（降单产物体积）
3. **sccache + 清理脚本**（卫生/加速）

确认的关键取舍：
- 测试合并力度：**分组合并**（6 个入口二进制），非全合一、非不动。
- 本地代码调试信息：**`line-tables-only`**（保留行号/堆栈，去掉变量级调试）。
- 落地范围：**三部分全做**并验证。

---

## 3. 第一部分：集成测试分组合并（核心）

### 3.1 分组方案

将 88 个文件按前缀归到 **6 个入口二进制**，原文件降级为 `mod`：

| 入口二进制 | 归入测试（按现有前缀） | 文件数 |
|-----------|----------------------|--------|
| `it_web` | `web_*` | 20 |
| `it_product` | `product_*` | 10 |
| `it_task_run` | `task_run_*` + `task_init_*` | 8 |
| `it_interactive` | `interactive_*` + `repl_*` + `daemon_*` | 13 |
| `it_provider` | `provider_*` + `recovery_*` + `execution_*` + `planning_*` + `final_*` | ~10 |
| `it_core` | 其余全部（artifact/design/spec/plan/openspec/risk/cli/context 等） | ~27 |

> 分组以"文件名前缀 + 功能域"为准，最终归类在实施时按实际依赖微调，但二进制总数保持 6 个。

### 3.2 目录改造方式

````
tests/
  it_web.rs            # 入口：仅写 mod 声明
  it_web/              # 子模块目录
    web_lifecycle_api.rs
    web_product_api.rs
    ...
  it_product.rs
  it_product/
    ...
  （其余 4 组同构）
  fixtures/            # 保持不动
  support/             # 共享 helper，保持不动
````

入口文件示例（`it_web.rs`）：

```rust
#[path = "it_web/web_lifecycle_api.rs"] mod web_lifecycle_api;
#[path = "it_web/web_product_api.rs"] mod web_product_api;
// ... 其余 web_* 文件
```

### 3.3 三个必须处理的坑（已验证）

1. **`include_str!` / 相对路径 fixture**
   - 问题：`include_str!("fixtures/...")` 是相对**当前源文件**的编译期路径。文件移入 `it_web/` 子目录后路径断裂。
   - 解法：将这类编译期相对路径统一改为
     `include_str!(concat!(env!("CARGO_MANIFEST_DIR"), "/tests/fixtures/..."))`。
   - 运行期相对路径（如 `read_document_model("tests/fixtures/artifacts/plan.md")`）**不受影响**，因为 `cargo test` 的 CWD 始终是 crate 根。

2. **顶层函数重名**
   - 各 `mod` 有独立命名空间，合并后不冲突（已确认 helper 多为文件私有 `fn`）。
   - `support` 模块：各入口按需 `mod support;`（指向 `tests/support/mod.rs`，用 `#[path]` 引）。

3. **`web_coding_ws_handler.rs` 内已有 `mod tests {}`**
   - 嵌套 mod 无问题，原样搬入即可。

### 3.4 收益与代价

- **收益**：测试二进制从 88 × ~125M ≈ 11G → 6 × ~150M ≈ 1G 量级。
- **代价**：改同组任一测试需重链整组（~150M 链接）；增量编译仍只重编改动文件。

## 4. 第二部分：profile 调优

在 `Cargo.toml` 末尾加入：

```toml
# 开发/测试构建：用更轻的调试信息
[profile.dev]
debug = "line-tables-only"   # 从默认 full 降到只保留行号表；保留断点/堆栈行号

# 依赖项不带本地调试信息，并带轻度优化
[profile.dev.package."*"]
debug = false
opt-level = 1
```

| 配置 | 效果 | 代价 |
|------|------|------|
| `debug = "line-tables-only"` | 调试信息体积大幅下降（debug 段常占 rlib 40–60%） | 失去变量级调试，保留行号/堆栈 |
| 依赖 `debug = false` | codex(336M)、starlark(204M) 等不再带 debuginfo | 无法单步进依赖源码 |
| 依赖 `opt-level = 1` | 依赖运行更快，测试更快 | 依赖首次编译略慢（只编一次） |

不改变任何代码行为，可随时回退（需深入调依赖时临时改 `debug = true`）。与第一部分叠加：6 个测试二进制各自再瘦身。

## 5. 第三部分：sccache + 清理脚本

### 5.1 sccache（编译缓存）

在 `.cargo/config.toml` 增加（不污染全局）：

```toml
[build]
rustc-wrapper = "sccache"
```

- 前提：本机已装 sccache。脚本探测，未装则跳过并提示（`cargo install sccache` 或包管理器）。
- 缓存默认 `~/.cache/sccache`，有大小上限（默认 10G）+ LRU 淘汰，不会无限增长。
- 定位：主要**加速重复编译 + 节省跨 worktree 重复产物**，对单 target 瞬时大小帮助有限，与第 1/2 部分互补。

#### 5.1.1 关键限制与取舍（评审补充）：sccache 对本 crate 默认不生效

sccache 有一条文档级硬限制：**当某次 rustc 调用开启了增量编译（incremental），sccache 会直接放弃缓存、回退到普通编译**（见 sccache README 的 "Rust" 一节）。而 cargo 的 `dev`/`test` profile **默认 `incremental = true`**，本 crate `libcadence_aria`（编译大头）正走这条路径。

后果：plan 原样接 sccache，**sccache 只对依赖（deps）命中，对最大的本 crate 几乎不缓存**；而依赖本就变动少、重编频率低，sccache 在这里的实际加速有限。同时设计把 incremental 13G 列为"根因 4"却只靠清理脚本删它——删了下次构建照样长回来，治标不治本。

由此引出一个 plan 未讨论的取舍：

| 方案 | incremental | 本 crate 是否被 sccache 缓存 | 13G 增量缓存 | 代价 |
|------|-------------|------------------------------|--------------|------|
| **A：plan 原样** | 开（默认） | 否（sccache 回退） | 靠脚本删，会再生 | 保留 Rust 原生增量快编 |
| **B：关 incremental + sccache** | 关（`CARGO_INCREMENTAL=0`） | 是（sccache 命中本 crate） | **从源头不再产生** | 失去原生增量；改动后整体重编，由 sccache 部分补偿 |

- 方案 B 把"缓存"职责从 Rust 原生 incremental 转交给 sccache：既让 sccache 真正缓存本 crate，又让那 13G 不再产生，与"根治"目标更一致。
- 方案 A 适合"频繁改少量代码、依赖本机原生增量快编"的日常迭代；代价是接受 sccache 对本 crate 形同虚设、且 incremental 缓存需定期清理。
- **未决项**：两方案的实测收益对比建议在 Task 0 基线之后做一次小实验确认（改一行代码触发重编、分别计时），再定。确认前 plan 的 Task 2 暂按方案 A，并保留切到方案 B 的余地（仅需在 `.cargo/config.toml` 的 `[build]` 加 `incremental = false` 或设环境变量）。

### 5.2 清理脚本 `scripts/clean-target.sh`

````bash
# 三档清理：
#   --stale   只删 deps/ 陈旧产物（保留最新一组）+ 清 incremental 缓存
#   --all     等价 cargo clean（全删）
#   --dry-run 只报告将删除多少，不动手
````

`--stale` 档解决"4543 个文件累积"：只回收旧 hash 产物和增量缓存，保留当前可用产物，避免下次全量重编。

## 6. 实施顺序与验证

1. 先做**第二部分（profile）** + **第三部分 5.1（sccache config）**——改动小、零风险、立即生效。
2. 再做**第一部分（测试合并）**——分 6 个入口逐组迁移，每迁一组跑 `cargo test --test it_xxx` 验证通过再继续。
3. 最后加**第三部分 5.2（清理脚本）**。
4. 全部完成跑一次完整 `cargo test`，确认 88 个测试逻辑全部保留、全绿。

### 验证基线

- 改造前：`cargo clean` → 完整构建（含全部测试）→ 记录 target 大小。
- 改造后：同流程对比。
- 衡量成败用**苹果对苹果的干净构建对比**（见第 7 节），而非当前 60G 这个虚高基数。

#### ✅ 实测对比（2026-05-30，干净构建 `cargo clean` + `cargo test --no-run`）

| 指标 | 改造前 | 改造后 | 降幅 |
|------|--------|--------|------|
| **target 总计** | **15G** | **2.6G** | **↓ 82%** |
| deps | 13G | 1.7G | ↓ 87% |
| incremental | 1.5G | 775M | ↓ 48% |
| build | 318M | 72M | ↓ 77% |
| 集成测试二进制数 | 88 | **6** | — |
| 集成测试二进制体积 | 9.42G | 0.29G | ↓ 97% |

实测 **2.6G < 设计目标下限 3G**，三条主线叠加效果：测试合并（88→6）消除重复链接，profile（`line-tables-only` + 依赖 `debug=false`）让每个产物再瘦身。全量 `cargo test` 553 passed / 0 failed，测试逻辑无损。

## 7. 收益拆账（重要：避免误读）

当前 60G **不是一次构建的成本，而是长期不清理累积出来的**。证据：deps 含 4543 个文件，`libcadence_aria` 同时存在 4+ 个不同 hash 旧版本（每个 130–217M）；incremental 有 10+ 个 `cadence_aria-*` 目录（每个 350–618M）。若现在（改造前代码）`cargo clean` 后干净构建一次，target 约只有 **15G 量级**，多出的 ~45G 是历史垃圾。

因此"从 60G 降下来"要分两笔性质不同的账：

| 来源 | 量级 | 性质 |
|------|------|------|
| **历史垃圾回收** | ~45G | 一次性。`cargo clean` / 清理脚本立刻拿回，但**不治本**——不改结构还会再涨回去 |
| **结构性根治** | 干净构建 ~15G → **3–5G** | 持久。本方案的核心目标，约省 10–12G/次，并让 target **不再涨回 60G** |

**真正衡量本方案成败的指标是干净构建的前后对比（~15G → 3–5G），不是 60G → 3–5G。** 后者把一次性垃圾回收的功劳混入了结构改造。结构改造的 ~12G 降幅主要来自：

1. 测试二进制 11G → ~1G（88 → 6 个入口）。
2. profile（line-tables-only + 依赖 debug=false）让 deps/产物各自瘦身约 40–60%。

> **数据性质说明**：上述 15G、3–5G 原为**基于实测文件大小的估算**。**✅ 已于 2026-05-30 实跑 Task 0 验证**：`cargo clean` 后 `cargo test --no-run` 干净构建，实测 **target = 15G**（deps 13G、incremental 1.5G、build 318M），与估算的 15G 吻合。其中 **87 个集成测试二进制 = 9.42G**（最大者 150M，主力 125–150M），lib/bin 单测二进制仅 0.26G。改造后目标 3–5G 仍待 Task 11 实测确认。

> **⚠️ 估算张力（评审补充，已被 Task 0 实测排除）**：评审期曾担心一个反证——**主仓库 target 仅 7.3G**，若主仓库已接近其干净构建量级，则 worktree 干净构建可能只有 7–8G，会动摇"结构改造省 ~12G"的论证。**Task 0 实测（15G）已排除此担忧**：worktree 干净构建确为 15G，主仓库 7.3G 的差异更可能源于主仓库未编译全部测试或测试集不同，而非本 worktree 虚高。**结论：测试合并性价比成立**——87→6 个入口预计省约 8G（9.42G→~1G），叠加 profile 后单个二进制再瘦，是 15G→3–5G 的主力来源。

**方案的核心价值不在"省了多少 G"，而在于把"每次构建留一堆垃圾、越攒越多"改成"每次构建产物本身就小，加清理脚本兜底"——让 target 不再悄悄涨回 60G。**

## 8. 风险与回退

| 风险 | 缓解 |
|------|------|
| 测试合并破坏 fixture 路径 | 统一改 `CARGO_MANIFEST_DIR` 编译期路径；逐组验证 |
| 合并后命名冲突 | 各 mod 独立命名空间，已验证无顶层冲突 |
| profile 影响调试体验 | line-tables-only 保留行号；可随时改回 |
| sccache 未安装 | 脚本探测跳过，不阻断构建 |
| 分组迁移引入回归 | 每组独立验证 + 末尾完整 `cargo test` 全绿 |
| **进程级全局状态串扰**（评审补充） | `web_test_controls.rs` 用 `env::set_var/remove_var("ARIA_E2E_TEST_CONTROLS")` 改进程级环境变量。合并进 `it_web` 后与同组其他 20 个测试**共享同一进程**，cargo 默认多线程并行跑测试 → 存在 set/remove 时序串扰风险。**缓解**：经核查该 env 仅被 `web_test_controls` 自身的 2 个测试读写（`src/web/test_controls.rs::is_e2e_enabled` 是唯一读取点，其余 web 测试不构建走该开关的 router），实际串扰面为零；但为稳健，迁移 `it_web` 后建议给这 2 个测试加 `#[serial]`（引入 `serial_test`）或合并为单个测试函数，消除并行隐患。端口已全部 `127.0.0.1:0` 随机分配，无冲突。 |

> **开发体验代价（评审补充，需正视）**：合并后**无法再单独编译/运行单个测试文件**。改 `it_web` 里任一测试，`cargo test --test it_web` 会重链整组（20 个文件）并拉起全组用例；想只跑某个用例只能靠 `cargo test --test it_web 测试函数名` 过滤（仍需先编整组）。对"高频改单个测试"的迭代，手感下降明显。这是测试合并的固有代价，与第 7 节"是否值得"的判断绑定——**Task 0 实测干净构建为 15G（测试二进制占 9.42G），合并省约 8G，体验代价被判定为值得付出**。

回退：profile/sccache 改 config 即回退；测试合并通过 git 可整体回滚。

## 9. 范围边界（YAGNI）

- **不做**：审视/裁剪 codex、starlark 等重型 git 依赖（收益不确定，本次不纳入）。
- **不做**：迁移到 Cargo workspace 多 crate 拆分（改动过大，超出本次范围）。
- **不做**：CI 层面的缓存策略（聚焦本地构建产物）。
- **不做（评审补充，记录备选）**：把 88 个里的纯函数/序列化单测（如 `*_schema_min_fields`、`product_coding_models`、`artifact_validate`、`runtime_units` 等不依赖完整 app/网络的用例）**下沉到 `src/` 对应模块的 `#[cfg(test)] mod tests`**。这类测试编进 lib 单测后**完全不产生独立测试二进制**，比"合并成 6 个入口"更彻底（6 个入口仍各链一份 crate+依赖）。
  - **为何本次不做**：① 工作量远大于分组合并（要逐个判断哪些测试不依赖集成环境、并改写其 `use cadence_aria::...` 为模块内路径）；② 与"分组合并成 6 个"目标重叠，边际收益递减；③ 下沉后这些测试会随 `cargo test --lib` 跑、不再是黑盒集成测试，**语义有变**（白盒访问私有项），需逐个评估是否可接受。
  - **何时重提**：若 Task 0/Task 11 实测显示 6 个入口二进制仍是 target 大头，且团队希望进一步压缩，可作为 v2.0 的独立议题，配合一次实测筛出"纯单测"清单再评估。
