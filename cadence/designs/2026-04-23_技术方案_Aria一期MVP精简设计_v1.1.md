## 1. 目标与范围

### 1.1 一期 MVP 定位

- **形态**：REPL client + 本地 daemon runtime
- **范围**：单用户、本机、单仓库、多任务并行（通过 worktree 隔离）
- **Provider 接入基线**：`spawn + CLI`（Claude Code 作为 orchestrator，Codex 作为 executor/reviewer）
- **隔离策略**：每任务独立 worktree
- **集成策略**：并行执行、串行集成（一期默认 FIFO）
- **人工介入**：保守策略，关键节点挂 approval gate
- **Daemon 生命周期**：daemon 常驻，REPL 只是客户端，可 attach/detach

### 1.2 文档定位与真相源边界

本文件的定位不是“重写上游协议”，而是：

1. 在**不改变上游协议真相源**的前提下，定义 Aria 一期如何实现。
2. 明确哪些能力一期完整实现、哪些简化实现、哪些延期到二期。
3. 明确为了降低工程复杂度，一期如何把多个协议节点折叠为较少的**实现单元**。

本文件**不替代**上游文档集中以下内容：

- 节点 ID、节点类型、上下游路由
- 横切能力 ID 与能力边界
- 产物最小字段与校验规则
- 全局协议章节与统一枚举值

**冲突裁定规则：**

- 若本文件与上游文档集存在冲突，以上游文档集为准。
- 本文件只允许做“实现分期裁剪”，不允许做“协议级重定义”。

### 1.3 一期不做的事（不代表删除协议定义）

| 项目 | 一期策略 | 说明 |
|------|---------|------|
| Risk Registry 完整生命周期 | 不完整实现 | 保留 `riskRegistry` 结构与 `riskId` 兼容入口，二期补完跨产物生命周期 |
| 各类循环独立阈值调优 | 不单独调优 | 保留命名计数器与 schema，一期先共用一组阈值配置 |
| Event Log compaction | 不启用后台 compaction | 保留 append-only、checksum、恢复机制；compaction 二期启用 |
| Gate TTL 自动升级 | 不启用超时扫描器 | 保留 TTL 字段和协议定义，一期默认 `null` |
| N26 自动补丁派生 | 不自动执行 | 保留协议分支与节点定义，一期默认 gate / manual intervention |
| TUI / GUI 目标态 | 不实现 | 保留 daemon/client 分层，二期再扩展前端形态 |

### 1.4 关键假设（显式声明）

1. **一期的主接入路径是 `spawn + CLI`**  
   这是一期最小可落地闭环的基线方案。SDK / app-server 是明确保留的二期升级路径，不视为不可行。
2. **daemon 常驻，REPL 只是客户端**  
   REPL 启动时 attach 到已有 daemon 或隐式启动新 daemon；REPL 退出时仅断开连接，daemon 继续运行；任务在后台持续执行。
3. **worktree 隔离优于 branch 隔离**  
   并发任务需要真正的文件系统隔离；只切 branch 无法满足并行编码与回归验证的需要。
4. **对话分叉与代码分叉分开处理**  
   会话管理靠 session/thread（`ProjectSession`、`Phase`），代码隔离靠 worktree（`WorktreeLease`）。两者是独立的生命周期：一个 session 可以对应多个 worktree，一个 worktree 的回收不影响 session 状态。

### 1.5 可扩展性保障

**前端形态扩展保障**

核心原则：**REPL 只负责交互，daemon 是运行时真相源**。无论前端形态是 REPL、TUI、GUI 还是 Web，它们都只是 daemon 的客户端，通过统一接口通信。

一期 REPL 与 daemon 通过 JSON over Unix socket 通信，Windows 环境使用 named pipe。二期做桌面端时，只替换客户端，不改变 daemon 核心状态机。

**Provider 接入扩展保障**

一期通过 Provider Adapter 把 daemon 与底层执行器解耦：

```text
daemon -> ProviderAdapter -> [CliAdapter | SdkAdapter | AppServerAdapter | ...]
```

一期只落地 `CliAdapter`。二期新增 `SdkAdapter` 或 `AppServerAdapter` 时，不应要求 daemon 重写协议层。

## 2. 节点模型与一期实现单元

### 2.1 双层模型

Aria 一期采用**双层模型**：

1. **协议层**：仍然是上游文档集中定义的 `N00-N28` 与 `X01-X09`
2. **实现层**：一期为了降低代码复杂度，使用较少的“实现单元”落地运行

关键约束：

- 实现单元不是新的协议节点 ID
- 协议层的节点定义、节点顺序、协议路由保持不变
- 即使多个协议节点由同一个实现单元处理，运行时仍应按协议边界产出对应的产物、checkpoint 和状态变迁记录

- **原有节点类型定义**：Aria 内部 / Agent 业务 / 混合

## 4. 横切能力的一期实现范围

### 4.1 基本原则

一期可以标注“完整实现 / 简化实现 / 延后子特性”，但：

- **不允许改横切能力 ID**
- **不允许把一个能力拆成两个新编号**
- **不允许把不同能力写成同一个编号**

### 4.2 能力矩阵

| 能力 ID | 能力名 | 一期状态 | 说明 |
|--------|-------|---------|------|
| `CC01` | `approval_gate` | 完整实现（无 TTL 扫描） | gate 创建/恢复必须；TTL 字段保留，默认 `null` |
| `CC02` | `policy_mode_and_override` | 简化实现 | 一期默认 `conservative`，保留 override 结构 |
| `CC03` | `provider_run_lifecycle` | 完整实现 | provider run 派发、收集、固化记录 |
| `CC04` | `artifact_validate` | 完整实现（L1/L2） | L3 语义性校验二期增强 |
| `CC05` | `checkpoint_and_recovery` | 完整实现（无 compaction） | checksum、恢复必须；compaction 二期启用 |
| `CC06` | `worktree_lifecycle` | 完整实现 | worktree 租约、创建、回收 |
| `CC07` | `integration_queue` | 简化实现 | 一期默认 FIFO；优先级/拓扑调度二期增强 |
| `CC08` | `manual_intervention` | 完整实现 | 终态兜底能力 |
| `CC10` | `provider_adapter_spec` | 完整实现 | 一期落地 CLI adapter，二期扩展 SDK/app-server |

### 4.3 一期简化细节

**`CC02 policy_mode_and_override`**

- 默认启用 `conservative`
- `balanced` / `aggressive` 的协议定义保留，但一期不要求完整激活
- 阶段级覆写数据结构保留，落地可先最小化

**`CC07 integration_queue`**

- 一期默认 FIFO
- **回流上限一期必须实现**：同一 WorkTask 连续 2 次集成失败后不再自动出队，升级到 `X08 manual_intervention`，reason 为 `integration_retry_limit_exceeded`。此机制与 `rework_counter` 类似，是防止集成死循环的兜底，实现成本低（计数器 + 判断），不涉及复杂的拓扑调度
- 队列持久化必须保留
- 紧急插队、依赖拓扑感知可延期，但不能删协议字段和扩展入口

**`CC01 approval_gate`**

- gate 基本生命周期必须实现
- TTL 字段和超时语义保留
- 一期不启用后台超时升级扫描

**`CC05 checkpoint_and_recovery`**

- checkpoint 落盘、checksum 校验、恢复逻辑必须实现
- Event log 保持 append-only
- compaction 规则保留为二期增强，不在一期调度器中主动执行

### 6.1 技术栈

### 6.2 REPL 架构（一期）

- **daemon** 是独立长驻进程

## 7. 交付边界与成功标准

### 7.1 交付边界

**本设计文档负责定义：**

- 一期如何在不改上游协议的前提下落地
- 哪些协议节点由哪些实现单元承载
- 哪些横切能力一期完整实现、简化实现、延期增强
- 一期技术栈、目录结构与运行边界

**本设计文档不负责定义：**

- 各节点 prompt 细节
- 具体 TDD 实现任务拆分
- 代码级提交粒度和研发排期

### 7.2 成功标准

一期成功，不仅要求“流程能跑”，还要求“协议不漂移”。

**必须同时满足：**

1. 系统可以完成一次真实需求的主流程：澄清 -> 规格 -> 设计 -> 计划 -> 分派 -> 编码 -> 测试 -> 评审 -> 集成 -> 总结
2. 所有输出产物都能按上游 canonical schema 校验通过
3. 即使使用实现单元合并，运行时仍能按协议边界记录 checkpoint / artifact / route
4. `N25 -> N26/N27`、`X09` 等协议定义仍然保留，只是一期运行策略可限制其自动化程度
5. daemon 可在 REPL 断开后继续运行，REPL 可重新 attach

**最低可接受标准：**

- 主流程跑通
- protocol/schema/ID 无冲突
- 集成阶段允许保守实现，但不得破坏上游协议字段和路由
