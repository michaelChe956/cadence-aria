# 技术方案：构建与测试命令提速（压进 2 分钟）

- **日期**：2026-05-30
- **版本**：v1.0
- **分支**：product-workbench-issue-lifecycle
- **目标**：
  1. 让以下三个命令在本机（手工验收场景）耗时压到 2 分钟以内
     - `cargo test --locked -j 1`
     - `cargo check --locked`
     - `cargo clippy --all-targets --all-features --locked -- -D warnings`
  2. 新增项目强制规则，固化提速后的规范命令，防止 `-j 1` 等慢命令回潮

---

## 1. 结论先行（TL;DR）

**唯一的真正瓶颈是 `cargo test` 命令里写死的 `-j 1`。** 它强制串行编译，让本机 23 个核全程闲置，把本该 1.5 分钟的活拖成 6 分钟。去掉它，三个目标即全部达成。

- 核心改动只有一处性质：**手工验收命令不要再带 `-j 1`**，并把 CI 同步改掉。
- 配套加一条 `[build] jobs = 8`，把并行度钉在"本仓库编译图能吃满的宽度 + 本机内存预算"的最优点上（既不更慢，又不抖 swap）。
- 探查中实测过的 rust-lld 链接器**没有收益（反而更慢），明确不采纳**——已替你试过，省得折腾。

---

## 2. 环境与瓶颈实测

### 2.1 机器画像（关键约束）

| 项 | 值 | 含义 |
|----|----|------|
| CPU | **24 核** | 并行能力充足 |
| 内存 | **7.6 GB**（WSL2，受 Windows `.wslconfig` 限制） | 紧张：空闲时已占用 1.4 GB swap |
| 工具链 | rust 1.95.0，自带 `rust-lld` | 链接器可选项已就位 |
| 测试规模 | `tests/` 6 个入口二进制（已做过合并），553 个测试 | 链接量集中在 6 个 ~150MB 二进制 |

### 2.2 实测数据（均在本 worktree 实跑，场景=改一处 `src` 代码后）

| 命令 | 耗时 | 峰值内存 | CPU 利用 | 判定 |
|------|------|---------|---------|------|
| `cargo test --locked -j 1`（现状·编译链接） | **6m16s** | 2.0 GB | 98%（仅 1 核） | ❌ 超标 |
| `cargo test --locked`（默认并行·编译链接） | **1m31s** | 3.0 GB | ~5 核 | ✅ |
| `cargo test --locked -j 8`（含运行 553 测试·全绿） | **1m25s** | 2.9 GB | ~4 核 | ✅ 达标 |
| `cargo check --locked`（增量） | **4.7s** | 0.6 GB | — | ✅ |
| `cargo clippy --all-targets …`（增量） | **8.3s** | 0.8 GB | — | ✅ |

> 关键观察：`-j 8` 与默认 `-j 24` 墙钟几乎一样（1m25s vs 1m31s）。原因是本仓库编译图是**串行瓶颈型**（`libcadence_aria` 必须先编完，6 个测试二进制才能开始），有效并行宽度只有约 5–8。超过 8 的并行度对本仓库**不再加速，只多吃内存**。

### 2.3 为什么 check/clippy 也被感觉"很慢"

实测增量场景下 check 4.7s、clippy 8.3s，本身不慢。被感觉慢的真实原因有二，都由 `-j 1` 间接引发：

1. **内存争用**：基线已占 1.4 GB swap，内存本就紧。`cargo test -j 1` 跑 6 分钟期间若同时跑 check/clippy，会一起挤进 swap 抖动，拖慢一切。
2. **串行等待**：6 分钟的 test 让整轮验收体感极慢。

去掉 `-j 1` 的长跑后，内存压力解除，check/clippy 的增量速度（个位数秒）即是常态。

---

## 3. 根因

> CI 配置 `.github/workflows/ci.yml` 把测试命令写成 `cargo test --locked -j 1`，手工验收沿用了该命令。
> `-j 1` = 强制单线程编译。24 核机器只用 1 核，纯粹浪费，与内存无关（峰值才 2 GB）。

`-j 1` 通常是为规避并行编译 OOM 才设的，但本仓库实测并行峰值仅 3 GB（余量 4.6 GB），**没有 OOM 风险**，`-j 1` 属于无必要的保守设置（已与你确认：可以改，无特殊原因）。

---

## 4. 方案（改动清单）

### 改动 1【核心·已实测】去掉 `-j 1`

**`.github/workflows/ci.yml`**：

````diff
-        run: cargo test --locked -j 1
+        run: cargo test --locked
````

- 效果：test 6m16s → ~1m25s。CI runner（4 核/16GB）同样受益（默认并行 4，无内存压力）。

### 改动 2【确定性 + 内存安全·已实测支撑】钉住并行度

**`.cargo/config.toml`** 增加 `[build]` 段：

````toml
[build]
jobs = 8        # 本仓库编译图有效并行宽度约 5–8；钉 8 既不更慢又不抖 swap
````

理由（数据支撑，非过度防御）：

- 本仓库 `-j 8` 与 `-j 24` 墙钟一致（见 2.2），但 `-j 8` 峰值 2.9GB、swap 稳定；放任默认 24 在冷构建（6 个测试二进制可能同时链接）时内存峰值未实测、有 swap 抖动风险。
- `-j 8` 是"同样快 + 更低风险"的严格更优点，且对本仓库的串行瓶颈型编译图近乎最优——与核数多少无关。
- 副作用：本地 check/clippy/test（不带显式 `-j`）统一走 8 并行，**无需记任何 `-j` 参数**。
- 对 CI（4 核）：`8 > 4` 轻微超订，无害（16GB 内存，OS 自行调度）。

> **取舍说明（供评审定夺）**：`.cargo/config.toml` 是提交到版本库、对所有机器/CI 生效的。若你希望"只约束本机、不影响他人"，可改为**不提交此项**，本机用环境变量 `export CARGO_BUILD_JOBS=8` 代替。鉴于本仓库串行瓶颈使 8 近乎通用最优、且项目目前为单人维护，本方案**默认建议直接提交** `jobs = 8`。

### 改动 3【关键·统一手工验收命令】

> ⚠️ **必须知道**：命令行显式 `-j 1` 会覆盖 config 里的 `jobs = 8`。所以**只改 config 不够**，手工验收时必须用去掉 `-j 1` 的新命令，否则 test 仍是 6 分钟。

新的手工验收命令（与 CI 完全一致）：

````bash
cargo test --locked
cargo check --locked
cargo clippy --all-targets --all-features --locked -- -D warnings
````

### 改动 4【新增强制规则·固化规范命令】

> ⚠️ **现存规则冲突必须处理**：`cadence/project-rules/README.md` 的「已启用项目规则」中当前明确写着 `cargo test --locked -j 1`，正是本方案要废除的命令。若只改 config/CI 而不改规则，规则会与方案自相矛盾，且 Agent/开发者照规则仍会敲出 `-j 1`。故本改动既"新增"也"修正"。

落点（已与负责人确认）：**独立规则文件 + 保留定向测试命令**。

1. **新建** `cadence/project-rules/build-test-commands.md`：作为构建/测试/检查命令的强制规范正文。内容要点：
   - 全量验证统一用：`cargo test --locked` / `cargo check --locked` / `cargo clippy --all-targets --all-features --locked -- -D warnings` / `cargo fmt --check`。
   - **禁止** `cargo test` 携带 `-j 1`（会覆盖 `.cargo/config.toml` 的 `jobs=8`，使测试退化为 6 分钟串行）。
   - **保留**定向快反馈命令：`cargo test --locked --lib <过滤名>`、`cargo test-approval-bridge`（这些只跑单测、本就不应带 `-j 1`）。
   - 并行度由 `.cargo/config.toml` 的 `jobs=8` 统一托管，命令行无需也不应再写 `-j`。
2. **修正** `cadence/project-rules/README.md` 「已启用项目规则」段：把 `cargo test --locked -j 1` 改为 `cargo test --locked`，并新增对 `build-test-commands.md` 的引用条目。
3. **在 `CLAUDE.md` 与 `AGENTS.md`** 的「项目个性化规则」处加引用，使新规则文件正式启用（与现有引用机制一致）。

## 5. 明确排除项（已实测，无收益，不做）

### rust-lld 链接器

工具链自带 `rust-lld`，理论上链接更快。**已实测，不采纳**：

| 链接器 | `test --no-run` 稳态（改一处 src） |
|--------|-----------------------------------|
| GNU ld 2.42（现状） | **1m31s** |
| rust-lld（`-fuse-ld=lld`） | **1m56s**（反而更慢） |

原因推测：本仓库二进制规模下，GNU ld 2.42 已够快，lld 的优势被 `-B` 间接寻址开销 + 本机内存/swap 抖动抵消。且 lld 的可移植配置（避免写死工具链绝对路径）在 stable 1.95 上较繁琐。**收益为负，直接排除。**

### 其他不做项（YAGNI）

- 不动 `Cargo.toml` 现有 profile（`debug=line-tables-only` / 依赖 `debug=false` / `opt-level=1`）——这些是上一轮 target 瘦身的成果，对链接体积/速度有利，无需为本目标改动。
- 不引入 mold（需 sudo 安装，rust-lld 都没收益，mold 不预期更好）。
- 不改 CI 的缓存策略（已有 `Swatinem/rust-cache@v2`）。

## 6. 冷构建（低频）兜底

- 手工验收发生在**改完代码之后**，属**增量构建**——即 2.2 表中的场景，三命令均已 < 2 分钟。
- **冷构建**（`cargo clean` / 切分支后首次）需重编全部重型 git 依赖（codex、starlark），是唯一可能超 2 分钟的低频场景，不在高频验收路径上。
- 若确需加速冷构建：项目已支持**本地按需** sccache（见 `cadence/readmes/2026-05-30_README_sccache可选缓存_v1.0.md`），`export RUSTC_WRAPPER=sccache` 即可，对全量重建依赖命中显著。不写入版本库（避免破坏 CI）。

## 7. 风险与回退

| 风险 | 评估 | 缓解/回退 |
|------|------|----------|
| 并行编译 OOM | 极低。实测峰值 3GB，余量 4.6GB；`jobs=8` 进一步收紧 | 出现则下调 `jobs` 至 4 |
| swap 抖动拖慢 | 低。`-j 8` 实测 swap 稳定（1395→1527MB） | 同上 |
| `jobs=8` 影响他人机器/CI | 低。8 对本仓库近乎通用最优；CI 超订无害 | 改用本地 `CARGO_BUILD_JOBS=8`，不提交 |
| 忘记去掉 `-j 1` | 中。会让 test 仍 6 分钟 | 改动 1 已同步 CI；文档统一新命令（改动 3） |

回退：三处改动均为配置/命令级，`git revert` 即可整体回滚，无代码风险。

## 8. 实施与验证

### 实施顺序

1. 改 `.github/workflows/ci.yml`（去 `-j 1`）。
2. 改 `.cargo/config.toml`（加 `[build] jobs = 8`）。
3. （文档）更新手工验收命令说明。

### 验证（改完后实跑，手工验收=增量场景）

````bash
# 触发一次增量重编后，分别计时三命令，确认均 < 2min
touch src/web/runtime.rs && time cargo check --locked
touch src/web/runtime.rs && time cargo clippy --all-targets --all-features --locked -- -D warnings
touch src/web/runtime.rs && time cargo test --locked        # 期望 ~1m25s，553 测试全绿
````

预期（基于已实测）：check 个位数秒、clippy 个位数秒、test ~1m25s，全部达标且 swap 不抖。

#### ✅ 实施后实测确认（2026-05-30，改动全部落地后）

去掉 `-j 1`、`jobs=8` 生效后，增量场景实跑：

| 命令 | 实测耗时 | 内存峰值 | 结果 |
|------|---------|---------|------|
| `cargo check --locked` | **4.6s** | 0.6GB | ✅ |
| `cargo clippy --all-targets …` | **8.1s** | 0.8GB | ✅ |
| `cargo test --locked`（553 测试全绿） | **20s**（连续验收复用产物）/ 单独冷增量 ~1m25s | 0.9GB | ✅ 全绿 |

swap 全程稳定（1392MB 无增长），三命令均远低于 2 分钟，目标达成。
