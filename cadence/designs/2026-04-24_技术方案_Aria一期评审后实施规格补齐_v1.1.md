## 1. 文档目标

本文件只解决评审中指出的“研发拿到文档后无法直接写代码”的缺口。

它不是替代以下真相源：

- `cadence/designs/2026-04-23_技术方案_Aria一期MVP精简设计_v1.2.md`
- `cadence/designs/2026-04-23_技术方案_Aria一期实现总契约_v1.0.md`
- `cadence/designs/2026-04-23_技术方案_Aria_IO协作协议与Provider契约_v1.0.md`

它是上述文档的 **implementation spec 补齐层**。当本文件与旧文档存在粒度差异时：

1. 节点、路由、产物语义仍以上游 canonical 协议为准。
2. 一期范围仍以 `Aria一期MVP精简设计_v1.2` 为准。
3. 代码级字段、路径、编译规则、fixture 和准入门槛以本文为准。

内容不丢失裁定：

1. 本文只追加 implementation spec，不删除 `2026-04-22_技术方案_Aria终端REPL与多Agent编排Runtime设计_v1.0.md` 指向的文档集目标、节点集合、横切能力和产物规范。
2. 一期可以折叠实现单元，但不得删除上游协议节点身份；折叠只影响 runtime unit，不影响 checkpoint、runtime_snapshot、event、traceability 和审计链。
3. 16 类业务产物仍按上游产物规范保留；一期额外把 `runtime_snapshot` 纳入 validator、存储和恢复链，因此实施口径为 **17 类一期产物 = 16 类业务产物 + runtime_snapshot**。
4. Provider、OpenSpec、Superpowers 只能提供候选输入或约束，不替代 Aria daemon 的 runtime truth。

## 5. Projection 编译规则

### 5.1 Markdown parser 裁定

一期固定采用：

- `pulldown-cmark`：Markdown tokenization 与 table/list/heading 识别。
- Aria `HeadingModel`：由源码行扫描补充 heading path、section line range、原始文本切片。
- `serde_json` golden fixture：锁定 projection 输出。

### 5.2 稳定 ID 生成

ID 生成顺序：

1. 若条目含显式 ID，使用显式 ID 并归一化为小写 snake/kebab 保持原前缀，例如 `REQ-001` -> `req-001`。
2. 若无显式 ID，按 `prefix + ordinal` 生成，例如 `req-001`、`story-001`、`decision-001`。
3. 同一 section 内插入新条目时，不重排既有显式 ID；无显式 ID 的条目可因顺序变化改变 ID，因此生产模板必须优先生成显式 ID。

一期模板必须要求 provider 在 Markdown 中写显式 ID。compiler 的自动 ID 只用于兼容手工输入。

4. 若文件缺失，bundle 标记为 `blocked` 并按 6.1 阻断节点。

## 7. Daemon 启动、发现与 replay

### 7.1 进程模式

一期 CLI 暴露三类入口：

```text
aria daemon run --workspace <repo_root> --socket <socket_path>
aria daemon status --workspace <repo_root>
aria repl --workspace <repo_root> [--no-start]
```

`aria repl` 默认行为：

1. 解析 workspace root。
2. 发现 daemon socket / named pipe。
3. 若未发现且没有 `--no-start`，自动启动 `aria daemon run`。
4. 完成 `hello -> attach -> subscribe`。

### 7.2 Socket / named pipe 路径

Unix:

```text
<repo_root>/.aria/runtime/daemon/<workspaceHash>/daemon.sock
```

Windows:

```text
\\.\pipe\aria-<workspaceHash>-v1
```

`workspaceHash` = repo root canonical path 的 SHA-256 前 12 位。

环境变量优先级：

1. `ARIA_DAEMON_SOCKET`
2. `--socket`
3. 默认路径

### 7.3 Lock 与元数据

daemon 必须写：

```text
<repo_root>/.aria/runtime/daemon/<workspaceHash>/daemon.lock
<repo_root>/.aria/runtime/daemon/<workspaceHash>/daemon.json
```

`daemon.json` 最小字段：

```json
{
  "daemonSessionId": "sess_xxx",
  "pid": 12345,
  "workspaceRoot": "/repo",
  "socketPath": "/repo/.aria/runtime/daemon/abc/daemon.sock",
  "startedAt": "2026-04-24T00:00:00Z",
  "protocolVersion": "aria.repl.v1"
}
```

若 lock 存在但 PID 不存在或 socket 不可连接，REPL 允许判定为 stale daemon 并重新启动。

### 7.4 Crash recovery

daemon 启动时按顺序恢复：

1. 读取 `.aria/runtime/session.json`。
2. 读取 task registry。
3. 读取每个 task 最新 runtime snapshot。
4. 重建 open gates、provider run 状态、worktree leases。
5. 从 event log 恢复 latest event id。

恢复失败时，daemon 不删除文件，返回 `recovery_failed`，由 REPL 打开 manual intervention gate。

### 7.5 Event replay 窗口

事件落盘路径：

```text
<repo_root>/.aria/runtime/events/<daemonSessionId>.jsonl
```

一期保留策略：

- 单 session 至少保留最近 `10_000` 条 event。
- 每个 task 至少保留最近 `1_000` 条 event。
- 历史 event 可压缩，但压缩后必须记录 `firstRetainedEventId`。

replay 判定：

1. REPL `hello` 带 `last_seen_event_id`。
2. 若 `last_seen_event_id >= firstRetainedEventId - 1`，daemon 从 `last_seen_event_id + 1` 补发。
3. 若 `last_seen_event_id < firstRetainedEventId - 1`，返回 `ok=false`、`error.code=replay_window_lost`。
4. REPL 收到后必须执行 `get_status` + `list_artifacts` 全量同步。

### 8.3 版本号策略

Artifact ID：

```text

```

Artifact ref：

```text

```

规则：

1. 同一 artifact 语义被更新时，`artifactId` 不变，`version` 递增。
2. 新语义产物使用新 `artifactId`。
3. `latest.json` 指向当前 active ref。
4. 被替代版本标记 `status=superseded`，不得物理删除。

| provider stdout/stderr | `.txt` |
| OpenSpec bundle / projection / binding | `.json` |

### 8.5 ExternalArtifactRef 生命周期

```text
candidate -> normalized -> superseded
candidate -> rejected
```

规则：

1. 外部来源首次发现时写 `candidate`。
2. 通过 canonical validator 后生成 Aria artifact，并回填 `normalizedArtifactRef`。
3. 无法解析、hash 不匹配、越权路径或 schema 不合法时写 `rejected` 与 `rejectionReason`。
4. 归一化后的 Aria artifact 更新版本时，旧 external ref 不重新指向新版本，只保留审计用途。

### 9.5 Provider structured output

CLI adapter 解析 stdout 时只接受最后一个完整 sentinel block：

```text
<ARIA_STRUCTURED_OUTPUT>
{ "artifactKind": "spec" }
</ARIA_STRUCTURED_OUTPUT>
```

规则：

1. sentinel 外的内容保留到 stdout raw artifact，但不参与结构化判断。
2. 缺 sentinel -> `provider_parse_error`。
3. sentinel 内 JSON 无法解析 -> `provider_parse_error`。
4. JSON 的 `artifactKind` 与节点期望不一致 -> `provider_incompatible_output`。
5. structured output 必须保存为 `provider-runs/<providerRunId>/structured-output.json` 并登记 `ExternalArtifactRef(sourceSystem=provider_raw_output)`。

## 10. `_aria.traceability_refs` 生成算法

### 10.1 输入优先级

1. `PlanProjection.work_packages[].traceability_refs`
2. 当前 WorkTask registry 中的 routing context
3. Provider structured output 中的候选 refs
4. 当前 artifact 自身显式 refs

### 10.2 生成流程

```text
artifactKind/report -> locate worktask_id
  -> load PlanProjection.work_packages[worktask_id]
  -> copy requirement/design/task/risk refs
  -> merge provider candidate refs if they are known IDs
  -> drop unknown IDs into conflict log
  -> write normalized _aria.traceability_refs
  -> emit ArtifactTraceabilityBinding
```

规则：

1. provider 候选 refs 不能扩大到未知 requirement/design/task ID。
2. provider 候选 refs 与 PlanProjection 冲突时，以 PlanProjection 为准。
3. 冲突必须写入 `ArtifactTraceabilityBinding.conflictReason`，并发 `traceability.updated` event。
4. report 类 JSON artifact 没有 `_aria.traceability_refs` 时，phase1 profile validator 必须失败。

### 10.3 Coverage 判定

| 状态 | 条件 |
|------|------|
| `closed` | requirement/design/task 至少被一个有效执行证据覆盖 |
| `uncovered` | projection 中存在 ID，但没有任何 report/integration/final evidence |
| `exempted` | 用户或 daemon 记录 manual exemption，且包含原因 |

## 12. Worktree 与并发锁

### 12.1 Worktree 路径

Git worktree 不放在 repo root 内，避免污染主工作区：

```text
<repo_parent>/.aria-worktrees/<repoName>-<workspaceHash>/<taskId>/<worktaskId>/
```

### 12.2 Lease

```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct WorktreeLease {
    pub lease_id: WorktreeLeaseId,
    pub task_id: TaskId,
    pub worktask_id: WorkTaskId,
    pub path: String,
    pub base_ref: String,
    pub branch_name: String,
    pub allowed_write_scope: Vec<String>,
    pub status: WorktreeLeaseStatus,
    pub acquired_at: IsoDateTime,
    pub released_at: Option<IsoDateTime>,
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum WorktreeLeaseStatus {
    Active,
    ReadyForIntegration,
    Integrating,
    Released,
    Failed,
}
```

### 12.3 锁规则

1. 同一 `worktaskId` 只能有一个 active lease。
2. 不同 worktask 可并行执行，但 `allowed_write_scope` 不能重叠；重叠时必须串行。
3. integration 阶段全局串行，一次只允许一个 `N23` 修改 integration branch。
4. lease 状态变更必须写 runtime snapshot。

## 13. N23 Git 集成操作

一期固定使用 candidate commit 集成，不使用裸 diff 直接合并。

### 13.1 Candidate commit

`N20 ready` 前，daemon 在 worktree 中执行：

1. 校验改动只在 `allowedWriteScope` 内。
2. `git add` 授权范围文件。
3. `git commit -m "aria: <worktaskId> candidate"`。
4. 记录 `candidateCommitSha`。

### 13.2 Integration branch

默认 branch：

```text
aria/integration/<taskId>
```

`N22` 记录 `preMergeSha`。

### 13.3 N23 操作

```text
git checkout aria/integration/<taskId>
git cherry-pick --no-commit <candidateCommitSha>
run integration pre-checks
git commit -m "aria: integrate <worktaskId>"
```

失败路由：

| 失败 | 操作 | 路由 |
|------|------|------|
| cherry-pick conflict | `git cherry-pick --abort` | `N19` |
| pre-check fail | `git reset --hard <preMergeSha>` | `N19` |
| commit fail | 保留 worktree，打开 gate | manual intervention |

### 13.4 Rollback

`N24` 失败且需回滚：

```text
git reset --hard <preMergeSha>
```

并写入 `integration_report.rollbackRef` 与 runtime snapshot。

## 14. Approval gate REPL 流

### 14.1 Gate open

daemon 写 gate registry 并发送：

```json
{
  "event_type": "gate.opened",
  "payload": {
    "gate_id": "gate_001",
    "gate_type": "approval",
    "blocking_node": "N26",
    "summary": "final_review requested followup patch",
    "options": ["approve", "reject", "reply"]
  }
}
```

### 14.2 REPL 命令

- `approve_gate`：允许继续到 `N26`
- `reject_gate`：路由到 `N27`，final summary 标记 followup rejected
- `reply_gate`：写入用户补充，daemon 重新计算 gate 状态；默认仍阻塞

### 14.3 N26 规则

`N25 overallDecision=followup` 时：

1. daemon 必须打开 gate。
2. 未收到 `approve_gate` 前不得创建 followup dispatch_package。
3. approval event 必须写入 `N26` runtime snapshot 的 `nodeSpecificFields.approvalGateId`。
