## 1. 文档目标

本文件只解决评审中指出的“研发拿到文档后无法直接写代码”的缺口。

它不是替代以下真相源：

- `cadence/designs/2026-04-23_技术方案_Aria一期MVP精简设计_v1.2.md`
- `cadence/designs/2026-04-23_技术方案_Aria一期实现总契约_v1.0.md`
- `cadence/designs/2026-04-26_技术方案_Aria_IO协作协议与Provider契约_v1.1.md`

它是上述文档的 **implementation spec 补齐层**。当本文件与旧文档存在粒度差异时：

1. 节点、路由、产物语义仍以上游 canonical 协议为准。
2. 一期范围仍以 `Aria一期MVP精简设计_v1.2` 为准。
3. 代码级字段、路径、编译规则、fixture 和准入门槛以本文为准。

内容不丢失裁定：

1. 本文只追加 implementation spec，不删除 `2026-04-22_技术方案_Aria终端REPL与多Agent编排Runtime设计_v1.0.md` 指向的文档集目标、节点集合、横切能力和产物规范。
2. 一期可以折叠实现单元，但不得删除上游协议节点身份；折叠只影响 runtime unit，不影响 checkpoint、runtime_snapshot、event、traceability 和审计链。
3. 16 类业务产物仍按上游产物规范保留；一期额外把 `runtime_snapshot` 纳入 validator、存储和恢复链，因此实施口径为 **17 类一期产物 = 16 类业务产物 + runtime_snapshot**。
4. Provider、OpenSpec、Superpowers 只能提供候选输入或约束，不替代 Aria daemon 的 runtime truth。

## 4. Rust 核心类型规格

本章类型可以直接作为 `src/protocol/*.rs` 的实施蓝本。字段命名裁定如下：

1. 所有跨进程序列化字段统一使用 `snake_case`，包括 REPL wire protocol envelope、runtime artifact、projection、bundle、provider DTO 和 JSON `_aria` 扩展字段。
2. 对应 Rust 类型必须使用 `serde(rename_all = "snake_case")`；Rust struct/type 名仍按 Rust 习惯使用 `PascalCase`，不属于序列化字段命名范围。
3. 枚举序列化值固定使用 `snake_case`，除非上游 canonical 协议已另有明确枚举表。
4. 时间字段一期统一用 ISO 8601 字符串，避免 chrono 序列化策略在模块间分叉。
5. 如本文件、MVP 或总契约中仍出现同一业务字段的 camelCase 表达，视为旧稿残留；研发实现、JSON schema、fixture 和 validator 一律按本条 snake_case 裁定执行。

#### 4.2.1 `allowed_write_scope` 格式裁定

`WorktreeRef.allowed_write_scope`、`NodeExecutionContract.allowed_write_scope`、`ProviderContextPackage.allowed_write_scope` 与 `WorktreeLease.allowed_write_scope` 使用同一格式：

1. 每个元素都是相对于 worktree root 的相对路径；禁止绝对路径、`..`、空字符串和 Windows drive prefix。
2. 支持 glob 表达式，推荐实现使用 `globset` 或等价库；glob 只在 worktree root 内匹配。
3. 以 `/` 结尾表示目录及其所有子内容，例如 `src/auth/` 覆盖 `src/auth/login.rs`。
4. 空数组 `[]` 表示只读，不允许任何写入。
5. 特殊值 `["*"]` 表示允许写入 worktree 内任意路径，但仍不得写 `.git/`、`.aria/`、外部 symlink 指向路径或 provider sandbox 禁止路径。
6. 写入校验必须在 daemon 侧执行；provider 自报的修改范围不能作为授权依据。

### 4.3 `CanonicalNodeInput`

```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct CanonicalNodeInput {
    pub session_id: SessionId,
    pub task_id: TaskId,
    pub node_id: NodeId,
    pub phase: Phase,
    pub effective_policy: PolicyRef,
    pub artifact_refs: Vec<ArtifactRef>,
    #[serde(default)]
    pub external_refs: Vec<ExternalArtifactRef>,
    pub worktree_ref: Option<WorktreeRef>,
    pub risk_registry_ref: RiskRegistryRef,
    #[serde(default)]
    pub loop_counters: BTreeMap<String, u32>,
    #[serde(default)]
    pub acceptance_targets: Vec<AcceptanceTarget>,
    #[serde(default)]
    pub input_validation_refs: Vec<ArtifactRef>,
}
```

必填裁定：

- `artifact_refs` 只能包含 `status=active` 的 Aria canonical artifact。
- `external_refs` 只能作为候选上下文，不允许被 routing / coverage 直接消费。
- `N16-N24` 的 `worktree_ref` 必填；其他节点可为空。
- `risk_registry_ref` 在 `N02` 之后必填。`N02/N03` 初始化 EpicTask 时必须创建空 Risk Registry sidecar，并生成 `RiskRegistryRef` 写入 task runtime state；后续 provider context builder 不允许用空字符串或裸路径占位。
- `RiskRegistryRef` 不属于 `ArtifactKind` 枚举中的 17 类 canonical artifact，也不复用 `ArtifactRef`。Risk Registry 的当前状态通过 `RuntimeSnapshot.risk_registry` 进入 checkpoint；sidecar ref 只用于 provider context、恢复和风险条目落盘定位。
- `loop_counters` 与 `input_validation_refs` 在业务语义上必填。Rust 结构体可以为兼容旧数据保留 `serde(default)`，但 `CanonicalNodeInput` validator 必须区分“字段缺失”和“字段存在但为空”，并按节点条件规则返回明确错误。
- `N16-N25` 的 `acceptance_targets` 条件必填；规划阶段可为空数组。
- `effective_policy.mode` 一期默认 `PolicyMode::Conservative`。若用户或外部策略请求 `Balanced` / `Aggressive`，daemon 必须降级为 conservative 行为，并写入 `policy_mode_degraded` event 与 `effective_policy` 审计字段；不得静默按高风险模式执行。

记录规则：

1. `provider_capability_ref` 记录本次运行前 adapter 探测到的 provider 命令、版本、输出模式和可用能力。
2. `adapter_compatibility_ref` 记录本次使用的 CLI / SDK 映射配置版本；一期必须至少支持 CLI adapter compatibility matrix。
3. `status` 必须随 provider run 生命周期更新；daemon crash recovery 后发现未完成 run 时，只能标记为 `recovered_pending` 或 `manual_intervention`，不得伪造为 `completed`。
4. `files_modified` 必须由 daemon 或 adapter 对 worktree diff 检测得出，不接受 provider 自报作为唯一来源。
5. `timeout_status` 固定枚举为 `not_timed_out` / `soft_timeout_terminated` / `hard_timeout_killed`。
6. `raw_artifact_refs` 必须覆盖 stdout、stderr、structured output 和 provider 产生的中间文件引用。
7. provider run 失败时，`error_code` 必须使用稳定错误码或 REPL 全局错误码，`error_details` 必须保留原始 stderr / parse error / compatibility matrix ref 的审计引用。
8. `stdout_ref`、`stderr_ref`、`structured_output_ref` 分别指向 `.aria/runtime/tasks/<task_id>/provider-runs/<provider_run_id>/stdout.txt`、`stderr.txt`、`structured-output.json` 对应的 artifact ref；对应文件即使为空也要落盘，便于审计和重放。
9. `raw_artifact_refs` 必须等于 `[stdout_ref, stderr_ref, structured_output_ref, ...provider 产生的其他中间文件 refs]` 去除空值后的有序列表；不得把 worktree 源码改动路径直接塞入 `raw_artifact_refs`。
10. `files_modified` 是 adapter 或 daemon 通过 worktree diff 检测出的相对路径列表，用于授权校验和审计；它与 `raw_artifact_refs` 是不同概念。
11. `AdapterOutput.stdout`、`AdapterOutput.stderr`、`AdapterOutput.structured_output` 是调用完成后的内联解析输入；adapter 或 daemon 必须把它们再物化到 provider run 目录并回填上述 refs。

#### 4.7.2 Provider 错误码注册表

Provider adapter 必须把 CLI / SDK / app-server 层错误归一化为稳定错误码，再交给 provider router 处理。

一期固定 provider 错误码：

| 错误码 | 触发条件 | 默认路由 |
|--------|----------|----------|
| `provider_command_missing` | 本机找不到配置的 provider command | gate 或 manual intervention，不自动安装 |
| `provider_unauthorized` | provider 可执行但未登录、token 失效或账号未授权 | gate 或 manual intervention，不自动重试 |
| `provider_permission_denied` | sandbox、文件系统或审批策略拒绝执行 | gate 或 manual intervention |
| `provider_incompatible_output` | provider 不支持节点要求的 structured output / output mode | gate，要求调整 compatibility matrix |
| `provider_timeout` | soft / hard timeout 后仍未完成 | 按 `max_retries` 重试，超过阈值后 manual intervention |
| `provider_parse_error` | stdout / structured output 无法按 `output_schema_ref` 解析 | 重试一次，再失败进入 gate |
| `provider_execution_failed` | provider 进程非零退出且不属于以上分类 | 按节点 `failure_routes` 处理 |

记录规则：

1. 错误码必须写入 `ProviderRunRecord`、`provider_run.failed` event payload 和失败 response 的 `error.details.provider_error_code`。
2. 不允许只把 stderr 原文透传给上层作为路由依据。
3. `provider_command_missing`、`provider_unauthorized`、`provider_permission_denied` 不得静默降级为 fake provider 成功。

运行控制字段裁定：

1. `NodeExecutionContract.timeout_sec` / `max_retries` 是 provider router 的默认执行策略；adapter 可以返回更细粒度的 `timeout_status`，但不能覆盖 contract。
2. `allowed_command_classes` 是 coarse-grained 能力边界；具体命令仍由 provider compatibility matrix 与 worktree sandbox 决定。
3. `ProviderContextPackage.worktree_path` 在 `N16-N24` 必填，在规划节点可为空。
4. `completion_criteria`、`forbidden_actions`、`verification_commands` 同时进入 context package 和 prompt 渲染，不新增到底层 `AdapterInput` 的强字段。
5. `ProviderRunRecord.provider_capability_ref` 与 `adapter_compatibility_ref` 必须指向本次运行实际使用的探测结果和 matrix 版本，不能只记录当前默认配置。
6. `approval_policy`、`sandbox_mode` 必须记录 provider run 实际执行时的策略快照，便于复盘权限、沙箱或审批导致的失败。

CLI adapter compatibility matrix 最小字段：

| 字段 | 说明 |
|------|------|
| `provider_type` | `claude_code` / `codex` / `fake` |
| `matrix_version` | matrix 版本，写入 `adapter_compatibility_ref` 对应记录 |
| `probe_command` | 判断命令是否可执行的命令模板 |
| `version_command` | 获取版本字符串的命令模板 |
| `auth_check_command` | 判断是否已登录/授权的命令模板 |
| `unauthorized_patterns[]` | 匹配未登录、token 失效、账号未授权输出 |
| `permission_denied_patterns[]` | 匹配文件系统、sandbox、审批策略拒绝输出 |
| `structured_output_mode` | provider 生成 sentinel JSON 的约定模式 |
| `prompt_input_mode` | stdin / arg / temp file 等 prompt 输入方式 |
| `output_parser` | sentinel block parser 或 provider 特定 parser |

真实 CLI adapter baseline 不能只靠 fixture 通过；研发必须实现上述命令与 pattern 的默认配置，并在用户机器缺失或未授权时返回稳定 provider error。

| `worktree_path` | `worktree_path` | `N16-N24` 必填，规划节点可为空 |
| `instructions + prompt_template + workflow_discipline + completion_criteria + forbidden_actions + verification_commands` | `prompt` | 由 context builder 渲染为单一 prompt |
| canonical artifact、projection、constraint bundle、context package 文件路径 | `context_files` | 传路径，不传内存对象 |
| `output_schema_ref` | `outputSchema` | 解析为 adapter 可使用的 schema 文本或 schema ref |
| `timeout_sec` | `timeout` | 直接映射 |
| `max_retries` | `max_retries` | 直接映射 |

记录规则：

1. `AdapterInput` 必须作为可审计 raw artifact 或 runtime record 落盘，并由 `ProviderRunRecord.adapter_input_ref` 引用。
2. `AdapterOutput` 必须完整记录 stdout、stderr、exit code、structured output、files modified、duration、timeout status，并由 `ProviderRunRecord.adapter_output_ref` 引用。
3. `files_modified` 必须由 adapter 或 daemon 基于 worktree diff 检测得出；provider 自报只能作为候选，不得作为唯一来源。
4. `structuredOutput` 解析失败时仍要保留 stdout/stderr，并把 provider run 标记为 `provider_parse_error`。
5. fake provider fixture 必须输出与真实 CLI adapter 相同的 `AdapterOutput` 形状，避免测试路径与真实路径分叉。

#### 4.7.4 `RuntimeUnit` 统一 handler 契约

作用：定义 daemon 状态机调用 `src/runtime_units/*.rs` 的唯一入口。每个 runtime unit 可以覆盖一个或多个上游协议节点，但 handler 签名必须一致。

```rust
pub type NodeRoute = String;

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct RuntimeEvent {
    pub event_type: String,
    pub payload: Value,
}

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct RuntimeUnitResult {
    pub protocol_steps: Vec<RuntimeProtocolStep>,
    pub next_route: Option<NodeRoute>,
    pub checkpoint_required: bool,
}

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct RuntimeProtocolStep {
    pub node_id: NodeId,
    pub artifacts: Vec<ArtifactRef>,
    pub events: Vec<RuntimeEvent>,
    pub node_specific_fields: BTreeMap<String, Value>,
    pub next_route: Option<NodeRoute>,
    pub checkpoint_required: bool,
    pub status: RuntimeStepStatus,
}

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct RuntimeUnitError {
    pub code: String,
    pub message: String,
    pub retryable: bool,
    pub route: Option<NodeRoute>,
    #[serde(default)]
    pub details: BTreeMap<String, Value>,
}

pub struct DaemonContext {
    pub workspace_root: String,
    pub artifact_store_root: String,
    pub session_id: SessionId,
    pub task_id: TaskId,
    pub now: IsoDateTime,
}

#[async_trait::async_trait]
pub trait RuntimeUnit {
    fn node_id(&self) -> NodeId;
    fn covered_protocol_nodes(&self) -> Vec<NodeId>;
    async fn execute(
        &self,
        input: CanonicalNodeInput,
        ctx: &mut DaemonContext,
    ) -> Result<RuntimeUnitResult, RuntimeUnitError>;
}
```

实现规则：

1. `node_id()` 必须返回协议层节点 ID，例如 `M10 plan_dispatch_impl` 返回 `N10`；不得返回 `M10` 或其他实现侧 ID。
2. `covered_protocol_nodes()` 必须列出被折叠覆盖的上游协议节点，例如 `["N10", "N11", "N12"]`。
3. 如代码需要记录实现侧稳定 ID，必须新增 `runtime_unit_id` 或等价 metadata 字段，不得复用 `NodeId`。
4. event、runtime snapshot、handoff、traceability 中的 `node_id` 只能使用 `N00-N28` 或 `X01-X09`，不能写入 `Mxx`。
5. `execute()` 的输入只能是已经通过 daemon 前置校验的 `CanonicalNodeInput`；handler 不得直接消费未登记的 provider stdout、Markdown 原文或外部文件作为 routing truth。
6. handler 需要调用 provider 时，必须通过 `ProviderContextPackage -> AdapterInput -> AdapterOutput -> ProviderRunRecord -> normalization -> artifact_validate` 统一执行链。
7. `RuntimeUnitResult.protocol_steps[]` 是折叠实现单元的协议边界输出。每个被覆盖协议节点必须有独立 step；未执行到的下游协议节点不得虚构 step。
8. `RuntimeProtocolStep.artifacts` 只能返回已归一化并通过校验的 canonical artifact ref；候选输出必须先落入 provider raw output 或 external refs。
9. `RuntimeProtocolStep.events` 由 daemon 发布前补齐 envelope 字段；handler 只返回语义 event type 与 payload。
10. `RuntimeProtocolStep.node_specific_fields` 是写入该协议节点 runtime snapshot 的最小字段来源；daemon 写 snapshot 时必须按 step 的 `node_id` 分别落盘。
11. `RuntimeUnitResult.next_route` 是整个实现单元完成后的建议路由；`RuntimeProtocolStep.next_route` 只表达该协议节点内部边界的下游建议。
12. daemon 状态机仍必须按实现总契约的阶段收口规则、gate、OpenSpec coverage、traceability coverage 做最终裁定。
13. `RuntimeUnitError.route` 用于明确失败后的 retry、gate、manual intervention 或回流目标；不得只返回字符串错误让上层猜测路由。
14. 每个正式推进的 `RuntimeProtocolStep` 成功或可恢复失败后必须写 checkpoint；`checkpoint_required=false` 只允许用于纯只读探测或测试 helper，不允许用于正式节点推进。
15. `M02` 必须分别返回 `N02`、`N03` step；`M10` 必须分别返回 `N10`、`N11`、`N12` step；`M20` 必须分别返回 `N20`、`N21`、`N22` step。
16. `RuntimeUnit::execute` 固定为 async trait。P1 必须引入 `async-trait` 或等价稳定方案，并在 tokio runtime 下执行 socket IO、provider spawn、git command wrapper 和文件 IO；不得在 handler 内用阻塞调用绕过 runtime。

## 5. Projection 编译规则

### 5.1 Markdown parser 裁定

一期固定采用：

- `pulldown-cmark`：Markdown tokenization 与 table/list/heading 识别。
- Aria `HeadingModel`：由源码行扫描补充 heading path、section line range、原始文本切片。
- `serde_json` golden fixture：锁定 projection 输出。

不使用正则直接解析 Markdown 正文。允许用正则从已切分好的 section 文本中提取显式 ID，例如 `[REQ-001]`。

`DocumentModel` 最小结构：

```rust

    pub source_path: String,
    pub sha256: Sha256Hex,
    pub sections: Vec<DocumentSection>,

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
```

### 5.2 稳定 ID 生成

ID 生成顺序：

1. 若条目含显式 ID，使用显式 ID 并归一化为小写 snake/kebab 保持原前缀，例如 `REQ-001` -> `req-001`。
2. 若无显式 ID，按 `prefix + ordinal` 生成，例如 `req-001`、`story-001`、`decision-001`。
3. 同一 section 内插入新条目时，不重排既有显式 ID；无显式 ID 的条目可因顺序变化改变 ID，因此生产模板必须优先生成显式 ID。

一期模板必须要求 provider 在 Markdown 中写显式 ID。compiler 的自动 ID 只用于兼容手工输入。

### 5.6 `projection_validator` 规则

`projection_validator` 是 P2 的独立 validator 层，只校验 projection record 与 projection payload，不校验 canonical artifact 最小字段，也不校验 `_aria` profile。

输入：

- `ArtifactProjectionRecord`
- 当前 task 的 artifact index
- 可选 golden JSON fixture 路径

输出：

- `ValidationResult { ok, errors[], warnings[], projection_id, source_artifact_ref }`
- 错误码必须稳定，至少包含 `projection_missing_field`、`projection_invalid_id`、`projection_source_not_found`、`projection_source_hash_mismatch`、`projection_reference_unknown`、`projection_payload_empty`

通用校验项：

| 校验项 | 规则 |
|--------|------|
| `projection_id` | 必填，格式为 `proj_<projection_kind>_<source_artifact_id>_<sequence4>`；`source_artifact_id` 来自 `source_artifact_ref.artifact_id`，不得用 `task_id` 替代 |
| `projection_kind` | 必须与 `payload.payload_kind` 匹配 |
| `source_artifact_ref` | 必须指向存在且 `status=active` 的 source artifact |
| `source_artifact_version` | 必须等于 source artifact 当前版本 |
| `source_artifact_hash` | 必须等于 source artifact 文件当前 sha256 |
| `compiled_at` / `compiled_by_node` | 必填，`compiled_by_node` 必须是产生该 projection 的协议节点 |
| `payload` | 必填，且对应 projection 的必填集合非空 |
| golden JSON | fixture 测试中输出必须与 `tests/fixtures/artifacts/golden/*.json` 字段逐项对齐 |

分类型校验项：

| Projection | 必填 payload | ID 与引用规则 |
|------------|--------------|----------------|
| `SpecProjection` | `functional_requirements[]`、`success_criteria[]` 非空 | `story_id`、`requirement_id`、`criterion_id` 必须稳定；`related_requirement_ids[]` 必须指向本 projection 已知 requirement |
| `DesignProjection` | `design_decisions[]` 非空，`risk_refs[]` 可为空但字段必须存在 | `design_decision_id` 必须稳定；`related_requirement_ids[]` 必须可在当前 `SpecProjection` 或 OpenSpec requirement constraints 中找到 |
| `PlanProjection` | `work_packages[]` 非空 | `work_package_id` 必须唯一；`dependencies[]` 两端必须指向已知 work package；`traceability_refs[]` 至少包含 requirement 或 design/task 之一 |

实现边界：

1. `projection_validator` 不直接读取 Markdown 原文；它只消费 compiler 输出与已登记 source artifact。
2. `projection_validator` 不允许把缺失的 requirement/design/task ID 自动补齐；缺失引用必须报错或 warning，并由上游节点修正。
3. P2 的 `tests/spec_projection.rs`、`tests/design_projection.rs`、`tests/plan_projection.rs` 必须分别覆盖正例、缺必填字段、未知引用和 golden JSON drift。

### 5.7 `phase1_profile_validator` 规则

`phase1_profile_validator` 校验 JSON artifact 的 `_aria` 扩展命名空间、traceability、projection refs 与 constraint refs，不校验 canonical 最小字段。

输入：

- JSON canonical artifact value
- artifact kind
- 当前 task 的 projection index、constraint bundle index、traceability binding index、provider run index

输出：

- `ValidationResult { ok, errors[], warnings[], artifact_ref, profile_version }`
- 错误码必须稳定，至少包含 `profile_missing_aria`、`profile_version_missing`、`profile_projection_ref_unknown`、`profile_constraint_ref_unknown`、`traceability_refs_missing`、`traceability_ref_unknown`、`worktask_routing_source_unknown`、`coverage_summary_missing`

通用 `_aria` 校验：

| 字段 | 规则 |
|------|------|
| `_aria.profile_version` | 所有 JSON artifact 必填，一期固定 `phase1.v1` |
| `_aria.provider_run_refs[]` | 若产物来自 provider，必须指向存在的 `ProviderRunRecord`；daemon 直接生成产物可为空 |
| `_aria.projection_refs[]` | 若产物声明依赖 projection，必须指向存在且 source artifact 未 superseded 的 projection |
| `_aria.constraint_check_ref` | 需要 OpenSpec 覆盖校验的产物必填，必须指向本次消费的 bundle check record |
| `_aria.traceability_refs[]` | report 类 artifact 必填，且每个 ID 必须能在 projection 或 OpenSpec bundle 中找到 |

分 artifact 校验：

| 产物 | 必须校验 |
|------|----------|
| `dispatch_package` | `_aria.worktask_routing[]` 必填；每项必须有 `worktask_id`、`source_work_package_id`、`execution_mode`、`allowed_write_scope`、`verification_commands`；`source_work_package_id` 必须能映射到 `PlanProjection.work_packages[]` |
| `coding_report` / `testing_report` / `code_review_report` / `integration_report` | `_aria.traceability_refs[]` 必填，必须由 daemon 归一化生成；provider 候选 refs 只能作为输入 |
| `final_review` | `_aria.coverage_summary` 必填，必须覆盖 closed / uncovered / exempted 三类集合 |
| `final_summary` | 不得引入 `final_review` 中不存在的新 coverage 结论；引用必须可回溯到已通过校验的 final review |

实现边界：

1. `phase1_profile_validator` 不接受 provider 自报的 `_aria.traceability_refs` 作为最终事实；report 类产物必须先走第 10 章归一化算法。
2. `projection_refs` 和 `constraint_check_ref` 是引用校验，不允许在 validator 内临时编译 projection 或 bundle。
3. fixture 必须包含至少一个 dispatch package、一个 report、一个 final review，并同时通过 canonical、projection 或 phase1 profile 对应层校验。

## 6. OpenSpec Bundle 编译规则

### 6.1 输入文件

一期固定读取：

```text
openspec/changes/<change_id>/proposal.md
openspec/changes/<change_id>/specs/main/spec.md
openspec/changes/<change_id>/design.md
openspec/changes/<change_id>/tasks.md
```

Scope 裁定：

1. 一期固定 `openspec_scope = "main"`，不在 P1 wire protocol 中新增 `requested_scope` 字段。
2. P2 bootstrap 创建 skeleton 时必须写入 `openspec/changes/<change_id>/specs/main/spec.md`。
3. Bundle compiler 只读取 `specs/main/spec.md`；若发现同一 change 下存在多个 `specs/*/spec.md`，返回 `openspec_multiple_scopes_unsupported`，进入 gate 或 manual intervention，不自动合并。
4. fixture 路径必须使用 `tests/fixtures/openspec/changes/sample-change/specs/main/spec.md`。
5. 多 scope / 多 capability spec 属于二期能力，不能在一期实现中临时自行派生 scope。

文件缺失处理：

| 缺失文件 | 阻断节点 | 错误码 |
|----------|----------|--------|
| `proposal.md` | `N05` | `openspec_source_missing` |
| `spec.md` | `N07` | `openspec_source_missing` |
| `design.md` | `N11` | `openspec_source_missing` |
| `tasks.md` | `N12/N16` | `openspec_source_missing` |

文件存在但内容为空或缺少关键约束时，不能视为 ready。compiler 必须输出 source readiness，并按消费节点阻断：

| 内容缺口 | 阻断节点 | 错误码 |
|----------|----------|--------|
| `specs/main/spec.md` 无 requirement id | `N07` | `openspec_requirement_constraints_empty` |
| `design.md` 无 design decision id 且无 component id | `N11` | `openspec_design_constraints_empty` |
| `tasks.md` 无 task id | `N12/N16` | `openspec_task_constraints_empty` |

P2 bootstrap 创建的 skeleton 初始状态只能用于 `bootstrap_pending/bootstrapped` 过渡，不代表后续阶段约束已就绪。`OpenSpecConstraintBundle.bundle_status=ready` 必须同时满足文件存在和当前消费节点所需约束非空。

`N06` 时序裁定：

1. `N06 spec_gate_review` 不消费 `requirement_constraints` 作为 pass 前置条件，因为通过 gate 的 Aria `spec` 尚未写回 `specs/main/spec.md`。
2. `N06` 的 daemon 判定顺序固定为：`artifact_validate(spec)` -> `SpecProjection` 编译与覆盖检查 -> `proposal_constraints` 覆盖检查 -> 生成 `spec_gate_decision`。
3. `N06 pass` 后，daemon 通过 Document Operation 将 canonical `spec` 写回 `openspec/changes/<change_id>/specs/main/spec.md`，记录 source manifest，标记当前 bundle `stale` 并立即重编译。
4. `N07` 进入前必须看到重编译后的 bundle 中 `requirement_constraints` 非空；若为空，阻断 `N07` 并返回 `openspec_requirement_constraints_empty`。

`compiled_from_projection_refs` 填充规则：

1. bundle 的主来源仍是 OpenSpec 文件。
2. 当 daemon 将 Aria `SpecProjection`、`DesignProjection` 或 `PlanProjection` 写回 OpenSpec 后触发重编译时，`compiled_from_projection_refs` 记录本次写回所依据的 projection refs。
3. 纯读取外部 OpenSpec 文件编译时，`compiled_from_projection_refs=[]`。
4. Projection 被 supersede 不直接让 bundle stale；只有该 projection 曾写回 OpenSpec，且对应 OpenSpec source manifest hash 变化后，才通过 source manifest 触发 stale/recompile。

### 6.2 `proposal.md` 映射

| Heading | 输出字段 |
|---------|----------|
| `## Why` / `## 背景` | `proposal_constraints.businessIntent[]` |
| `## What Changes` / `## 范围` | `proposal_constraints.scope[]` |
| `## Non-Goals` / `## 不做` | `proposal_constraints.nonGoals[]` |
| `## Impact` / `## 影响` | `proposal_constraints.impactedAreas[]` |

### 6.3 `spec.md` 映射

支持 OpenSpec 常见结构：

```markdown
### ADDED Requirements

#### Requirement: REQ-001 Create task from REPL

##### Scenario: SCN-001 New task succeeds
- WHEN user sends new_task
- THEN daemon returns task_id
```

映射规则：

- `#### Requirement:` 后的显式 ID -> `requirement_constraints.requirementIds[]`
- `##### Scenario:` 后的显式 ID -> `requirement_constraints.scenarioIds[]`
- `THEN` 行中含 `[AC-xxx]` 时 -> `requirement_constraints.success_criteriaIds[]`

### 6.4 `design.md` 映射

| Heading | 输出字段 |
|---------|----------|
| `## Decisions` / `## 设计决策` | `design_constraints.designDecisionIds[]` |
| `## Components` / `## 组件` | `design_constraints.componentIds[]` |
| `## Risks` / `## 风险` | `design_constraints.risk_ids[]` |

### 6.5 `tasks.md` 映射

支持格式：

```markdown
- [ ] TASK-001 建立 REPL wire schema。Reqs: REQ-001; Designs: DD-001; Acceptance: AC-001
- [ ] TASK-002 实现 daemon handshake。Reqs: REQ-002; Designs: DD-001; Acceptance: AC-002
```

映射规则：

- checkbox 行显式 ID -> `task_constraints.task_ids[]`
- 出现顺序 -> `task_constraints.taskSequence[]`
- `Reqs:` 后的 requirement ID -> `task_constraints.related_requirement_ids_by_task[task_id]`
- `Designs:` 后的 design decision ID -> `task_constraints.related_design_decision_ids_by_task[task_id]`
- `Acceptance:` 后的 acceptance target / criterion ID -> `task_constraints.acceptance_target_ids_by_task[task_id]`

### 6.6 Bundle 失效规则

每次编译记录 `source_manifest[].sha256`。进入任意依赖 OpenSpec 的节点前：

1. 重新计算 manifest。
2. 若任一文件 hash 变化，当前 bundle 标记为 `stale`。
3. daemon 重新编译新 bundle。
4. 若文件缺失，bundle 标记为 `blocked` 并按 6.1 阻断节点。
5. 一期不启用文件系统 watch；外部修改检测完全依赖节点进入前的 source manifest hash 比对。
6. 如果用户在节点执行期间修改 OpenSpec 文件，daemon 在下一个依赖 OpenSpec 的节点进入前发现变化并重编译；正在执行的 provider run 不被中途抢占，但其输出归一化时必须再次做 constraint check，发现 bundle stale 则阻断推进。

## 7. Daemon 启动、发现与 replay

### 7.1 进程模式

一期 CLI 暴露三类入口：

```text
aria daemon run --workspace <repo_root> --socket <socket_path>
aria daemon status --workspace <repo_root>
aria repl --workspace <repo_root> [--no-start]
```

`aria repl` 默认行为：

1. 解析 workspace root。
2. 发现 daemon socket / named pipe。
3. 若未发现且没有 `--no-start`，自动启动 `aria daemon run`。
4. 完成 `hello -> attach -> subscribe`。

### 7.2 Socket / named pipe 路径

Unix:

```text
<repo_root>/.aria/runtime/daemon/<workspace_hash>/daemon.sock
```

Windows:

```text
\\.\pipe\aria-<workspace_hash>-v1
```

`workspace_hash` = repo root canonical path 的 SHA-256 前 12 位。

环境变量优先级：

1. `ARIA_DAEMON_SOCKET`
2. `--socket`
3. 默认路径

### 7.3 Lock 与元数据

daemon 必须写：

```text
<repo_root>/.aria/runtime/daemon/<workspace_hash>/daemon.lock
<repo_root>/.aria/runtime/daemon/<workspace_hash>/daemon.json
```

`daemon.json` 最小字段：

```json
{
  "daemon_session_id": "sess_xxx",
  "pid": 12345,
  "workspace_root": "/repo",
  "socketPath": "/repo/.aria/runtime/daemon/abc/daemon.sock",
  "startedAt": "2026-04-24T00:00:00Z",
  "protocol_version": "aria.repl.v1"
}
```

若 lock 存在但 PID 不存在或 socket 不可连接，REPL 允许判定为 stale daemon 并重新启动。

### 7.4 Crash recovery

daemon 启动时按顺序恢复：

1. 读取 `.aria/runtime/session.json`。
2. 读取 task registry。
3. 读取每个 task 最新 runtime snapshot。
4. 重建 open gates、provider run 状态、worktree leases。
5. 从 event log 恢复 latest event id。

恢复失败时，daemon 不删除文件，返回 `recovery_failed`，由 REPL 打开 manual intervention gate。

### 7.5 Event replay 窗口

事件落盘路径：

```text
<repo_root>/.aria/runtime/events/<daemon_session_id>.jsonl
```

一期保留策略：

1. 所有 event 存储在同一个 `<daemon_session_id>.jsonl` 文件中，session 级 event 和 task 级 event 通过 payload 中是否含 `task_id` / `node_id` 区分。
2. 单 session 至少保留最近 `10_000` 条 event，含 session 级与 task 级 event。
3. 每个 task 至少保留最近 `1_000` 条 task event。执行截断时，daemon 先保护每个 task 的最近 1,000 条，再从未受保护的旧 event 中截断。
4. 若 task 数过多导致受保护 event 总数超过 10,000，允许保留窗口大于 10,000；不得为了满足全局窗口而丢弃某个 task 的最近 1,000 条。
5. 历史 event 可压缩，但压缩后必须记录 `firstRetainedEventId`、`firstRetainedEventIdByTask` 和压缩段摘要。

事件 ID 与索引结构：

1. `event_id` 是 daemon session 内单调递增的 `u64`，从 `1` 开始；跨 daemon session 不保证连续，跨 session 排序使用 `(daemon_session_id, event_id)`。
2. 每行 event 必须包含 `event_id`、`event_type`、`created_at`、`payload`；task 级 event 的 `payload.task_id` 必填。
3. `.aria/runtime/events/index.json` 必须至少保存：

```json
{
  "daemon_session_id": "sess_20260426_001",
  "latest_event_id": 1200,
  "first_retained_event_id": 201,
  "first_retained_event_id_by_task": {
    "task_001": 301
  },
  "compacted_segments": [
    {
      "from_event_id": 1,
      "to_event_id": 200,
      "summary_ref": "events/compacted/sess_20260426_001_0001_0200.json"
    }
  ]
}
```

replay 判定：

1. REPL `hello` 带 `last_seen_event_id`。
2. 若 `last_seen_event_id >= first_retained_event_id - 1`，daemon 从 `last_seen_event_id + 1` 补发。
3. 若 `last_seen_event_id < first_retained_event_id - 1`，返回 `ok=false`、`error.code=replay_window_lost`。
4. REPL 收到后必须执行 `get_status` + `list_artifacts` 全量同步。

## 8. Artifact 存储约定

### 8.1 根目录

Aria runtime 元数据默认写入：

```text
<repo_root>/.aria/runtime/
```

该目录不作为源代码产物提交。后续若需要审计归档，由 `aria export` 命令导出。

### 8.2 目录结构

```text
.aria/runtime/
├── session.json
├── daemon/
├── events/
│   ├── <daemon_session_id>.jsonl
│   └── index.json
├── tasks/
│   └── <task_id>/
│       ├── task.json
│       ├── risk-registry/
│       │   ├── registry.json
│       │   └── refs/<risk_registry_ref_id>.json
│       ├── artifacts/
│       │   └── <artifact_kind>/
│       │       ├── <artifact_id>_v0001.md
│       │       ├── <artifact_id>_v0002.json
│       │       └── latest.json
│       ├── projections/
│       │   └── <projection_kind>/<projection_id>.json
│       ├── constraints/
│       │   ├── <constraint_bundle_id>.json
│       │   └── checks/<constraint_check_id>.json
│       ├── traceability/
│       │   └── <binding_id>.json
│       ├── snapshots/
│       │   └── <node_id>_<eventId>.json
│       ├── provider-runs/
│       │   └── <provider_run_id>/
│       │       ├── record.json
│       │       ├── stdout.txt
│       │       ├── stderr.txt
│       │       └── structured-output.json
│       └── external-refs/
│           └── <external_ref_id>.json
└── indexes/
    ├── artifact_index.json
    └── task_index.json
```

目录裁定：

1. `risk-registry/` 是 task 级 runtime sidecar，不是 `artifacts/<artifact_kind>/` 下的 canonical artifact kind。
2. `risk-registry/registry.json` 存储当前 `RiskRegistrySnapshot`；`refs/<risk_registry_ref_id>.json` 存储可被 `CanonicalNodeInput.risk_registry_ref` 引用的 `RiskRegistryRef`。
3. `constraints/checks/<constraint_check_id>.json` 存储 `ConstraintCheckRecord`，供 `_aria.constraint_check_ref` 校验使用。
4. `latest.json` 只存 active `ArtifactRef` metadata，不复制 artifact 内容。最小 schema：

```json
{
  "artifact_ref_id": "ref_art_spec_task_001_0001_v0001",
  "artifact_id": "art_spec_task_001_0001",
  "artifact_kind": "spec",
  "version": 1,
  "path": ".aria/runtime/tasks/task_001/artifacts/spec/art_spec_task_001_0001_v0001.md",
  "sha256": "...",
  "status": "active"
}
```

### 8.3 版本号策略

Artifact ID：

```text
art_<artifact_kind>_<task_id>_<sequence4>
```

Artifact ref：

```text
ref_<artifact_id>_v<version4>
```

规则：

1. “同一 artifact 语义”定义为同一个 `ArtifactKind` + 同一个 `TaskId` + 同一个逻辑产物槽位，例如同一个 task 的当前 `spec` 文档修订。
2. 同一 artifact 语义被更新时，`artifact_id` 不变，`version` 按该 artifact 独立递增。
3. 新语义产物使用新 `artifact_id`，version 从 `1` 开始。
4. 被 superseded 后，如果新内容仍是同一语义更新，继续使用原 `artifact_id` 并递增 version；如果是回流后重新生成的全新逻辑产物，使用新 `artifact_id`，version 从 `1` 开始，并把旧 ref 写入 `superseded_artifact_refs`。
5. `latest.json` 指向当前 active ref。
6. 被替代版本标记 `status=superseded`，不得物理删除。
7. projection 的 `source_artifact_version` 必须等于被编译 artifact 的版本号。

### 8.4 文件扩展名

| Artifact kind | 扩展名 |
|---------------|--------|
| `spec` / `design` / `plan` | `.md` |
| report / decision / snapshot / dispatch | `.json` |
| provider stdout/stderr | `.txt` |
| OpenSpec bundle / projection / binding | `.json` |

### 8.5 ExternalArtifactRef 生命周期

```text
candidate -> normalized -> superseded
candidate -> rejected
```

规则：

1. 外部来源首次发现时写 `candidate`。
2. 通过 canonical validator 后生成 Aria artifact，并回填 `normalizedArtifactRef`。
3. 无法解析、hash 不匹配、越权路径或 schema 不合法时写 `rejected` 与 `rejectionReason`。
4. 归一化后的 Aria artifact 更新版本时，旧 external ref 不重新指向新版本，只保留审计用途。

### 9.5 Provider structured output

CLI adapter 解析 stdout 时只接受最后一个完整 sentinel block：

```text
<ARIA_STRUCTURED_OUTPUT>
{ "artifact_kind": "spec" }
</ARIA_STRUCTURED_OUTPUT>
```

规则：

1. sentinel 外的内容保留到 stdout raw artifact，但不参与结构化判断。
2. 缺 sentinel -> `provider_parse_error`。
3. sentinel 内 JSON 无法解析 -> `provider_parse_error`。
4. JSON 的 `artifact_kind` 与节点期望不一致 -> `provider_incompatible_output`。
5. structured output 必须保存为 `provider-runs/<provider_run_id>/structured-output.json` 并登记 `ExternalArtifactRef(source_system=provider_raw_output)`。

## 10. `_aria.traceability_refs` 生成算法

### 10.1 输入优先级

1. `PlanProjection.work_packages[].traceability_refs`
2. 当前 WorkTask registry 中的 routing context
3. Provider structured output 中的候选 refs
4. 当前 artifact 自身显式 refs

### 10.2 生成流程

```text
artifact_kind/report -> locate worktask_id
  -> locate source_work_package_id from dispatch_package._aria.worktask_routing[]
  -> load PlanProjection.work_packages[source_work_package_id]
  -> copy requirement/design/task/risk refs
  -> merge provider candidate refs if they are known IDs
  -> drop unknown IDs into conflict log
  -> write normalized _aria.traceability_refs
  -> emit ArtifactTraceabilityBinding
```

规则：

1. provider 候选 refs 不能扩大到未知 requirement/design/task ID。
2. provider 候选 refs 与 PlanProjection 冲突时，以 PlanProjection 为准。
3. 冲突必须写入 `ArtifactTraceabilityBinding.conflictReason`，并发 `traceability.updated` event。
4. report 类 JSON artifact 没有 `_aria.traceability_refs` 时，phase1 profile validator 必须失败。
5. `worktask_id` 与 `work_package_id` 不默认等同。`N12 dispatch_authoring` 生成 WorkTask 时必须在 `dispatch_package._aria.worktask_routing[]` 写入 `source_work_package_id`；如果团队选择让两者相同，也必须显式写入该字段。
6. 找不到 `source_work_package_id` 或对应 PlanProjection work package 时，归一化失败，错误码为 `traceability_work_package_not_found`。
7. 每个 report 在归一化时立即生成 `ArtifactTraceabilityBinding`，不等待所有 reports 完成后再批量生成。
8. `ArtifactTraceabilityBinding` 与归一化后的 artifact ref 必须在同一个 checkpoint 前落盘；checkpoint 写入前若 binding 失败，则该 report 不得进入 active。

### 10.3 Coverage 判定

| 状态 | 条件 |
|------|------|
| `closed` | requirement/design/task 至少被一个有效执行证据覆盖 |
| `uncovered` | projection 中存在 ID，但没有任何 report/integration/final evidence |
| `exempted` | 用户或 daemon 记录 manual exemption，且包含原因 |

## 12. Worktree 与并发锁

### 12.1 Worktree 路径

Git worktree 不放在 repo root 内，避免污染主工作区：

```text
<repo_parent>/.aria-worktrees/<repo_name>-<workspace_hash>/<task_id>/<worktask_id>/
```

Integration worktree 固定使用独立目录，不复用 repo root，也不复用任一 worktask 执行 worktree：

```text
<repo_parent>/.aria-worktrees/<repo_name>-<workspace_hash>/<task_id>/integration/
```

所有 `N22/N23/N24` git 命令必须由 daemon command wrapper 在 integration worktree 下执行。实现和测试都不得假设当前进程工作目录是 repo root；测试必须断言 repo root 的 branch、index 和 working tree 不因 `N23/N24` 被修改。

### 12.2 Lease

```rust
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub struct WorktreeLease {
    pub lease_id: WorktreeLeaseId,
    pub task_id: TaskId,
    pub worktask_id: WorkTaskId,
    pub path: String,
    pub base_ref: String,
    pub branch_name: String,
    pub allowed_write_scope: Vec<String>,
    pub status: WorktreeLeaseStatus,
    pub acquired_at: IsoDateTime,
    pub released_at: Option<IsoDateTime>,
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum WorktreeLeaseStatus {
    Active,
    ReadyForIntegration,
    Integrating,
    Released,
    Failed,
}
```

### 12.3 锁规则

1. 同一 `worktask_id` 只能有一个 active lease。
2. 不同 worktask 可并行执行，但 `allowed_write_scope` 不能重叠；重叠时第二个 lease 申请必须进入等待队列。
3. integration 阶段全局串行，一次只允许一个 `N23` 修改 integration branch。
4. lease 状态变更必须写 runtime snapshot。
5. scope 重叠定义：归一化为相对路径后，两个 scope 中任意一对路径存在前缀包含关系即视为重叠；目录 scope 覆盖其所有子路径。例如 `src/auth/` 与 `src/auth/login.rs` 重叠。
6. glob scope 必须先展开为可匹配规则后判断可能重叠；无法静态证明不重叠时按重叠处理。
7. 第二个重叠 worktask 申请 lease 时进入等待队列，不直接失败；等待期间写 runtime snapshot，原因码为 `worktree_scope_conflict_waiting`，并记录阻塞它的 `lease_id`。
8. 前一个 lease 释放后，等待队列按 FIFO 重试；获得 lease 后发 `worktree.lease_acquired` event。

路径规范化规则：

1. 所有 scope、diff path、provider reported path 在校验前必须先基于 worktree root 做 canonicalize/realpath。
2. symlink 解析后若目标跳出 worktree root，直接拒绝并返回 `worktree_scope_symlink_escape`。
3. 大小写不敏感文件系统上，normalizer 必须使用同一大小写折叠策略缓存路径；Linux 默认保持大小写敏感。
4. `allowed_write_scope`、`files_modified` 和 `git diff --name-only` 的比较必须使用同一个 normalizer。
5. normalizer 测试至少覆盖目录前缀、glob、symlink 跳出、大小写碰撞、`.git/` 与 `.aria/` 禁写路径。

## 13. N23 Git 集成操作

一期固定使用 candidate commit 集成，不使用裸 diff 直接合并。

### 13.1 Candidate commit

`N20 ready` 前，daemon 在 worktree 中执行：

1. 校验改动只在 `allowed_write_scope` 内。
2. `git add` 授权范围文件。
3. `git commit -m "aria: <worktask_id> candidate"`。
4. 记录 `candidate_commit_sha`。

执行裁定：

1. Candidate commit 由 daemon 直接执行，不经过 provider；`N20` 是 daemon 内部集成准备节点，不是 Agent 产物生成节点。
2. `git add` 只能覆盖授权范围内已通过 diff 校验的文件；禁止 `git add .`。
3. daemon 读取新 commit SHA 后，必须写入 worktask runtime state、`N20` runtime snapshot 的 `node_specific_fields.candidate_commit_sha`、以及传给 `N22` 的 handoff package。
4. candidate commit 创建失败时，`N20` 不得进入 ready；必须返回稳定错误码并按 policy 进入 gate 或 manual intervention。
5. `N22` 只能消费 `N20` 生成的 `candidate_commit_sha`，不得临时创建 candidate commit。

稳定错误码：

| 错误码 | 判定条件 | 默认路由 |
|--------|----------|----------|
| `candidate_commit_empty_diff` | 授权范围内没有可提交 diff | 回到 `N16/N19` 或关闭空 worktask |
| `candidate_commit_author_missing` | git author/email 缺失导致 commit 失败 | manual intervention |
| `candidate_commit_hook_failed` | pre-commit 或 commit-msg hook 失败 | `N19`，保留 hook 输出 |
| `candidate_commit_signing_failed` | GPG/SSH 签名失败 | manual intervention |
| `candidate_commit_submodule_dirty` | 授权范围涉及 submodule 且 submodule dirty 或 detached 状态不明 | manual intervention |
| `candidate_commit_lfs_failed` | LFS filter/smudge/clean 失败 | manual intervention |

### 13.2 Integration branch

默认 branch：

```text
aria/integration/<task_id>
```

`N22` 记录 `pre_merge_sha`。

Integration branch 只能在 12.1 定义的 integration worktree 中 checkout 或创建。daemon 必须先确保 integration worktree 已绑定 `aria/integration/<task_id>`；如果 branch 不存在，则从当前任务约定的 base ref 创建。禁止在 repo root 执行 checkout、cherry-pick、reset 或 commit。

### 13.3 N23 操作

```text
(cwd=<integration_worktree>) git checkout aria/integration/<task_id>
(cwd=<integration_worktree>) git cherry-pick --no-commit <candidate_commit_sha>
(cwd=<integration_worktree>) run integration pre-checks
(cwd=<integration_worktree>) git commit -m "aria: integrate <worktask_id>"
```

失败路由：

| 失败 | 操作 | 路由 |
|------|------|------|
| cherry-pick conflict | `(cwd=<integration_worktree>) git cherry-pick --abort` | `N19` |
| pre-check fail | `(cwd=<integration_worktree>) git reset --hard <pre_merge_sha>` | `N19` |
| commit fail | 保留 worktree，打开 gate | manual intervention |

### 13.4 Rollback

`N24` 失败且需回滚：

```text
(cwd=<integration_worktree>) git reset --hard <pre_merge_sha>
```

并写入 `integration_report.rollback_ref` 与 runtime snapshot。

## 14. Approval gate REPL 流

### 14.1 Gate open

daemon 写 gate registry 并发送：

```json
{
  "event_type": "gate.opened",
  "payload": {
    "gate_id": "gate_001",
    "gate_type": "approval",
    "blocking_node": "N26",
    "summary": "final_review requested followup patch",
    "options": ["approve", "reject", "reply"]
  }
}
```

### 14.2 REPL 命令

- `approve_gate`：允许继续到 `N26`
- `reject_gate`：路由到 `N27`，final summary 标记 followup rejected
- `reply_gate`：写入用户补充，daemon 重新计算 gate 状态；默认仍阻塞

### 14.3 N26 规则

`N25 overall_decision=followup` 时：

1. daemon 必须打开 gate。
2. 未收到 `approve_gate` 前不得创建 followup dispatch_package。
3. approval event 必须写入 `N26` runtime snapshot 的 `node_specific_fields.approval_gate_id`。
4. `N26` provider 只允许产出候选 patch task delta 或候选 dispatch routing 意图，不得直接修改 OpenSpec Markdown，也不得直接生成最终 active `dispatch_package`。
5. provider 输出 patch task delta 时，daemon 必须先将 delta 通过 Document Operation 应用到 `openspec/changes/<change_id>/tasks.md`；不允许只保存一个脱离 OpenSpec source manifest 的独立 delta 文件后继续推进。
6. OpenSpec `tasks.md` 更新完成后，daemon 必须将当前 `OpenSpecConstraintBundle` 标记为 `stale` 并重编译。
7. daemon 基于重编译后的 `task_constraints` 和 provider 候选意图生成新的 canonical `dispatch_package`。
8. 新 `dispatch_package` 必须绑定重编译后的 `task_constraints`，并通过 OpenSpec 约束覆盖校验；校验通过后才能推进到 `N13`。
9. 若 provider 直接输出完整候选 `dispatch_package`，daemon 仍必须先抽取其中的 task delta 写回 OpenSpec，再按第 5-8 条重新生成 active `dispatch_package`；候选包只作为审计输入保留。
