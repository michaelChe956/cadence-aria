# 真实流式 Provider 接入设计

## 0. 评审结论

本设计经过代码与背景文档复核后可以进入实施计划阶段。相对初稿，关键修正如下：

- 保留旧 `ProviderAdapter` / `CliProviderAdapter` / sentinel 结构化输出链路作为兼容层。本次只替换对话式 Workspace WebSocket 的 `FakeStreamingProvider`，不直接删除仍被 `task_run`、`WebRuntime`、历史测试覆盖的同步 provider 路径。
- 将真实 provider 运行语义限定为当前 WebSocket 连接作用域：连接关闭或用户 abort 时取消 provider 进程并回到可恢复的会话快照。后台持续运行与断线后重新桥接正在运行的 provider 需要独立设计，不纳入本次实施范围。
- Claude Code 接入方式对齐 vibe-kanban：使用 `stream-json` + stdin 控制协议 + permission hook，而不是仅靠 stdout 解析 `control_request`。
- Codex 接入方式对齐 vibe-kanban：使用 `app-server` JSON-RPC、`thread/start`、`turn/start` 与 server-initiated approval requests，不手写虚构的 `"stream_chunk"` / `"approval_request"` notification 名称。
- 权限确认抽象为 provider 无关的 `ApprovalBridge`，Auto 模式自动批准，Supervised 模式经 WebSocket 推送确认卡片并等待前端返回。

## 2. 目标与非目标

### 2.1 目标

- 在对话式 Workspace WebSocket 路径接入真实 Claude Code 与 Codex CLI。
- 将 `FakeStreamingProvider` 从生产 WebSocket 路径替换为基于 provider 选择的 `ProviderRegistry`。
- 支持双向权限确认：
  - Auto：自动批准，不向前端展示确认卡片。
  - Supervised：provider 请求权限时通过 WebSocket 推送确认卡片，前端允许或拒绝后回写 provider。
- provider 进程必须能被 abort、WebSocket 断连、运行失败可靠清理。
- 真实 provider 不再依赖 sentinel 结构化输出作为 Workspace 对话流的主协议。
- 保持现有 task_run、WebRuntime、旧 provider 测试兼容。

### 2.2 非目标

- 不删除 `src/cross_cutting/cli_adapter.rs`、`src/cross_cutting/provider_adapter.rs`、`src/cross_cutting/adapter_compatibility.rs`。
- 不迁移 `task_run` 到新 streaming provider。
- 不实现 provider 在 WebSocket 断连后继续后台运行。
- 不实现 Claude/Codex 历史 session resume；回滚后通过截断后的 Aria 消息历史重建 prompt，并启动新的 CLI 进程。
- 不实现 Coding Workspace 的完整代码变更 diff、测试 tab、rework 循环。

## 3. 总体架构

```text
┌───────────────────────────────────────────────────────────┐
│                  workspace_ws_handler.rs                   │
│  WsInMessage ⇄ WsOutMessage，连接关闭时取消活动 run         │
└──────────────────────────┬────────────────────────────────┘
                           │
┌──────────────────────────▼────────────────────────────────┐
│                    WorkspaceEngine                         │
│  - 构建 StreamingProviderInput                             │
│  - 从 ProviderRegistry 选择 provider                       │
│  - ProviderEvent → EngineEvent → WsOutMessage              │
│  - PermissionResponse → ProviderCommand                    │
└──────────────────────────┬────────────────────────────────┘
                           │
┌──────────────────────────▼────────────────────────────────┐
│              StreamingProviderAdapter session API          │
│  start(input, cancel) -> ProviderSession                   │
│  ProviderSession = events_rx + commands_tx                 │
└───────────────┬──────────────────────────┬────────────────┘
                │                          │
┌───────────────▼──────────────┐ ┌─────────▼────────────────┐
│      ClaudeCodeProvider       │ │        CodexProvider      │
│ stream-json + control stdin    │ │ app-server JSON-RPC       │
│ permission hook bridge         │ │ server request bridge      │
└───────────────┬──────────────┘ └─────────┬────────────────┘
                │                          │
┌───────────────▼──────────────────────────▼────────────────┐
│                共享基础设施                                │
│ ProcessManager | JsonLinePeer | JsonRpcPeer | ApprovalBridge│
│ VersionChecker | ProviderRegistry                          │
└───────────────────────────────────────────────────────────┘
```

核心原则：

- Workspace 对话路径只消费 `ProviderEvent`，不理解 Claude/Codex 私有协议。
- provider 私有协议只存在于 `claude_code_provider.rs` 与 `codex_provider.rs`。
- 权限请求统一通过 `ApprovalBridge` 转为 Aria WebSocket 消息。
- 旧 sentinel adapter 保持原样，作为另一路兼容运行时。

## 4. 核心类型

### 4.1 StreamingProviderInput

```rust
pub struct StreamingProviderInput {
    pub provider_type: ProviderType,
    pub role: AdapterRole,
    pub prompt: String,
    pub working_dir: PathBuf,
    pub session_id: Option<String>,
    pub permission_mode: ProviderPermissionMode,
    pub env_vars: BTreeMap<String, String>,
    pub timeout_secs: u64,
}

pub enum ProviderPermissionMode {
    Auto,
    Supervised,
}
```

`WorkspaceEngine` 从当前 `WorkspaceSession` 构建 `StreamingProviderInput`。`working_dir` 对文档类 Workspace 默认使用仓库 workspace root；Coding Workspace 后续可切到 worktree path。

### 4.2 ProviderEvent

```rust
pub enum ProviderEvent {

        message: String,

pub enum ProviderStatus {
    Starting,
    Running,
    WaitingApproval,
    Completed,
    Failed,

}
```

### 4.3 ProviderCommand

```rust
pub enum ProviderCommand {
    PermissionResponse {
        id: String,
        approved: bool,
        reason: Option<String>,

}
```

### 4.4 ProviderSession 与 Adapter

```rust
pub struct ProviderSession {
    pub events: mpsc::Receiver<ProviderEvent>,
    pub commands: mpsc::Sender<ProviderCommand>,

#[async_trait]
pub trait StreamingProviderAdapter: Send + Sync {
    fn provider_type(&self) -> ProviderType;

```

## 6. ProviderRegistry 与可用性探测

`ProviderRegistry` 位于 `src/cross_cutting/provider_registry.rs`，职责：

- 根据 `ProviderName` 返回 `Arc<dyn StreamingProviderAdapter>`。
- 启动时探测 `claude`、`codex` 是否可执行。
- provider 不可用时不注册，并在前端 provider 列表中展示 unavailable 状态。
- `fake` 只在 fake runtime 或测试中注册；生产 WebSocket 默认优先使用 session 记录中的真实 provider。

可用性探测策略：

- `claude --version` 或配置覆盖命令返回 0 即视为可用，版本号解析失败不阻塞注册，但记录 warning。
- `codex --version` 或配置覆盖命令返回 0 即视为可用，版本号解析失败不阻塞注册，但记录 warning。
- 命令不存在时返回 `ProviderAdapterError::command_missing`，WebSocket 推送 `error`。

命令配置：

- 默认 Claude 命令：`claude`
- 默认 Codex 命令：`codex`
- 后续可以通过环境变量覆盖：
  - `ARIA_CLAUDE_COMMAND`
  - `ARIA_CODEX_COMMAND`

## 7. ProcessManager

新增 `src/cross_cutting/process_manager.rs`。

实现要求：

- 使用 `tokio::process::Command` 启动 provider。
- 引入 `command-group = { version = "5.0", features = ["with-tokio"] }`，用 `AsyncGroupChild` 统一管理进程组。
- stdin/stdout/stderr 必须全部 pipe。
- `CancellationToken` 触发时先发送 provider 协议级 interrupt；若超时未退出，再 kill 进程组。
- stdout 与 stderr 分开读取：stdout 用于协议解析，stderr 聚合为错误上下文。
- child 退出非 0 时发送 `ProviderEvent::Failed`。

## 8. ApprovalBridge

新增 `src/cross_cutting/approval_bridge.rs`。

职责：

- 为 provider 创建权限请求 ID。
- Auto 模式直接返回 approved。
- Supervised 模式发送 `ProviderEvent::PermissionRequest`，并等待 `ProviderCommand::PermissionResponse`。
- 等待期间发送 `ProviderStatus::WaitingApproval`。
- 用户拒绝时返回 denied，并携带 reason。
- cancel 触发时停止等待并返回 aborted。

```rust
pub struct ApprovalBridge {
    mode: ProviderPermissionMode,
    events: mpsc::Sender<ProviderEvent>,
    pending: Arc<Mutex<HashMap<String, oneshot::Sender<PermissionDecision>>>>,
}

pub struct PermissionDecision {
    pub approved: bool,
    pub reason: Option<String>,
}
```

`WorkspaceEngine` 的 event bridge 负责把 `WsInMessage::PermissionResponse` 转发给当前 `ProviderSession.commands`。

## 9. ClaudeCodeProvider

新增 `src/cross_cutting/claude_code_provider.rs`。

### 9.1 启动参数

Claude Code 参数基于 vibe-kanban 当前实现：

```text
-p
--verbose
--output-format=stream-json
--input-format=stream-json
--include-partial-messages
--replay-user-messages
```

权限控制：

- Auto 与 Supervised 均使用 stdin control protocol。
- Supervised 增加 `--permission-prompt-tool=stdio`，并设置 permission mode 为 `default`。
- Auto 设置 permission mode 为 `bypassPermissions`，并由 `ApprovalBridge` 自动批准可见的 hook 请求。

### 9.2 stdin 控制流程

启动后顺序写入：

1. `control_request initialize`，携带 hooks。
2. `control_request set_permission_mode`。
3. `user` message，内容为 `StreamingProviderInput.prompt`。

### 9.3 stdout 解析

逐行读取 stream-json：

- assistant text / partial message → `ProviderEvent::TextDelta`
- `control_request.can_use_tool` → 进入 `ApprovalBridge`
- `result` → `ProviderEvent::Completed`
- error-like JSON 或 stderr 退出 → `ProviderEvent::Failed`

Claude control request 的 response 必须写回 stdin：

- approved → `PermissionResult::Allow`
- rejected → `PermissionResult::Deny`

## 10. CodexProvider

新增 `src/cross_cutting/codex_provider.rs`、`src/cross_cutting/json_rpc_peer.rs`。

### 10.1 依赖选择

为避免手写 Codex app-server 私有 payload，本设计选择引入与 vibe-kanban 一致的协议 crate：

```toml
codex-protocol = { git = "https://github.com/openai/codex.git", package = "codex-protocol", tag = "rust-v0.124.0" }
codex-app-server-protocol = { git = "https://github.com/openai/codex.git", package = "codex-app-server-protocol", tag = "rust-v0.124.0" }
```

`JsonRpcPeer` 只负责 JSON-RPC envelope、request id、pending response map、server request 分发；业务 payload 使用上述 crate 的类型。

### 10.2 启动参数

```text
app-server
```

线程参数：

- `cwd = StreamingProviderInput.working_dir`
- Auto：`sandbox = WorkspaceWrite`，`approval_policy = Never`
- Supervised：`sandbox = WorkspaceWrite`，`approval_policy = UnlessTrusted`

### 10.3 RPC 流程

1. 发送 `initialize`
2. 发送 `initialized`
3. 发送 `thread/start`
4. 发送 `turn/start`，内容为 `StreamingProviderInput.prompt`
5. 处理 notification 与 server request：
   - item completed / assistant message delta → `ProviderEvent::TextDelta`
   - turn completed → `ProviderEvent::Completed`
   - file change approval request → `ApprovalBridge`
   - command execution approval request → `ApprovalBridge`
   - tool request user input → 暂映射为 `ProviderEvent::Failed`，提示当前版本不支持问答型权限

### 10.4 拒绝策略

用户拒绝权限时：

- command execution：有 reason 时返回 cancel + feedback，无 reason 时返回 decline。
- file change：有 reason 时返回 cancel + feedback，无 reason 时返回 decline。
- provider 后续是否继续由 Codex 自身决定，Aria 继续消费事件直到 completed、failed 或 cancel。

## 11. WorkspaceEngine 集成

`WorkspaceEngine` 改造点：

- 构造 `StreamingProviderInput`，从 session provider 字段选择 provider。
- 调用 `ProviderRegistry::get(provider_name)` 获取 provider。
- 启动 provider session 后用 `tokio::select!` 同时处理：
  - provider events
  - WebSocket command channel
  - cancel token
- `ProviderEvent::TextDelta` 转为 `EngineEvent::StreamChunk`。
- `ProviderEvent::PermissionRequest` 转为 `EngineEvent::PermissionRequest`。
- `ProviderEvent::Completed` 后创建 assistant message、artifact、checkpoint，并进入 `human_confirm`。
- `ProviderEvent::Failed` 后推送 error，并回到 `prepare_context`。
- `Abort` 时取消 token，丢弃未完成 assistant message，不创建 checkpoint。

当前阶段仍按已有实现走：

```text
prepare_context -> running -> human_confirm -> completed
```

`cross_review` 暂不在真实 provider 接入中补齐。进入真实 cross review 需要单独实现 reviewer provider 调度与多轮 artifact revision。

## 13. 错误处理

| 场景 | 行为 |
|------|------|
| provider 命令不存在 | `ProviderEvent::Failed` + WebSocket `error` |
| provider stdout JSON 解析失败 | 保留原始行到错误上下文，发送 `ProviderEvent::Failed` |
| provider stderr 有内容但 exit 0 | 不判失败，作为诊断附加到日志 |
| provider 非 0 退出 | 发送 `ProviderEvent::Failed`，包含 exit code 与 stderr 摘要 |
| WebSocket 断连 | 取消当前 provider 进程组 |
| 用户 abort | 取消当前 provider，丢弃 streaming buffer，不创建 checkpoint |
| 权限等待时 abort | `ApprovalBridge` 返回 aborted，provider session 结束 |
| 用户拒绝权限 | 将拒绝写回 provider，后续由 provider 协议决定继续或失败 |
