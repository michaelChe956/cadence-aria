# 项目管理工作台与 Issue 全生命周期技术方案

## 文档信息

- 文档类型：技术方案
- 版本：v1.0
- 日期：2026-05-14
- 适用仓库：`cadence-aria`
- 需求来源：`cadence/prds/2026-05-14_概要需求_项目管理工作台与Issue全生命周期_v1.1.md`
- 设计目标：把 PRD v1.1 中的项目工作台、Issue 全生命周期、硬 Gate、Provider 执行透明化、外置 worktree 调度和回退机制拆成可实施的工程边界。

## 1. 背景

当前 Aria 已具备 Web 工作台、workspace registry、issue registry、runtime task store、provider adapter、SSE event hub、checkpoint rollback、worktree lease 等基础能力。但这些能力仍围绕“单 workspace / 单 task 执行”组织：

- Web 侧已有 `WorkspaceRegistry`、`IssueRegistry`、`WebRuntime`、`EventHub`，状态主要写在仓库内 `.aria/runtime/...`。
- Runtime 侧已有 `TaskRunStore`、provider run record、checkpoint、projection、runtime units 和执行链。
- Worktree 侧已有写范围校验与 lease 管理能力，但产品入口还没有把 work item DAG、并行调度和外置 worktree 串起来。

PRD v1.1 的核心调整是引入产品层级：`Project → Repository → Issue`，并保持 `{repo}/.aria/runtime` 作为执行真相源。因此本方案不重写 runtime，而是在其上增加产品索引、Issue-runtime 绑定、硬 Gate 状态机和工作台交互层。

## 2. 目标与非目标

### 2.1 目标

1. 新增产品索引层，支持 Project、Repository、Issue、IssueRuntimeBinding、ExecutionRecord、GateRecord 的持久化。
2. 保留仓库内 `{repo}/.aria/runtime` 作为 task、checkpoint、provider runs、artifacts、projection 的执行真相源。
3. 建立 Issue 与现有 runtime task 的绑定关系，使新工作台能启动、恢复、展示和回退 Issue 执行过程。
4. 新增硬 Gate 机制，支持无条件暂停、人工确认、要求修改、终止和重启恢复。
5. 新增 Provider 选择策略：静态节点契约声明推荐 executor 和允许 executor，本次执行记录实际选择。
6. 增强 SSE 和 ProviderExecutionPanel，提供输入摘要、流式输出、artifact 展示、review 轮次和 gate 操作。
7. 建立外置 worktree 调度方案，使用依赖 DAG 与 `allowed_write_scope` 冲突检测决定可并行集合。
8. 明确双层回退：产品索引 soft-delete + runtime checkpoint / worktree git reset。

### 2.2 非目标

- 不迁移或删除已有 `{repo}/.aria/runtime` 数据。
- 不把 `Issue` 直接替代 runtime task。
- 不在本版本实现测试验收阶段状态机；`Acceptance` 只作为 UI 占位。
- 不在本版本实现自动合并和 integration worktree 的完整闭环。
- 不改变现有 provider adapter 的底层进程执行模型，只增加必要的事件、记录和选择策略。

## 4. 模块一：产品索引与兼容数据层

### 4.1 新增模块

建议新增后端模块：

```text
src/product/
├── mod.rs
├── app_paths.rs
├── project_store.rs
├── repository_store.rs
├── issue_store.rs
├── runtime_binding_store.rs
├── execution_record_store.rs
├── gate_store.rs
└── compatibility_scan.rs
```

模块职责：

- `app_paths.rs`：解析 `~/.aria` 根目录，支持测试注入临时目录。
- `project_store.rs`：读写 `Project` 与 `last_opened_at`。
- `repository_store.rs`：读写 `Repository`，计算 canonical path hash 和 `runtime_root`。
- `issue_store.rs`：读写 `Issue`、phase、status、active binding。
- `runtime_binding_store.rs`：维护 `IssueRuntimeBinding`。
- `execution_record_store.rs`：追加 `execution-records.jsonl`，支持按 issue / binding 查询与 soft-delete。
- `gate_store.rs`：读写 `GateRecord`，支持恢复 open gates。
- `compatibility_scan.rs`：从 `{repo}/.aria/runtime/tasks` 重建最小产品索引。

### 4.2 文件布局

产品索引层使用 PRD v1.1 定义路径：

```text
~/.aria/
├── config.json
├── state/last-project.json
└── projects/{project_id}/
    ├── project.json
    ├── repos.json
    └── issues/{issue_id}/
        ├── issue.json
        ├── bindings/{binding_id}.json
        ├── execution-records.jsonl
        ├── gates/{gate_id}.json
        ├── story-spec.v{n}.json
        ├── design-spec-backend.v{n}.json
        ├── design-spec-frontend.v{n}.json
        └── work-items/{work_item_id}.json
```

### 4.3 兼容读取

当 `~/.aria` 没有项目索引，但 repo 已有 `.aria/runtime` 时：

1. 创建默认 project。
2. 为 repo 创建 repository record。
3. 扫描 `{repo}/.aria/runtime/tasks/*/state.json`。
4. 为每个 task 创建可恢复 issue，标题优先取 `request_text`，否则使用 `change_id` 或 `task_id`。
5. 创建 `IssueRuntimeBinding` 指向 task root。
6. 不复制 runtime artifacts，只保存引用。

兼容读取必须是幂等的：同一 repo path、task id 重复扫描不生成重复 Issue。

### 4.4 测试

- 创建/读取 Project、Repository、Issue。
- Repository canonical path hash 稳定。
- IssueRuntimeBinding 能正确指向 `{repo}/.aria/runtime/tasks/{task_id}`。
- 重复扫描同一 runtime task 不重复创建 issue。
- 产品索引缺失时可从 runtime 重建最小首页数据。

## 5. 模块二：Issue 状态机与硬 Gate

### 5.1 Issue 阶段

本版本只允许状态机进入：

```rust
Phase::Clarification
Phase::Development
```

`Phase::Acceptance` 只用于 UI 占位，不作为自动路由目标。

### 5.2 硬 Gate 语义

硬 Gate 节点：

- `N06.5`：Story Spec 人工确认。
- `N09`：Design Spec 人工确认。
- `N10.5`：Work Item 人工确认。
- `N11.5`：Plan 人工确认。

硬 Gate 行为：

1. 创建 `GateRecord { status: Open }`。
2. 创建或更新 `ExecutionRecord`，记录 `gate_id` 和 `before_state`。
3. 发布 `gate.opened` SSE。
4. 暂停当前 binding 的流程推进。
5. 等待 API 决策。

### 5.3 Gate API

新增 API：

```text
POST /api/issues/{issue_id}/gates/{gate_id}/confirm
POST /api/issues/{issue_id}/gates/{gate_id}/request-change
POST /api/issues/{issue_id}/gates/{gate_id}/terminate
```

请求体：

```json
{
  "comment": "用户说明，可为空",
  "requested_change": "要求修改时必填"
}
```

返回：

```json
{
  "issue_id": "issue-id",
  "gate_id": "gate-id",
  "node_id": "N09",
  "decision": "confirmed",
  "next_node": "N10"
}
```

### 5.4 Gate 决策路由

| 决策 | Gate 状态 | Issue / binding 行为 | 下一步 |
|------|-----------|----------------------|--------|
| confirm | `Confirmed` | 继续当前 binding | 当前 gate 后继节点 |
| request-change | `ChangeRequested` | 保留历史，追加执行记录 | 回到对应 generate 节点 |
| terminate | `Terminated` | 当前 binding 标记 `Blocked` 或 `Detached` | 停止推进 |

同一 binding 同一 node 只允许一个 open hard gate。若重复触发，应返回已有 gate 而不是新建。

### 5.5 恢复

服务启动时：

1. 遍历 `~/.aria/projects/*/issues/*/gates/*.json`。
2. 找到 `status == Open` 的 gate。
3. 验证 binding 指向的 runtime task root 是否仍存在。
4. 发布 projection update，使前端恢复 GateActionBar。

### 5.6 测试

- hard gate 不受 `PolicyPreset` 自动执行影响。
- confirm 后只能继续一次。
- request-change 必须保存用户修改意见并路由回生成节点。
- terminate 不删除 execution record。
- 重启恢复 open gate。

## 6. 模块三：Web API、SSE 与 Provider 透明化

### 6.1 API 分组

新增或重构 API：

```text
GET  /api/projects
POST /api/projects
GET  /api/projects/{project_id}
POST /api/projects/{project_id}/open

GET  /api/projects/{project_id}/repositories
POST /api/projects/{project_id}/repositories

GET  /api/projects/{project_id}/issues
POST /api/projects/{project_id}/issues
GET  /api/issues/{issue_id}
POST /api/issues/{issue_id}/start-clarification
POST /api/issues/{issue_id}/start-development

GET  /api/issues/{issue_id}/execution-records
GET  /api/issues/{issue_id}/provider-inputs/{input_ref}
GET  /api/issues/{issue_id}/artifacts/{artifact_ref}

GET  /api/events?cursor={cursor}
```

现有 `/api/workspaces`、`/api/issues` 可先保留兼容，内部逐步改为产品索引层实现。

### 6.2 SSE 事件

新增事件类型：

```rust
provider.input_prepared
provider.output_stream
provider.output_completed
review.round_started
review.round_completed
gate.opened
gate.resolved
execution.record_appended
issue.phase_changed
workitem.status_changed
worktree.status_changed
```

事件 envelope：

```rust
struct ProductWebEvent {
    cursor: u64,
    event_type: String,
    project_id: Option<String>,
    issue_id: Option<String>,
    binding_id: Option<String>,
    task_id: Option<String>,
    payload: serde_json::Value,
}
```

### 6.3 输入全文引用

SSE 只发送：

- `input_summary`
- `input_full_ref`
- `redaction_applied`

完整输入写入 runtime task root：

```text
{task_root}/provider-inputs/{provider_run_id}.json
```

展示规则：

- 默认展示摘要。
- 用户展开时调用 `GET /api/issues/{issue_id}/provider-inputs/{input_ref}`。
- 后端读取前验证 input ref 属于当前 issue binding。
- 返回前做 secret redaction。
- 单次返回默认限制 200KB，超出返回截断标记和下载/分页引用。

### 6.4 provider 输出持久化

Provider stdout/stderr 流式写入：

```text
{task_root}/logs/provider-output.jsonl
```

每行包含：

```json
{
  "provider_run_id": "run-id",
  "node_id": "N16",
  "stream": "stdout",
  "text": "chunk",
  "timestamp": 0
}
```

SSE 内存 replay 只作为实时体验；历史回放必须读取持久化日志。

### 6.5 测试

- SSE cursor 能从指定位置 replay。
- provider input SSE 不包含完整 prompt/context。
- input ref 不能越权读取其他 issue 的输入。
- redaction 覆盖 token、API key、私钥、Authorization header。
- provider output 刷新后可从 jsonl 恢复。

### 7.2 前端状态

在现有 `workbench-store` 基础上拆分状态：


- Acceptance 阶段灰显，不触发执行按钮。

|------|------|
| `ProjectSwitcher` | 切换最近项目、创建项目、更新 last_opened |

| `GateActionBar` | confirm/request-change/terminate |

### 7.4 UX 状态

必须覆盖：

- 无项目：显示项目创建/选择页。
- 有最近项目但路径失效：提示重新选择或移除。
- repo 不存在或非 git：禁止创建 repository。
- agent CLI 不可用：Agent Picker 灰显并说明原因。
- hard gate open：主操作区显示 GateActionBar。
- provider 正在运行：执行卡片流式输出。
- request-change 后：显示修改意见和回流节点。
- worktree dirty 回退：二次确认。

### 7.5 测试

- 项目创建、切换、最近项目恢复。
- 代码库管理弹窗添加 git repo。
- Issue 创建并绑定 repo。
- hard gate action bar 三个操作。
- ProviderExecutionPanel 可展示 input summary、stream、artifact。
- SSE 断线重连后恢复事件。

## 8. 模块五：外置 Worktree 调度与回退

### 8.1 路径

外置 worktree 路径：

```text
<repo_parent>/.aria-worktrees/<repo_name>-<repo_hash>/<issue_short_id>/<work_item_short_id>/
```

预留 integration 路径：

```text
<repo_parent>/.aria-worktrees/<repo_name>-<repo_hash>/<issue_short_id>/integration/
```

本版本不自动合并 integration，只保留路径约定。

### 8.2 分支

```text
aria/{issue_short_id}/{work_item_short_id}
```

创建前检查：

- repo path 存在。
- repo 是 git worktree。
- base ref 可解析。
- 目标 branch 不与已有 branch 冲突；若存在，检查是否已绑定同一 work item。

### 8.3 调度输入

每个 `WorkItem` 必须提供：

- `depends_on`
- `allowed_write_scope`
- `execution_mode`
- `worktree_path`
- `worktree_branch`

### 8.4 可执行集合计算

算法：

```text
ready = work_items where status == Pending and dependencies completed
active_scopes = active work_items allowed_write_scope

for item in ready:
  if execution_mode != Agent:
    skip
  if scope overlaps active_scopes:
    wait
  else:
    acquire lease and start
```

写范围无法判断时按冲突处理。等待队列按 FIFO 重试。

### 8.5 回退

开发阶段回退流程：

1. 读取目标 `ExecutionRecord.before_state`。
2. 标记目标及之后 execution records 为 dropped。
3. 若存在 `runtime_checkpoint_id`，调用现有 checkpoint restore 恢复 runtime state/projection。
4. 在对应 worktree path 执行 `git reset --hard before_head_commit`。
5. 若 worktree dirty 且用户未确认 force，拒绝回退并返回 preview。
6. 发布 rollback completed 事件。

禁止在 repo root 执行 reset。

### 8.6 测试

- 外置 worktree 路径生成稳定。
- 依赖未完成的 work item 不进入 ready。
- 写范围重叠进入等待队列。
- 写范围不重叠可并行。
- 回退只 reset work item worktree，不修改 repo root。
- dirty worktree 未 force 时拒绝回退。

## 9. 实施分期

### Phase 1：产品索引和兼容绑定

目标：新工作台能列出项目、代码库、Issue，并能绑定现有 runtime task。

交付：

- `src/product/*`
- Project / Repository / Issue / Binding store
- 兼容扫描 `{repo}/.aria/runtime/tasks`
- 最小 API：projects、repositories、issues
- 前端项目切换和代码库管理骨架

验收：

- 新建项目和 repo 后落盘到 `~/.aria`。
- 已有 runtime task 可被重建为 IssueRuntimeBinding。
- 刷新 Web 页面后项目和 Issue 列表可恢复。

### Phase 2：需求澄清链路和硬 Gate

目标：Story Spec / Design Spec 可生成、review、硬 Gate 确认。

交付：

- spec/design runtime units 或现有节点适配。
- GateRecord store。
- Gate API 和 SSE。
- Issue 详情 spec 展示。
- GateActionBar。

验收：

- N06.5 / N09 必定暂停，不受 policy 自动运行影响。
- confirm 后继续后续节点。
- request-change 回流到生成节点。
- 重启后 open gate 恢复。

### Phase 3：Work Item、Plan 和开发执行

目标：从确认后的 spec/design 生成 work items、plan，并执行 coding/testing/review/rework。

交付：

- WorkItem / Plan store。
- N10/N11 适配。
- Agent Picker 与 NodeExecutorPolicy。
- 外置 worktree 创建。
- work item DAG 和 `allowed_write_scope` 冲突检测。

验收：

- Work item 人工确认后进入 plan gate。
- worktree 创建在 repo root 外。
- 重叠写范围不会并行。
- N16/N17/N18/N19 能记录 ExecutionRecord。

### Phase 4：Provider 执行透明化和回退

目标：完整展示 provider 输入摘要、输出流、产物和回退能力。

交付：

- provider input ref 持久化。
- provider output jsonl。
- SSE cursor resume。
- ProviderExecutionPanel。
- RollbackDialog 增强。

验收：

- SSE 不发送完整 prompt/context。
- 刷新后可恢复 provider output 历史。
- 回退 preview 显示 dropped records、runtime checkpoint、worktree path。
- 开发阶段回退只影响目标 worktree。

## 10. 风险与约束

| 风险 | 影响 | 缓解 |
|------|------|------|
| `~/.aria` 与 repo runtime 状态不一致 | 首页显示错误或无法恢复 | 以 runtime 为执行真相源；产品索引可重建 |
| 硬 Gate 与现有 `PausedForApproval` 混淆 | policy 绕过人工确认 | GateType 分层，HardGate 不走 policy |
| Provider 全量输入泄露敏感信息 | 安全风险 | SSE 只传 ref，读取时 redaction + 大小限制 |
| worktree reset 误操作 repo root | 丢失用户变更 | reset 前校验 worktree path，不允许 repo root |
| 并行写范围判断不准 | 文件冲突或误阻塞 | 无法判断时保守串行；精确 glob 匹配作为独立增强，不阻塞本方案 |
| 现有 workspace API 兼容期过长 | 维护成本增加 | Phase 1 明确兼容 facade，后续逐步迁移调用方 |
