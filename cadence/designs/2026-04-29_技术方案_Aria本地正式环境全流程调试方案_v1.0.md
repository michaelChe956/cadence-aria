## 1. 目标与边界

本方案用于在本地正式环境测试 Aria 的全流程能力。被测对象是 `cadence-aria`，`naruto` 是 Aria 的目标工作区。

正式端到端用例为：让 Aria 在 `naruto` 的隔离 worktree 中完成一个最小全栈用户登录功能，包含前端页面、后端服务、JWT 鉴权和内存用户存储。

关键边界如下：

- 登录功能的需求拆解、OpenSpec 产物、计划、代码开发、测试、评审、集成和最终总结都应由 Aria 工作流触发。
- 外部人工不直接替 Aria 开发或验证 `naruto` 的业务功能。
- 外部人工只审计 Aria 是否完成职责，包括 runtime 产物、provider 记录、测试报告、OpenSpec 写回、final review 和 final summary。
- 如果需要复跑 `naruto` 测试，只能作为故障定位辅助，不能替代 Aria 工作流内的测试证据。

## 2. 推荐总方案

采用“宿主机 Rust 增量开发 + Docker 最终门禁 + `naruto` 测试 worktree + 真实 provider E2E”的混合方案。

宿主机 Rust 工具链用于日常快速反馈，包括 `cargo check`、定向 `cargo test`、`cargo run` 和 watch 类增量编译。Docker 用于最终一致性验证，必须按仓库已启用的 Rust/Docker 规则执行 `fmt`、`clippy`、`check`、`test`。

IDE 断点调试不是主流程。它只在排查 daemon socket、异步状态机、provider spawn、stdout/stderr 解析等问题时作为可选工具。

## 3. 环境基线

两台机器都应具备以下能力：

- Rust toolchain：遵循仓库 `rust-toolchain.toml`，当前为 `1.95.0`。
- Docker：用于最终门禁和跨机器一致性验证。
- Git：支持 worktree、分支、commit、diff。
- Node.js 与 `pnpm`：供 `naruto` 中的前后端项目使用。
- Claude Code CLI：`claude` 可执行、已登录、可在目标工作区运行。
- Codex CLI：`codex` 可执行、已登录、可在目标工作区运行。
- OpenSpec：`naruto/openspec/config.yaml` 存在，Aria 能创建或更新 change 产物。
- superpowers：`naruto` 的 Claude/Codex 环境能识别并执行相关 skill 规则。

当前 macOS Intel 已确认存在 `claude`、`codex` 和 `docker`，宿主机没有 `cargo`。因此 macOS 的第一步应安装 Rust，而不是依赖 IDE 调试。

## 5. 分层调试模型

### 5.1 L0：Aria 自身快速反馈

目标是确认源码修改后的最小反馈闭环稳定。

主要验证：

- `cargo check --locked`
- 定向 `cargo test --test <name>`
- `cargo run --locked -- daemon status --workspace <workspace>`
- 可选 watch 工具，如 `cargo watch` 或 `bacon`

这一层只验证 Aria 自身，不进入真实 Claude/Codex。

### 5.2 L1：daemon 与 REPL 通信

目标是确认 CLI、daemon socket、wire protocol 和基础 runtime 存储正常。

主要覆盖：

- `aria daemon status`
- `aria daemon run --serve-one`
- `aria repl --workspace <workspace>`
- `hello`
- `attach`
- `new_task`
- `get_status`
- `list_artifacts`
- `detach`

验收证据：

- daemon lock 与 metadata 正常创建和清理。
- Unix socket 可连接。
- response envelope 字段稳定。
- `.aria/runtime/session.json` 与 task 初始化产物存在。

### 5.3 L2：fake provider 全链路 smoke

目标是剥离真实 provider 的不确定性，确认 Aria 的 planning、execution、integration、final closure 内部链路稳定。

主要覆盖：

- planning chain
- execution setup
- coding/testing/review/rework chain
- integration prepare/execute/verify
- final review/final summary

验收证据：

- phase1 smoke 测试通过。
- protocol steps 顺序符合预期。
- artifact validation 与 phase1 profile validation 通过。
- OpenSpec bundle、projection、traceability 逻辑稳定。

### 5.4 L3：真实 provider preflight

目标是确认 Aria 能真实调用 Claude Code 和 Codex，并能解析其输出。

主要覆盖：

- provider capability probe
- auth check
- version check
- CLI spawn
- stdin prompt 输入
- stdout/stderr 捕获
- sentinel JSON 解析
- provider error 分类

验收证据：

- Claude Code 与 Codex 均能被探测。
- 未出现 `provider_command_missing`、`provider_unauthorized`、`provider_permission_denied`。
- 对测试 prompt 能输出合法 `<ARIA_STRUCTURED_OUTPUT>` sentinel JSON。
- `ProviderRunRecord` 完整记录 stdout、stderr、structured output refs。

### 5.5 L4：正式 E2E 登录用例

目标是用真实 Claude Code、Codex、superpowers、OpenSpec 在 `naruto` 测试 worktree 中完成登录功能闭环。

输入需求：

> 做一个用户登录功能，具备前端页面，后端服务，用 JWT 即可。不需要 SQLite，不需要数据库，内存处理即可。

目标技术栈约束：

- 前端：Vite + React
- 后端：Node.js + Express
- 包管理：pnpm
- 鉴权：JWT
- 存储：内存用户存储
- API：至少 `POST /api/login`，可选 `GET /api/me`
- 页面：登录表单、错误态、登录成功态
- 测试：由 Aria 工作流触发并记录

关键要求：

- Aria 必须生成或更新 OpenSpec change。
- Aria 必须生成计划并分派 worktask。
- Aria 必须调用真实 Claude/Codex 完成代码变更。
- Aria 必须在 testing 或 integration verify 阶段运行 `naruto` 的验证命令。
- Aria 必须记录测试命令、结果、失败详情和最终覆盖关系。

## 6. 正式 E2E 验收证据

正式 E2E 不以人工手工验证页面为主，而以 Aria 产物为准。

必须检查以下证据：

- `.aria/runtime/session.json`
- `.aria/runtime/tasks/<task_id>/`
- runtime checkpoints
- provider run records
- provider raw stdout/stderr refs
- structured output refs
- OpenSpec proposal/design/tasks/spec 写回
- spec/design/plan projections
- `testing_report`
- `code_review_report`
- `integration_report` 或 `integration_verify` 结果
- `final_review`
- `final_summary`
- `naruto` 测试 worktree 中的代码 diff

`testing_report` 至少应说明：

- Aria 触发了哪些测试或构建命令。
- 命令在哪个 worktree 中运行。
- stdout/stderr 或摘要在哪里。
- 测试是否通过。
- 如失败，失败原因和后续路由是什么。

## 7. 执行顺序

### 阶段 1：环境准备

在 macOS Intel 与 Win11 WSL Ubuntu 分别准备 Rust、Docker、Node、pnpm、Claude Code、Codex、OpenSpec 和 superpowers。

完成判定：

- `cargo --version` 可用，并自动使用 `rust-toolchain.toml` 中的 `1.95.0`。
- `docker --version` 可用。
- `pnpm --version` 可用。
- `claude --version` 可用且已登录。
- `codex --version` 可用且已登录。
- `naruto` 工作区干净。

### 阶段 2：Aria 快速开发反馈

在宿主机运行 Aria 的快速验证命令。

完成判定：

- `cargo check --locked` 通过。
- 与当前调试范围相关的定向测试通过。
- `cargo run --locked -- daemon status --workspace <workspace>` 输出符合预期。

### 阶段 3：Docker 最终门禁

按仓库规则使用 Docker 运行：

- `cargo fmt --check`
- `cargo clippy --all-targets --all-features --locked -- -D warnings`
- `cargo check --locked`
- `cargo test --locked -j 1`

完成判定：

- 全部通过。
- Docker 运行未产生 root-owned 文件。

### 阶段 4：创建 `naruto` 测试 worktree

从 `naruto` 创建隔离测试 worktree 和测试分支。

完成判定：

- `naruto` 主工作区保持干净。
- 测试 worktree 位于独立目录。
- 测试分支用于本次 E2E。
- `openspec/config.yaml`、`AGENTS.md`、`CLAUDE.md`、`.codex/config.toml` 等配置在测试 worktree 中可见。

### 阶段 5：Aria daemon/REPL 基础链路

以 `naruto` 测试 worktree 为 `--workspace` 启动 Aria daemon 或 REPL。

完成判定：

- daemon 可启动。
- REPL 可连接。
- `new_task` 能创建 task。
- `get_status` 能看到 task 状态。
- `.aria/runtime` 初始化正确。

### 阶段 6：fake provider 全链路 smoke

在 Aria 仓库内运行 fake provider 的 phase1 smoke。

完成判定：

- planning 到 final summary 的内部链路通过。
- 节点顺序与 phase1 预期一致。
- artifact、projection、OpenSpec bundle、profile validation 均通过。

### 阶段 7：真实 provider preflight

让 Aria 通过真实 CLI adapter 分别调用 Claude Code 和 Codex。

完成判定：

- provider probe 通过。
- auth check 通过。
- 能输出合法 sentinel JSON。
- provider run record 记录完整。

### 阶段 8：正式 JWT 登录 E2E

向 Aria 输入登录功能需求，由 Aria 完成 OpenSpec、计划、编码、测试、评审、集成和最终总结。

完成判定：

- `naruto` 测试 worktree 中产生合理代码变更。
- OpenSpec change 被创建或更新。
- Aria 工作流内部触发并记录测试。
- final review 覆盖所有关键需求。
- final summary 为成功收口或明确说明未收口原因。

### 阶段 9：只读审计

外部人工只读检查 Aria 产物和代码 diff。

完成判定：

- Aria 产物链完整。
- provider run refs 可追溯。
- 测试证据来自 Aria 工作流内部。
- 无人工绕过 Aria 的业务实现或测试。

## 8. 失败诊断矩阵

| 层级 | 典型失败 | 首查对象 | 处理方向 |
|------|----------|----------|----------|
| 环境 | Rust、Docker、pnpm、claude、codex 不可用 | preflight 输出 | 修环境，不修 Aria |
| 鉴权 | provider unauthorized | provider stderr、auth check | 重新登录 provider |
| CLI spawn | command missing、permission denied | `ProviderAdapterError` | 修 PATH、权限、WSL/macOS 差异 |
| 输出解析 | 缺 sentinel、JSON 非法、artifact_kind 不匹配 | raw stdout、structured output ref | 修 prompt 或 adapter 兼容矩阵 |
| OpenSpec | change 缺失、bundle stale/blocked | OpenSpec bundle、tasks.md | 修 OpenSpec 写回与编译逻辑 |
| worktree | 分支冲突、写入范围错误、权限问题 | git status、worktree lease | 修 worktree lifecycle |
| 测试 | `testing_report` 失败 | Aria 记录的测试命令和 stdout/stderr | 让 Aria 进入 rework，不人工修业务代码 |
| 评审 | code review blocking | `code_review_report` | 让 Aria 进入 rework |
| 集成 | integration verify fail | integration snapshot、git diff | 修 Aria 集成链路或让 Aria 返工 |
| 收口 | final coverage 不完整 | final review、traceability refs | 修 coverage/traceability 或补任务 |

## 9. 跨平台注意事项

macOS Intel：

- Unix socket 路径长度要保持较短。
- Claude/Codex 的 PATH 以交互 shell 和非交互进程分别确认。
- Docker volume 使用当前仓库路径挂载。

Win11 WSL Ubuntu：

- Aria、`naruto`、Claude/Codex、Node/pnpm 应尽量全部运行在 WSL 文件系统内。
- 避免在 `/mnt/c/...` 上做高频 Rust 编译和 Node 安装。
- provider 登录态必须在 WSL 内可见，而不是仅 Windows 宿主可见。
- Docker Desktop 与 WSL 集成需确认 volume 权限和 UID/GID。

## 10. 日常增量调试建议

高频修改 Aria 源码时：

- 优先宿主机 `cargo check --locked`。
- 对相关测试使用 `cargo test --test <name>`。
- daemon/REPL 问题用最小 workspace 复现。
- provider 问题先用 preflight prompt 复现，不直接跑完整 E2E。
- 每次正式结论前必须跑 Docker 门禁。

可选 watch 策略：

- 使用 `cargo watch -x 'check --locked'` 做持续编译。
- 或使用 `bacon` 做交互式 Rust 反馈。

这些工具提升反馈速度，但不改变最终验收标准。

## 12. 后续建议

第一轮只执行成功路径 B：真实小功能闭环。成功后再追加故障链路矩阵，包括：

- provider 未登录
- provider 输出格式错误
- provider timeout
- OpenSpec bundle stale
- testing failure 后 rework
- code review blocking 后 rework
- integration verify failure
- final coverage 不完整

故障链路应作为第二阶段测试计划，不应阻塞第一轮正式成功路径验证。
