# 技术方案：Aria npx 本地化分发

- 文档类型：技术方案设计
- 版本：v1.0
- 创建日期：2026-05-31
- 适用分支：`product-workbench-issue-lifecycle`
- 目标读者：项目开发者、维护者、发布负责人

## 1. 目标与背景

把当前 `cadence-aria` 项目打包成一个可通过 `npx` 一键运行的软件，让用户无需安装 Rust 工具链、无需联网编译，即可在本地启动 Aria Web 工作台。

### 1.1 现状（已勘查确认）

- **架构**：Rust 后端二进制 `aria`（edition 2024 / rustc 1.95.0）+ React19/Vite6/TypeScript 前端（`web/`，pnpm 管理）。
- **运行方式**：`aria web --workspace <path> [--host <h>] [--port <p>]` 启动 axum Web 服务（默认 host `127.0.0.1`，未指定端口时绑定 `0`，由内核自动分配），并以前端构建产物作为 fallback 静态资源服务。另有 `task run`、`daemon`、`repl` 等子命令。
- **入口透传**：`src/main.rs` 将 `std::env::args().skip(1)` 整体交给 `run_cli_async`，参数解析集中在 `src/cli.rs`。
- **绑定地址输出契约**：`serve_web` 在 `src/web/app.rs` 中执行 `eprintln!("aria web listening on http://{bound_addr}")`，输出到 stderr。

### 1.2 打包成 npx 的关键障碍（已定位）

1. **前端资源路径硬编码**：`src/web/static_assets.rs:6` 使用 `env!("CARGO_MANIFEST_DIR").join("web/dist")`，指向编译期源码目录。分发后该路径不存在，是分发的核心硬伤。该引用点唯一（另有两处 `CARGO_MANIFEST_DIR` 位于 `codex_provider.rs:839`、`claude_code_provider.rs:966`，均为测试 fixture，不影响运行时）。
2. **aria 是原生二进制，npx 属 JS 生态**：需要 JS launcher + 二进制分发方案。
3. **依赖外部 codex/claude CLI**：真实 provider 通过 spawn 外部进程运行，默认调用 PATH 中的 `claude`、`codex`（`src/cross_cutting/adapter_compatibility.rs:74-114`）。
4. **git 依赖**：`codex-protocol` 等来自 GitHub tag，冷编译较慢，不适合让终端用户本地编译。

## 2. 决策摘要（已与用户确认）

| 决策项 | 选定方案 |
|--------|---------|
| 分发策略 | 预编译二进制 + 平台子包（esbuild/swc/biome 同款的 optionalDependencies 模式） |
| 平台矩阵 | 3 目标：`linux-x64`、`darwin-x64`、`darwin-arm64`（`linux-arm64` 作为后续增量） |
| 前端资源 | 编译期嵌入二进制（`rust-embed`），平台子包只含单个二进制文件 |
| 外部 CLI | 用户自行安装 codex/claude；`aria web` 启动时检测 PATH 并友好提示，但不阻断启动 |
| 默认行为 | `npx @cadence-aria/cli` 无参时启动 web 并自动打开默认浏览器 |
| 包名 | 主包 `@cadence-aria/cli`，bin 别名 `aria`；平台子包 `@cadence-aria/cli-<platform>` |
| 发布自动化 | Git `v*` tag 触发 GitHub Actions 全自动多平台构建并发布到 npm |
| CLI 面 | 完整透传所有子命令（`web`/`task run`/`daemon`/`repl`），无参注入默认 `web` |
| 交付范围 | 一次性完整交付（Rust 改造 + launcher + 本地 npm pack 验证 + 多平台 release workflow） |

## 3. npm 包拓扑

采用业界验证的「主包 + 平台子包 + optionalDependencies」结构（esbuild、@swc/core、@biomejs/biome 同款）：

```
@cadence-aria/cli                      ← 主包（用户 npx 的目标）
├── package.json                       ← bin: { "aria": "bin/aria.js" }
│                                         optionalDependencies 列出全部平台子包
├── bin/aria.js                        ← JS launcher（纯 JS，无原生代码）
└── README.md                          ← 体积极小，无二进制

@cadence-aria/cli-linux-x64            ← 平台子包（含预编译二进制）
├── package.json                       ← os:["linux"], cpu:["x64"]
└── bin/aria                           ← release 二进制（已嵌入前端 dist）

@cadence-aria/cli-darwin-x64           ← os:["darwin"], cpu:["x64"]
└── bin/aria

@cadence-aria/cli-darwin-arm64         ← os:["darwin"], cpu:["arm64"]
└── bin/aria
```

### 3.1 工作原理

- npm 安装主包时会尝试安装所有 `optionalDependencies`；每个平台子包通过 `os`/`cpu` 字段声明适用平台，npm 仅真正安装与当前机器匹配的那一个，其余静默跳过（不报错）。
- launcher 通过 `require.resolve("@cadence-aria/cli-<platform>/bin/aria")` 定位已装好的二进制。
- 因为前端 `web/dist` 已在编译期嵌入 `aria` 二进制（见第 4 节），平台子包内只有一个二进制文件，无额外静态资源目录。

### 3.2 版本一致性

主包与所有平台子包版本号严格一致；主包 `optionalDependencies` 中各平台子包固定为「等于主包版本」（exact，非 range），由 release 流程统一注入，避免版本漂移。

## 4. Rust 侧改造

### 4.1 改造 1：前端资源嵌入二进制（核心）

将 `src/web/static_assets.rs` 从磁盘读取改为编译期嵌入：

- 引入 `rust-embed`，把 `web/dist` 目录在编译期打进二进制。
- 运行时默认从嵌入资源读取；SPA 路由 fallback 到嵌入的 `index.html`。
- 保留开发逃生口：环境变量 `ARIA_WEB_DIST` 若设置为一个磁盘目录，则覆盖嵌入资源、改从该目录读取（供本地调试、playwright e2e、前端热更新场景使用）。
- 资源响应需正确设置 `Content-Type`（复用现有 `mime_guess` 依赖）。
- 构建顺序约束：`aria` 二进制构建前，`web/dist` 必须已由 `pnpm -C web build` 生成。通过 `build.rs` 校验 `web/dist/index.html` 存在，缺失时给出清晰的编译期错误（指明应先执行前端构建），避免嵌入空目录。

**接口契约**：`static_dist_service`（或其替代）对外行为保持「精确命中返回对应文件、未命中返回 index.html、正确 Content-Type」，使 `src/web/app.rs:221` 的调用点改动最小。

### 4.2 改造 2：启动检测提示

`aria web` 启动时探测 PATH 中是否存在 `claude`、`codex`（program 名取自 `adapter_compatibility` 的 `provider_command`/probe，避免硬编码漂移）：

- 缺失任一命令：打印友好的中文提示（说明该命令支撑哪些 provider 功能、给出安装指引链接占位），但**不阻断**启动——工作台 UI 本身可用，FakeProvider 演示路径不依赖外部 CLI。
- 两者齐备：可选地打印一行已检测到的提示（或静默）。
- 探测逻辑独立成可单测的函数（输入：PATH 查找结果；输出：提示文案与是否缺失），不在 `serve_web` 内联，便于 TDD。

### 4.3 改造 3：默认行为支撑（Rust 侧最小改动）

默认「启动 web + 开浏览器」主要由 JS launcher 实现（见第 5 节）。Rust 侧只需保证绑定地址输出契约稳定：`aria web` 启动成功后向 stderr 输出可被 launcher 解析的 `listening on http://<host>:<port>` 行。当前 `src/web/app.rs` 已具备该输出。本节仅作契约确认，必要时为该行格式补充测试以防回归。

## 5. JS Launcher（`bin/aria.js`）

纯 JS（CommonJS 或 ESM，依主包 `type` 字段定），无原生代码。职责：

1. **定位二进制**：
   - 由 `process.platform` + `process.arch` 映射子包名：`linux/x64 → cli-linux-x64`、`darwin/x64 → cli-darwin-x64`、`darwin/arm64 → cli-darwin-arm64`。
   - `require.resolve("@cadence-aria/cli-<platform>/bin/aria")` 取二进制路径。
   - 未匹配/未安装：报清晰错误，列出当前 `platform-arch` 与已支持矩阵，提示该平台暂无预编译包（可走源码构建）。

2. **参数透传与默认子命令**：
   - 无参（仅 `aria`）：注入默认 `web` 子命令（workspace 默认取当前工作目录，与 Rust 侧 `parse_workspace` 缺省一致）。
   - 有参：原样透传给二进制（`task run`、`daemon`、`repl`、`web --port 3000` 等）。
   - launcher 自身消费的标志（如 `--no-open`）从透传参数中剥离，不传给二进制。

3. **自动开浏览器**（仅默认 web 模式）：
   - 监听子进程 stderr，匹配 `listening on http://<host>:<port>`，提取地址。
   - 调用系统命令打开浏览器：macOS `open`、Linux `xdg-open`（不可用时降级为仅打印地址）。
   - `--no-open` 关闭自动打开，仅打印地址（适配远程/无头环境）。
   - 非默认 web 模式（用户显式传子命令）不触发开浏览器。

4. **进程与信号转发**：
   - 用 `child_process.spawn` 拉起二进制，`stdio` 透传（stdin/stdout/stderr 贯通，stderr 同时被 launcher 监听以抓地址）。
   - 转发 `SIGINT`/`SIGTERM` 给子进程，保证 Ctrl-C 干净退出。
   - 子进程退出码透传为 launcher 退出码。

## 6. 发布自动化（GitHub Actions，Tag 触发）

`v*` tag push 触发，工作流分阶段：

1. **多平台构建（矩阵）**：
   - `ubuntu-latest` → `x86_64-unknown-linux-gnu`；
   - `macos-13`（Intel）→ `x86_64-apple-darwin`；
   - `macos-14`（Apple Silicon）→ `aarch64-apple-darwin`。
   - 均为**原生编译**（在对应 OS/arch runner 上直接 `cargo build --release`），规避交叉编译 git 重型依赖的坑。
   - 每个构建 job 前置：`pnpm -C web install` + `pnpm -C web build` 生成 `web/dist`，再 `cargo build --release`（前端经 `rust-embed` 嵌入二进制）。

2. **组包**：
   - 脚本将各 release 二进制放入对应平台子包目录的 `bin/aria`。
   - 由模板生成各平台子包 `package.json`（注入版本号、`os`、`cpu`、`bin`）。
   - 校验二进制可执行、架构正确（`file` 命令断言）。

3. **发布**：
   - 先 `npm publish` 全部平台子包，再 `npm publish` 主包（确保主包安装时子包已可解析）。
   - 主包 `optionalDependencies` 各子包版本注入为当前发布版本（exact）。
   - 使用 `npm publish --access public`（scoped 包默认 private，需显式 public）。

4. **发布前置条件（执行发布时由维护者提供）**：
   - npm 上 `@cadence-aria` 组织/scope 已创建。
   - CI 配置 `NPM_TOKEN` secret，具备 `@cadence-aria` scope 的发布权限。
   - 版本号管理策略：tag 即版本来源（如 `v0.1.0` → 包版本 `0.1.0`），由 workflow 从 tag 解析并写入各 `package.json`。

## 7. 测试策略（遵循 TDD）

按项目规则先写测试再写实现，新功能须有对应测试。

### 7.1 Rust 单测

- `static_assets` 嵌入资源：精确命中已知文件返回正确内容与 Content-Type；未命中路径 fallback 到 `index.html`；`ARIA_WEB_DIST` 设置时改从磁盘目录读取并覆盖嵌入资源。
- 启动检测：PATH 中存在/缺失 `claude`、`codex` 时，检测函数返回正确的「缺失列表 + 提示文案」；验证缺失时不阻断（函数返回提示而非错误）。
- 绑定地址输出行格式回归测试（防止 launcher 解析契约被破坏）。
- 验证命令遵循 `cadence/project-rules/build-test-commands.md`：`cargo fmt --check`、`cargo clippy --all-targets --all-features --locked -- -D warnings`、`cargo check --locked`、`cargo test --locked`；禁止 `-j 1`。

### 7.2 Launcher 单测

- 平台映射：三平台 arch 组合映射到正确子包名；未支持平台返回清晰错误。
- 参数透传：无参注入 `web`；有参原样透传；`--no-open` 被剥离不传给二进制。
- 开浏览器决策：默认 web 模式触发、显式子命令不触发、`--no-open` 关闭。
- 用 node 内置测试（`node:test`）+ 子进程桩（fake 二进制脚本）验证，无需真实 aria 二进制。

### 7.3 本地集成验证（发布前冒烟基线）

- `pnpm -C web build` → `cargo build --release` → 组当前平台（linux-x64）子包 → `npm pack` 主包与子包。
- 在干净临时目录 `npx <主包 tgz>`：验证二进制被定位、web 启动、stderr 打印地址、（有桌面环境时）浏览器打开工作台。
- `npx <主包> --no-open`：仅打印地址、不开浏览器。
- `npx <主包> web --port <固定端口>`：参数透传生效。

### 7.4 现有 e2e 兼容

- 确认前端嵌入改造后，`web/e2e/` 下 playwright 用例仍绿；e2e 启动 aria 时通过 `ARIA_WEB_DIST` 指向 `web/dist`（或验证嵌入路径），保持现有 dev/e2e 工作流不破。

## 8. 影响范围与风险

- **改动面**：Rust 侧集中在 `static_assets.rs`、新增启动检测模块、`build.rs`、`Cargo.toml`（新增 `rust-embed`）；新增 npm 包目录（主包 + 平台子包模板）；新增 release workflow。`src/cli.rs`、`src/main.rs` 基本不动。
- **二进制体积**：嵌入前端 dist 后二进制增大（量级取决于前端产物，预计数 MB）；release profile 已有体积优化经验（见 `Cargo.toml` profile 配置思路），可后续按需收敛。
- **构建耦合**：前端必须先于二进制构建，已由 `build.rs` 校验兜底。
- **平台局限**：本期不含 `linux-arm64` 与 Windows 原生；非支持平台 launcher 给出清晰提示，不静默失败。
- **外部依赖**：codex/claude CLI 由用户自装；缺失仅影响真实 provider 执行，工作台与 Fake 演示不受影响。
- **安全**：Web 默认绑定 `127.0.0.1`（仅本机），无对外暴露；不引入鉴权变更。launcher 调用系统 `open`/`xdg-open` 仅传入本地 `http://127.0.0.1:<port>` 地址，不拼接用户可控内容。
