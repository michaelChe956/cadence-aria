## 1. 目标与范围

### 1.1 一期 MVP 定位

- **形态**：REPL client + 本地 daemon runtime
- **范围**：单用户、本机、单仓库、多任务并行（通过 worktree 隔离）
- **Provider 接入**：spawn + CLI（Claude Code 作为 orchestrator，Codex 作为 executor/reviewer）
- **隔离策略**：每任务独立 worktree
- **集成策略**：并行执行、串行集成（FIFO 队列，一期不做优先级调度）
- **人工介入**：保守策略，关键节点挂 approval gate
- **Daemon 生命周期**：daemon 常驻，REPL 只是客户端，可 attach/detach

### 1.2 一期不做的事（明确排除）

| 排除项 | 原因 |
|--------|------|
| Risk Registry | 一期用产物内嵌风险列表，跨产物追踪延期到二期 |
| 5 类循环计数器 | 一期用 1 个通用 `retry_counter`，各场景统一阈值 |
| Event Log compaction | 一期 append-only，不清理；延期到二期 |
| Gate TTL / 超时 | 一期 gate 永久等待，不自动升级 |
| 补丁派生（N26） | 完整流程保留节点定义，但实现上标记为"二期扩展点" |
| TUI/GUI 目标态 | 完全从一期设计移除，不引入未来假设 |

### 1.3 关键假设（显式声明）

1. **spawn + CLI 是唯一 provider 接入方式** — 已调研确认 SDK/API 均不可行（详见调研笔记）
2. **daemon 常驻，REPL 只是客户端** — REPL 启动时 attach 到已有 daemon 或隐式启动新 daemon；REPL 退出时仅断开连接，daemon 继续运行；任务在后台持续执行
3. **worktree 隔离优于 branch 隔离** — 并发任务需要真正的文件系统隔离，git branch 无法满足并行编码需求

### 1.4 可扩展性保障

**TUI/GUI 扩展保障**

核心原则：**REPL 只负责交互，daemon 是运行时真相源**。无论前端形态是什么（REPL / TUI / GUI / Web），它们都只是 daemon 的客户端，通过统一接口通信。

一期 REPL 与 daemon 通过 JSON over Unix socket（Windows 用 named pipe）通信。daemon 本身不感知前端形态。二期做桌面端时，替换 REPL 为新的客户端实现，daemon 不动。

**Provider SDK 切换保障**

一期设计了 Provider Adapter 接口层。daemon 只调用 adapter 的统一接口，不关心底层实现：

```
daemon -> ProviderAdapter::dispatch() -> [CliAdapter | SdkAdapter | ...]
```

一期实现 `CliAdapter`，二期可以并行实现 `SdkAdapter`，两者共存，通过配置切换。daemon 代码零改动。

### 2.3 合并节点内部子阶段（细节不丢失）

**N02 task_init（原 N02 + N03）**
- 子阶段 A：`epic_task_create` — 创建 EpicTask，分配 epic_ 前缀 ID，绑定 session
- 子阶段 B：`policy_resolve` — 解析全局策略 + 阶段级覆写，生成 effectivePolicy（一期只支持 conservative）
- 内部状态：子阶段 A 完成后产出 `taskCreated` 中间状态，子阶段 B 完成后产出最终 runtime_snapshot
- 失败处理：任一字阶段失败，整个 N02 失败，进入 X08 manual_intervention

**N10 plan_dispatch（原 N10 + N11 + N12）**
- 子阶段 A：`readiness_check` — 调用 Claude Code 检查 design + design_review 产物完整性，判断阻塞项
- 子阶段 B：`plan_authoring` — 调用 Claude Code 编写执行计划（plan 产物）
- 子阶段 C：`dispatch_authoring` — 基于 plan 拆分子任务，生成 dispatch_package
- 子阶段依赖：A -> B -> C 串行
- 回流失效：若 readiness_check 失败回流到 N07，plan 和 dispatch_package 标记为 superseded

**N13 execution_setup（原 N13 + N14 + N15）**
- 子阶段 A：`worktask_register` — 为 dispatch_package 中的每个子任务注册 WorkTask，分配 work_ 前缀 ID
- 子阶段 B：`worktree_prepare` — 为每个 WorkTask 创建/分配 worktree，生成 WorktreeLease
- 子阶段 C：`execution_route_resolve` — 解析每个 WorkTask 的首次执行节点（通常是 N16 coding）
- 并行性：子阶段 A 和 B 可并行，C 依赖 A+B 完成
- 产物：一个 runtime_snapshot 包含 `registeredWorktasks`、`worktreeLeases`、`routing` 三个字段组

**N20 integration_prepare（原 N20 + N21 + N22）**
- 子阶段 A：`ready_for_integration` — 检查 WorkTask 前置产物状态，标记 ready
  - daemon 做结构化预检（字段值、枚举值合规）
  - 当 code_review_report.review_decision = conditional_pass 时，调用 Codex 判断条件是否已满足
- 子阶段 B：`integration_enqueue` — 将 ready 的 WorkTask 按 FIFO 入队
- 子阶段 C：`integration_prepare` — 出队后准备集成：记录 baseRefBeforeIntegration，执行 git precheck
- 队列管理：子阶段 B 维护 queuePosition 和 enqueueTime；子阶段 C 处理出队
- 冲突处理：precheck 发现冲突时标记 conflictRisk，由 N23 处理

### 2.4 横切节点（8 个）

| 节点 ID | 节点名 | 一期范围 |
|---------|--------|---------|
| X01 | approval_gate_open | 必须 |
| X02 | approval_gate_resume | 必须 |
| X03 | provider_run_dispatch | 必须 |
| X04 | provider_run_collect | 必须 |
| X05 | artifact_validate | 必须 |
| X06 | checkpoint_persist | 必须 |

**移除：** X09 policy_override_apply 不独立为节点，其功能合并到 N02 task_init 的子阶段 B 中。

### 2.5 调整说明：spec_gate_review 改为 Agent 节点

N06 spec_gate_review 从内部节点改为 Agent 业务节点，由 **Codex** 执行：

- 评审内容：规格完整性、需求清晰度、agent 可执行性评估
- daemon 职责：进入前执行 X05 artifact_validate（L1/L2 格式校验），收集 Codex 评审结果并路由
- 使用 Codex 而非 Claude Code：与 design_review（N08）保持"编写者+评审者"分离模式

### 2.6 调整说明：ready_for_integration 改为混合执行

N20 子阶段 A（ready_for_integration）从纯内部改为混合：
- daemon 做结构化预检（字段值合规）
- 当 review_decision = conditional_pass 时，调用 Codex 判断条件是否已满足
- 语义判断交给 Agent，daemon 只做调度

## 3. 产物规范精简

### 3.1 精简策略

不删减产物种类（16 种全部保留，不丢失字段），但按产物类型分组组织规范文档，减少文档数量和重复章节。

原设计：16 种产物 x 每份独立规范文档 = 16 份文档
精简后：3 份类型规范 + 1 份产物总目录 = 4 份文档

### 3.2 分组方案

| 规范文档 | 涵盖产物 | 说明 |
|---------|---------|------|
| **用户可读文档规范** | intake_brief, spec, design, plan, dispatch_package, final_summary | Markdown 格式，统一章节结构、版本标注 |
| **评审决策产物规范** | spec_gate_decision, design_review, design_revision_record, readiness_check, code_review_report, testing_report, coding_report, integration_report, final_review | JSON 格式，统一枚举值引用、决策语义 |
| **Runtime Snapshot 通用规范** | 所有内部节点产出 | 一份规范覆盖 N00/N02/N13/N20/N28 等全部内部节点 |
| **产物总目录** | 全部 16 种 | 索引文档，列产物 ID、类型、格式、所属节点 |

### 3.3 一期范围

| 产物 | 一期范围 | 说明 |
|------|---------|------|
| intake_brief | 必须 | 用户原始需求记录 |
| clarification_record | 必须 | 需求澄清记录 |

### 3.4 Runtime Snapshot 通用规范

所有内部节点共用一份规范，节点特有字段通过 `nodeSpecificFields` 扩展：

```json
{
  "snapshotId": "snap_<nodeId>_<sequence>",
  "sessionId": "sess_xxx",
  "taskId": "epic_xxx / work_xxx",
  "nodeId": "Nxx",
  "phase": "string",
  "timestamp": "ISO8601",
  "effectivePolicy": {},
  "artifactRefs": [],
  "providerRunRefs": [],
  "worktreeRef": "wt_xxx",
  "retryCounter": 0,
  "nodeSpecificFields": {}
}
```

| 节点 | nodeSpecificFields 关键字段 |
|------|---------------------------|
| N00 | bootstrapMode, repoRoot |
| N02 | epicTaskId, effectivePolicy |
| N13 | registeredWorktasks, worktreeLeases, routing |
| N20 | queuePosition, enqueueTime, baseRefBeforeIntegration |
| N28 | completedAt, releasedResources |

**校验规则（一期只实现 L1 + L2）：**
- L1：必需字段存在且非空
- L2：字段类型正确、ID 格式合规
- L3（语义性）：标注为二期增强

### 3.5 产物格式规则

- **用户可读文档** -> Markdown（`.md`）
- **评审决策产物** -> JSON（`.json`）
- **Runtime Snapshot** -> JSON（`.json`，存储于 `.aria/runtime/checkpoints/`）

## 4. 横切能力精简

### 4.1 精简策略

横切能力不删减种类，但每份能力文档标注一期实现范围：必须实现 / 简化实现 / 保留定义但延期。

### 4.2 一期必须实现（6 个）

| 横切能力 | 一期范围 | 核心功能 |
|---------|---------|---------|
| **CC10 provider_adapter_spec** | 必须 | spawn + CLI 统一接口；并发池（默认 4）；超时策略（默认 5min）；输出结构化提取 |
| **CC03 provider_run_lifecycle** | 必须 | dispatch -> execute -> collect -> 固化记录；run_ 前缀 ID 分配 |
| **CC04 worktree_lifecycle** | 必须 | 创建 worktree、绑定 WorkTask、释放租约；支持并发创建 |
| **CC05 artifact_validate** | 必须 | L1 存在性校验 + L2 结构性校验；L3 标注为二期 |
| **CC06 checkpoint_persist** | 必须 | snapshot 落盘到 `.aria/runtime/checkpoints/`；按 task 隔离存储；**写入后计算 SHA-256 checksum** |
| **CC07 runtime_recover** | 必须 | daemon 启动时从 checkpoint 恢复状态；checksum 校验失败时尝试上一个 checkpoint |
| **CC01 approval_gate** | 必须 | gate 创建、挂起、恢复；**无 TTL**（一期 gate 永久等待） |
| **CC08 manual_intervention** | 必须 | 无法自动决策时进入终态；生成 intervention record；向 REPL 暴露摘要；等待用户接管 |

### 4.3 一期简化实现（2 个）

| 横切能力 | 一期范围 | 简化内容 | 二期扩展 |
|---------|---------|---------|---------|
| **CC02 policy_mode_and_override** | 简化 | 只实现 `conservative` 策略 | `balanced` / `aggressive` |
| **CC08 integration_queue** | 简化 | 只实现 FIFO 串行队列 | 优先级调度、依赖拓扑感知 |

**policy_mode_and_override 简化细节：**
- 策略行为映射表只填写 conservative 列
- 所有 approval gate 默认开启
- retry 阈值固定为 3 次
- 阶段级策略覆写：一期只支持"全局策略"和"当前任务策略"两级

**integration_queue 简化细节：**
- 出队顺序：严格 FIFO
- 不支持插队
- 队列持久化：通过 checkpoint 实现（daemon 重启时从 checkpoint 恢复队列状态）

### 4.4 保留定义但一期不实现（3 个）

| 横切能力 | 保留理由 | 一期替代方案 | 二期扩展路径 |
|---------|---------|------------|-------------|
| **Event Log compaction** | 性能优化 | 一期 append-only，不清理；恢复依赖 checkpoint | event log 写入格式一期即确定，compaction 是纯只读后处理 |
| **Gate TTL / 超时** | 防卡住 | 一期 gate 永久等待 | gate 记录结构预留 `ttl` 字段（置 null），二期加定时检查 |
| **Risk Registry** | 跨产物追踪 | 一期各产物内嵌风险列表 | 各产物保留风险字段，二期提取为全局 Registry，产物改为 riskId 引用 |

## 5. 全局协议精简

### 5.1 精简策略

全局协议从 17 章压缩到 8 章，移除或合并的章节标注"一期简化 / 二期扩展"。

### 5.2 一期全局协议（8 章）

| 章节 | 内容 | 一期范围 |
|------|------|---------|
| **1. 核心对象** | ProjectSession, EpicTask, WorkTask, Phase, Artifact, ApprovalGate, ProviderRun, WorktreeLease | 必须，8 个对象全部保留 |
| **2. ID 规则** | 前缀规范（sess_/epic_/work_/gate_/run_/wt_/art_/evt_/Nxx/Xxx） | 必须 |
| **3. 状态语义** | Task 7 状态 + Node 6 结果 | 必须 |
| **4. 交接包规则** | 9 个必含字段 | 必须 |
| **5. 产物分类** | managed_doc / runtime_snapshot / validation_record / handoff_package | 必须 |
| **6. 存储规则** | cadence/ 用户文档 + .aria/runtime/ 机器状态 | 必须 |
| **7. 完成判定** | 6 条全局规则 | 必须 |
| **8. 失败与回流** | 可重试失败 -> 有限重试 -> 人工介入 | 必须，但**循环边界简化** |

### 5.3 合并/延期章节

| 原章节 | 处理方式 | 说明 |
|--------|---------|------|
| 人工介入全局规则 | 合并到 8 | 全局规则中说明 gate 和 manual_intervention 的兜底关系 |
| Provider 接入全局规则 | 合并到 1 | 在 ProviderRun 对象说明中标注"一期 spawn+CLI" |
| 循环边界控制 | 简化入 8 | 保留"循环计数 + 阈值升级"原则，但统一为 1 个计数器 |
| 回流失效 | **保留** | 不简化，回流标记 superseded 是数据一致性基础 |
| 枚举值注册表 | **保留** | 不简化，跨文档一致性依赖此表 |
| Risk Registry | 延期到二期 | 一期产物内嵌风险列表，不启用全局 Registry |
| Runtime Snapshot | **保留** | 不简化，但统一为通用规范 |
| 统一术语表 | **保留** | 不简化 |
| 关键假设 | **新增** | 显式声明 spawn+CLI、daemon 生命周期、worktree 选择理由 |

### 5.4 循环边界简化（重点）

**一期统一规则：**
- 每个 WorkTask 维护一个 `retry_counter`，初始为 0
- 任意节点回流（包括 rework、设计修订、澄清补充、集成失败）时 `retry_counter += 1`
- 超过阈值（默认 3，可配置）后：不再自动回流，进入 X08 manual_intervention
- 补丁派生（原 N26）不计入 retry_counter，一期不做自动补丁派生

**二期扩展：** 按场景细分计数器（rework_counter / design_revision_counter / clarification_counter 等）

## 6. 技术栈与目录结构

### 6.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 运行时 | **Rust** | daemon 核心：调度、状态机、worktree 管理、checkpoint |
| Provider 适配 | Rust + 进程调用 | `std::process::Command` spawn Claude Code / Codex CLI |
| REPL | **Rust**（一期） | 基于 `rustyline` 或类似库，通过 Unix socket 与 daemon 通信 |
| 通信协议 | JSON over Unix socket | 语言无关，二期桌面端可直接对接 |
| 序列化 | JSON | 产物、snapshot、checkpoint 统一用 JSON |
| 版本控制 | git + git2 crate | worktree 操作、集成合并 |

### 6.2 REPL 架构（方案 C）

- **daemon** 是独立进程，暴露 JSON over Unix socket 接口
- **REPL** 是一期客户端，用 Rust 实现，通过 socket 与 daemon 通信
- **二期桌面端** 用 TypeScript/Electron 或 Tauri 写新客户端，替换 Rust REPL，daemon 完全不动

通信协议基于 JSON-RPC 2.0 或自定义 JSON 协议：

```json
// REPL -> Daemon
{
  "method": "task.submit",
  "params": { "intake": "实现用户登录功能" },
  "id": 1
}

// Daemon -> REPL
{
  "result": { "epicTaskId": "epic_001", "phase": "clarification" },
  "id": 1
}
```

## 附录 C：关键规则摘要

### C.1 回流失效规则摘要

**核心原则：** 当节点回流跨越一个或多个中间节点时，中间节点产出的产物必须标记为失效，不允许后续流程引用失效产物。

**失效标记：**
- 回流发生时，Aria 将回流路径上所有中间节点的产物标记为 `superseded`
- `superseded` 标记写入 runtime_snapshot，包含：被失效产物引用列表、回流原因、发起/目标节点 ID、失效时间戳

**失效引用规则：**
- 任何节点执行时，不得引用状态为 `superseded` 的产物
- Aria 在节点进入前必须校验输入契约中的所有产物引用，确保无 `superseded` 状态
- 如果输入契约中存在 `superseded` 产物，节点不得进入执行

**重新执行规则：**
- 回流目标节点重新执行后，其产出的新产物自动覆盖同类型的 `superseded` 产物
- 后续节点只能引用最新产物，不允许引用历史版本

**Checkpoint 记录：**
- 回流事件必须写入 checkpoint
- checkpoint 中包含完整的失效范围（哪些节点、哪些产物被标记为 superseded）
- daemon 重启后恢复时，必须还原失效标记

**常见回流路径与失效范围：**

| 回流路径 | 跨越节点 | 失效产物 |
|---------|---------|---------|
| N12 -> N07 | N08, N09, N10, N11 | design_review, design_revision_record, readiness_check, plan |
| N11 -> N07 | N08, N09, N10 | design_review, design_revision_record, readiness_check |
| N10 -> N07 | N08, N09 | design_review, design_revision_record |
| N06 -> N04 | N05 | spec |
| N23 -> N19 | N20, N21, N22 | ready_for_integration snapshot, queue record, prepare snapshot |
| N24 -> N19 | — | 无中间节点 |
