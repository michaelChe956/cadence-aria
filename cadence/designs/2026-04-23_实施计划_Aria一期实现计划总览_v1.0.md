# Aria Phase 1 Implementation Plan Index

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 Aria 一期实现拆成研发可以分工执行、按里程碑验收的总计划与 4 个子计划。

**Architecture:** 一期实现按 4 条主线推进：基础骨架与 REPL/daemon 协议、产物与约束、provider 与规划链、执行与最终收口。总 plan 负责定义依赖和里程碑，具体执行以子 plan 为准。

**Tech Stack:** Rust、Cargo、JSON、Markdown、Unix socket / named pipe、git worktree、本地 CLI provider。

---

## 1. 为什么拆成多个 plan

本次 implementation plan **必须拆成多个子 plan**，原因不是文档管理，而是工程边界：

1. REPL/daemon wire protocol 是独立接口面，先定好才能并行开发。
2. canonical artifact / projection / `_aria` / OpenSpec bundle / traceability 是独立的数据面，不能散落在节点实现里。
3. provider contract 与 planning nodes 是独立执行面，可以先用 fake provider 做通路验证。
4. execution / integration / final closure 是独立收口面，依赖前面三层但本身可以作为单独里程碑验收。

如果只写一个总 plan，结果会是：

- 所有负责人都抢同一份文档
- 依赖边界变模糊
- 实施顺序和并行边界说不清

所以采用：

- **1 份总览 plan**
- **4 份子 plan**

## 3. 推荐执行顺序

### 3.1 基线顺序

1. P1
2. P2
3. P3
4. P4

### 3.2 允许的受控并行

若团队有足够人手，可以按以下边界受控并行：

1. **P2 可以在 P1 Task 2 完成后启动**
   - 前提：`repl_wire.rs`、基础 protocol 类型、Cargo 工程骨架已建立
2. **P3 可以在 P2 Task 3 完成后启动**
   - 前提：projection、`_aria`、phase1 profile validator 已可用
3. **P4 可以在 P3 Task 4 完成后启动**
   - 前提：`dispatch_package` 与 `dispatch_package._aria.worktask_routing[]` 已稳定

禁止的并行方式：

1. 在没有 wire protocol 的情况下先做 REPL 与 daemon UI 行为
2. 在没有 projection / `_aria` 的情况下直接写 planning / execution 节点
3. 在没有 `dispatch_package._aria.worktask_routing[]` 的情况下直接做 execution nodes

## 4. 共享前置约束

所有子计划都必须遵守以下前置约束：

1. 以上游 canonical 协议文档为节点、路由、最小字段真相源
2. 以 `Aria一期MVP精简设计_v1.2` + `Aria一期实现总契约_v1.0` 为 phase1 implementation profile 真相源
3. 一期必须坚持：
   - Aria 单一正式收口
   - OpenSpec 全程约束、非可选
   - Superpowers 只约束 provider 内部工作纪律
4. 所有机器可消费信息必须走：
   - projection
   - `_aria`
   - runtime record
5. 不允许直接靠 Markdown 原文或 provider stdout 做 routing / coverage 判定

## 5. 共享测试基线

四个子计划都应统一使用以下测试层次：

1. **unit tests**
   - protocol / wire schema
   - projection compiler
   - bundle compiler
   - traceability binder
   - provider context builder
2. **integration tests**
   - REPL <-> daemon handshake
   - planning chain with fake provider
   - execution / integration / final closure with fake provider
3. **fixture-based tests**
   - fixture OpenSpec change
   - fixture canonical artifacts
   - fixture dispatch package / reports

建议测试目录：

```text
tests/
├── unit/
│   ├── protocol/
│   ├── projections/
│   ├── constraints/
│   ├── traceability/
│   └── provider/
└── integration/
    ├── repl_daemon/
    ├── planning_chain/
    ├── execution_chain/
    └── final_closure/
```
