# Chat 对话框五项体验优化设计

> 版本：v1.0 | 日期：2026-05-26

## 1. 概述

### 1.1 问题清单

| # | 问题 | 影响 |
|---|------|------|
| 1 | 执行事件（tool calls）作为独立消息打断作者流式输出 | 用户阅读作者思考过程被频繁打断 |
| 2 | 审核者消息不是流式输出 | 审核过程无实时反馈，体验割裂 |
| 3 | 作者使用 brainstorming skill 时无法与用户多轮交互 | provider 无法暂停等待用户选择 |
| 4 | 权限确认按钮点击无法生效 | 用户无法授权 provider 执行操作 |
| 5 | Artifact 包含 provider 思考过程前缀 | 产物不干净，需要手动裁剪 |

### 1.2 设计目标

- 作者消息气泡内嵌执行事件，支持折叠展开
- 审核者消息正确流式展示
- 新增 `choice_request` / `choice_response` 协议支持 provider 运行中多轮交互
- 修复权限确认按钮失效 bug
- 通过协议标记让后端精确提取 artifact 内容

## 2. 问题1：消息分组与内嵌折叠

### 2.1 现状分析

当前 `useWorkspaceWs.ts` 的 `handleMessage` 对每个 `execution_event` 和 `permission_request` 都调用 `appendChatEntry` 创建独立 entry。`ChatEntryList` 按 entry 顺序平铺渲染，导致执行事件打断作者消息的连续性。

### 2.2 方案：消息分组渲染

引入"消息组"概念：同一个 `node_id` 下的连续 entries 在渲染层分组为一个视觉单元。

#### 2.2.1 数据模型

不改变 `ChatEntry` 数据结构和 store 逻辑。分组仅在渲染层完成：

```typescript
interface MessageGroup {
  id: string;
  node_id: string;
  role: ChatEntryRole;
  primaryEntry: ChatEntry;          // provider_stream entry
  inlineEvents: ChatEntry[];        // execution_event entries（内嵌）
  interruptEntries: ChatEntry[];    // permission_request / choice_request（独立卡片）
}
```

#### 2.2.2 分组规则

1. 遍历 `chatEntries`，按 `node_id` 连续聚合
2. 同一 `node_id` 下：
   - `provider_stream` → 作为 `primaryEntry`
   - `execution_event` → 收入 `inlineEvents`
   - `permission_request` / `choice_request` → 收入 `interruptEntries`
3. 没有 `node_id` 或 `node_id` 变化时，开始新组
4. `stage_change` / `artifact_update` / `review_verdict` 等系统消息不参与分组，独立渲染

#### 2.2.3 渲染结构

```
┌─ MessageGroup ─────────────────────────────────┐
│ [作者] Claude Code                              │
│                                                 │
│ 正在分析需求，生成 Story Spec...                  │
│                                                 │
│ ┌─ [折叠] Read src/models/user.rs ────────────┐ │
│ └─────────────────────────────────────────────┘ │
│ ┌─ [折叠] Bash: cargo test ───────────────────┐ │
│ └─────────────────────────────────────────────┘ │
│                                                 │
│ 根据代码分析，用户模型需要...                      │
│                                                 │
│ ┌─ [权限请求] Write cadence/specs/... ────────┐ │
│ │ [允许] [拒绝]                                │ │
│ └─────────────────────────────────────────────┘ │
│                                                 │
│ 继续生成中...                                    │
└─────────────────────────────────────────────────┘
```

#### 2.2.4 折叠行为

- 执行事件默认折叠，只显示一行摘要：图标 + 标题（如 "Read src/auth.rs"）
- 点击展开后显示：命令、输出（带 max-height 滚动）
- 权限请求和选择请求始终展开（需要用户交互）

#### 2.2.5 实现要点

| 文件 | 改动 |
|------|------|
| `web/src/components/chat-workspace/ChatEntryList.tsx` | 渲染前调用 `groupEntries()` 分组 |
| 新建 `web/src/components/chat-workspace/MessageGroup.tsx` | 消息组容器组件 |
| 新建 `web/src/components/chat-workspace/InlineEventRow.tsx` | 可折叠的内嵌事件行 |
| `web/src/components/chat-workspace/ChatEntryRenderer.tsx` | 保留，用于渲染 interrupt entries |
| `web/src/components/chat-workspace/entries/ProviderStreamEntry.tsx` | 改为接收 inlineEvents 并穿插渲染 |

#### 2.2.6 流式追加时的分组

流式输出期间，`provider_stream` entry 持续更新 content。新到达的 `execution_event` 追加到当前组的 `inlineEvents`。前端通过 `activeStreamEntryId` 判断当前是否在流式中，流式中的组始终展示最新内容。

## 3. 问题2：审核者消息流式输出

### 3.1 现状分析

后端 reviewer 运行时确实通过 `EngineEvent::StreamChunk { role: "reviewer", ... }` 发送流式内容。前端 `handleStreamChunk` 也正确处理了 reviewer role。

可能的问题点：
1. **fake provider** 在测试时可能一次性输出完整内容而非逐步流式
2. **rebuildChatEntries** 在 `session_state` 恢复时，reviewer 的内容从 `detail.streaming_content` 或 `detail.messages` 重建，此时已经是完整文本，不再有流式效果
3. 前端 `ProviderStreamEntry` 对 reviewer 和 author 使用相同渲染逻辑，但 reviewer 的 node 可能在 `buildChatEntries` 中没有正确生成 `provider_stream` entry

### 3.2 方案

1. **确认 real provider（claude_code / codex）的 reviewer 确实流式输出** — 如果是 fake provider 的问题则不需要修前端
2. **修复 `buildChatEntries`** — 确保 `reviewer_run` 类型的 node 也生成 `provider_stream` entry（当前代码已覆盖，需验证实际运行时 `detail.streaming_content` 是否有值）
3. **MessageGroup 对 reviewer 同样适用** — reviewer 的消息组也支持内嵌折叠

### 3.3 调查步骤

在 Plan 实施时：
1. 启动 dev server，使用 real provider 运行一次带 reviewer 的 workspace
2. 在浏览器 DevTools 中观察 WebSocket 消息，确认是否收到 `stream_chunk` with `role: "reviewer"`
3. 如果收到但前端不流式展示，则是前端 bug；如果没收到，则是后端 provider adapter 问题

## 4. 问题3：多轮交互协议（choice_request / choice_response）

### 4.1 需求

Provider 在运行中（如使用 brainstorming skill）需要暂停执行，向用户提出选择题或开放式问题，等待用户回答后继续。

### 4.2 协议设计

#### 4.2.1 新增 WebSocket 消息类型

**服务端 → 客户端：**

```rust
// workspace_ws_types.rs
WsOutMessage::ChoiceRequest {
    id: String,                    // 唯一请求 ID
    node_id: Option<String>,       // 关联的 timeline node
    prompt: String,                // 问题文本
    options: Vec<ChoiceOption>,    // 选项列表（可为空，表示开放式问题）
    multi_select: bool,            // 是否允许多选
}

struct ChoiceOption {
    label: String,                 // 选项标签
    description: Option<String>,   // 选项描述
}
```

**客户端 → 服务端：**

```rust
// workspace_ws_types.rs
WsInMessage::ChoiceResponse {
    id: String,                         // 对应的请求 ID
    selected_indices: Vec<usize>,       // 选中的选项索引
    text_input: Option<String>,         // 用户自由输入（"Other" 场景）
}
```

#### 4.2.2 Provider 侧接口

```rust
// streaming_provider.rs
pub enum ProviderEvent {
    // ... 现有事件 ...
    ChoiceRequest(ChoiceRequestData),
}

pub struct ChoiceRequestData {
    pub id: String,
    pub prompt: String,
    pub options: Vec<ChoiceOptionData>,
    pub multi_select: bool,
}

pub struct ChoiceOptionData {
    pub label: String,
    pub description: Option<String>,
}

pub enum ProviderCommand {
    // ... 现有命令 ...
    ChoiceResponse {
        id: String,
        selected_indices: Vec<usize>,
        text_input: Option<String>,
    },
}
```

#### 4.2.3 Engine 处理流程

```
Provider 发出 ChoiceRequest
    → Engine 持久化到 node_detail
    → Engine 发送 EngineEvent::ChoiceRequest
    → WS Handler 转发 WsOutMessage::ChoiceRequest 到前端
    → Provider 进入等待状态（监听 command_rx）

用户在前端选择
    → 前端发送 WsInMessage::ChoiceResponse
    → WS Handler 通过 command_tx 发送 ProviderCommand::ChoiceResponse
    → Provider 收到响应，继续执行
```

#### 4.2.4 前端实现

| 组件 | 职责 |
|------|------|
| `ChatEntryType` 新增 `"choice_request"` / `"choice_response"` | 类型定义 |
| 新建 `ChoiceRequestEntry.tsx` | 渲染选择卡片（单选/多选/开放式） |
| 新建 `ChoiceResponseEntry.tsx` | 渲染用户已选择的结果 |
| `ChatEntryRenderer.tsx` | 新增 case 分支 |
| `useWorkspaceWs.ts` | 处理 `choice_request` 消息，新增 `sendChoiceResponse` |
| `ChatWorkspacePage.tsx` | 传递 `onChoiceResponse` 回调 |

#### 4.2.5 ChoiceRequestEntry 渲染

```
┌─ [系统] 选择请求 ──────────────────────────────┐
│                                                 │
│ 请选择你期望的技术方案：                           │
│                                                 │
│ ○ 方案A：使用 Redis 缓存                         │
│   适合高并发场景，需要额外基础设施                   │
│                                                 │
│ ○ 方案B：使用内存缓存                             │
│   简单直接，适合单实例部署                          │
│                                                 │
│ ○ 方案C：使用文件缓存                             │
│   无额外依赖，性能一般                             │
│                                                 │
│ [自定义输入...]                                   │
│                                                 │
│                              [确认选择]           │
└─────────────────────────────────────────────────┘
```

#### 4.2.6 超时处理

- 与 permission_request 类似，设置超时（默认 5 分钟）
- 超时后 provider 收到 timeout 信号，可选择使用默认选项继续或终止

## 5. 问题4：权限确认按钮失效

### 5.1 现状分析

代码路径：
1. 用户点击「允许」→ `handlePermissionResponse(entry, true)`
2. → `requestIdFromEntry(entry)` 从 `entry.metadata.request_id` 提取 ID
3. → `respondPermission(requestId, true, undefined)` 发送 WebSocket 消息
4. → 后端 `WsInMessage::PermissionResponse` → `command_tx.send(ProviderCommand::PermissionResponse)`

可能的失效原因：

**原因A：`current_run` 已结束**
```rust
let command_tx = {
    current_run.lock().await.as_ref().map(|run| run.command_tx.clone())
};
if let Some(command_tx) = command_tx { ... }
```
如果 provider 已经超时或完成，`current_run` 为 `None`，`command_tx` 拿不到，消息被静默丢弃。

**原因B：前端 `requestIdFromEntry` 返回 null**
```typescript
function requestIdFromEntry(entry: ChatEntry) {
  const metadata = entry.metadata;
  const requestId = metadata?.request_id;
  return typeof requestId === "string" ? requestId : null;
}
```
如果 `metadata.request_id` 不存在或不是 string，返回 null，`handlePermissionResponse` 直接 return。

**原因C：entry 已被标记为 resolved**
`PermissionRequestEntry` 的按钮只在 `onRespond` 存在时渲染。但 `ChatEntryRenderer` 始终传递 `onPermissionResponse`，所以按钮应该始终可见。问题可能是点击后 `resolvePermissionRequest` 从 `pendingPermissions` 移除了请求，但 entry 本身没有标记 `resolved`，导致按钮仍然可见但再次点击无效。

### 5.2 修复方案

1. **前端：点击后标记 entry 为 resolved**
   - `respondPermission` 成功发送后，更新对应 entry 的 `resolved` 字段
   - `PermissionRequestEntry` 在 `resolved === true` 时显示结果标签而非按钮

2. **前端：添加 loading 状态**
   - 点击按钮后立即 disable，防止重复点击
   - 显示 loading 指示器直到收到确认

3. **后端：添加响应确认**
   - 当 `command_tx` 为 None 时（provider 已结束），返回 `ProtocolError` 告知前端
   - 前端收到错误后显示"Provider 已结束，无法处理权限请求"

4. **前端：permission timeout 处理**
   - 后端发送 `PermissionTimeout` 事件时，前端标记对应 entry 为 resolved（超时）

### 5.3 实现要点

| 文件 | 改动 |
|------|------|
| `web/src/hooks/useWorkspaceWs.ts` | `respondPermission` 成功后标记 entry resolved |
| `web/src/state/workspace-ws-store.ts` | 新增 `resolvePermissionEntry(id, approved)` action |
| `web/src/components/chat-workspace/entries/PermissionRequestEntry.tsx` | 支持 resolved 状态展示 |
| `src/web/workspace_ws_handler.rs` | `current_run` 为 None 时返回 protocol_error |

## 6. 问题5：Artifact 协议标记与内容裁剪

### 6.1 现状分析

当前 `update_artifact` 直接使用 provider 的 `full_content`（完整输出文本）作为 artifact markdown。Provider 输出中包含思考过程、skill 声明等非结构化前缀，这些不应出现在 artifact 中。

### 6.2 方案：协议标记

#### 6.2.1 标记格式

Provider 在输出 artifact 内容时使用 XML 标记包裹：

```
（provider 思考过程、skill 声明等...）

<artifact>
# Story Spec 标题

## 范围
...

## 验收标准
...
</artifact>
```

#### 6.2.2 后端提取逻辑

```rust
fn extract_artifact_content(full_output: &str) -> String {
    // 优先：检测 <artifact> 标记
    if let Some(start) = full_output.find("<artifact>") {
        let content_start = start + "<artifact>".len();
        if let Some(end) = full_output[content_start..].find("</artifact>") {
            return full_output[content_start..content_start + end].trim().to_string();
        }
        // 只有开始标记没有结束标记：取标记之后的所有内容
        return full_output[content_start..].trim().to_string();
    }

    // Fallback：检测第一个 markdown 一级标题
    if let Some(heading_pos) = full_output.find("\n# ") {
        return full_output[heading_pos..].trim().to_string();
    }
    if full_output.starts_with("# ") {
        return full_output.to_string();
    }

    // 最终 fallback：返回原始内容
    full_output.to_string()
}
```

#### 6.2.3 Prompt 工程

在 provider 的 system prompt 中明确要求：

```
当你生成最终产物（Story Spec / Design Spec / Work Item）时，必须使用 <artifact> 标签包裹产物内容：

<artifact>
你的产物 Markdown 内容
</artifact>

标签外的内容（思考过程、分析、说明）不会出现在最终产物中。
```

#### 6.2.4 流式 Artifact 更新

当前 artifact 在 provider 完成后一次性提取。对于流式场景：
- 后端在 `full_content` 中检测到 `<artifact>` 开始标记后，开始将后续 stream_chunk 同时追加到 artifact
- 检测到 `</artifact>` 结束标记后停止追加
- 这样 artifact pane 可以实时展示生成中的产物内容

#### 6.2.5 实现要点

| 文件 | 改动 |
|------|------|
| `src/product/workspace_engine.rs` | `run_author_provider` 中 artifact 提取使用 `extract_artifact_content` |
| `src/product/workspace_engine.rs` | 流式阶段检测 `<artifact>` 标记，实时发送 `ArtifactUpdate` |
| Provider prompt 模板 | 添加 `<artifact>` 标记使用说明 |

## 8. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 消息分组逻辑在流式追加时可能出现闪烁 | 使用 `useMemo` + 稳定的 group id，避免不必要的重渲染 |
| choice_request 超时后 provider 行为不确定 | 定义明确的超时策略：使用默认选项或终止 |
| artifact 标记在流式中被分割到多个 chunk | 后端维护 buffer，检测完整标记后再切换状态 |
| 权限按钮修复可能影响已有的 permission 流程 | 添加回归测试覆盖正常 approve/deny 路径 |

## 9. 验收标准

1. **问题1**：作者消息气泡内嵌执行事件，默认折叠只显示标题，展开显示详情
2. **问题2**：审核者消息实时流式展示，与作者消息体验一致
3. **问题3**：Provider 可发出选择请求，前端展示选择卡片，用户选择后 provider 继续执行
4. **问题4**：权限确认按钮点击后立即生效，显示 loading → resolved 状态
5. **问题5**：Artifact pane 中只展示 `<artifact>` 标记内的纯净内容，无思考过程前缀
