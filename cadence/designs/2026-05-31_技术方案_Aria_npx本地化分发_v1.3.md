## 1. 目标与背景

把当前 `cadence-aria` 项目打包成一个可通过 `npx` 一键运行的软件，让用户无需安装 Rust 工具链、无需联网编译，即可在本地启动 Aria Web 工作台。

### 1.1 现状（已勘查确认）

- **架构**：Rust 后端二进制 `aria`（edition 2024 / rustc 1.95.0）+ React19/Vite6/TypeScript 前端（`web/`，pnpm 管理）。
- **运行方式**：`aria web --workspace <path> [--host <h>] [--port <p>]` 启动 axum Web 服务（默认 host `127.0.0.1`，未指定端口时绑定 `0`，由内核自动分配），并以前端构建产物作为 fallback 静态资源服务。另有 `task run`、`daemon`、`repl` 等子命令。
- **入口透传**：`src/main.rs` 将 `std::env::args().skip(1)` 整体交给 `run_cli_async`，参数解析集中在 `src/cli.rs`。
- **绑定地址输出契约**：`serve_web` 在 `src/web/app.rs` 中执行 `eprintln!("aria web listening on http://{bound_addr}")`，输出到 stderr。
- **静态资源服务现状**：`src/web/static_assets.rs` 的 `static_dist_service()` 返回 `ServeDir<ServeFile>`，由 `src/web/app.rs:221` 通过 `.fallback_service(...)` 挂载。`ServeDir::new` 是运行时构造，`env!("CARGO_MANIFEST_DIR")` 是编译期取路径字符串——因此当前实现即使 `web/dist` 缺失也能 `cargo check` 通过，仅运行时白屏，属静默失败风险点。
- **前端产物体积**：`web/dist/assets/index-*.js` 约 **472KB**（未压缩），加 CSS 约 27KB、二级 JS 约 15KB。
- **`web/dist` 的 git 状态**：被 `web/.gitignore:2`（`dist/`）忽略，**未纳入版本控制**。任何干净 checkout 后该目录不存在，必须由前端构建生成。
- **release profile 现状**：`Cargo.toml` 仅有 `[profile.dev]` 系列优化，**无 `[profile.release]`**。

### 1.2 打包成 npx 的关键障碍（已定位）

1. **前端资源路径硬编码**：`src/web/static_assets.rs:6` 使用 `env!("CARGO_MANIFEST_DIR").join("web/dist")`，指向编译期源码目录。分发后该路径不存在，是分发的核心硬伤。该引用点唯一（另有两处 `CARGO_MANIFEST_DIR` 位于 `codex_provider.rs:839`、`claude_code_provider.rs:966`，均为测试 fixture，不影响运行时）。
2. **aria 是原生二进制，npx 属 JS 生态**：需要 JS launcher + 二进制分发方案。
3. **依赖外部 codex/claude CLI**：真实 provider 通过 spawn 外部进程运行，默认调用 PATH 中的 `claude`、`codex`（`src/cross_cutting/adapter_compatibility.rs:74-114`）。
4. **git 依赖**：`codex-protocol` 等来自 GitHub tag，冷编译较慢，不适合让终端用户本地编译。

## 2. 决策摘要（已与用户确认）

| 决策项 | 选定方案 |
|--------|---------|
| 分发策略 | 预编译二进制 + 平台子包（esbuild/swc/biome 同款的 optionalDependencies 模式） |
| 平台矩阵 | 3 目标：`linux-x64`、`darwin-x64`、`darwin-arm64`（`linux-arm64` 作为后续增量） |
| 前端资源 | 编译期嵌入二进制（`rust-embed`，统一嵌入模式），平台子包只含单个二进制文件 |
| 嵌入覆盖 | 运行时统一从嵌入资源读取；仅当显式设置 `ARIA_WEB_DIST` 时改读该磁盘目录（开发/e2e 逃生口） |
| 外部 CLI | 用户自行安装 codex/claude；`aria web` 启动时检测 PATH 并友好提示，但不阻断启动 |
| 默认行为 | `npx @cadence-aria/cli` 无参时启动 web 并自动打开默认浏览器 |
| 端口策略 | launcher 主动探测空闲端口，以 `web --port <p>` 传入，直接用该端口开浏览器（无 stderr 反解竞态） |
| 包名 | 主包 `@cadence-aria/cli`，bin 别名 `aria`；平台子包 `@cadence-aria/cli-<platform>` |
| 发布自动化 | Git `v*` tag 触发 GitHub Actions 全自动多平台构建并发布到 npm |
| CLI 面 | 完整透传所有子命令（`web`/`task run`/`daemon`/`repl`），无参注入默认 `web` |
| 交付范围 | 一次性完整交付（Rust 改造 + launcher + 本地 npm pack 验证 + 多平台 release workflow） |

## 3. npm 包拓扑

采用业界验证的「主包 + 平台子包 + optionalDependencies」结构（esbuild、@swc/core、@biomejs/biome 同款）：

```
@cadence-aria/cli                      ← 主包（用户 npx 的目标）
├── package.json                       ← bin: { "aria": "bin/aria.js" }
│                                         optionalDependencies 列出全部平台子包
├── bin/aria.js                        ← JS launcher（纯 JS，无原生代码）
└── README.md                          ← 体积极小，无二进制

@cadence-aria/cli-linux-x64            ← 平台子包（含预编译二进制）
├── package.json                       ← os:["linux"], cpu:["x64"]
└── bin/aria                           ← release 二进制（已嵌入前端 dist）

@cadence-aria/cli-darwin-x64           ← os:["darwin"], cpu:["x64"]
└── bin/aria

@cadence-aria/cli-darwin-arm64         ← os:["darwin"], cpu:["arm64"]
└── bin/aria
```

### 3.1 工作原理

- npm 安装主包时会尝试安装所有 `optionalDependencies`；每个平台子包通过 `os`/`cpu` 字段声明适用平台，npm 仅真正安装与当前机器匹配的那一个，其余静默跳过（不报错）。
- launcher 通过 `require.resolve("@cadence-aria/cli-<platform>/bin/aria")` 定位已装好的二进制。
- 因为前端 `web/dist` 已在编译期嵌入 `aria` 二进制（见第 4 节），平台子包内只有一个二进制文件，无额外静态资源目录。

### 3.2 版本一致性

主包与所有平台子包版本号严格一致；主包 `optionalDependencies` 中各平台子包固定为「等于主包版本」（exact，非 range），由 release 流程统一注入，避免版本漂移。

## 4. Rust 侧改造

### 4.1 改造 1：前端资源嵌入二进制（核心）

将 `src/web/static_assets.rs` 从磁盘读取改为编译期嵌入，**采用「统一嵌入」模式**（debug 与 release 行为一致），避免 rust-embed 默认双模带来的「开发态读磁盘、发布态读嵌入」分裂：

- 引入 `rust-embed`，对 `web/dist` 用 `#[derive(RustEmbed)] #[folder = "web/dist"]`，并**启用 `debug-embed` feature**，使 debug 构建同样嵌入（保证 e2e/本地 `cargo run` 验证的就是发布形态）。
- **运行时统一从嵌入资源读取**；SPA 路由未命中时 fallback 到嵌入的 `index.html`；资源响应用 `mime_guess` 按扩展名设置 `Content-Type`（`mime_guess` 已在依赖中）。
- **唯一逃生口**：环境变量 `ARIA_WEB_DIST` 若指向一个存在的磁盘目录，则覆盖嵌入资源、改从该目录读取（供前端热更新调试、playwright e2e 使用）。未设置或目录不存在时一律走嵌入资源。逃生口边界清晰、单一开关，不与 rust-embed 内建 debug 模式叠加。
- **接口形态变化**：`static_dist_service()` 当前返回 `ServeDir<ServeFile>` 并经 `.fallback_service` 挂载。改造后改为「返回一个统一的 fallback handler/Service」，内部按 `ARIA_WEB_DIST` 是否设置二选一（嵌入 service 或 `ServeDir`）。`src/web/app.rs:221` 调用点相应适配（仍是单一挂载点，改动局部）。对外行为契约不变：精确命中返回对应文件与正确 Content-Type、未命中返回 `index.html`。
- **e2e 适配**：`web/e2e/start-api.mjs` 用 `cargo run` 起后端、cwd 在仓库根。改造后为其注入 `ARIA_WEB_DIST=<repo>/web/dist` 环境变量，使 e2e 与现有 dev 工作流不破（前端改动无需重编 Rust）。

### 4.2 改造 1 配套：构建顺序契约与 build.rs 硬校验

因 `web/dist` 被 `web/.gitignore` 忽略、不在版本控制中，**任何干净 checkout（CI、新克隆、切分支）下该目录都不存在**。必须确立显式构建契约：

- **构建顺序契约**：`aria` 二进制（含嵌入）构建前，`web/dist` 必须已由 `pnpm -C web install && pnpm -C web build` 生成。本地开发、CI release 均遵循。
- **build.rs 硬校验**：新增 `build.rs`，编译期检查 `web/dist/index.html` 存在且 `web/dist` 非空；缺失/为空则 `panic!` 终止编译，错误信息明确指示「请先运行 `pnpm -C web build`」。**这是必需的兜底**——rust-embed 对不存在/空目录在部分版本下会静默嵌入空集合，产出「能启动但全站 404 白屏」的二进制，硬校验杜绝该静默失败。
- build.rs 同时声明 `cargo:rerun-if-changed=web/dist`，使前端产物变化触发重嵌入。

### 4.3 改造 2：启动检测提示

`aria web` 启动时探测 PATH 中是否存在 `claude`、`codex`（program 名取自 `adapter_compatibility` 的 `provider_command`/probe，避免硬编码漂移）：

- 缺失任一命令：打印友好的中文提示（说明该命令支撑哪些 provider 功能、给出安装指引链接占位），但**不阻断**启动——工作台 UI 本身可用，FakeProvider 演示路径不依赖外部 CLI。
- 两者齐备：可选地打印一行已检测到的提示（或静默）。
- 探测逻辑独立成可单测的函数（输入：PATH 查找结果；输出：提示文案与是否缺失），不在 `serve_web` 内联，便于 TDD。

### 4.4 改造 3：默认行为支撑（Rust 侧最小改动）

默认「启动 web + 开浏览器」主要由 JS launcher 实现（见第 5 节）。Rust 侧只需保证绑定地址输出契约稳定：`aria web` 启动成功后向 stderr 输出 `listening on http://<host>:<port>` 行。当前 `src/web/app.rs` 已具备该输出。本节仅作契约确认，并为该行格式补充回归测试以防破坏 launcher 兜底解析。

### 4.5 改造 4：release profile 体积优化

新增 `[profile.release]`，控制嵌入前端后的二进制体积：

- `strip = true`（去符号）、`lto = true`（链接期优化）、`opt-level = "z"` 或 `"s"`（体积优先，二选一在实现时实测取较优）、`codegen-units = 1`、`panic = "abort"`（若与现有 panic 处理兼容，需验证不破坏测试）。
- 目标：将含 git 重依赖 + 嵌入前端（约 0.5MB 解压后）的 release 二进制收敛到可接受的 npm 单包体积。
- **体积验证作为发布前 gate**（见第 7.3 节），不是「后续按需」。

## 5. JS Launcher（`bin/aria.js`）

纯 JS（CommonJS 或 ESM，依主包 `type` 字段定），无原生代码。职责：

1. **定位二进制**：
   - 由 `process.platform` + `process.arch` 映射子包名：`linux/x64 → cli-linux-x64`、`darwin/x64 → cli-darwin-x64`、`darwin/arm64 → cli-darwin-arm64`。
   - `require.resolve("@cadence-aria/cli-<platform>/bin/aria")` 取二进制路径。
   - 未匹配/未安装：报清晰错误，列出当前 `platform-arch` 与已支持矩阵，提示该平台暂无预编译包（可走源码构建）。

2. **参数透传与默认子命令**：
   - 无参（仅 `aria`）：注入默认 `web` 子命令（workspace 默认取当前工作目录，与 Rust 侧 `parse_workspace` 缺省一致）。
   - 有参：原样透传给二进制（`task run`、`daemon`、`repl`、`web --port 3000` 等）。
   - launcher 自身消费的标志（如 `--no-open`）从透传参数中剥离，不传给二进制。

3. **端口与自动开浏览器**（仅默认 web 模式）：
   - launcher 用 node（如 `net.createServer` 探测）选取一个空闲 TCP 端口 `<p>`。
   - 以 `web --port <p>`（host `127.0.0.1`）传给二进制，**端口由 launcher 掌握**，无需从 stderr 反解内核随机端口，消除竞态。
   - 二进制就绪后，launcher 直接用 `http://127.0.0.1:<p>` 调用系统命令打开浏览器：macOS `open`、Linux `xdg-open`（不可用时降级为仅打印地址）。
   - **就绪判定**：可对 `/api/health` 轮询（e2e 已用该端点）或读到 stderr 的 `listening on` 行；二者皆可，端口已知，仅用于「何时开浏览器」的时机，无端口解析依赖。
   - `--no-open` 关闭自动打开，仅打印地址（适配远程/无头环境）。
   - 用户显式传 `web --port <自定义>` 或其它子命令时，尊重用户参数、不再自选端口、不触发开浏览器（非默认 web 模式）。

4. **进程与信号转发**：
   - 用 `child_process.spawn` 拉起二进制，stdin/stdout/stderr `inherit` 透传给终端（stderr 不被 pipe 截留，用户能看到完整日志）。
   - 转发 `SIGINT`/`SIGTERM` 给子进程，保证 Ctrl-C 干净退出。
   - 子进程退出码透传为 launcher 退出码。

## 6. 发布自动化（GitHub Actions，Tag 触发）

`v*` tag push 触发。工作流分为「多平台构建」与「归集发布」两阶段：构建 job 按平台并行（fan-out），各自上传产物为 workflow artifact；归集 job 等全部构建完成后（needs 依赖）统一组包、发布 npm、创建 GitHub Release。这样 npm publish 与 Release 创建只在所有平台都构建成功后发生一次，避免半成品发布。

### 6.1 多平台构建（矩阵，并行）

- `ubuntu-latest` → `x86_64-unknown-linux-gnu`；
- `macos-13`（Intel）→ `x86_64-apple-darwin`；
- `macos-14`（Apple Silicon）→ `aarch64-apple-darwin`。
- 均为**原生编译**（在对应 OS/arch runner 上直接 `cargo build --release`），规避交叉编译 git 重型依赖的坑。本地开发机为 WSL（linux-x64），无法产出原生 macOS 二进制，故 macOS 目标必须依赖 GitHub 托管 runner——这是选用 Actions 的不可替代理由。
- **`macos-13` 退役风险**：GitHub 正逐步淘汰 Intel macOS runner，`macos-13` 未来可能下线导致 darwin-x64 构建中断。届时备选：用 arm64 runner 交叉编译 `x86_64-apple-darwin`（但会触碰本方案刻意规避的 git 重依赖交叉编译坑，需额外验证）。本期沿用 `macos-13`，并在 workflow 注释标注该风险点，便于后续替换。
- 每个构建 job 前置（**强制顺序**）：`pnpm -C web install` + `pnpm -C web build` 生成 `web/dist`，再 `cargo build --release`（前端经 `rust-embed` 嵌入二进制）。build.rs 硬校验作为兜底，若顺序错误则编译期失败。
- 每个 job 末尾：将产物二进制**用 `tar`（如 `aria-<platform>.tar.gz`）打包后**再 `actions/upload-artifact` 上传。**这是必须的**——`actions/upload-artifact` 内部以 zip 存储、**不保留 Unix 可执行位**，若直接上传裸二进制，归集 job 取回的 `aria` 会丢失 `+x`，launcher spawn 时 `permission denied`。tar 保留文件权限，规避该 GitHub Actions 已知陷阱。

### 6.2 归集与组包（单 job，needs 全部构建）

- `actions/download-artifact` 取回三平台 tar 包并解包（`tar` 解包恢复执行位）。
- 解包后对每个二进制显式 `chmod +x`（双保险，不依赖 tar 元数据完好）。
- 脚本将各 release 二进制放入对应平台子包目录的 `bin/aria`。
- 由模板生成各平台子包 `package.json`（注入 `name`、`version`、`os`、`cpu`、`files`）。**子包不声明 `bin`**——`bin` 仅由主包声明并指向 launcher（`bin/aria.js`）；子包只是携带二进制的容器，launcher 用 `require.resolve("@cadence-aria/cli-<platform>/bin/aria")` 定位其中的二进制。若子包也声明 `bin: aria`，会与主包 bin 名冲突，与 6.3「执行位保全链」一致。
- 校验：二进制**具备可执行位**（`test -x` 断言，防止权限丢失流到发布）、架构正确（`file` 命令断言）、**体积在阈值内**（见 7.3）。

### 6.3 发布 npm

- 先 `npm publish` 全部平台子包，再 `npm publish` 主包（确保主包安装时子包已可解析）。
- 主包 `optionalDependencies` 各子包版本注入为当前发布版本（exact）。
- 使用 `npm publish --access public`（scoped 包默认 private，需显式 public）。
- **执行位保全链**：归集 job 中 `bin/aria` 已 `chmod +x`（见 6.2）；`npm publish` 的 tarball 保留文件 mode，用户 `npm install`/`npx` 解包后执行位仍在。launcher 通过 `require.resolve` 定位并 spawn 子包二进制（子包不声明 `bin`，故不依赖 npm 的 bin chmod 机制），因此该执行位必须在归集阶段就正确——6.1/6.2 的 tar + chmod 即为此保障。
- **publish 非原子性注意**：多包发布无事务，若中途失败（如 3 个子包发了 2 个），npm 版本号**不可重发覆盖**。处理策略：**不要**尝试覆盖同版本，改用新 patch 版本（如 `0.1.1`）整体重发。workflow 应在任一 publish 失败时整体失败并清晰报错，避免遗留半个版本。

### 6.4 创建 GitHub Release 并附带二进制（v1.2 新增）

- npm 发布成功后，由同一归集 job 创建对应 `v<version>` 的 GitHub Release（用 `softprops/action-gh-release` 或 `gh release create`）。
- **附件**：上传三平台二进制（建议打包为 `aria-<platform>.tar.gz`，含可执行文件 + LICENSE + 简短 README），并附 **SHA256 校验和文件**（`SHA256SUMS`），便于用户校验完整性。
- Release notes：可由 tag message 或自动生成的 changelog 填充（实现时择一，不阻塞）。
- 定位：GitHub Release 是 npm 之外的**可追溯下载备份**，面向内网用户、不便用 npm 的环境、或需直接取二进制的场景。它**不是** launcher 的运行时下载源（本方案不走 postinstall 下载，见 1.2 决策），仅作分发冗余与审计留痕。

### 6.5 工作流权限与前置条件（执行发布时由维护者提供）

- **权限**：创建 GitHub Release 需 job 具备 `permissions: contents: write`（用默认 `GITHUB_TOKEN` 即可，无需额外 PAT）。
- npm 上 `@cadence-aria` 组织/scope 已创建。
- CI 配置 `NPM_TOKEN` secret，具备 `@cadence-aria` scope 的发布权限。
- 版本号管理策略：tag 即版本来源（如 `v0.1.0` → 包版本 `0.1.0`），由 workflow 从 tag 解析并写入各 `package.json` 与 Release 名称。

## 7. 测试策略（遵循 TDD）

按项目规则先写测试再写实现，新功能须有对应测试。

### 7.1 Rust 单测

- `static_assets` 嵌入资源：精确命中已知文件返回正确内容与 Content-Type；未命中路径 fallback 到 `index.html`；`ARIA_WEB_DIST` 指向存在目录时改从磁盘读取并覆盖嵌入；`ARIA_WEB_DIST` 未设置/目录不存在时走嵌入。
- 启动检测：PATH 中存在/缺失 `claude`、`codex` 时，检测函数返回正确的「缺失列表 + 提示文案」；验证缺失时不阻断（函数返回提示而非错误）。
- 绑定地址输出行格式回归测试（防止 launcher 解析兜底契约被破坏）。
- 验证命令遵循 `cadence/project-rules/build-test-commands.md`：`cargo fmt --check`、`cargo clippy --all-targets --all-features --locked -- -D warnings`、`cargo check --locked`、`cargo test --locked`；禁止 `-j 1`。
- 注意：因 build.rs 硬校验要求 `web/dist` 存在，本地跑 Rust 测试前需已构建前端（或在测试环境保证 `web/dist` 存在）。该前置写入开发文档。

### 7.2 Launcher 单测

- 平台映射：三平台 arch 组合映射到正确子包名；未支持平台返回清晰错误。
- 参数透传：无参注入 `web`；有参原样透传；`--no-open` 被剥离不传给二进制；显式 `web --port` 时不自选端口。
- 空闲端口选取：返回可用端口；并发选取不撞已占用端口。
- 开浏览器决策：默认 web 模式触发、显式子命令不触发、`--no-open` 关闭。
- 用 node 内置测试（`node:test`）+ 子进程桩（fake 二进制脚本）验证，无需真实 aria 二进制。

### 7.3 本地集成验证（发布前冒烟基线）

- `pnpm -C web build` → `cargo build --release` → 组当前平台（linux-x64）子包 → `npm pack` 主包与子包。
- 在干净临时目录 `npx <主包 tgz>`：验证二进制被定位、web 启动、launcher 选端口、（有桌面环境时）浏览器打开工作台。
- `npx <主包> --no-open`：仅打印地址、不开浏览器。
- `npx <主包> web --port <固定端口>`：参数透传生效、不自选端口、不开浏览器。
- **执行位验证**：`npm pack` 子包后解开 tarball，断言 `bin/aria` 仍具 `+x`（验证 publish tarball 保权限链；CI 路径上的 upload-artifact 丢权限问题由 6.1/6.2 的 tar+chmod 覆盖，超出本地冒烟范围，靠 6.2 的 `test -x` gate 拦截）。
- **体积 gate**：记录 release 二进制与组包后 npm tarball 体积，设定阈值（实现时据实测定，初步建议二进制 ≤ 80MB、tarball ≤ 30MB；超阈则在发布前显式确认）。

### 7.4 现有 e2e 兼容

- 确认前端嵌入改造后，`web/e2e/` 下 playwright 用例仍绿；e2e 通过 `start-api.mjs` 注入 `ARIA_WEB_DIST=<repo>/web/dist` 指向源码树 dist，保持现有 dev/e2e 工作流不破。

## 8. 影响范围与风险

- **改动面**：Rust 侧集中在 `static_assets.rs`、`src/web/app.rs:221` 挂载点适配、新增启动检测模块、新增 `build.rs`、`Cargo.toml`（新增 `rust-embed` 依赖 + `[profile.release]`）；`web/e2e/start-api.mjs` 注入环境变量；新增 npm 包目录（主包 + 平台子包模板）；新增 release workflow。`src/cli.rs`、`src/main.rs` 基本不动。
- **二进制体积**：嵌入前端 dist（JS 约 472KB 等）+ git 重依赖，release 二进制预计数十 MB；已通过 `[profile.release]` 优化与体积 gate 管控（第 4.5、7.3 节）。
- **构建耦合**：前端必须先于二进制构建，且 `web/dist` 不在版本控制中——已由构建顺序契约 + build.rs 硬校验兜底（第 4.2 节）。
- **平台局限**：本期不含 `linux-arm64` 与 Windows 原生；非支持平台 launcher 给出清晰提示，不静默失败。
- **外部依赖**：codex/claude CLI 由用户自装；缺失仅影响真实 provider 执行，工作台与 Fake 演示不受影响。
- **安全**：Web 默认绑定 `127.0.0.1`（仅本机），无对外暴露；不引入鉴权变更。launcher 调用系统 `open`/`xdg-open` 仅传入本地 `http://127.0.0.1:<port>` 地址，端口为自选整数、不拼接用户可控内容。
