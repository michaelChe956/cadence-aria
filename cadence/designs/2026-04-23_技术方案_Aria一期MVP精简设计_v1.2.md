## 1. 目标与范围

### 1.1 一期 MVP 定位

- **形态**：REPL client + 本地 daemon runtime
- **范围**：单用户、本机、单仓库、多任务并行（通过 worktree 隔离）
- **Provider 接入基线**：BYO 本地 CLI + `spawn + CLI`（用户本机安装 Claude Code / Codex；Claude Code 作为 orchestrator，Codex 作为 executor/reviewer）
- **隔离策略**：每任务独立 worktree
- **集成策略**：并行执行、串行集成（一期默认 FIFO）
- **人工介入**：保守策略，关键节点挂 approval gate
- **Daemon 生命周期**：daemon 常驻，REPL 只是客户端，可 attach/detach

### 1.2 文档定位与真相源边界

本文件的定位不是“重写上游协议”，而是：

1. 在**不改变上游协议真相源**的前提下，定义 Aria 一期如何实现。
2. 明确哪些能力一期完整实现、哪些简化实现、哪些延期到二期。
3. 明确为了降低工程复杂度，一期如何把多个协议节点折叠为较少的**实现单元**。
4. 明确 `v1.2 + 实现总契约` 这两份文档如何共同组成一期的实现真相源。

本文件**不替代**上游文档集中以下内容：

- 节点 ID、节点类型、上下游路由
- 横切能力 ID 与能力边界
- 产物最小字段与校验规则
- 全局协议章节与统一枚举值

一期研发落地时，真相源分三层：

| 层级 | 真相源 | 负责内容 |
|------|--------|----------|
| canonical 协议层 | 上游协议文档集 | 节点 ID、节点路由、最小字段、最小校验、统一枚举 |
| phase1 implementation profile | 本文件 + `2026-04-23_技术方案_Aria一期实现总契约_v1.0.md` | 一期必须实现的 additive fields、projection、wire protocol、OpenSpec 生命周期、traceability 规则 |
| runtime 落盘层 | daemon 实际写入的 artifact / projection / checkpoint / provider run record | 实际运行时审计、恢复、回放 |

**冲突裁定规则：**

1. 若问题是 canonical 字段名、最小必填字段、节点 ID、节点路由、统一枚举，以上游文档集为准。
2. 若问题是一期必须如何承载补充字段、如何编译 projection、REPL <-> daemon wire protocol 如何定义、OpenSpec 约束如何编译和失效、traceability 如何落盘，以 `v1.2 + 实现总契约` 为准。
3. 一期允许在不删除 canonical 最小字段的前提下，增加 additive fields 或 projection；这属于 implementation profile，不属于协议漂移。
4. 开发人员不得跳过 implementation profile，自行决定“补充字段写进 canonical artifact、projection 还是 runtime record”。

### 1.3 一期不做的事（不代表删除协议定义）

| 项目 | 一期策略 | 说明 |
|------|---------|------|
| Risk Registry 完整生命周期 | 不完整实现 | 保留 `riskRegistry` 结构与 `riskId` 兼容入口，二期补完跨产物生命周期 |
| 各类循环独立阈值调优 | 不单独调优 | 保留命名计数器与协议默认阈值；实现代码可复用同一套 counter/threshold 机制，但不得用单一阈值覆盖不同环路 |
| Event Log compaction | 不启用后台 compaction | 保留 append-only、checksum、恢复机制；compaction 二期启用 |
| Gate TTL 自动升级 | 不启用超时扫描器 | 保留 TTL 字段和协议定义，一期默认 `null` |
| N26 自动补丁派生 | 不自动执行 | 保留协议分支与节点定义，一期默认 gate / manual intervention |
| TUI / GUI 目标态 | 不实现 | 保留 daemon/client 分层，二期再扩展前端形态 |

### 1.4 关键假设（显式声明）

1. **一期的主接入路径是 `spawn + CLI`**
   这是一期最小可落地闭环的基线方案。SDK / app-server 是明确保留的二期升级路径，不视为不可行。
2. **daemon 常驻，REPL 只是客户端**
   REPL 启动时 attach 到已有 daemon 或隐式启动新 daemon；REPL 退出时仅断开连接，daemon 继续运行；任务在后台持续执行。
3. **worktree 隔离优于 branch 隔离**
   并发任务需要真正的文件系统隔离；只切 branch 无法满足并行编码与回归验证的需要。
4. **对话分叉与代码分叉分开处理**
   会话管理靠 session/thread（`ProjectSession`、`Phase`），代码隔离靠 worktree（`WorktreeLease`）。两者是独立的生命周期：一个 session 可以对应多个 worktree，一个 worktree 的回收不影响 session 状态。
5. **OpenSpec 在一期是强约束、非可选**
   每个 EpicTask 必须绑定 `changeId`，并在进入规划/执行节点前拥有可编译的 OpenSpec 约束 bundle；禁止静默降级为“只靠聊天历史推进”。

### 1.5 可扩展性保障

**前端形态扩展保障**

核心原则：**REPL 只负责交互，daemon 是运行时真相源**。无论前端形态是 REPL、TUI、GUI 还是 Web，它们都只是 daemon 的客户端，通过统一接口通信。

一期 REPL 与 daemon 通过 JSON over Unix socket 通信，Windows 环境使用 named pipe。二期做桌面端时，只替换客户端，不改变 daemon 核心状态机。

**Provider 接入扩展保障**

一期通过 Provider Adapter 把 daemon 与底层执行器解耦：

```text
daemon -> ProviderAdapter -> [CliAdapter | SdkAdapter | AppServerAdapter | ...]
```

一期只落地 `CliAdapter`。二期新增 `SdkAdapter` 或 `AppServerAdapter` 时，不应要求 daemon 重写协议层。

一期 Claude Code / Codex 接入方式是用户本机 BYO CLI：用户负责安装、登录和授权；Aria 不内置 provider、不托管远程模型、不自动下载安装 CLI。daemon 只负责探测 CLI 能力、选择 compatibility matrix、受控 spawn 和记录 `ProviderRunRecord`。

| `final_review` 结果 | 一期路由 | 说明 |
|---------------------|---------|------|
| `overallDecision = pass` 且 `followupNeeded = false` | `N27` | 正常进入最终总结 |
| `overallDecision = followup` 且补丁边界清晰、`patch_round_counter` 未超过协议阈值 | `X01 approval_gate_open` | gate 中展示补丁范围；用户显式确认后，才允许通过命令触发 `N26`，daemon 不自动派生 |

| `overallDecision = followup` 但 `patch_round_counter` 超过协议阈值 | `X08 manual_intervention_hold` | reason 为 `patch_round_limit_exceeded` |
| `overallDecision = fail` | `X08 manual_intervention_hold` | reason 为 `final_review_failed` |

该规则保证：

- `N26` 协议定义没有被删除。
- 一期不出现“final review 后静默自动派生补丁任务”的行为。
- 是否进入 `N26` 必须由用户显式动作触发，便于审计和避免无限补丁循环。

## 3. 产物规范与 Schema 兼容策略

### 3.1 基本原则

一期允许对**代码组织方式**做精简，但不允许重写上游产物 schema。

具体来说：

- **16 份上游产物规范文档继续是规范真相源**
- 一期实现中，代码可以按少量 schema 族来组织校验器与序列化器
- 本文件中的产物摘要只做索引，不替代原始产物规范

### 3.2 一期实现视角下的 Schema 分组

| 分组 | 涵盖产物 | 说明 |
|------|---------|------|
| **Markdown 用户文档** | `intake_brief`, `spec`, `design`, `plan`, `final_summary` | 用户可读文档 |
| **JSON 决策/运行记录** | `clarification_record`, `spec_gate_decision`, `design_review`, `design_revision_record`, `readiness_check`, `coding_report`, `testing_report`, `code_review_report`, `integration_report`, `final_review` | 机器消费的结构化记录 |
| **JSON 交接包** | `dispatch_package` | 交接包，不应改成 Markdown |
| **JSON 运行快照** | `runtime_snapshot` | Aria 内部状态快照 |

#### 3.2.1 一期 Phase1 Profile 承载策略

一期为了做到“canonical 不漂移，但研发可直接实现”，额外引入两类实现承载：

1. **Projection**：把 Markdown canonical artifact 编译成机器可消费的结构化投影。
2. **Reserved extension namespace**：对 JSON artifact 允许增加 `_aria` 顶层扩展对象，承载一期 routing / traceability / constraint-check 信息。

承载规则如下：

| 产物族 | 机器必需的补充信息 | 一期承载方式 | 直接消费者 |
|--------|-------------------|--------------|------------|
| `spec` | story / requirement / success criteria / open item 的稳定 ID 与结构化索引 | `SpecProjection` | `N06/N07/N11/N25`、dashboard projection |
| `design` | design decision、shared component / module、data entity、API entry、risk ref | `DesignProjection` | `N08/N10/N11/N16/N18/N25`、dashboard projection |
| `plan` | work package 的 `execution_mode`、`human_required_reason`、`traceability_refs`、`acceptance_targets` | `PlanProjection` | `N12/N13/N16-N20`、dashboard projection |
| `dispatch_package` | worktask routing、write scope、traceability refs、constraint check refs | `dispatch_package._aria` | `N13/N16-N20/N23/N25` |
| `coding/testing/review/integration/final review` | traceability refs、constraint check refs、provider run refs | 各 JSON report 的 `_aria` | daemon、traceability、dashboard projection |
| `runtime_snapshot` | canonical 字段之外的恢复和回放元数据 | `nodeSpecificFields` + runtime sidecar | recovery、event replay |

#### 3.2.2 一期 validator 分层

一期实现必须有两层 validator：

1. **canonical validator**：只校验上游文档定义的最小字段与最小结构。
2. **phase1 profile validator**：校验 projection / `_aria` 扩展 / traceability completeness。

只有两层 validator 通过，且 OpenSpec 覆盖校验通过，节点才允许正式收口。

### 3.4 Runtime Snapshot 的兼容性要求

`runtime_snapshot` 在一期中统一定义为 **daemon 在每个协议节点完成后写入的 checkpoint envelope**。

因此需要区分两层概念：

- **节点业务产物**：节点对外提供给后继节点消费的主产物，例如 `spec`、`design_review`、`integration_report`
- **runtime snapshot**：daemon 为恢复、审计、路由和回流一致性维护的运行时快照

节点总目录中的“产物类型”以业务产物为主，不代表只有 Aria 内部节点才能产生 runtime snapshot。无论协议节点是 Aria 内部、Agent 业务还是混合节点，只要节点完成并发生状态迁移，daemon 都必须写入 checkpoint 和对应 runtime snapshot。

一期统一沿用上游全局协议中的 canonical 字段：

- `snapshotId`
- `sessionId`
- `taskId`
- `nodeId`
- `phase`
- `timestamp`
- `effectivePolicy`
- `artifactRefs`
- `providerRunRefs`
- `worktreeRef`
- `reworkCounter`
- `riskRegistry`
- `loopCounters`
- `supersededArtifactRefs`
- `nodeSpecificFields`

**一期兼容策略：**

| 字段/概念 | 一期要求 |
|----------|---------|
| `reworkCounter` | 保留，不改名为其他字段 |
| `loopCounters` | 保留对象结构；一期可复用同一套计数机制，但默认阈值必须按协议分别配置 |
| `riskRegistry` | 保留并可最小化实现；至少支持落盘、恢复、riskId 引用占位 |
| `supersededArtifactRefs` | 必须保留；回流一致性依赖该字段，不能删除 |
| `nodeSpecificFields` | 继续按节点扩展 |

### 3.5 允许的实现期补充信息

一期允许在不破坏 canonical schema 的前提下增加**补充信息**，但这些补充信息不能替代最小字段。

| 补充信息 | 允许方式 | 不允许的方式 |
|---------|---------|-------------|
| Markdown artifact 的机器索引 | 编译到 projection | 直接假设 Markdown 原文可稳定机读 |
| JSON artifact 的 routing / traceability / coverage 元数据 | 写入 `_aria` 命名空间 | 把 `_aria` 字段混入 canonical 最小字段清单 |
| `spec` 的 agent 可执行性评估 | 作为附加章节，且同步进入 `SpecProjection` | 替代 `functional_requirements` / `open_items` |
| `design` 的 `architecture_overview` / `api_design` 表达 | 作为展示性标题映射到 `architecture_summary` / `api_contracts`，并同步进入 `DesignProjection` | 改写 canonical 字段名 |
| `plan` 的 `timeline` / `milestones` / `execution_mode` / `human_required_reason` | 文档可附加叙述，但机器消费以 `PlanProjection` 为准 | 替代 `work_packages` / `acceptance_checks` |
| `dispatch_package` 的 routing 元数据 | 仅写入 `dispatch_package._aria` | 破坏 `worktasks/dependencies/parallel_groups/acceptance_targets` canonical 结构 |
| `final_summary` 的 `recommendations` | 作为 `next_steps` 的扩展说明 | 替代 `next_steps` canonical 字段 |

#### 3.5.1 `_aria` 扩展命名空间规则

所有 JSON artifact 的实现扩展字段统一放在 `_aria` 顶层对象中，避免与 canonical 字段混淆。

`_aria` 的一期通用字段：

- `profile_version`
- `constraint_check_ref`
- `traceability_refs`
- `provider_run_refs`
- `projection_refs`

其中：

- `dispatch_package._aria` 额外必须包含 `worktask_routing`
- `coding_report/testing_report/code_review_report/integration_report/final_review` 至少必须包含 `traceability_refs`
- `final_review._aria` 额外必须包含 `coverage_summary`

#### 3.5.2 Projection 规则

Markdown artifact 不追加隐藏 front matter。
daemon 在 artifact validate 通过后，立即编译 projection 并单独落盘：

- `spec` -> `SpecProjection`
- `design` -> `DesignProjection`
- `plan` -> `PlanProjection`

后续任何 routing、coverage、dashboard、traceability 逻辑，一律消费 projection，不回头直接解析 Markdown 原文。

### 3.6 产物格式规则

- **Markdown 用户文档**：`intake_brief`, `spec`, `design`, `plan`, `final_summary`
- **JSON 决策/运行记录**：`clarification_record`, `spec_gate_decision`, `design_review`, `design_revision_record`, `readiness_check`, `coding_report`, `testing_report`, `code_review_report`, `integration_report`, `final_review`
- **JSON 交接包**：`dispatch_package`
- **JSON 运行快照**：`runtime_snapshot`

## 4. 横切能力的一期实现范围

### 4.1 基本原则

一期可以标注“完整实现 / 简化实现 / 延后子特性”，但：

- **不允许改横切能力 ID**
- **不允许把一个能力拆成两个新编号**
- **不允许把不同能力写成同一个编号**

### 4.2 能力矩阵

| 能力 ID | 能力名 | 一期状态 | 说明 |
|--------|-------|---------|------|
| `CC01` | `approval_gate` | 兼容实现（无 TTL 扫描） | gate 创建/恢复必须；TTL 字段保留，默认 `null`；TTL 自动升级二期启用 |
| `CC02` | `policy_mode_and_override` | 兼容实现 | 一期默认 `conservative`；非 conservative 覆写先降级处理并记录 |
| `CC03` | `provider_run_lifecycle` | 完整实现 | provider run 派发、收集、固化记录 |
| `CC04` | `artifact_validate` | 兼容实现（L1/L2） | L1/L2 必须实现；L3 语义性校验二期增强 |
| `CC05` | `checkpoint_and_recovery` | 兼容实现（无 compaction） | checkpoint、checksum、恢复必须；compaction 二期启用 |
| `CC06` | `worktree_lifecycle` | 完整实现 | worktree 租约、创建、回收 |
| `CC07` | `integration_queue` | 简化实现 | 一期默认 FIFO；优先级/拓扑调度二期增强 |
| `CC08` | `manual_intervention` | 完整实现 | 终态兜底能力 |
| `CC10` | `provider_adapter_spec` | 完整实现 | 一期落地 CLI adapter，二期扩展 SDK/app-server |

### 4.3 一期简化细节

**`CC02 policy_mode_and_override`**

- 默认启用 `conservative`
- `balanced` / `aggressive` 的协议定义保留，但一期不要求完整激活
- 阶段级覆写数据结构保留，落地可先最小化
- 若用户配置 `balanced` / `aggressive`，一期运行时必须显式降级为 `conservative` 行为，并在 `effectivePolicy` 或 event 中记录 `policy_mode_degraded`
- 不允许出现字段接受了 `balanced` / `aggressive`，但实际行为既不执行映射表、也不记录降级原因的情况

**`CC07 integration_queue`**

- 一期默认 FIFO
- **回流上限一期必须实现**：同一 WorkTask 连续 2 次集成失败后不再自动出队，升级到 `X08 manual_intervention`，reason 为 `integration_retry_limit_exceeded`。此机制与 `rework_counter` 类似，是防止集成死循环的兜底，实现成本低（计数器 + 判断），不涉及复杂的拓扑调度
- 队列持久化必须保留
- 紧急插队、依赖拓扑感知可延期，但不能删协议字段和扩展入口

**`CC01 approval_gate`**

- gate 基本生命周期必须实现
- TTL 字段和超时语义保留
- 一期不启用后台超时升级扫描

**`CC05 checkpoint_and_recovery`**

- checkpoint 落盘、checksum 校验、恢复逻辑必须实现
- Event log 保持 append-only
- compaction 规则保留为二期增强，不在一期调度器中主动执行

## 6. 技术栈与目录结构

### 6.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 运行时核心 | Rust | daemon、状态机、恢复、队列、worktree |
| Provider 基线接入 | Rust + 进程调用 | BYO 本地 CLI，`spawn + CLI` |
| REPL 客户端 | Rust | 一期 REPL 前端 |
| 通信协议 | JSON over Unix socket / named pipe + tokio | daemon 与客户端通信；评审后裁定一期统一使用 tokio async transport |
| 序列化 | JSON + Markdown | 结构化记录用 JSON，用户文档用 Markdown |
| 文档操作 | `pulldown-cmark` + Aria HeadingModel + serde JSON/YAML + 可选 ast-grep | Markdown canonical artifact 走 Markdown tokenization 与 heading model；JSON/YAML 走结构化 parser；代码结构搜索/重写可接 ast-grep |
| 版本控制 | git + git2 crate | worktree 与集成 |
| 升级路径 | SDK / app-server | 二期增强，不是一期前提 |

> **关于技术栈演进**：上游总览将 provider adapter 和未来 UI/TUI/桌面壳定位为 TypeScript 职责。一期选择全 Rust 是基于以下工程判断：一期只用 CLI spawn，Rust `std::process::Command` 即可完成，无 SDK 依赖；混合语言栈增加构建复杂度（Cargo + npm/pnpm + FFI 桥接），对一期不划算。二期接入 Claude Agent SDK / Codex SDK 时，这些 SDK 均为 JS/TS 优先发布，届时应重新评估是否引入 TypeScript adapter 层。此决策不影响协议层——daemon 与 provider adapter 通过 `CC10` 定义的统一接口解耦，更换 adapter 实现语言不需要重写 daemon 核心状态机。

> **评审后实施规格补齐**：Rust 类型定义、`AdapterInput` / `AdapterOutput`、`RuntimeUnit`、projection/OpenSpec 编译规则、daemon 发现协议、artifact 路径、prompt 模板、worktree 集成和 fixture 以 `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.3.md` 为准。P2 不得在该文档定义的 Projection、OpenSpec bundle、Markdown parser 和 fixture 规则未锁定前启动。

**Provider Adapter 边界补充**

- `CC10` 的稳定真相源是 `AdapterInput` / `AdapterOutput`、provider run lifecycle、结构化记录和错误语义
- Aria 一期不内置、不托管、不自动安装 Claude Code / Codex；用户必须在本机提供可执行 CLI，daemon 只做 capability probe、compatibility matrix 和受控 spawn
- 具体 CLI 命令名、参数名、session flag、输出解析细节属于 adapter compatibility matrix，不应作为协议级不可变约束
- 一期可以在 Rust CLI adapter 中内置默认映射，但实现上必须允许后续通过配置或新 adapter 替换
- 若 CLI 实际能力与文档示例不同，以 adapter 探测结果和运行记录为准，不得让 CLI 输出直接成为系统真相源
- 一期真实 CLI adapter baseline 必须覆盖 capability probe、compatibility matrix、spawn、timeout、stdout/stderr、structured output、files modified 记录；业务闭环仍可用 fake provider fixture 固化

**Document Operation 边界补充**

- Aria 需要自己创建、更新、归一化和校验 `cadence/` 下的 canonical artifacts 与 OpenSpec 文档；provider 只能提供候选内容
- Markdown 用户文档不得靠正则或裸字符串替换维护结构，必须经过 Markdown AST / heading model 做章节级读取、插入、替换和 projection 编译
- JSON artifact 与 `_aria` 扩展必须通过 serde 类型、JSON schema 或结构化 patch 操作，不允许手写字符串拼 JSON
- OpenSpec 的 `proposal.md` / `design.md` / `tasks.md` 使用 Markdown AST 做文档结构操作；其中 YAML / JSON / 代码块按各自语言 parser 处理
- ast-grep 可作为可选工具能力，用于 worktree 中代码结构搜索、lint、codemod，或 JSON/YAML 等支持语言的结构化查询；它不是 Markdown canonical artifact 的主编辑引擎

### 6.2 REPL 架构（一期）

- **daemon** 是独立长驻进程
- **REPL** 是客户端，负责 attach / detach 与用户交互
- 一期采用自定义 JSON envelope，字段风格兼容 JSON-RPC 2.0
- 二期桌面端只替换客户端，不重写 daemon 状态机

一期必须把 REPL <-> daemon wire protocol 视为**正式实现接口**，不得在 REPL 和 daemon 中各写一套隐式约定。

一期最小协议要求：

| 项目 | 一期要求 |
|------|---------|
| envelope | 固定字段：`protocol_version`、`message_type`、`request_id`、`command`、`event_id`、`event_type`、`sent_at`、`occurred_at`、`ok`、`payload`、`error`；具体必填条件以实现总契约为准 |
| request/response | request 必须有 `command`；response 必须回显 `request_id`，并明确 `ok/error` |
| event | 必须有 `event_id`、`event_type`、`occurred_at`；`task_id` / `node_id` 按事件语义条件必填，具体以实现总契约为准 |
| handshake | 必须实现 `hello -> attach -> subscribe` 三段式握手 |
| reconnect | 必须基于 `reconnect_token + last_seen_event_id` 恢复 |
| error model | 必须实现统一错误码，而不是靠 stderr 文本猜测 |

wire protocol 的字段级定义、命令 payload、事件 payload、错误码注册表以实现总契约为准。

## 7. 交付边界与成功标准

### 7.1 交付边界

**本设计文档负责定义：**

- 一期如何在不改上游协议的前提下落地
- 哪些协议节点由哪些实现单元承载
- 哪些横切能力一期完整实现、简化实现、延期增强
- 一期技术栈、目录结构与运行边界

**本设计文档不负责定义：**

- 各节点 prompt 细节
- 具体 TDD 实现任务拆分
- 代码级提交粒度和研发排期

### 7.2 成功标准

一期成功，不仅要求“流程能跑”，还要求“协议不漂移”。

**必须同时满足：**

1. 系统可以完成一次真实需求的主流程：澄清 -> 规格 -> 设计 -> 计划 -> 分派 -> 编码 -> 测试 -> 评审 -> 集成 -> 总结
2. 所有输出产物都能按上游 canonical schema 校验通过
3. 即使使用实现单元合并，运行时仍能按协议边界记录 checkpoint / artifact / route
4. `N25 -> N26/N27`、`X09` 等协议定义仍然保留，只是一期运行策略可限制其自动化程度
5. daemon 可在 REPL 断开后继续运行，REPL 可重新 attach
6. 每个协议节点完成后，daemon 都能写入包含 canonical 字段的 runtime snapshot，即使该节点的业务产物不是 `runtime_snapshot`
7. `spec/design/plan` 都能编译出稳定 projection，供 routing / coverage / dashboard projection 使用
8. 所有需要机器消费的 JSON artifact 都实现 `_aria` 扩展命名空间，并通过 phase1 profile validator
9. REPL 与 daemon 仅通过正式 wire protocol 通信，不允许共享进程内临时对象绕过协议
10. OpenSpec 约束 bundle、artifact projection、traceability binding 都有稳定落盘和失效重编译规则
11. Markdown / JSON / YAML 文档创建、更新、归一化和校验统一走 Document Operation 层；ast-grep 只作为可选结构搜索、lint、codemod 能力
12. Claude Code / Codex 真实调用基于用户本机 BYO CLI；CLI 缺失、未登录或权限不足时必须可诊断失败，不允许自动安装或静默降级为假成功

**最低可接受标准：**

- 主流程跑通
- protocol/schema/ID 无冲突
- 集成阶段允许保守实现，但不得破坏上游协议字段和路由

## 附录 C：关键规则摘要

### C.0 Runtime Snapshot 规则摘要

**核心原则：** `runtime_snapshot` 是 daemon 的 checkpoint envelope，不是替代业务产物的节点输出。

**一期要求：**

- 每个协议节点完成后都写入 runtime snapshot。
- runtime snapshot 必须包含全局协议定义的 canonical 字段。
- 节点目录中的“产物类型”以业务产物为主；业务产物与 runtime snapshot 不能互相替代。
- Agent 节点的 provider 输出不能直接成为系统真相源，必须经 artifact validate、checkpoint 和 handoff 固化。

### C.1 回流失效规则摘要

**核心原则：** 当节点回流跨越一个或多个中间节点时，中间节点产出的产物必须标记为 `superseded`，后续流程不得继续引用。

**一期要求：**

- 回流发生时写入 `supersededArtifactRefs`
- 节点进入前校验输入引用不得包含 `superseded`
- daemon 重启恢复时必须恢复失效标记
