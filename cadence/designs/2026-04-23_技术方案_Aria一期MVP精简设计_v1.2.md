## 1. 目标与范围

### 1.1 一期 MVP 定位

- **形态**：REPL client + 本地 daemon runtime
- **范围**：单用户、本机、单仓库、多任务并行（通过 worktree 隔离）
- **Provider 接入基线**：BYO 本地 CLI + `spawn + CLI`（用户本机安装 Claude Code / Codex；Claude Code 作为 orchestrator，Codex 作为 executor/reviewer）
- **隔离策略**：每任务独立 worktree
- **集成策略**：并行执行、串行集成（一期默认 FIFO）
- **人工介入**：保守策略，关键节点挂 approval gate
- **Daemon 生命周期**：daemon 常驻，REPL 只是客户端，可 attach/detach

### 1.2 文档定位与真相源边界

本文件的定位不是“重写上游协议”，而是：

1. 在**不改变上游协议真相源**的前提下，定义 Aria 一期如何实现。
2. 明确哪些能力一期完整实现、哪些简化实现、哪些延期到二期。
3. 明确为了降低工程复杂度，一期如何把多个协议节点折叠为较少的**实现单元**。
4. 明确 `v1.2 + 实现总契约 + 评审后实施规格补齐` 如何共同组成一期的实现真相源。

本文件**不替代**上游文档集中以下内容：

- 节点 ID、节点类型、上下游路由
- 横切能力 ID 与能力边界
- 产物最小字段与校验规则
- 全局协议章节与统一枚举值

### 1.4 关键假设（显式声明）

1. **一期的主接入路径是 `spawn + CLI`**
   这是一期最小可落地闭环的基线方案。SDK / app-server 是明确保留的二期升级路径，不视为不可行。
2. **daemon 常驻，REPL 只是客户端**
   REPL 启动时 attach 到已有 daemon 或隐式启动新 daemon；REPL 退出时仅断开连接，daemon 继续运行；任务在后台持续执行。
3. **worktree 隔离优于 branch 隔离**
   并发任务需要真正的文件系统隔离；只切 branch 无法满足并行编码与回归验证的需要。
4. **对话分叉与代码分叉分开处理**
   会话管理靠 session/thread（`ProjectSession`、`Phase`），代码隔离靠 worktree（`WorktreeLease`）。两者是独立的生命周期：一个 session 可以对应多个 worktree，一个 worktree 的回收不影响 session 状态。
5. **OpenSpec 在一期是强约束、非可选**
   每个 EpicTask 必须绑定 `change_id`，并在进入规划/执行节点前拥有可编译的 OpenSpec 约束 bundle；禁止静默降级为“只靠聊天历史推进”。
   约束 bundle 不是一次性全量 ready，而是随阶段逐步成熟：`N06 pass` 后才要求 `requirement_constraints` 非空，`N08 pass/conditional_pass` 后才要求 `design_constraints` 非空，`N11/N26` 写回 tasks 后才要求 `task_constraints` 非空。

### 1.5 可扩展性保障

**前端形态扩展保障**

核心原则：**REPL 只负责交互，daemon 是运行时真相源**。无论前端形态是 REPL、GUI 还是 Web，它们都只是 daemon 的客户端，通过统一接口通信。

一期 REPL 与 daemon 通过 JSON over Unix socket 通信，Windows 环境使用 named pipe。二期做桌面端时，只替换客户端，不改变 daemon 核心状态机。

**Provider 接入扩展保障**

一期通过 Provider Adapter 把 daemon 与底层执行器解耦：

```text
daemon -> ProviderAdapter -> [CliAdapter | SdkAdapter | AppServerAdapter | ...]
```

一期只落地 `CliAdapter`。二期新增 `SdkAdapter` 或 `AppServerAdapter` 时，不应要求 daemon 重写协议层。

一期 Claude Code / Codex 接入方式是用户本机 BYO CLI：用户负责安装、登录和授权；Aria 不内置 provider、不托管远程模型、不自动下载安装 CLI。daemon 只负责探测 CLI 能力、选择 compatibility matrix、受控 spawn 和记录 `ProviderRunRecord`。

## 2. 节点模型与一期实现单元

### 2.1 双层模型

Aria 一期采用**双层模型**：

1. **协议层**：仍然是上游文档集中定义的 `N00-N28` 与 `X01-X09`
2. **实现层**：一期为了降低代码复杂度，使用较少的“实现单元”落地运行

- 实现单元不是新的协议节点 ID
- 协议层的节点定义、节点顺序、协议路由保持不变
- 即使多个协议节点由同一个实现单元处理，运行时仍应按协议边界产出对应的产物、checkpoint 和状态变迁记录

#### 2.3.1 `N00/N01` pre-task 规则

`N00 session_bootstrap` 与 `N01 intake_capture` 发生在 `N02 epic_task_create` 之前，不能伪造 `task_id`。

一期裁定：

1. `N00/N01` 写入 session-level checkpoint / runtime snapshot，`task_id` 条件为空或缺省。
2. 从 `N02` 创建 EpicTask 成功开始，所有 task-scoped runtime snapshot 必须包含真实 `task_id`。
3. 若 `N00/N01` 也复用 `RuntimeUnit` 调度入口，daemon 必须提供 pre-task input wrapper；不得把空字符串、临时 task 或用户请求 hash 填入 `task_id`。
4. Event envelope 中 session 级事件可不带 `task_id/node_id`；一旦事件语义关联具体 task 或 node，必须在 envelope 或 payload 中提供对应 ID。

### 2.5 N06 的一期策略：增强评审，不改协议身份

原协议中，[`N06 spec_gate_review`](./aria-repl-runtime-docs/nodes/2026-04-22_技术方案_节点N06_spec_gate_review_v1.0.md:1) 是 **Aria 内部节点**。一期不改这个定义。

一期允许的增强实现是：

- daemon 先执行 `artifact_validate`
- 若需要更强的规格可执行性判断，可调用 Codex 产出**advisory review**
- 最终 `spec_gate_decision` 仍由 daemon 按固定协议字段生成并路由

也就是说：

- **可以增强评审输入**
- **不可以把 N06 改写成纯 Agent 节点**

### 2.6 N20 的一期策略：daemon 决策，可选 advisory

原协议中，[`N20 ready_for_integration`](./aria-repl-runtime-docs/nodes/2026-04-22_技术方案_节点N20_ready_for_integration_v1.0.md:1) 是 **Aria 内部节点**。一期保留这一点。

允许的一期增强是：

- daemon 负责结构化前置校验
- 当 `code_review_report.review_decision = conditional_pass` 时，可调用 Codex 做补充判断
- 最终 ready / block / rework 的状态变迁仍由 daemon 决定

**N20/N23/N24 Git 集成事实边界**

一期固定使用 candidate commit + integration branch，不允许裸 diff 直接合并：

1. `N20 ready` 前，daemon 校验 diff 只在 `allowed_write_scope` 内，然后在 WorkTask worktree 中创建 `git commit -m "aria: <worktask_id> candidate"`。
2. `candidate_commit_sha` 写入 WorkTask runtime state、`N20` runtime snapshot 和传给 `N22` 的 handoff package。
3. `N22` 记录 integration worktree 的 `pre_merge_sha`，不得临时创建 candidate commit。
4. `N23` 只在 integration worktree 中执行 checkout / cherry-pick / pre-check / integration commit，禁止在 repo root 执行 checkout、cherry-pick、reset 或 commit。
5. `N23` 成功后必须形成 integration commit，并写入 `integration_report` 和 runtime snapshot；失败时按 `pre_merge_sha` rollback 或进入 `N19` / manual intervention。
6. `N24` 验证失败且需要回滚时，执行 `git reset --hard <pre_merge_sha>`，并写入 `integration_report.rollback_ref` 与 runtime snapshot。

### 2.7 N26 的一期策略：保留协议路由，关闭静默自动派生

原协议中，[`N25 -> N26/N27`](./aria-repl-runtime-docs/nodes/2026-04-22_技术方案_节点N26_patch_followup_dispatch_v1.0.md:1) 是正式定义的分支。

一期不删除这个分支，只做运行策略限制：

- `N25 final_review` 仍可得出 `overall_decision = followup`
- 协议上仍认为系统**可以**进入 `N26`
- 一期实现 `M26/N26`，但禁止静默自动派生补丁任务；必须先进入：
  - `approval_gate`，或
  - `manual_intervention`

**一期确定性路由规则**

| `final_review` 结果 | 一期路由 | 说明 |
|---------------------|---------|------|
| `overall_decision = pass` 且 `followup_needed = false` | `N27` | 正常进入最终总结 |
| `overall_decision = followup` 且补丁边界清晰、`patch_round_counter` 未超过协议阈值 | `X01 approval_gate_open` | gate 中展示补丁范围；用户 `approve_gate` 后进入 `M26/N26`，daemon 不静默自动派生 |
| `overall_decision = followup` 且用户 `reject_gate` | `N27` | `final_summary` 必须标记 `closed_with_rejected_followup` 或 `manual_terminated`，不得伪造 `final_review.overall_decision=pass` |
| `overall_decision = followup` 但补丁边界不清、缺口严重度无法裁决或缺少验收目标 | `X08 manual_intervention_hold` | reason 为 `followup_scope_unclear` |
| `overall_decision = followup` 但 `patch_round_counter` 超过协议阈值 | `X08 manual_intervention_hold` | reason 为 `patch_round_limit_exceeded` |
| `overall_decision = fail` | `X08 manual_intervention_hold` | reason 为 `final_review_failed` |

该规则保证：

- `N26` 协议定义没有被删除。
- 一期不出现“final review 后静默自动派生补丁任务”的行为。
- 是否进入 `N26` 必须由用户显式动作触发，便于审计和避免无限补丁循环。
- rejected followup 是非成功或人工关闭路径，不能被当成 pass 路径。

**final closure 状态语义**

| 路径 | `final_review` | gate 行为 | `final_summary` 语义 |
|------|----------------|-----------|----------------------|
| 正常通过 | `overall_decision=pass` 且 `followup_needed=false` | 不开 followup gate | `closed_successfully` |
| followup approve | `overall_decision=followup` | 用户 `approve_gate` 后进入 `M26/N26`，生成新 `dispatch_package`，回到 `N13` | 当前轮不关闭；新 followup worktask 完成后重新进入 final review |
| followup reject | `overall_decision=followup` | 用户 `reject_gate` | `closed_with_rejected_followup` 或 `manual_terminated`，必须保留未闭合 coverage / risk |
| final fail | `overall_decision=fail` | 进入 manual intervention | `manual_terminated` 或等待人工接管 |

### 2.8 X09 的一期策略：定义保留，可不独立暴露

[`X09 policy_override_apply`](./aria-repl-runtime-docs/2026-04-22_技术方案_Aria节点总目录_v1.0.md:47) 在协议层保留。

- 但 event、snapshot、effective_policy 语义必须与协议兼容

## 3. 产物规范与 Schema 兼容策略

### 3.1 基本原则

一期允许对**代码组织方式**做精简，但不允许重写上游产物 schema。

具体来说：

- **16 份上游业务产物规范文档继续是规范真相源**
- `runtime_snapshot` 是全局协议定义的运行快照，作为第 17 类一期必须落地的 runtime artifact 进入 validator 和 checkpoint 流程
- 一期实现中，代码可以按少量 schema 族来组织校验器与序列化器
- 本文件中的产物摘要只做索引，不替代原始产物规范

### 3.2 一期实现视角下的 Schema 分组

| 分组 | 涵盖产物 | 说明 |
|------|---------|------|
| **Markdown 用户文档** | `intake_brief`, `spec`, `design`, `plan`, `final_summary` | 用户可读文档 |
| **JSON 决策/运行记录** | `clarification_record`, `spec_gate_decision`, `design_review`, `design_revision_record`, `readiness_check`, `coding_report`, `testing_report`, `code_review_report`, `integration_report`, `final_review` | 机器消费的结构化记录 |
| **JSON 交接包** | `dispatch_package` | 交接包，不应改成 Markdown |
| **JSON 运行快照** | `runtime_snapshot` | Aria 内部状态快照 |

#### 3.2.1 一期 Phase1 Profile 承载策略

一期为了做到“canonical 不漂移，但研发可直接实现”，额外引入两类实现承载：

1. **Projection**：把 Markdown canonical artifact 编译成机器可消费的结构化投影。
2. **Reserved extension namespace**：对 JSON artifact 允许增加 `_aria` 顶层扩展对象，承载一期 routing / traceability / constraint-check 信息。

承载规则如下：

| 产物族 | 机器必需的补充信息 | 一期承载方式 | 直接消费者 |
|--------|-------------------|--------------|------------|
| `spec` | story / requirement / success criteria / open item 的稳定 ID 与结构化索引 | `SpecProjection` | `N06/N07/N11/N25`、dashboard projection |
| `design` | design decision、shared component / module、data entity、API entry、risk ref | `DesignProjection` | `N08/N10/N11/N16/N18/N25`、dashboard projection |
| `plan` | work package 的 `execution_mode`、`human_required_reason`、`traceability_refs`、`acceptance_targets` | `PlanProjection` | `N12/N13/N16-N20`、dashboard projection |
| `dispatch_package` | worktask routing、write scope、traceability refs、constraint check refs | `dispatch_package._aria` | `N13/N16-N20/N23/N25` |
| `coding/testing/review/integration/final review` | traceability refs、constraint check refs、provider run refs | 各 JSON report 的 `_aria` | daemon、traceability、dashboard projection |
| `runtime_snapshot` | canonical 字段之外的恢复和回放元数据 | `node_specific_fields` + `projection_refs` + `constraint_bundle_refs` + runtime sidecar | recovery、event replay |

#### 3.2.2 一期 validator 与 coverage 矩阵

一期实现必须区分 canonical validator、projection validator、phase1 profile validator 和 OpenSpec coverage。它们职责不同，不能互相替代。

| 校验对象 | 校验入口 | 一期范围 | 完成条件 |
|---------|---------|---------|---------|
| 16 类业务产物 + `runtime_snapshot` | canonical validator | 全部一期产物 | 只校验上游 canonical 最小字段与最小结构 |
| `spec` / `design` / `plan` | projection validator | 仅这 3 类 Markdown canonical artifact | 编译出稳定 `SpecProjection` / `DesignProjection` / `PlanProjection`，golden fixture 与 Rust 类型一致 |
| 需要 `_aria` 的 JSON artifact | phase1 profile validator | `dispatch_package`、执行/评审/report 类 JSON、`final_review` 等 | `_aria.profile_version`、constraint refs、traceability refs、provider run refs、projection refs 等满足一期 profile |
| OpenSpec 约束覆盖 | OpenSpec coverage check | 消费约束的阶段收口 | 当前节点所需 constraints 非空且覆盖关系闭合、豁免或转人工 |

节点正式收口必须同时通过其适用的 canonical validator、projection validator、phase1 profile validator 和 OpenSpec coverage check。P2 实现时不得把 17 类产物都强行做 projection，也不得把 `_aria` profile 校验混入 canonical 最小字段校验。

### 3.4 Runtime Snapshot 的兼容性要求

`runtime_snapshot` 在一期中统一定义为 **daemon 在每个协议节点完成后写入的 checkpoint envelope**。

因此需要区分两层概念：

- **节点业务产物**：节点对外提供给后继节点消费的主产物，例如 `spec`、`design_review`、`integration_report`
- **runtime snapshot**：daemon 为恢复、审计、路由和回流一致性维护的运行时快照

节点总目录中的“产物类型”以业务产物为主，不代表只有 Aria 内部节点才能产生 runtime snapshot。无论协议节点是 Aria 内部、Agent 业务还是混合节点，只要节点完成并发生状态迁移，daemon 都必须写入 checkpoint 和对应 runtime snapshot。

一期统一沿用上游全局协议中的 canonical 字段：

- `snapshot_id`
- `session_id`
- `task_id`
- `node_id`
- `phase`
- `timestamp`
- `effective_policy`
- `artifact_refs`
- `provider_run_refs`
- `worktree_ref`
- `rework_counter`
- `risk_registry`
- `loop_counters`
- `superseded_artifact_refs`
- `node_specific_fields`

一期 additive fields：

- `projection_refs`
- `constraint_bundle_refs`

这两个字段用于 recovery、event replay、coverage 和 profile validator，不属于上游 canonical 最小字段，也不得替代 `artifact_refs`、`provider_run_refs` 或 `node_specific_fields`。

`N00/N01` pre-task 例外：`N00/N01` 写 session-level runtime snapshot，`task_id` 条件为空；从 `N02` 开始，task-scoped runtime snapshot 必须包含真实 `task_id`。

**一期兼容策略：**

| 字段/概念 | 一期要求 |
|----------|---------|
| `rework_counter` | 保留，不改名为其他字段 |
| `loop_counters` | 保留对象结构；一期可复用同一套计数机制，但默认阈值必须按协议分别配置 |
| `risk_registry` | 保留并可最小化实现；至少支持落盘、恢复、risk_id 引用占位 |
| `superseded_artifact_refs` | 必须保留；回流一致性依赖该字段，不能删除 |
| `node_specific_fields` | 继续按节点扩展 |

### 3.5 允许的实现期补充信息

一期允许在不破坏 canonical schema 的前提下增加**补充信息**，但这些补充信息不能替代最小字段。

| 补充信息 | 允许方式 | 不允许的方式 |
|---------|---------|-------------|
| Markdown artifact 的机器索引 | 编译到 projection | 直接假设 Markdown 原文可稳定机读 |
| JSON artifact 的 routing / traceability / coverage 元数据 | 写入 `_aria` 命名空间 | 把 `_aria` 字段混入 canonical 最小字段清单 |
| `spec` 的 agent 可执行性评估 | 作为附加章节，且同步进入 `SpecProjection` | 替代 `functional_requirements` / `open_items` |
| `design` 的 `architecture_overview` / `api_design` 表达 | 作为展示性标题映射到 `architecture_summary` / `api_contracts`，并同步进入 `DesignProjection` | 改写 canonical 字段名 |
| `plan` 的 `timeline` / `milestones` / `execution_mode` / `human_required_reason` | 文档可附加叙述，但机器消费以 `PlanProjection` 为准 | 替代 `work_packages` / `acceptance_checks` |
| `dispatch_package` 的 routing 元数据 | 仅写入 `dispatch_package._aria` | 破坏 `worktasks/dependencies/parallel_groups/acceptance_targets` canonical 结构 |
| `final_summary` 的 `recommendations` | 作为 `next_steps` 的扩展说明 | 替代 `next_steps` canonical 字段 |

#### 3.5.1 `_aria` 扩展命名空间规则

所有 JSON artifact 的实现扩展字段统一放在 `_aria` 顶层对象中，避免与 canonical 字段混淆。

`_aria` 的一期通用字段：

- `profile_version`
- `constraint_check_ref`
- `traceability_refs`
- `provider_run_refs`
- `projection_refs`

其中：

- `dispatch_package._aria` 额外必须包含 `worktask_routing`
- `coding_report/testing_report/code_review_report/integration_report/final_review` 至少必须包含 `traceability_refs`
- `final_review._aria` 额外必须包含 `coverage_summary`

`dispatch_package._aria.worktask_routing[]` 最小字段：

| 字段 | 说明 |
|------|------|
| `worktask_id` | 对应 canonical `dispatch_package.worktasks[]` 中的 WorkTask |
| `source_work_package_id` | 回查 `PlanProjection.work_packages[]` 的稳定来源；不得默认等同于 `worktask_id` |
| `execution_mode` | `agent_only` / `human_assisted` / `human_required` |
| `human_required_reason` | `execution_mode=human_required` 时必填 |
| `allowed_write_scope` | 当前 WorkTask 可写路径范围 |
| `verification_commands` | 当前 WorkTask 的最小验证命令 |
| `traceability_refs` | requirement / design decision / task / risk 的稳定引用 |
| `acceptance_targets` | 执行、测试、评审节点消费的验收目标 |

report 类 artifact 的 `_aria.traceability_refs` 不能直接信任 provider 输出。daemon 必须按以下优先级归一化：

1. `PlanProjection.work_packages[].traceability_refs`
2. `dispatch_package._aria.worktask_routing[].source_work_package_id`
3. Provider structured output 中的候选 refs
4. 当前 artifact 自身显式 refs

provider 候选 refs 与 PlanProjection 冲突时，以 PlanProjection 和 daemon 归一化结果为准；冲突必须写入 `ArtifactTraceabilityBinding.conflict_reason`，并发 `traceability.updated` event。找不到 `source_work_package_id` 或对应 work package 时，report 不得进入 active。

#### 3.5.2 Projection 规则

Markdown artifact 不追加隐藏 front matter。
daemon 在 artifact validate 通过后，立即编译 projection 并单独落盘：

- `spec` -> `SpecProjection`
- `design` -> `DesignProjection`
- `plan` -> `PlanProjection`

后续任何 routing、coverage、dashboard、traceability 逻辑，一律消费 projection，不回头直接解析 Markdown 原文。

### 3.6 产物格式规则

- **Markdown 用户文档**：`intake_brief`, `spec`, `design`, `plan`, `final_summary`
- **JSON 决策/运行记录**：`clarification_record`, `spec_gate_decision`, `design_review`, `design_revision_record`, `readiness_check`, `coding_report`, `testing_report`, `code_review_report`, `integration_report`, `final_review`
- **JSON 交接包**：`dispatch_package`
- **JSON 运行快照**：`runtime_snapshot`

## 4. 横切能力的一期实现范围

### 4.1 基本原则

一期可以标注“完整实现 / 简化实现 / 延后子特性”，但：

- **不允许改横切能力 ID**
- **不允许把一个能力拆成两个新编号**
- **不允许把不同能力写成同一个编号**

状态口径：

| 状态 | 含义 |
|------|------|
| 完整实现 | 按一期 implementation profile 完整实现；若上游能力含二期子特性，则该子特性必须在说明中明确延期 |
| 兼容实现 | 保留协议字段、状态、事件和恢复入口，但某些后台扫描、语义校验或高级策略延期 |
| 简化实现 | 保留扩展入口和审计字段，一期只启用最小安全策略 |

因此 `CC10 provider_adapter_spec` 的“完整实现”指一期 CLI adapter、ProviderRunRecord、compatibility matrix、错误码和 raw output 审计完整，不表示 SDK/app-server adapter 一期也要落地。

### 4.2 能力矩阵

| 能力 ID | 能力名 | 一期状态 | 说明 |
|--------|-------|---------|------|
| `CC01` | `approval_gate` | 兼容实现（无 TTL 扫描） | gate 创建/恢复必须；TTL 字段保留，默认 `null`；TTL 自动升级二期启用 |
| `CC02` | `policy_mode_and_override` | 兼容实现 | 一期默认 `conservative`；非 conservative 覆写先降级处理并记录 |
| `CC03` | `provider_run_lifecycle` | 完整实现 | provider run 派发、收集、固化记录 |
| `CC04` | `artifact_validate` | 兼容实现（L1/L2） | L1/L2 必须实现；L3 语义性校验二期增强 |
| `CC05` | `checkpoint_and_recovery` | 兼容实现（无 compaction） | checkpoint、checksum、恢复必须；compaction 二期启用 |
| `CC06` | `worktree_lifecycle` | 完整实现 | worktree 租约、创建、回收 |
| `CC07` | `integration_queue` | 简化实现 | 一期默认 FIFO；优先级/拓扑调度二期增强 |
| `CC08` | `manual_intervention` | 完整实现 | 终态兜底能力 |
| `CC10` | `provider_adapter_spec` | 完整实现 | 一期落地 CLI adapter，二期扩展 SDK/app-server |

### 4.3 一期简化细节

**`CC02 policy_mode_and_override`**

- 默认启用 `conservative`
- `balanced` / `aggressive` 的协议定义保留，但一期不要求完整激活
- 阶段级覆写数据结构保留，落地可先最小化
- 若用户配置 `balanced` / `aggressive`，一期运行时必须显式降级为 `conservative` 行为，并在 `effective_policy` 或 event 中记录 `policy_mode_degraded`
- 不允许出现字段接受了 `balanced` / `aggressive`，但实际行为既不执行映射表、也不记录降级原因的情况

**`CC07 integration_queue`**

- 一期默认 FIFO
- **回流上限一期必须实现**：同一 WorkTask 连续 2 次集成失败后不再自动出队，升级到 `X08 manual_intervention`，reason 为 `integration_retry_limit_exceeded`。此机制与 `rework_counter` 类似，是防止集成死循环的兜底，实现成本低（计数器 + 判断），不涉及复杂的拓扑调度
- 队列持久化必须保留
- 紧急插队、依赖拓扑感知可延期，但不能删协议字段和扩展入口

**`CC01 approval_gate`**

- gate 基本生命周期必须实现
- TTL 字段和超时语义保留
- 一期不启用后台超时升级扫描

**`CC05 checkpoint_and_recovery`**

- checkpoint 落盘、checksum 校验、恢复逻辑必须实现
- Event log 保持 append-only
- compaction 规则保留为二期增强，不在一期调度器中主动执行

| 12. 循环边界控制 | 兼容实现 | 保留命名计数器和协议默认阈值；实现机制可共用，阈值取值不可共用覆盖 |
| 13. 回流失效 | 完整实现 | `superseded` 规则必须保留 |
| 14. 枚举值注册表 | 完整实现 | 不允许另起一套枚举 |
| 15. Risk Registry | 兼容实现 | 最小创建/落盘/恢复先落地，完整生命周期二期增强 |
| 16. Runtime Snapshot | 完整实现 | canonical schema 保留 |
| 17. 统一术语表 | 完整实现 | 保留 |

### 5.3 一期兼容策略

**循环边界**

- 保留 `rework_counter` 与 `loop_counters`
- 一期可以让多个循环共用同一套计数器实现机制和配置读取代码
- 各循环的默认阈值必须沿用上游协议：例如 `patch_round_counter` 默认 2，`rework_counter` / `design_revision_counter` / `clarification_round_counter` 默认 3
- 不再新增非 canonical 的 `retry_counter` 字段替代原 schema

**Risk Registry**

- 一期至少支持：
  - task 创建时初始化空 Risk Registry sidecar
  - 生成稳定 `risk_registry_ref`
  - `risk_id` 分配
  - runtime_snapshot 落盘
  - 重启恢复
  - 产物中的风险引用可被解析
- Risk Registry 是 task 级 runtime sidecar，不是 17 类 canonical artifact，不复用 `ArtifactRef`
- ProviderRunRecord 不直接持有 `risk_registry_ref`；风险关联通过 `CanonicalNodeInput.risk_registry_ref`、`RuntimeSnapshot.risk_registry`、`ArtifactTraceabilityBinding.related_risk_ids` 建立
- 二期再补完：
  - `accepted` / `escalated` 的完整流程
  - 更复杂的跨产物追踪分析

**OpenSpec 生命周期**

一期 OpenSpec 是强约束、非可选，但约束内容随阶段逐步成熟：

| 阶段 | OpenSpec 动作 | 后继约束 |
|------|---------------|----------|
| `N02/N03` | 固化 `change_id`，初始化 `openspec_bootstrap_status=bootstrap_pending` | P2 bootstrap skeleton |
| P2 bootstrap | 创建或读取 `proposal.md`、`specs/main/spec.md`、`design.md`、`tasks.md`，编译初始 bundle | 可进入 `N04/N05` |
| `N06 pass` | daemon 将通过 gate 的 Aria `spec` 写回 `specs/main/spec.md`，bundle stale/recompile | `N07` 前 `requirement_constraints` 必须非空 |
| `N08 pass/conditional_pass` | daemon 将稳定 `design` 写回 `design.md`，bundle stale/recompile | `N10/N11` 可消费 `design_constraints` |
| `N11` | daemon 将 plan 中的 task 约束写回 `tasks.md`，bundle stale/recompile | `N12/N16` 前 `task_constraints` 必须非空 |
| `N26 approve` | daemon 将 followup task delta 写回 `tasks.md`，bundle stale/recompile | 新 `dispatch_package` 必须绑定新 task constraints |

固定规则：

1. `N06` 的 pass 前置检查源是 Aria `spec`、`SpecProjection` 与 `proposal_constraints`，不是尚未写回的 `requirement_constraints`。
2. `N07` 只强制 `requirement_constraints`，不得要求尚未写回的 `design_constraints`。
3. `N10/N11` 消费 `design_constraints`，但 `task_constraints` 由 `N11` 写回后才可要求非空。
4. `tasks.md` 缺失不得进入 `N12/N16`；必须回到 `N11` 或 gate/manual intervention。
5. OpenSpec 写回与 bundle 重编译是原子操作；重编译失败或关键约束为空时，不得带着半更新的 OpenSpec 继续推进。

**Provider 接入**

- 一期：CLI adapter
- 二期：SDK / app-server adapter
- 该分期是**实现路径差异**，不是协议否定
- Provider raw output 包括 stdout、stderr、structured output、候选中间文件和 skill 过程草稿，统一登记为 `ExternalArtifactRef(source_system=provider_raw_output)`；不得复用 canonical `ArtifactRef`，不得直接驱动 routing / coverage / 状态推进
- CLI adapter 解析结构化输出时以补齐规格中的 sentinel block 规则为准；sentinel 外内容只作为 raw artifact 审计保留

### 5.4 一期绝不允许的“简化”

以下内容不属于可接受的 MVP 简化：

1. 改节点 ID、横切能力 ID、枚举值
2. 改 canonical 产物字段名
3. 删除协议中正式存在的路由分支
4. 把结构化 JSON 交接包改写成 Markdown 文档
5. 用“实现层折叠”取代“协议层定义”
6. 用单一循环阈值覆盖上游协议已经定义的不同默认阈值

## 6. 技术栈与目录结构

### 6.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 运行时核心 | Rust | daemon、状态机、恢复、队列、worktree |
| Provider 基线接入 | Rust + 进程调用 | BYO 本地 CLI，`spawn + CLI` |
| REPL 客户端 | Rust | 一期 REPL 前端 |
| 通信协议 | JSON over Unix socket / named pipe + tokio | daemon 与客户端通信；评审后裁定一期统一使用 tokio async transport |
| 序列化 | JSON + Markdown | 结构化记录用 JSON，用户文档用 Markdown |
| 文档操作 | `pulldown-cmark` + Aria HeadingModel + serde JSON/YAML + 可选 ast-grep | Markdown canonical artifact 走 Markdown tokenization 与 heading model；JSON/YAML 走结构化 parser；代码结构搜索/重写可接 ast-grep |
| 版本控制 | git + git2 crate | worktree 与集成 |
| 升级路径 | SDK / app-server | 二期增强，不是一期前提 |

P1 crate 基线以实施计划为准，至少包含 `serde`、`serde_json`、`thiserror`、`tokio`、`async-trait`、`anyhow`、`uuid`、`chrono`。本文只定义技术方向，具体 Cargo 依赖、feature 和测试命令以 P1-P4 计划为准。

> **关于技术栈演进**：上游总览将 provider adapter 和未来 UI/桌面壳定位为 TypeScript 职责。一期选择全 Rust 是基于以下工程判断：一期只用 CLI spawn，Rust `std::process::Command` 即可完成，无 SDK 依赖；混合语言栈增加构建复杂度（Cargo + npm/pnpm + FFI 桥接），对一期不划算。二期接入 Claude Agent SDK / Codex SDK 时，这些 SDK 均为 JS/TS 优先发布，届时应重新评估是否引入 TypeScript adapter 层。此决策不影响协议层——daemon 与 provider adapter 通过 `CC10` 定义的统一接口解耦，更换 adapter 实现语言不需要重写 daemon 核心状态机。

> **评审后实施规格补齐**：Rust 类型定义、`AdapterInput` / `AdapterOutput`、`RuntimeUnit`、projection/OpenSpec 编译规则、daemon 发现协议、artifact 路径、prompt 模板、worktree 集成和 fixture 以 `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.4.md` 为准。P2 不得在该文档定义的 Projection、OpenSpec bundle、Markdown parser 和 fixture 规则未锁定前启动。

**Provider Adapter 边界补充**

- `CC10` 的稳定真相源是 `AdapterInput` / `AdapterOutput`、provider run lifecycle、结构化记录和错误语义
- Aria 一期不内置、不托管、不自动安装 Claude Code / Codex；用户必须在本机提供可执行 CLI，daemon 只做 capability probe、compatibility matrix 和受控 spawn
- 具体 CLI 命令名、参数名、session flag、输出解析细节属于 adapter compatibility matrix，不应作为协议级不可变约束
- 一期可以在 Rust CLI adapter 中内置默认映射，但实现上必须允许后续通过配置或新 adapter 替换
- 若 CLI 实际能力与文档示例不同，以 adapter 探测结果和运行记录为准，不得让 CLI 输出直接成为系统真相源
- 一期真实 CLI adapter baseline 必须覆盖 capability probe、compatibility matrix、spawn、timeout、stdout/stderr、structured output、files modified 记录；业务闭环仍可用 fake provider fixture 固化
- stdout、stderr、structured output 和 provider 中间文件统一作为 provider raw output 登记为 `ExternalArtifactRef(source_system=provider_raw_output)`；`ProviderRunRecord.raw_artifact_refs/stdout_ref/stderr_ref/structured_output_ref` 使用 external ref，不复用 canonical `ArtifactRef`
- structured output 只作为候选 DTO；daemon 必须执行 normalization、artifact_validate、phase1 profile validator、OpenSpec coverage 和 checkpoint 后，才能形成 active canonical artifact

**Document Operation 边界补充**

- Aria 需要自己创建、更新、归一化和校验 `cadence/` 下的 canonical artifacts 与 OpenSpec 文档；provider 只能提供候选内容
- Markdown 用户文档不得靠正则或裸字符串替换维护结构，必须经过 Markdown AST / heading model 做章节级读取、插入、替换和 projection 编译
- JSON artifact 与 `_aria` 扩展必须通过 serde 类型、JSON schema 或结构化 patch 操作，不允许手写字符串拼 JSON
- OpenSpec 的 `proposal.md` / `design.md` / `tasks.md` 使用 Markdown AST 做文档结构操作；其中 YAML / JSON / 代码块按各自语言 parser 处理
- ast-grep 可作为可选工具能力，用于 worktree 中代码结构搜索、lint、codemod，或 JSON/YAML 等支持语言的结构化查询；它不是 Markdown canonical artifact 的主编辑引擎

Document Operation owner：

| 文件 | 责任 | 禁止事项 |
|------|------|----------|
| `src/protocol/document_ops.rs` | `DocumentModel`、`DocumentSection`、`DocumentBlock`、`HeadingPath`、`DocumentPatch`、`DocumentPatchResult` 等纯类型 | 文件 IO、Markdown parser 调用、sha256、写盘逻辑 |
| `src/cross_cutting/document_ops.rs` | `read_document_model`、`create_document`、`upsert_section`、`extract_projection_source`、`apply_json_patch`、YAML/JSON structured writer、sha256 | 重复定义 canonical artifact 身份、绕过 protocol 类型 |

研发不得新增第三个 Document Operation 入口；projection compiler、OpenSpec bundle compiler、artifact validator 和 runtime units 都必须通过上述 owner 模块读写文档。

### 6.2 REPL 架构（一期）

- **daemon** 是独立长驻进程
- **REPL** 是客户端，负责 attach / detach 与用户交互
- 一期采用自定义 JSON envelope，字段风格兼容 JSON-RPC 2.0
- 二期桌面端只替换客户端，不重写 daemon 状态机

一期必须把 REPL <-> daemon wire protocol 视为**正式实现接口**，不得在 REPL 和 daemon 中各写一套隐式约定。

一期最小协议要求：

| 项目 | 一期要求 |
|------|---------|
| envelope | 固定字段：`protocol_version`、`message_type`、`request_id`、`command`、`event_id`、`event_type`、`sent_at`、`occurred_at`、`ok`、`payload`、`error`；具体必填条件以实现总契约为准 |
| request/response | request 必须有 `command`；response 必须回显 `request_id`，并明确 `ok/error` |
| event | 必须有 `event_id`、`event_type`、`occurred_at`；`task_id` / `node_id` 按事件语义条件必填，具体以实现总契约为准 |
| handshake | 必须实现 `hello -> attach -> subscribe` 三段式握手 |
| reconnect | 必须基于 `reconnect_token + last_seen_event_id` 恢复 |
| error model | 必须实现统一错误码，而不是靠 stderr 文本猜测 |

wire protocol 的字段级定义、命令 payload、事件 payload、错误码注册表以实现总契约为准。

P1 最小命令面：

| command | 一期用途 |
|---------|---------|
| `hello` | 建立协议版本、能力与 replay 起点 |
| `attach` | 绑定 daemon session，返回可见 task / gate / latest event |
| `subscribe` | 建立事件流 |
| `new_task` | 创建 `intake_brief`、EpicTask runtime state、`change_id` |
| `get_status` | 查询 session、task、queue、gate、provider run 摘要 |
| `list_artifacts` | 查询 canonical artifacts、projections、constraint bundles、traceability bindings |
| `approve_gate` | 批准 gate，允许后续路由继续 |
| `reject_gate` | 拒绝 gate，按 gate 类型进入 `N27` 或 manual intervention |
| `reply_gate` | 补充用户输入，默认仍保持 gate 阻塞，直到 daemon 重算 gate 状态 |
| `detach` | REPL 断开，daemon 继续运行 |

Daemon discovery 与 replay 一期必须实现：

1. `aria repl --workspace <repo_root> [--no-start]` 默认发现 daemon；未发现且没有 `--no-start` 时自动启动 `aria daemon run`。
2. socket / named pipe 路径、`daemon.json`、`daemon.lock`、stale daemon 判定以补齐规格第 7 章为准。
3. reconnect 时 `hello` 携带 `last_seen_event_id`；daemon 根据 replay 窗口补发事件或返回 `replay_window_lost`。
4. event log 落盘字段可使用内部 `created_at`，wire event envelope 必须映射为 `occurred_at`。

## 7. 交付边界与成功标准

### 7.1 交付边界

**本设计文档负责定义：**

- 一期如何在不改上游协议的前提下落地
- 哪些协议节点由哪些实现单元承载
- 哪些横切能力一期完整实现、简化实现、延期增强
- 一期技术栈、目录结构与运行边界

**本设计文档不负责定义：**

- 各节点 prompt 细节
- 具体 TDD 实现任务拆分
- 代码级提交粒度和研发排期

### 7.2 成功标准

一期成功，不仅要求“流程能跑”，还要求“协议不漂移”。

**必须同时满足：**

1. 系统可以完成一次真实需求的主流程：澄清 -> 规格 -> 设计 -> 计划 -> 分派 -> 编码 -> 测试 -> 评审 -> 集成 -> 总结
2. 所有输出产物都能按上游 canonical schema 校验通过
3. 即使使用实现单元合并，运行时仍能按协议边界记录 checkpoint / artifact / route
4. `N25 -> N26/N27`、`X09` 等协议定义仍然保留，只是一期运行策略可限制其自动化程度
5. daemon 可在 REPL 断开后继续运行，REPL 可重新 attach
6. 每个协议节点完成后，daemon 都能写入包含 canonical 字段的 runtime snapshot，即使该节点的业务产物不是 `runtime_snapshot`
7. `spec/design/plan` 都能编译出稳定 projection，供 routing / coverage / dashboard projection 使用
8. 所有需要机器消费的 JSON artifact 都实现 `_aria` 扩展命名空间，并通过 phase1 profile validator
9. REPL 与 daemon 仅通过正式 wire protocol 通信，不允许共享进程内临时对象绕过协议
10. OpenSpec 约束 bundle、artifact projection、traceability binding 都有稳定落盘和失效重编译规则
11. Markdown / JSON / YAML 文档创建、更新、归一化和校验统一走 Document Operation 层；ast-grep 只作为可选结构搜索、lint、codemod 能力
12. Claude Code / Codex 真实调用基于用户本机 BYO CLI；CLI 缺失、未登录或权限不足时必须可诊断失败，不允许自动安装或静默降级为假成功

**最低可接受标准：**

- 主流程跑通
- protocol/schema/ID 无冲突
- 集成阶段允许保守实现，但不得破坏上游协议字段和路由

**成功标准到测试入口的映射：**

| 成功标准 | 主要计划 | 测试入口 |
|---------|---------|---------|
| REPL / daemon 连接、重连、事件回放 | P1 | `tests/repl_wire.rs`、`tests/repl_daemon_handshake.rs`、`tests/reconnect_replay.rs`、`tests/daemon_discovery.rs` |
| `new_task -> intake_brief -> EpicTask -> effective_policy/change_id` | P1 | `tests/task_init_and_intake.rs` |
| runtime snapshot canonical + additive fields | P1/P2 | `tests/runtime_snapshot_schema.rs` |
| 17 类产物 canonical validator | P2 | `tests/artifact_validate.rs`、`tests/artifact_schema_min_fields.rs` |
| `spec/design/plan` projection | P2 | `tests/spec_projection.rs`、`tests/design_projection.rs`、`tests/plan_projection.rs` |

## 附录 C：关键规则摘要

### C.0 Runtime Snapshot 规则摘要

**核心原则：** `runtime_snapshot` 是 daemon 的 checkpoint envelope，不是替代业务产物的节点输出。

**一期要求：**

- 每个协议节点完成后都写入 runtime snapshot。
- runtime snapshot 必须包含全局协议定义的 canonical 字段。
- `N00/N01` 是 pre-task 节点，写 session-level snapshot，`task_id` 条件为空；从 `N02` 开始 task-scoped snapshot 必须包含真实 `task_id`。
- 一期 runtime snapshot 额外保留 `projection_refs` 与 `constraint_bundle_refs`，用于 recovery、event replay、coverage 和 profile validator。
- 节点目录中的“产物类型”以业务产物为主；业务产物与 runtime snapshot 不能互相替代。
- Agent 节点的 provider 输出不能直接成为系统真相源，必须经 artifact validate、checkpoint 和 handoff 固化。

### C.1 回流失效规则摘要

**核心原则：** 当节点回流跨越一个或多个中间节点时，中间节点产出的产物必须标记为 `superseded`，后续流程不得继续引用。

**一期要求：**

- 回流发生时写入 `superseded_artifact_refs`
- 节点进入前校验输入引用不得包含 `superseded`
- daemon 重启恢复时必须恢复失效标记
