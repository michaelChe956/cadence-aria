## 1. 设计目标

本文件只解决一件事：把已有 Aria 协议文档拼成一份**研发可直接落地**的实现契约。

本文件必须明确以下问题：

1. Aria 如何与用户交互，直到形成正式输入、正式输出和正式收口。
2. Aria 如何驱动 Claude Code / Codex，明确到开发人员可以实现 `contract registry`、`context builder`、`provider router`。
3. Claude Code / Codex 如何使用 Superpowers 和 OpenSpec，明确到开发人员可以实现节点级约束。
4. Aria、OpenSpec、Superpowers、provider raw output 的输入输出物如何组合、追踪、校验和收口。
5. story、design、plan、风险、执行状态等内容后续将用于展示看板 / 大屏，因此当前正式产物必须预留稳定结构和追踪关系；但展示端本身不属于一期、二期实现范围。

## 2. 文档定位与裁定规则

### 2.1 双层真相源

Aria 一期采用双层真相源：

| 层级 | 作用 | 真相源 |
|------|------|--------|
| **协议真相源** | 定义 canonical 节点、产物、横切能力、枚举值、回流规则 | 上游协议文档集 |
| **实现真相源** | 定义对象模型、调用链、约束矩阵、阶段收口、模块边界 | 本文档 |

### 2.2 裁定规则

1. 若本文与上游协议文档冲突，以上游协议文档为准。
2. 若开发时发现上游协议文档在实现层缺少必要约束，以本文为实现入口，但必须记录“待回改上游文档项”。
3. 开发人员不得自行发明绕开本文和上游协议的运行路径。

### 2.3 本文不做的事

1. 不重定义 `N00-N28`、`X01-X09`、`CC01-CC10`。
2. 不重定义 canonical artifact 字段名。
3. 不取代 implementation plan。
4. 不把 OpenSpec 或 Superpowers 升格为 Aria runtime truth。

## 3. 三套系统的准确分工

### 3.1 一句话模型

**Aria 单一收口，OpenSpec 全程约束，Superpowers 内部纪律。**

### 3.2 角色分工

| 系统 | 角色 | 是否可生成正式收口 | 是否可推进 Aria 状态机 |
|------|------|--------------------|------------------------|
| **Aria** | 唯一正式输入输出与运行收口系统 | 是 | 是，且只能由 daemon 推进 |
| **OpenSpec** | 全流程约束与覆盖校验框架 | 否 | 否，只能限制能否推进 |
| **Superpowers** | Claude Code / Codex 的内部工作纪律 | 否 | 否，只能影响 provider 内部工作方式 |

### 3.3 核心原则

1. **所有正式输入输出物都归 Aria。**
2. **OpenSpec 不替代 Aria 正式产物，只提供约束和覆盖校验。**
3. **Superpowers 不直接形成系统真相源，只形成 provider run 内部纪律。**
4. **Claude Code / Codex 只能产出候选结果，真正落盘与推进由 daemon 完成。**

## 4. 正式产物、约束物、工作纪律

### 4.1 Aria 正式产物

Aria 正式产物分为两类：

| 类型 | 用途 | 是否可驱动节点推进 |
|------|------|--------------------|
| **canonical artifact** | 用户可读文档、结构化记录、运行报告 | 是 |
| **runtime artifact** | checkpoint、snapshot、gate、queue、provider run record | 是 |

正式产物示例：

- `intake_brief`
- `clarification_record`
- `spec`
- `spec_gate_decision`
- `design`
- `design_review`
- `design_revision_record`
- `readiness_check`
- `plan`
- `dispatch_package`
- `coding_report`
- `testing_report`
- `code_review_report`
- `integration_report`
- `final_review`
- `final_summary`
- `runtime_snapshot`

### 4.2 OpenSpec 约束物

OpenSpec 在一期不作为正式产物源，而作为**约束物来源**。

OpenSpec 原始文件：

- `proposal.md`
- `specs/**/spec.md`
- `design.md`
- `tasks.md`

Aria 不直接以这些 Markdown 作为 runtime truth，而是将它们归一化为 `OpenSpecConstraintBundle`，供后续节点校验使用。

### 4.3 Superpowers 纪律物

Superpowers 的作用是约束 Claude Code / Codex 在 provider run 内部如何工作，例如：

- `brainstorming`
- `writing-plans`
- `test-driven-development`
- `verification-before-completion`
- `systematic-debugging`
- `receiving-code-review`

这些 skill 会影响 provider run 的内部过程，但不会直接形成 Aria 正式产物。

### 4.4 provider raw output

provider raw output 包括：

- stdout
- stderr
- structured extraction
- 中间文件
- 计划草稿
- skill 运行过程中生成的候选文档

它们统一登记为 `ExternalArtifactRef`，在经过归一化、校验、落盘前，不得驱动下一节点。

## 5. 用户关心内容在 Aria 中的物理落位

### 5.1 story / design 内容必须物理存在于 Aria

下表定义用户最关心的内容，在 Aria 中必须如何物理存在：

| 内容 | Aria 正式落位 | 说明 |
|------|---------------|------|
| 用户故事 | `spec.user_stories` | 正式需求定义 |
| 功能需求 | `spec.functional_requirements` | 正式需求定义 |
| 成功标准 | `spec.success_criteria` | 正式验收入口 |
| 待确认项 | `spec.open_items` | 需求未决项 |
| 非功能需求 | `spec.non_functional_requirements` | 性能、安全、可用性等 |
| 数据模型 | `design.data_model` | 正式设计内容 |
| API 契约 | `design.api_contracts` | 正式设计内容 |
| 公共组件 | `design.frontend_design.shared_components` / `design.backend_design.shared_modules` | 以 design 章节或结构化子字段落位 |
| 风险点 | `design.risk_list` + `riskRegistry` | 设计层与运行层双落位 |
| 设计缺口 | `design.open_items` 或 `design_revision_record.remaining_risks` | 设计未决项 |
| 是否可由 agent 实现 | `plan.work_packages[].execution_mode` | 供 daemon 路由 |
| 必须人工实现的原因 | `plan.work_packages[].human_required_reason` | 供 gate / manual intervention 使用 |
| worktask 级执行方式 | `dispatch_package.worktasks[].execution_mode` | `agent_only` / `human_assisted` / `human_required` |
| 验收目标 | `dispatch_package.acceptance_targets` | 供执行、测试、评审节点消费 |

### 5.2 必须遵守的规则

1. story / design 内容必须在 Aria 正式产物中物理存在。
2. “能否交给 agent 实现”不能只写在自然语言说明里，必须进入可路由字段。
3. OpenSpec 或 Superpowers 中出现的类似内容，只能作为候选上下文，不能替代上述 Aria 正式落位。

### 5.3 展示看板 / 大屏预留规则

后续会基于 Aria 正式产物构建 story / design / execution 展示看板或大屏，因此当前产物设计必须满足“可投影展示”的要求。

**核心约束：**

1. 展示看板的数据来源必须是 Aria 正式产物和 runtime record，而不是 provider raw output。
2. 一期、二期**不实现**展示端、不新增展示专用正式产物。
3. 当前必须保证后续可以从正式产物中稳定提取展示所需信息。

**后续展示的数据来源约定：**

| 展示主题 | 未来展示数据来源 | 当前必须保证的结构 |
|----------|------------------|--------------------|
| story 看板 | `spec.user_stories`, `functional_requirements`, `success_criteria`, `open_items` | 稳定字段名、稳定顺序、可追踪 requirement IDs |
| design 看板 | `design.data_model`, `api_contracts`, `frontend_design`, `backend_design`, `risk_list` | 组件边界、接口、风险条目可结构化读取 |
| plan / task 看板 | `plan.work_packages`, `dispatch_package.worktasks`, `acceptance_targets` | 每个 work item 有稳定 ID、状态、执行模式、验收目标 |
| 执行状态大屏 | `coding_report`, `testing_report`, `code_review_report`, `integration_report`, `final_review` | report 可映射到 task / requirement / design decision |
| runtime 运行大屏 | `runtime_snapshot`, `ProviderRunRecord`, gate / queue / traceability bindings | 必须保留时间戳、状态、节点、关联关系 |

**当前落地要求：**

1. 所有未来可能上屏的核心对象都必须有稳定 ID。
2. 关键对象必须有稳定状态字段、时间戳字段、关联字段。
3. `ArtifactTraceabilityBinding` 必须足以支持未来做“需求 -> 设计 -> 任务 -> 执行结果”的链路展示。
4. 后续展示层应基于 Aria 正式产物构建只读 projection / read model，而不是回头解析 Markdown 原文或 CLI 输出。

## 6. 核心对象模型

### 6.1 `CanonicalNodeInput`

作用：定义当前节点进入执行时，Aria 已正式确认的输入真相。

最小字段：

- `sessionId`
- `taskId`
- `nodeId`
- `phase`
- `effectivePolicy`
- `artifactRefs`
- `worktreeRef`
- `riskRegistryRef`
- `loopCounters`
- `acceptanceTargets`
- `inputValidationRefs`

### 6.2 `OpenSpecConstraintBundle`

作用：把 OpenSpec 原始文档编译成 Aria 可消费的结构化约束包。

最小字段：

- `constraintBundleId`
- `changeId`
- `proposalConstraints`
- `requirementConstraints`
- `designConstraints`
- `taskConstraints`
- `traceabilityRequirements`
- `sourceOpenSpecRefs`
- `compiledAt`
- `compiledByNode`

约束说明：

1. `proposalConstraints` 约束 scope / non-goals / business intent。
2. `requirementConstraints` 约束 requirement IDs / scenario IDs / success criteria。
3. `designConstraints` 约束 design decisions / component boundaries / risk IDs。
4. `taskConstraints` 约束 task IDs / task sequence / acceptance mappings。

### 6.3 `NodeExecutionContract`

作用：定义某个节点如何驱动 provider。

最小字段：

- `nodeId`
- `providerType`
- `role`
- `requiredCanonicalInputs`
- `requiredConstraintKinds`
- `allowedExternalInputs`
- `allowedWriteScope`
- `allowedCommandClasses`
- `forbiddenActions`
- `expectedOutputs`
- `failureRoutes`
- `advisoryOnly`

### 6.4 `WorkflowDisciplineSpec`

作用：定义某个节点内 Claude Code / Codex 必须如何使用 Superpowers。

最小字段：

- `nodeId`
- `superpowersRequired`
- `superpowersOptional`
- `enforcementLevel`
- `workflowViolationRoute`

### 6.5 `ProviderContextPackage`

作用：Aria 实际发给 Claude Code / Codex 的执行包。

定义：

`ProviderContextPackage = CanonicalNodeInput + OpenSpecConstraintBundle + NodeExecutionContract + WorkflowDisciplineSpec + NodePromptTemplateRef`

最小字段：

- `providerType`
- `role`
- `nodeId`
- `canonicalInputs`
- `constraintBundle`
- `workflowDiscipline`
- `worktreePath`
- `allowedWriteScope`
- `instructions`
- `outputSchemaRef`
- `completionCriteria`
- `forbiddenActions`
- `verificationCommands`

### 6.6 `ProviderRunRecord`

作用：记录一次 provider 调用的完整审计信息。

最小字段：

- `providerRunId`
- `nodeId`
- `providerType`
- `role`
- `contextPackageRef`
- `adapterInputRef`
- `exitCode`
- `stdoutRef`
- `stderrRef`
- `structuredOutputRef`
- `filesModified`
- `durationMs`
- `timeoutStatus`
- `retryCount`
- `approvalPolicy`
- `sandboxMode`

### 6.7 `ArtifactTraceabilityBinding`

作用：把 Aria 正式产物与 OpenSpec 约束对象、provider 结果建立可追踪绑定。

最小字段：

- `bindingId`
- `canonicalArtifactRef`
- `constraintBundleRef`
- `relatedRequirementIds`
- `relatedDesignDecisionIds`
- `relatedTaskIds`
- `providerRunRefs`
- `bindingStatus`

## 7. Aria 与用户的交互契约

### 7.1 交互原则

1. 用户只直接与 REPL 交互。
2. REPL 只负责输入、展示和回复 gate，不持有 runtime truth。
3. daemon 是唯一运行时真相源。

### 7.2 最小命令面

一期至少实现以下语义命令：

| 命令语义 | 作用 |
|----------|------|
| `new <request>` | 创建新的 EpicTask |
| `status` | 查看 session、task、phase、queue、gate、provider run |
| `artifacts` | 查看正式产物、约束包、traceability bindings |
| `approve <gateId>` | 通过 gate |
| `reject <gateId>` | 拒绝 gate |
| `reply <gateId> <text>` | 对澄清或人工介入补充说明 |
| `attach` | attach 到现有 daemon |
| `detach` | 断开当前 REPL，不终止 daemon |

### 7.3 REPL <-> daemon 消息分类

一期消息至少分三类：

| 类型 | 用途 |
|------|------|
| `request` | REPL 发起命令 |
| `response` | daemon 对命令同步响应 |
| `event` | daemon 异步推送 task/gate/provider run 状态变化 |

### 7.4 用户交互的正式收口

用户输入只有在形成下列正式产物后，才算进入系统：

- `intake_brief`
- `clarification_record`
- `spec`
- `design`
- `plan`
- `final_summary`

用户在 REPL 中看到的任何摘要，都必须来自正式产物或 runtime record，而不是临时聊天输出。

## 8. Aria 驱动 Claude Code / Codex 的统一执行链

### 8.1 统一执行链

对于所有 Agent 节点，Aria 必须按以下固定链路执行：

1. daemon 读取当前 `nodeId`
2. 查询 `NodeExecutionContract`
3. 组装 `CanonicalNodeInput`
4. 编译或读取当前 `OpenSpecConstraintBundle`
5. 解析 `WorkflowDisciplineSpec`
6. 组装 `ProviderContextPackage`
7. 映射为 `AdapterInput`
8. 通过 `ProviderAdapter` 调用 Claude Code 或 Codex
9. 收集 `ProviderRunRecord`
10. 将 raw output 登记为 `ExternalArtifactRef`
11. 提取候选输出
12. 执行 `artifact_validate`
13. 执行 OpenSpec 约束校验
14. 写入 Aria 正式产物
15. 写 checkpoint / runtime snapshot
16. 根据 `failureRoutes` 或协议路由推进节点

### 8.2 不允许发生的事

1. provider 直接推进 Aria 状态机。
2. provider 直接决定节点完成。
3. OpenSpec 原始 Markdown 直接作为 runtime truth。
4. skill 中间过程直接作为正式产物。

## 10. OpenSpec 作为全流程约束器

### 10.1 OpenSpec 的准确用途

OpenSpec 在一期的准确定位是：

1. 约束 scope，不允许需求扩散。
2. 约束 requirement coverage，不允许实现漏项。
3. 约束 design boundary，不允许设计跑偏。
4. 约束 task decomposition，不允许 Aria 拆出无来源工作项。
5. 约束 final coverage，不允许收口时留下悬空 requirement / task / risk。

### 10.2 OpenSpec 不做的事

1. 不直接决定 `Nxx -> Nyy`。
2. 不直接宣布任务完成。
3. 不替代 Aria 正式产物。

### 10.3 OpenSpec 编译规则

OpenSpec 原始文档不应在每个节点中原样全文喂给 provider。  
一期应在 `N01-N03` 或每次关键规划节点进入前，将 OpenSpec 原始文档编译为 `OpenSpecConstraintBundle`。

编译结果至少包含：

- `scope_constraints`
- `non_goal_constraints`
- `requirement_ids`
- `scenario_ids`
- `design_decision_ids`
- `risk_ids`
- `task_ids`
- `task_acceptance_map`
- `traceability_map`

## 11. 阶段级收口规则

### 11.1 统一原则

任何阶段收口，都必须同时通过两层检查：

1. **Aria 正式产物校验**
2. **OpenSpec 约束覆盖校验**

只有双校验都通过，daemon 才能推进状态。

### 11.2 需求阶段收口

正式收口物：

- `spec`
- `spec_gate_decision`

收口条件：

1. `spec` 通过 Aria schema 校验
2. `spec_gate_decision` 已生成
3. `spec.user_stories`、`functional_requirements`、`success_criteria` 覆盖 OpenSpec `proposalConstraints` / `requirementConstraints`
4. `open_items` 中的剩余未决项符合策略要求
5. checkpoint 写入完成

### 11.3 设计阶段收口

正式收口物：

- `design`
- `design_review`
- 必要时 `design_revision_record`

收口条件：

1. `design` 通过 Aria schema 校验
2. `design_review` 为 `pass` 或 `conditional_pass`
3. `design.data_model`、`api_contracts`、`shared_components`、`risk_list` 覆盖 OpenSpec design / requirement constraints
4. 风险已进入 `riskRegistry`
5. checkpoint 写入完成

### 11.4 计划阶段收口

正式收口物：

- `readiness_check`
- `plan`
- `dispatch_package`

收口条件：

1. `plan` 和 `dispatch_package` 通过 Aria schema 校验
2. `plan.work_packages` 全量映射到 OpenSpec `task_ids`
3. `dispatch_package.worktasks` 不得存在无来源工作项
4. 每个 `worktask` 必须绑定 requirement / task / acceptance target
5. checkpoint 写入完成

### 11.5 WorkTask 执行阶段收口

正式收口物：

- `coding_report`
- `testing_report`
- `code_review_report`
- `N20 runtime_snapshot`

收口条件：

1. 三个 report 都通过 Aria schema 校验
2. 三个 report 都能回溯到对应 requirement / design decision / task
3. 未越权修改 worktree 外文件
4. `N20` 判定 ready
5. checkpoint 写入完成

### 11.6 集成阶段收口

正式收口物：

- `integration_report`

收口条件：

1. `integration_report` 通过校验
2. `nextDecision` 明确
3. 若 `verificationStatus = fail`，必须明确回流、回滚或人工介入路径
4. checkpoint 写入完成

### 11.7 最终阶段收口

正式收口物：

- `final_review`
- `final_summary`
- `N28 runtime_snapshot`

收口条件：

1. `final_review.overallDecision = pass`
2. `final_summary` 通过校验
3. 所有 OpenSpec constraints 均已被覆盖、豁免或显式转人工
4. 不存在悬空 requirement / task / risk
5. `N28` checkpoint 写入完成

## 12. 输入输出物关系矩阵

### 12.1 总矩阵

| 产物类型 | 生产者 | 主要用途 | 可否指导下一步 | 可否作为正式收口 | 可否直接推进状态机 |
|----------|--------|----------|----------------|------------------|--------------------|
| Aria canonical artifact | Aria + provider 归一化后 | 正式输入输出 | 是 | 是 | 是，由 daemon 推进 |
| Aria runtime artifact | daemon | checkpoint / gate / queue / snapshot / run record | 是 | 是 | 是 |
| OpenSpec 原始文档 | 用户 / Aria 辅助生成 | 约束来源 | 否，需先编译 | 否 | 否 |
| OpenSpecConstraintBundle | daemon 编译 | 约束校验 | 是，作为约束输入 | 否 | 否 |
| Superpowers 中间产物 | provider run 内部 | 工作纪律辅助 | 否，需先归一化 | 否 | 否 |
| provider raw output | Claude / Codex | 候选输出 | 否，需先归一化 | 否 | 否 |

### 12.2 组合规则

1. **方案指导输入**  
   `Aria canonical artifacts + OpenSpecConstraintBundle + WorkflowDisciplineSpec`
2. **正式收口输入**  
   `Aria 正式产物 + OpenSpec 覆盖校验结果 + checkpoint`
3. **禁止的组合**  
   - OpenSpec 原始 Markdown 直接驱动下一节点
   - skill 中间过程直接作为正式收口
   - raw stdout 直接作为 canonical artifact

### 12.3 追踪链

一期必须形成以下追踪链：

`proposal / requirement -> spec -> design -> plan -> dispatch_package -> worktask -> coding/testing/review -> integration -> final_review/final_summary`

任何正式收口节点都必须能回答：

1. 当前产物覆盖了哪些 requirement IDs？
2. 当前产物关联了哪些 design decision IDs？
3. 当前产物完成了哪些 task IDs？
4. 当前产物处理了哪些 risk IDs？
