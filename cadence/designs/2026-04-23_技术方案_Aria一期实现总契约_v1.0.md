## 1. 设计目标

本文件只解决一件事：把已有 Aria 协议文档拼成一份**研发可直接落地**的实现契约。

本文件必须明确以下问题：

1. Aria 如何与用户交互，直到形成正式输入、正式输出和正式收口。
2. Aria 如何驱动 Claude Code / Codex，明确到开发人员可以实现 `contract registry`、`context builder`、`provider router`。
3. Claude Code / Codex 如何使用 Superpowers 和 OpenSpec，明确到开发人员可以实现节点级约束。
4. Aria、OpenSpec、Superpowers、provider raw output 的输入输出物如何组合、追踪、校验和收口。
5. story、design、plan、risk、执行状态等内容后续将用于展示看板 / 大屏，因此当前正式产物必须预留稳定结构和追踪关系；但展示端本身不属于一期、二期实现范围。

## 2. 文档定位与裁定规则

### 2.1 双层真相源

Aria 一期采用双层真相源：

| 层级 | 作用 | 真相源 |
|------|------|--------|
| **协议真相源** | 定义 canonical 节点、产物、横切能力、枚举值、回流规则 | 上游协议文档集 |
| **实现真相源** | 定义对象模型、调用链、约束矩阵、阶段收口、模块边界 | 本文档 |

### 2.2 裁定规则

1. 若本文与上游协议文档冲突，以上游协议文档为准。
2. 若开发时发现上游协议文档在实现层缺少必要约束，以本文为实现入口，但必须记录“待回改上游文档项”。
3. 开发人员不得自行发明绕开本文和上游协议的运行路径。

### 2.3 本文不做的事

1. 不重定义 `N00-N28`、`X01-X09`、`CC01-CC10`。
2. 不重定义 canonical artifact 字段名。
3. 不取代 implementation plan。
4. 不把 OpenSpec 或 Superpowers 升格为 Aria runtime truth。

## 3. 三套系统的准确分工

### 3.1 一句话模型

**Aria 单一收口，OpenSpec 全程约束，Superpowers 内部纪律。**

### 3.2 角色分工

| 系统 | 角色 | 是否可生成正式收口 | 是否可推进 Aria 状态机 |
|------|------|--------------------|------------------------|
| **Aria** | 唯一正式输入输出与运行收口系统 | 是 | 是，且只能由 daemon 推进 |
| **OpenSpec** | 全流程约束与覆盖校验框架 | 否 | 否，只能限制能否推进 |
| **Superpowers** | Claude Code / Codex 的内部工作纪律 | 否 | 否，只能影响 provider 内部工作方式 |

### 3.3 核心原则

1. **所有正式输入输出物都归 Aria。**
2. **OpenSpec 不替代 Aria 正式产物，只提供约束和覆盖校验。**
3. **Superpowers 不直接形成系统真相源，只形成 provider run 内部纪律。**
4. **Claude Code / Codex 只能产出候选结果，真正落盘与推进由 daemon 完成。**

## 4. 正式产物、约束物、工作纪律

### 4.1 Aria 正式产物

Aria 正式产物分为三类：

| 类型 | 用途 | 是否可驱动节点推进 |
|------|------|--------------------|
| **canonical artifact** | 用户可读文档、结构化记录、运行报告 | 是 |
| **projection artifact** | 从 canonical artifact 编译出的机器可消费索引 | 是 |
| **runtime artifact** | checkpoint、snapshot、gate、queue、provider run record | 是 |

正式产物示例：

- `intake_brief`
- `clarification_record`
- `spec`
- `spec_gate_decision`
- `design`
- `design_review`
- `design_revision_record`
- `readiness_check`
- `plan`
- `dispatch_package`
- `coding_report`
- `testing_report`
- `code_review_report`
- `integration_report`
- `final_review`
- `final_summary`
- `runtime_snapshot`

一期还必须额外落盘以下 projection artifact：

- `SpecProjection`
- `DesignProjection`
- `PlanProjection`

一期还必须支持 JSON artifact 的 `_aria` 扩展命名空间：

- `_aria.profile_version`
- `_aria.constraint_check_ref`
- `_aria.traceability_refs`
- `_aria.provider_run_refs`
- `_aria.projection_refs`

### 4.2 OpenSpec 约束物

OpenSpec 在一期不作为正式产物源，而作为**约束物来源**。

OpenSpec 原始文件：

- `proposal.md`
- `specs/main/spec.md`
- `design.md`
- `tasks.md`

Aria 不直接以这些 Markdown 作为 runtime truth，而是将它们归一化为 `OpenSpecConstraintBundle`，供后续节点校验使用。

### 4.3 Superpowers 纪律物

Superpowers 的作用是约束 Claude Code / Codex 在 provider run 内部如何工作，例如：

- `brainstorming`
- `writing-plans`
- `test-driven-development`
- `verification-before-completion`
- `systematic-debugging`
- `receiving-code-review`

这些 skill 会影响 provider run 的内部过程，但不会直接形成 Aria 正式产物。

### 4.4 provider raw output

provider raw output 包括：

- stdout
- stderr
- structured extraction
- 中间文件
- 计划草稿
- skill 运行过程中生成的候选文档

它们统一登记为 `ExternalArtifactRef`，在经过归一化、校验、落盘前，不得驱动下一节点。

## 5. 用户关心内容在 Aria 中的物理落位

### 5.1 story / design 内容必须物理存在于 Aria

下表定义用户最关心的内容，在 Aria 中必须如何物理存在：

| 内容 | Aria 正式落位 | 说明 |
|------|---------------|------|
| 用户故事 | `spec` + `SpecProjection.user_stories[]` | 文档保留在 `spec`，机器索引在 projection |
| 功能需求 | `spec` + `SpecProjection.functional_requirements[]` | requirement ID 由 projection 固化 |
| 成功标准 | `spec` + `SpecProjection.success_criteria[]` | 正式验收入口 |
| 待确认项 | `spec` + `SpecProjection.open_items[]` | 需求未决项 |
| 非功能需求 | `spec` + `SpecProjection.non_functional_requirements[]` | 性能、安全、可用性等 |
| 数据模型 | `design` + `DesignProjection.data_entities[]` | 文档与结构化索引双落位 |
| API 契约 | `design` + `DesignProjection.api_entries[]` | 文档与结构化索引双落位 |
| 公共组件 | `design` + `DesignProjection.shared_components[]` / `shared_modules[]` | 规划文档 + 机器索引 |
| 风险点 | `design.risk_list` + `DesignProjection.risk_refs[]` + `risk_registry` | 设计层与运行层双落位 |
| 设计缺口 | `design_revision_record.remaining_risks` + `DesignProjection.open_items[]` | 设计未决项 |
| 是否可由 agent 实现 | `PlanProjection.work_packages[].execution_mode` + `dispatch_package._aria.worktask_routing[]` | 供 daemon 路由 |
| 必须人工实现的原因 | `PlanProjection.work_packages[].human_required_reason` + `dispatch_package._aria.worktask_routing[]` | 供 gate / manual intervention 使用 |
| worktask 级执行方式 | `dispatch_package._aria.worktask_routing[]` | `agent_only` / `human_assisted` / `human_required` |
| 验收目标 | `dispatch_package.acceptance_targets` + `PlanProjection.work_packages[].acceptance_targets[]` | 供执行、测试、评审节点消费 |

### 5.2 必须遵守的规则

1. story / design 内容必须在 Aria 正式产物中物理存在。
2. “能否交给 agent 实现”不能只写在自然语言说明里，必须进入可路由字段。
3. Markdown canonical artifact 的机器索引必须通过 projection 提供，禁止让 daemon 或 dashboard 直接回头解析 Markdown 原文。
4. JSON artifact 的 routing / traceability / coverage 元数据一律放在 `_aria` 命名空间，禁止把实现扩展字段混入 canonical 最小字段集合。
5. OpenSpec 或 Superpowers 中出现的类似内容，只能作为候选上下文或约束来源，不能替代上述 Aria 正式落位。

### 5.3 展示看板 / 大屏预留规则

后续会基于 Aria 正式产物构建 story / design / execution 展示看板或大屏，因此当前产物设计必须满足“可投影展示”的要求。

**核心约束：**

1. 展示看板的数据来源必须是 Aria 正式产物和 runtime record，而不是 provider raw output。
2. 一期、二期**不实现**展示端、不新增展示专用正式产物。
3. 当前必须保证后续可以从正式产物中稳定提取展示所需信息。

**后续展示的数据来源约定：**

| 展示主题 | 未来展示数据来源 | 当前必须保证的结构 |
|----------|------------------|--------------------|
| story 看板 | `SpecProjection` | 稳定字段名、稳定顺序、可追踪 requirement IDs |
| design 看板 | `DesignProjection` | 组件边界、接口、风险条目可结构化读取 |
| plan / task 看板 | `PlanProjection` + `dispatch_package._aria.worktask_routing[]` | 每个 work item 有稳定 ID、状态、执行模式、验收目标 |
| 执行状态大屏 | `coding/testing/review/integration/final` report 的 `_aria.traceability_refs` | report 可映射到 task / requirement / design decision |
| runtime 运行大屏 | `runtime_snapshot`, `ProviderRunRecord`, gate / queue / traceability bindings | 必须保留时间戳、状态、节点、关联关系 |

**当前落地要求：**

1. 所有未来可能上屏的核心对象都必须有稳定 ID。
2. 关键对象必须有稳定状态字段、时间戳字段、关联字段。
3. `ArtifactTraceabilityBinding` 必须足以支持未来做“需求 -> 设计 -> 任务 -> 执行结果”的链路展示。
4. 后续展示层应基于 Aria 正式产物构建只读 projection / read model，而不是回头解析 Markdown 原文或 CLI 输出。

### 5.4 文档操作与结构化编辑规则

Aria 一期必须具备自己的 Document Operation 层。该层负责创建、更新、归一化和校验 Aria canonical artifacts 与 OpenSpec 文档；Claude Code / Codex 只能输出候选内容，不能直接把 provider raw output 变成正式文档。

文档操作分工如下：

| 文档/文件类型 | 主处理方式 | 可选辅助工具 | 不允许的方式 |
|---------------|------------|--------------|--------------|
| Markdown canonical artifacts（`spec`、`design`、`plan`、`final_summary`） | Markdown AST / heading model | Markdown formatter | 正则或裸字符串替换核心章节 |
| OpenSpec Markdown（`proposal.md`、`design.md`、`tasks.md`） | Markdown AST / heading model | Markdown formatter | 直接拼接全文后跳过结构校验 |
| JSON artifacts 与 `_aria` | serde 类型、JSON schema、结构化 patch | JSON diff / patch | 手写字符串拼 JSON |
| YAML / JSON 配置或规则文件 | serde YAML/JSON parser | ast-grep（若语言支持且有规则） | 无 schema 的文本替换 |
| 代码文件 | 语言 parser / 编译器 / 测试 | ast-grep 结构搜索与重写 | 只靠 provider 文字说明改代码 |

Document Operation 层必须提供以下能力：

1. `read_document_model(path)`：把 Markdown 文档解析成章节、列表、表格、代码块等结构模型。
2. `upsert_section(path, heading_path, content)`：按标题路径插入或替换章节，保留非目标章节。
3. `extract_projection_source(path)`：为 projection compiler 提供稳定结构化输入。
4. `apply_json_patch(path, patch)`：对 JSON artifact / `_aria` 做结构化更新。
5. `write_with_hash(path, content)`：写入后生成 hash，供 artifact version、source manifest 和 traceability 使用。

ast-grep 使用边界：

1. ast-grep 是可选 tool adapter，不是 Aria runtime truth。
2. ast-grep 适合在 worktree 内做代码结构搜索、lint、codemod，也可用于官方支持语言中的 JSON/YAML 等结构化查询。
3. ast-grep 不作为 Markdown canonical artifact 的主编辑引擎；Markdown 正文结构仍以 Markdown AST / heading model 为准。
4. 若本机未安装 ast-grep，Aria 不应阻断一期主流程；只记录 capability missing，并回退到对应语言的基础 parser 或人工介入。

## 6. 核心对象模型

### 6.1 `CanonicalNodeInput`

作用：定义当前节点进入执行时，Aria 已正式确认的输入真相。

最小字段：

- `session_id`
- `task_id`
- `node_id`
- `phase`
- `effective_policy`
- `artifact_refs`
- `external_refs`
- `worktree_ref`
- `risk_registry_ref`
- `loop_counters`
- `acceptance_targets`
- `input_validation_refs`

字段必填规则：

| 字段 | 必填规则 | 说明 |
|------|----------|------|
| `session_id` / `task_id` / `node_id` / `phase` | 必填 | 当前节点执行边界 |
| `effective_policy` | 必填 | 已解析后的运行策略，不允许 provider 自行解释原始配置 |
| `artifact_refs` | 必填 | 当前节点允许消费的 Aria canonical artifact refs，不得包含 `superseded` |
| `external_refs` | 可选 | 已登记的 OpenSpec / Superpowers / provider raw 外部引用，只能作为候选上下文 |
| `worktree_ref` | 条件必填 | `N16-N24` 必填；规划阶段可为空 |
| `risk_registry_ref` | 必填 | 当前 Risk Registry sidecar 快照引用；使用独立 `RiskRegistryRef`，不复用 `ArtifactRef` |
| `loop_counters` | 必填 | 包含 `clarification_round_counter`、`design_revision_counter`、`rework_counter`、`patch_round_counter` 等命名计数器 |
| `acceptance_targets` | 条件必填 | `N16-N25` 必填；规划阶段可为空数组 |
| `input_validation_refs` | 必填 | 节点进入前输入校验记录 |

实现定稿规则：

1. `CanonicalNodeInput` 的 Rust 类型和 JSON schema 必须以本节为准，同时兼容 IO 协议中的 `external_refs` 字段。
2. `external_refs` 不得提升为 `artifact_refs`，除非已经通过归一化生成 Aria canonical artifact。
3. `artifact_refs`、`worktree_ref`、`risk_registry_ref` 必须使用结构化 ref 类型，不得只用裸路径字符串。
4. Risk Registry 是 task 级 runtime sidecar，当前快照仍进入 `RuntimeSnapshot.risk_registry`；不得把 Risk Registry 临时加入 `ArtifactKind` 枚举来规避类型缺口。

### 6.2 `ArtifactProjectionRecord`

作用：把 Markdown canonical artifact 编译成机器可消费的结构化 projection，供 routing、coverage、traceability、dashboard 使用。

最小字段：

- `projection_id`
- `projection_kind`：`spec_projection` / `design_projection` / `plan_projection`
- `source_artifact_ref`
- `source_artifact_version`
- `source_artifact_hash`
- `compiled_at`
- `compiled_by_node`
- `payload`

一期固定 payload：

1. `SpecProjection`
   - `user_stories[]`：`story_id`、`title`、`related_requirement_ids`
   - `functional_requirements[]`：`requirement_id`、`text`、`priority`
   - `success_criteria[]`：`criterion_id`、`text`、`related_requirement_ids`
   - `open_items[]`：`item_id`、`text`、`resolution_mode`
   - `non_functional_requirements[]`
2. `DesignProjection`
   - `design_decisions[]`：`decision_id`、`summary`、`related_requirement_ids`
   - `shared_components[]` / `shared_modules[]`
   - `data_entities[]`
   - `api_entries[]`
   - `risk_refs[]`
   - `open_items[]`
3. `PlanProjection`
   - `work_packages[]`：`work_package_id`、`description`、`execution_mode`、`human_required_reason`、`traceability_refs`、`acceptance_targets`
   - `dependencies[]`
   - `parallelism_groups[]`

Projection ID 与编译规则：

1. `projection_id` 格式固定为 `proj_<projection_kind>_<sourceArtifactId>_<sequence>`。
2. `source_artifact_hash` 使用 source artifact 正文的 SHA-256。
3. `compiled_by_node` 必须写入触发编译的协议节点 ID，例如 `N05`、`N07`、`N11`。
4. 同一 source artifact 内容不变时，重复编译必须生成等价 `payload`；若 source artifact 版本或 hash 变化，必须生成新 projection record。
5. projection compiler 可以使用 Markdown parser，但 compiler 输出必须由 fixture golden test 固化，后续 routing / coverage 不得再次解析 Markdown 原文。

#### 6.2.1 JSON Artifact `_aria` 扩展对象

作用：为 JSON canonical artifact 提供统一的 phase1 implementation profile 元数据。

通用字段：

- `profile_version`
- `constraint_check_ref`
- `traceability_refs`
- `provider_run_refs`
- `projection_refs`

`dispatch_package._aria` 额外字段：

- `worktask_routing[]`
  - `worktask_id`
  - `source_work_package_id`
  - `execution_mode`
  - `human_required_reason`
  - `allowed_write_scope`
  - `traceability_refs`

`final_review._aria` 额外字段：

- `coverage_summary`
- `uncovered_items`
- `manual_exemptions`

### 6.3 `OpenSpecConstraintBundle`

作用：把 OpenSpec 原始文档编译成 Aria 可消费的结构化约束包。

最小字段：

- `constraint_bundleId`
- `bundleVersion`
- `bundle_status`：`bootstrap_pending` / `ready` / `stale` / `blocked`
- `change_id`
- `proposal_constraints`
- `requirement_constraints`
- `design_constraints`
- `task_constraints`
- `traceability_requirements`
- `coverage_model`
- `source_manifest`
- `compiled_from_projection_refs`
- `compiled_at`
- `compiled_by_node`

约束说明：

1. `proposal_constraints` 约束 scope / non-goals / business intent。
2. `requirement_constraints` 约束 requirement IDs / scenario IDs / success criteria。
3. `design_constraints` 约束 design decisions / component boundaries / risk IDs。
4. `task_constraints` 约束 task IDs / task sequence / acceptance mappings。
5. `source_manifest` 必须记录每个 OpenSpec 文件的 `path`、`kind`、`sha256`，用于失效检测。

#### 6.3.1 `OpenSpecConstraintBundle` 序列化字段裁定

一期实现中，`OpenSpecConstraintBundle` 的 JSON 顶层字段名以 6.3 的最小字段为准，固定使用 snake_case：

- `proposal_constraints`
- `requirement_constraints`
- `design_constraints`
- `task_constraints`
- `traceability_requirements`
- `coverage_model`

实现中可以存在 `scope_constraints`、`requirement_ids`、`task_ids`、`traceability_map` 等内部 helper 输出，但这些名称不得替代 bundle 的顶层序列化字段名。

固定规则：

1. Rust struct、JSON schema、fixture golden JSON 必须共享同一套字段名。
2. `scope_constraints` 等 snake_case 名称只能作为子字段或内部中间模型，不作为跨模块契约。
3. P2 必须提供 `openspec_bundle_schema` 测试，锁定序列化字段名，避免 P3/P4 按不同命名消费 bundle。

### 6.4 `NodeExecutionContract`

作用：定义某个节点如何驱动 provider。

最小字段：

- `node_id`
- `provider_type`
- `runtime_role`：`orchestrator` / `executor` / `reviewer` / `advisory_reviewer`
- `adapter_role`：`orchestrator` / `executor` / `reviewer`
- `required_canonical_inputs`
- `required_projection_kinds`
- `required_constraint_kinds`
- `allowed_external_inputs`
- `allowed_write_scope`
- `allowed_command_classes`
- `forbidden_actions`
- `expected_outputs`

## 7. Aria 与用户的交互契约

### 7.1 交互原则

1. 用户只直接与 REPL 交互。
2. REPL 只负责输入、展示和回复 gate，不持有 runtime truth。
3. daemon 是唯一运行时真相源。

### 7.2 Envelope 定义

一期所有 REPL <-> daemon 消息都使用同一套 JSON envelope：

| 字段 | 必填 | 说明 |
|------|------|------|
| `protocol_version` | 是 | 一期固定为 `aria.repl.v1` |
| `message_type` | 是 | `request` / `response` / `event` |
| `request_id` | `request/response` 必填 | request-response 相关 ID |
| `event_id` | `event` 必填 | 单调递增事件 ID |
| `command` | `request/response` 必填 | 请求命令名；response 必须回显请求命令 |
| `event_type` | `event` 必填 | 事件类型，例如 `task.created` |
| `daemon_session_id` | 否 | daemon 当前会话 ID |
| `task_id` | 否 | 关联 task |
| `node_id` | 否 | 关联 node |
| `sent_at` | 是 | ISO 8601 时间戳 |
| `occurred_at` | `event` 必填 | 事件实际发生时间；response 不使用该字段 |
| `ok` | `response` 必填 | 成功为 `true`，失败为 `false`；request/event 不使用该字段 |
| `payload` | 是 | 业务负载 |
| `error` | 否 | 仅失败 response、event 使用 |

Envelope 判定规则：

1. `request` 必须同时具备 `request_id`、`command`、`payload`。
2. `response` 必须同时具备 `request_id`、`command`、`ok`；`ok=false` 时必须包含 `error`。
3. `event` 必须同时具备 `event_id`、`event_type`、`occurred_at`；`event_id` 在 daemon session 内单调递增。
4. `sent_at` 表示消息发送时间，`occurred_at` 表示事件发生时间；两者不得混用。
5. REPL / daemon 的 schema round-trip 测试必须覆盖成功 response、失败 response、event 三类 envelope。

`task_id` / `node_id` 的裁定规则：

1. envelope 层保持可选，因为 session 级事件或系统级错误可能没有具体 task 或 node。
2. `7.5` 表中列出的 task / node 相关事件，若语义上关联 task 或 node，则必须在 envelope 或 payload 中提供对应 ID。
3. P1 测试必须覆盖：session 级 event 可不带 `task_id/node_id`；`task.created`、`task.phase_changed`、`provider_run.started` 等事件必须能定位到 task 或 node。

### 7.3 命令面与 payload

一期必须实现以下命令：

| `command` | 请求 `payload.args` | 成功响应 `payload` |
|-----------|--------------------|------------------|
| `hello` | `client_id`、`client_capabilities`、`last_seen_event_id?` | `daemon_session_id`、`supported_commands`、`heartbeat_interval_ms`、`reconnect_token` |
| `attach` | `reconnect_token?`、`requested_task_id?` | `attached=true`、`visible_tasks[]`、`open_gates[]`、`latest_event_id` |
| `subscribe` | `task_ids?`、`event_types?` | `subscribed=true`、`subscription_id` |
| `new_task` | `request_text`、`requested_change_id?` | `task_id`、`phase`、`intake_ref`、`change_id` |
| `get_status` | `task_id?` | `session`、`tasks[]`、`queue`、`gates[]`、`provider_runs[]` |
| `list_artifacts` | `task_id`、`artifact_types?` | `canonical_artifacts[]`、`projections[]`、`constraint_bundles[]`、`traceability_bindings[]` |
| `approve_gate` | `gate_id`、`comment?` | `gate_status=approved`、`next_route` |
| `reject_gate` | `gate_id`、`comment?` | `gate_status=rejected`、`next_route` |
| `reply_gate` | `gate_id`、`reply_text` | `gate_status`、`reply_ref` |
| `detach` | 空 | `detached=true` |

命令 payload 固定结构：

```json
{
  "command": "get_status",
  "args": {}
}
```

1. request envelope 的 `command` 字段与 `payload.command` 必须一致。
2. response envelope 的 `payload` 只包含命令结果，不再重复包一层 `command`。
3. 非法命令返回 `ok=false`，`error.code=unsupported_command`。
4. 参数字段缺失或类型错误返回 `ok=false`，`error.code=invalid_request`。

CLI 命令语义与 `command` 的对应关系如下：

| CLI 命令语义 | `command` |
|-------------|-----------|
| `new <request>` | `new_task` |
| `status` | `get_status` |
| `artifacts` | `list_artifacts` |
| `approve <gateId>` | `approve_gate` |
| `reject <gateId>` | `reject_gate` |
| `reply <gateId> <text>` | `reply_gate` |
| `attach` | `attach` |
| `detach` | `detach` |

一期命令 registry 裁定：

1. 上表命令是 P1 必须实现的最小命令面。
2. `import openspec <change_id>` / `export openspec <task_id>` 属于 P2 之后可实现的扩展命令，不能作为 P1 通信验收阻塞项。
3. OpenSpec 强约束不依赖 `import openspec` 命令存在；`new_task.requested_change_id?` 与 daemon 生成的 `chg_<task_id>` 已经足以完成一期 `change_id` 绑定。

### 7.4 握手、订阅与重连

一期连接建立流程固定为：

1. REPL 建立 socket / named pipe 连接。
2. REPL 发送 `hello`。
3. daemon 返回 `daemon_session_id`、`reconnect_token`、`latest_event_id`。
4. REPL 发送 `attach`。
5. daemon 返回当前可见任务、gate、最新快照摘要。
6. REPL 发送 `subscribe`，建立事件流。

重连规则：

1. REPL 必须持久化 `reconnect_token` 与 `last_seen_event_id`。
2. 断连后重新 `hello` 时带上 `last_seen_event_id`。
3. daemon 若事件仍可回放，则从 `last_seen_event_id + 1` 开始补发。
4. 若事件窗口已丢失，daemon 返回 `error.code = replay_window_lost`，REPL 必须重新请求 `get_status` 和 `list_artifacts` 做全量同步。

### 7.5 事件与错误模型

一期必须实现以下事件：

| `event_type` | 最小 payload 字段 |
|--------------|-------------------|
| `task.created` | `task_id`、`phase` |
| `task.phase_changed` | `task_id`、`from_phase`、`to_phase` |
| `artifact.materialized` | `artifact_ref`、`artifact_type`、`producer_node` |
| `projection.compiled` | `projection_ref`、`projection_kind`、`source_artifact_ref` |
| `constraint_bundle.compiled` | `constraint_bundle_ref`、`change_id`、`bundle_status` |
| `traceability.updated` | `binding_refs[]`、`coverage_status` |
| `gate.opened` | `gate_id`、`gate_type`、`blocking_node` |
| `gate.resolved` | `gate_id`、`resolution`、`next_route` |
| `provider_run.started` | `provider_run_id`、`node_id`、`provider_type` |
| `provider_run.completed` | `provider_run_id`、`exit_code`、`duration_ms` |
| `provider_run.failed` | `provider_run_id`、`error_code`、`retryable` |

一期必须实现以下错误码：

- `invalid_request`
- `unsupported_command`
- `not_attached`
- `task_not_found`
- `gate_not_found`
- `state_conflict`
- `constraint_violation`
- `provider_busy`
- `replay_window_lost`
- `internal_error`

错误对象固定字段：

- `code`
- `message`
- `retryable`
- `details`

### 7.6 用户交互的正式收口

用户输入只有在形成下列正式产物后，才算进入系统：

- `intake_brief`
- `clarification_record`
- `spec`
- `design`
- `plan`
- `final_summary`

用户在 REPL 中看到的任何摘要，都必须来自正式产物或 runtime record，而不是临时聊天输出。

## 8. Aria 驱动 Claude Code / Codex 的统一执行链

### 8.0 `N00-N03` 入口链

Agent 节点执行链开始前，daemon 必须先完成入口链：

1. `N00 session_bootstrap`：创建或恢复 daemon session，初始化 event log、gate registry、task registry，并写入 `N00` runtime snapshot。
2. `N01 intake_capture`：把 `new_task.request_text` 归一化为 `intake_brief`，记录 `origin_type=user_repl`，并返回 `intake_ref`。
3. `N02 epic_task_create`：创建 EpicTask runtime state，绑定 `task_id`、初始 phase、`intake_ref`。
4. `N03 policy_resolve`：计算 `effective_policy`，解析或生成 `change_id`，并写入 task runtime state。

`change_id` 规则：

1. 若用户通过 `new_task.requested_change_id` 提供显式 ID，daemon 必须校验并固化。
2. 若用户未提供，则 daemon 生成 `chg_<task_id>`。
3. P1 负责固化 `change_id` 与 `openspec_bootstrap_status=bootstrap_pending`。
4. P2 在 Document Operation 层可用后负责创建 OpenSpec skeleton、更新 `openspec_bootstrap_status=bootstrapped` 并编译 `OpenSpecConstraintBundle`。
5. `change_id` 一旦进入 task runtime state，后续节点不得替换；如需更换，必须创建新的 EpicTask 或走人工介入。

### 8.1 统一执行链

对于所有 Agent 节点，Aria 必须按以下固定链路执行：

1. daemon 读取当前 `node_id`
2. 查询 `NodeExecutionContract`
3. 组装 `CanonicalNodeInput`
4. 读取或编译当前所需 `ArtifactProjectionRecord`
5. 编译或读取当前 `OpenSpecConstraintBundle`
6. 解析 `WorkflowDisciplineSpec`
7. 解析 `NodePromptTemplateRef`
8. 组装 `ProviderContextPackage`
9. 映射为 `AdapterInput`
10. 通过 `ProviderAdapter` 调用 Claude Code 或 Codex
11. 收集 `ProviderRunRecord`
12. 将 raw output 登记为 `ExternalArtifactRef`
13. 提取候选输出
14. 执行 canonical `artifact_validate`
15. 执行 phase1 profile validator（projection / `_aria` / traceability）
16. 执行 OpenSpec 约束校验
17. 写入 Aria 正式产物；必要时编译/更新 projection 与 `_aria`
18. 写 checkpoint / runtime snapshot
19. 根据 `failure_routes` 或协议路由推进节点

### 8.2 不允许发生的事

1. provider 直接推进 Aria 状态机。
2. provider 直接决定节点完成。
3. OpenSpec 原始 Markdown 直接作为 runtime truth。
4. skill 中间过程直接作为正式产物。

## 10. OpenSpec 作为全流程约束器

### 10.1 OpenSpec 的准确用途

OpenSpec 在一期的准确定位是：

1. 约束 scope，不允许需求扩散。
2. 约束 requirement coverage，不允许实现漏项。
3. 约束 design boundary，不允许设计跑偏。
4. 约束 task decomposition，不允许 Aria 拆出无来源工作项。
5. 约束 final coverage，不允许收口时留下悬空 requirement / task / risk。

### 10.2 OpenSpec 不做的事

1. 不直接决定 `Nxx -> Nyy`。
2. 不直接宣布任务完成。
3. 不替代 Aria 正式产物。
4. 不单独持有 runtime 状态，不和 daemon 并行维护第二套状态机。

### 10.3 OpenSpec 编译规则

OpenSpec 原始文档不应在每个节点中原样全文喂给 provider。  
一期对 OpenSpec 的要求是**强约束、非可选**。每个 EpicTask 都必须绑定一个 `change_id`，并按 P1/P2 分工完成以下动作：

1. `N02/N03` 解析用户输入中的显式 `change_id`；若没有，则由 daemon 生成 `chg_<task_id>`。
2. `N02/N03` 将 `change_id` 写入 task runtime state，后续节点不得替换。
3. `N02/N03` 将 `openspec_bootstrap_status` 初始化为 `bootstrap_pending`。
4. P2 的 OpenSpec bootstrap service 检查仓库中是否存在 `openspec/changes/<change_id>/`。
5. 若不存在，则通过 Document Operation 层创建最小骨架：
   - `proposal.md`
   - `specs/main/spec.md`
   - `design.md`
   - `tasks.md`
6. skeleton 创建成功后，将 `openspec_bootstrap_status` 更新为 `bootstrapped`，再编译 `OpenSpecConstraintBundle`。

OpenSpec scope 裁定：

1. 一期固定 `openspec_scope = "main"`，不在 P1 wire protocol 中新增 `requested_scope` 字段。
2. P2 bootstrap 创建 skeleton 时必须写入 `openspec/changes/<change_id>/specs/main/spec.md`。
3. Bundle compiler 只读取 `specs/main/spec.md`；若发现同一 change 下存在多个 `specs/*/spec.md`，返回 `openspec_multiple_scopes_unsupported`，进入 gate 或 manual intervention。
4. 多 scope / 多 capability spec 属于二期能力。

一期应在以下时机编译 `OpenSpecConstraintBundle`：

- `N03` 初始上下文准备完成后
- `N06` pass 后写入 `specs/main/spec.md`、`N08` pass 或 conditional_pass 后写入 `design.md`、`N11/N26` 写入 `tasks.md` 后
- 用户手动修改 OpenSpec 文件后
- 任一消费约束的节点进入前，若检测到 bundle `stale`

编译结果至少包含：

- `scope_constraints`
- `non_goal_constraints`
- `requirement_ids`
- `scenario_ids`
- `design_decision_ids`
- `risk_ids`
- `task_ids`
- `task_acceptance_map`
- `traceability_map`

编译结果顶层序列化字段固定使用 6.3.1 裁定的 snake_case 字段名：

- `proposal_constraints`
- `requirement_constraints`
- `design_constraints`
- `task_constraints`
- `traceability_requirements`
- `coverage_model`

内部编译过程中可以产生 `scope_constraints`、`non_goal_constraints`、`requirement_ids`、`scenario_ids`、`design_decision_ids`、`risk_ids`、`task_ids`、`task_acceptance_map`、`traceability_map` 等中间输出；这些名称只能作为内部 helper 或具体 payload 子字段，不得替代 `OpenSpecConstraintBundle` 的顶层字段职责。

`N06` 时序裁定：

1. `N06 spec_gate_review` 的 pass 前置检查源是 Aria `spec`、`SpecProjection` 与 `proposal_constraints`，不是尚未写回的 `requirement_constraints`。
2. `N06 pass` 后 daemon 才将通过 gate 的 canonical `spec` 写入 `specs/main/spec.md`，并立即标记 bundle `stale` 后重编译。
3. `N07` 进入前必须要求重编译后的 `requirement_constraints` 非空；若为空，返回 `openspec_requirement_constraints_empty` 并回到 `N05/N06`。
4. 研发不得通过把 skeleton `spec.md` 当成 ready 来绕过此时序。

固定规则：

1. Rust struct、JSON schema、fixture golden JSON 必须共享同一套字段名。
2. `scope_constraints` 等 snake_case 名称只能作为子字段或内部中间模型，不作为跨模块契约。
3. P2 必须提供 `openspec_bundle_schema` 测试，锁定序列化字段名，避免 P3/P4 按不同命名消费 bundle。

### 10.4 OpenSpec 生命周期、失效与降级规则

一期禁止“静默降级”。

1. `proposal.md` 缺失：不得进入 `N05`，必须回到 `N04/N05` 补齐。
2. `specs/main/spec.md` 缺失：不得进入 `N07`，必须回到 `N05`。
3. `design.md` 缺失：不得进入 `N11`，必须回到 `N07`。
4. `tasks.md` 缺失：不得进入 `N12/N16`，必须回到 `N11`。

bundle 失效规则：

1. 任一 `source_manifest.sha256` 变化，bundle 状态置为 `stale`。
2. 若回流跨越 `N05/N07/N11/N26` 中任一 OpenSpec 写节点，则 bundle 状态置为 `stale`。
3. Projection 被 supersede 不直接让 bundle stale；只有该 projection 曾写回 OpenSpec，且对应 OpenSpec source manifest hash 变化后，才通过 source manifest 触发 stale / recompile。

单写者规则：

1. 只有 daemon 授权的规划节点允许修改 OpenSpec 文件。
2. 执行节点 `N16-N24` 默认只读 OpenSpec。
3. 多 worktree 并行执行时，OpenSpec change 由主 task worktree 统一写入，worker worktree 只读挂载或复制快照。

## 11. 阶段级收口规则

### 11.1 统一原则

任何阶段收口，都必须同时通过三层检查：

1. **Aria canonical artifact 校验**
2. **phase1 profile 校验**（projection / `_aria` / traceability）
3. **OpenSpec 约束覆盖校验**

只有三层校验都通过，daemon 才能推进状态。

### 11.2 需求阶段收口

正式收口物：

- `spec`
- `spec_gate_decision`

收口条件：

1. `spec` 通过 Aria schema 校验
2. `spec_gate_decision` 已生成
3. `SpecProjection` 已生成并带稳定 `story_id` / `requirement_id` / `criterion_id`
4. `SpecProjection` 覆盖 OpenSpec `proposal_constraints`，并已在 `N06 pass` 后写回 `specs/main/spec.md` 生成非空 `requirement_constraints`
5. `open_items` 中的剩余未决项符合策略要求
6. checkpoint 写入完成

### 11.3 设计阶段收口

正式收口物：

- `design`
- `design_review`
- 必要时 `design_revision_record`

收口条件：

1. `design` 通过 Aria schema 校验
2. `design_review` 为 `pass` 或 `conditional_pass`
3. `DesignProjection` 已生成并带稳定 `design_decision_id` / `component_id` / `api_id`
4. `DesignProjection` 覆盖 OpenSpec design / requirement constraints
5. 风险已进入 `risk_registry`
6. checkpoint 写入完成

### 11.4 计划阶段收口

正式收口物：

- `readiness_check`
- `plan`
- `dispatch_package`

收口条件：

1. `plan` 和 `dispatch_package` 通过 Aria schema 校验
2. `PlanProjection` 已生成，并为每个 work package 补齐 `execution_mode`、`human_required_reason`、`traceability_refs`
3. `PlanProjection.work_packages` 全量映射到 OpenSpec `task_ids`
4. `dispatch_package._aria.worktask_routing[]` 已生成，且 `worktasks` 不得存在无来源工作项
5. 每个 `worktask` 必须绑定 requirement / design decision / task / acceptance target
6. checkpoint 写入完成

### 11.5 WorkTask 执行阶段收口

正式收口物：

- `coding_report`
- `testing_report`
- `code_review_report`
- `N20 runtime_snapshot`

收口条件：

1. 三个 report 都通过 Aria schema 校验
2. 三个 report 的 `_aria.traceability_refs` 都存在
3. 三个 report 都能回溯到对应 requirement / design decision / task
4. 未越权修改 worktree 外文件
5. `N20` 判定 ready
6. checkpoint 写入完成

### 11.6 集成阶段收口

正式收口物：

- `integration_report`

收口条件：

1. `integration_report` 通过校验
2. `integration_report._aria.traceability_refs` 存在
3. `next_decision` 明确
4. 若 `verification_status = fail`，必须明确回流、回滚或人工介入路径
5. checkpoint 写入完成

### 11.7 最终阶段收口

正式收口物：

- `final_review`
- `final_summary`
- `N28 runtime_snapshot`

收口条件：

1. `final_review.overall_decision = pass`
2. `final_summary` 通过校验
3. `final_review._aria.coverage_summary` 存在
4. 所有 OpenSpec constraints 均已被覆盖、豁免或显式转人工
5. 不存在悬空 requirement / task / risk
6. `N28` checkpoint 写入完成

## 12. 输入输出物关系矩阵

### 12.1 总矩阵

| 产物类型 | 生产者 | 主要用途 | 可否指导下一步 | 可否作为正式收口 | 可否直接推进状态机 |
|----------|--------|----------|----------------|------------------|--------------------|
| Aria canonical artifact | Aria + provider 归一化后 | 正式输入输出 | 是 | 是 | 是，由 daemon 推进 |
| Aria projection artifact | daemon 编译 | routing / coverage / dashboard / traceability | 是 | 是，作为 profile 校验对象 | 否，需配合 canonical artifact |
| Aria runtime artifact | daemon | checkpoint / gate / queue / snapshot / run record | 是 | 是 | 是 |
| OpenSpec 原始文档 | 用户 / Aria 辅助生成 | 约束来源 | 否，需先编译 | 否 | 否 |
| OpenSpecConstraintBundle | daemon 编译 | 约束校验 | 是，作为约束输入 | 否 | 否 |
| Superpowers 中间产物 | provider run 内部 | 工作纪律辅助 | 否，需先归一化 | 否 | 否 |
| provider raw output | Claude / Codex | 候选输出 | 否，需先归一化 | 否 | 否 |

### 12.2 组合规则

1. **方案指导输入**  
   `Aria canonical artifacts + projection artifacts + OpenSpecConstraintBundle + WorkflowDisciplineSpec`
2. **正式收口输入**  
   `Aria canonical artifacts + projection artifacts + JSON artifact._aria + OpenSpec 覆盖校验结果 + checkpoint`
3. **禁止的组合**  
   - OpenSpec 原始 Markdown 直接驱动下一节点
   - skill 中间过程直接作为正式收口
   - raw stdout 直接作为 canonical artifact
   - 仅凭 Markdown 原文或 report 原文做 routing / coverage 判定

### 12.3 追踪链

一期必须形成以下追踪链：

`proposal / requirement -> spec -> SpecProjection -> design -> DesignProjection -> plan -> PlanProjection -> dispatch_package._aria -> worktask -> report._aria -> integration -> final_review/final_summary`

任何正式收口节点都必须能回答：

1. 当前产物覆盖了哪些 requirement IDs？
2. 当前产物关联了哪些 design decision IDs？
3. 当前产物完成了哪些 task IDs？
4. 当前产物处理了哪些 risk IDs？

### 12.4 Traceability 生成规则

一期 traceability 不靠人工汇总，必须按以下规则自动生成：

1. `SpecProjection` 是 requirement ID 真相源。
2. `DesignProjection` 是 design decision / shared component / API / data entity ID 真相源。
3. `PlanProjection` 是 work package 级执行模式和 requirement/design/task 绑定真相源。
4. `dispatch_package._aria.worktask_routing[]` 是 worktask 级 traceability 与 write scope 真相源。
5. `coding/testing/review/integration/final_review` 的 `_aria.traceability_refs` 由 daemon 在归一化时生成或校正，provider 只负责提供候选引用。
6. `ArtifactTraceabilityBinding` 以 projection 和 `_aria.traceability_refs` 为输入统一生成；若两者冲突，以 daemon 归一化结果为准并记录冲突日志。

## 14. 一期验收标准

### 14.1 实现验收

1. 任意 Agent 节点进入前都能构建 `ProviderContextPackage`。
2. 任意关键规划节点都能构建 `OpenSpecConstraintBundle`。
3. `spec/design/plan` 在写入后都能编译出稳定 projection。
4. `N06/N08-N12/N20/N25/N28` 的收口都必须经过 canonical 校验、phase1 profile 校验和 OpenSpec 覆盖校验。
5. `dispatch_package._aria.worktask_routing[]` 必须存在 `execution_mode`、`traceability_refs`、`allowed_write_scope`。
6. 每个 WorkTask 都能追踪到 requirement / design decision / task。
7. REPL 与 daemon 可基于正式 wire protocol 独立开发并互通。
8. provider 不能绕过 daemon 直接推进状态。
9. Document Operation 层可结构化读写 Markdown / JSON / YAML，projection 和 OpenSpec bundle compiler 不直接裸读 Markdown 原文或手写字符串改文档。
10. CLI adapter baseline 可运行，且 provider capability / compatibility matrix 会写入 `ProviderRunRecord`。
11. Claude Code / Codex 真实调用基于用户本机 BYO CLI；CLI 缺失、未登录或权限不足时可诊断失败，不自动安装、不静默降级。
12. `N25 overall_decision=followup/fail` 的一期路由有测试覆盖：followup 先进入 gate 或 manual intervention，不允许静默自动派生补丁任务。

### 14.2 研发可落地验收

1. 哪些内容必须写进 Aria 正式产物。
2. Aria 在某节点调用 Claude Code / Codex 时到底传什么。
3. OpenSpec 在该节点如何限制不跑偏。
4. Superpowers 在该节点如何约束 provider 内部工作方式。
5. Aria 自己如何结构化读写 canonical artifact、OpenSpec 文档和 JSON / YAML 配置。
6. 阶段收口到底看哪些对象和哪些校验结果。

### 14.3 实施前精确化门槛

评审后代码级实施规格以 `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.4.md` 为准。该文档补齐 Rust 类型定义、`AdapterInput` / `AdapterOutput`、`RuntimeUnit`、Projection / OpenSpec 编译规则、daemon 发现协议、artifact 路径、prompt 模板、traceability 生成算法、worktree / N23 集成规则、provider 运行控制 / 审计字段和 fixture 内容规范。

7. P3 必须同时覆盖 fake provider 和真实 CLI adapter baseline；真实 CLI 不要求模型效果，但必须验证 spawn、超时、stdout/stderr、structured output、files modified 记录，以及 CLI missing / unauthorized / permission denied 诊断。
8. P3 必须用 provider error registry 和 prompt template manifest 锁定错误路由与 `N04-N12` prompt 模板。
9. P4 必须覆盖 `N25 pass/followup/fail` 三类最终评审路径，其中 followup 路径必须验证 gate 确认后才允许进入 `N26`。
