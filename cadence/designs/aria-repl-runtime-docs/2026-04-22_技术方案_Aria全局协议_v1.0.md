# 设计文档：Aria 全局协议

**文档信息**
- **设计编号**：DES-2026-04-22-ARIA-GLOBAL-PROTOCOL
- **创建日期**：2026-04-22
- **版本**：v1.0
- **负责人**：Codex

## 1. 协议目的

本协议定义 Aria 文档集中的统一规则，确保不同节点文档、横切能力文档和产物规范文档在同一组对象模型、ID 规则、状态语义和交接语义上工作。

## 2. 核心对象

| 对象 | 作用 | 真相源 |
|------|------|--------|
| `ProjectSession` | 绑定单仓库的一次主会话 | daemon runtime |
| `EpicTask` | 用户原始需求对应的主任务 | daemon runtime |
| `WorkTask` | 从计划中拆出的执行子任务 | daemon runtime |
| `Phase` | 任务当前所在主阶段 | daemon runtime |
| `Artifact` | 文档产物或运行快照 | daemon runtime + 文件系统 |
| `ApprovalGate` | 待人工处理的闸门 | daemon runtime |
| `ProviderRun` | 一次 Claude/Codex 调用 | daemon runtime |
| `WorktreeLease` | 任务对 worktree 的持有关系 | daemon runtime |
| `EventRecord` | 运行事件日志条目 | daemon runtime |

## 3. ID 规则

| 对象 | ID 前缀 | 示例 |
|------|---------|------|
| ProjectSession | `sess_` | `sess_repo_001` |
| EpicTask | `epic_` | `epic_login_flow_001` |
| WorkTask | `work_` | `work_api_schema_003` |
| ApprovalGate | `gate_` | `gate_design_014` |
| ProviderRun | `run_` | `run_codex_092` |

| EventRecord | `evt_` | `evt_phase_transition_5001` |
| 节点 | `Nxx` / `Xxx` | `N07`、`X04` |

## 4. 状态语义

### 4.1 Task 状态

- `pending`：已注册，未执行
- `running`：当前节点执行中
- `blocked`：被审批、依赖、恢复或冲突阻塞
- `recovering`：daemon 重启或异常恢复中
- `failed`：当前节点失败且未完成回流
- `completed`：已完成并通过交付校验
- `terminated`：人工终止

### 4.2 Node 结果

- `success`
- `retryable_failure`
- `gate_required`
- `manual_intervention_required`
- `rework_required`
- `terminal_failure`

## 5. 交接包规则

任意节点交接给后继节点时，Aria 必须生成统一交接包，至少包含：

- `sessionId`
- `taskId`
- `nodeId`
- `phase`
- `effectivePolicy`
- `artifactRefs`
- `providerRunRefs`
- `worktreeRef`
- `latestCheckpointRef`
- `handoffSummary`

没有完整交接包，不允许进入后继节点。

## 6. 产物分类

- `managed_doc`
  用户可审阅文档，存放在 `cadence/` 下
- `runtime_snapshot`
  运行快照，供 Aria 内部恢复和判定使用
- `validation_record`
  校验结果记录
- `handoff_package`
  节点交接载荷

## 7. 存储规则

- 用户可读文档：按文档类型写入 `cadence/prds`、`cadence/designs`、`cadence/plans`、`cadence/reports`
- 机器运行状态：写入仓库本地 `.aria/runtime/`
- worktree 元数据：写入 `.aria/runtime/worktrees/`
- event log：写入 `.aria/runtime/events/`
- checkpoint：写入 `.aria/runtime/checkpoints/`

## 8. 完成判定全局规则

任意节点判定完成必须同时满足：

1. 必需动作已执行完毕
2. 必需产物已生成
3. 必需产物通过最小结构校验
4. 若节点配置了人工闸门，则闸门已处理
5. 交接包已生成
6. checkpoint 已写入

## 9. 失败与回流全局规则

- 可自动重试的失败：最多有限重试，超过阈值后转人工闸门
- 会影响设计或代码正确性的失败：优先回流到明确上游节点
- 无法自动决策的失败：进入 `manual_intervention_required`
- 任何失败都必须保留失败上下文，不允许静默覆盖

## 10. 人工介入全局规则

- 默认策略：`conservative`
- 支持全局策略与阶段级覆写
- 任意节点在以下情况下必须挂 gate：
  - 边界信息缺失
  - 设计冲突无法裁决
  - 集成冲突需要业务判断
  - 连续失败超过阈值

## 11. Provider 接入全局规则

- 一期所有 provider 调用均通过 `spawn + CLI`
- provider adapter 只负责接入，不负责核心状态机
- 角色抽象先于 provider 绑定
- CLI 输出不是系统真相源，只有结构化运行记录与产物才是真相源
