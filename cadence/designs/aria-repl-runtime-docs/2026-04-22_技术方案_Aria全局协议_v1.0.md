# 设计文档：Aria 全局协议

**文档信息**
- **设计编号**：DES-2026-04-22-ARIA-GLOBAL-PROTOCOL
- **创建日期**：2026-04-22
- **版本**：v1.1
- **负责人**：Codex

## 1. 协议目的

本协议定义 Aria 文档集中的统一规则，确保不同节点文档、横切能力文档和产物规范文档在同一组对象模型、ID 规则、状态语义和交接语义上工作。

## 2. 核心对象

| 对象 | 作用 | 真相源 |
|------|------|--------|
| `ProjectSession` | 绑定单仓库的一次主会话 | daemon runtime |
| `EpicTask` | 用户原始需求对应的主任务 | daemon runtime |
| `WorkTask` | 从计划中拆出的执行子任务 | daemon runtime |
| `Phase` | 任务当前所在主阶段 | daemon runtime |
| `Artifact` | 文档产物或运行快照 | daemon runtime + 文件系统 |
| `ApprovalGate` | 待人工处理的闸门 | daemon runtime |
| `ProviderRun` | 一次 Claude/Codex 调用 | daemon runtime |
| `WorktreeLease` | 任务对 worktree 的持有关系 | daemon runtime |
| `EventRecord` | 运行事件日志条目 | daemon runtime |

## 3. ID 规则

| 对象 | ID 前缀 | 示例 |
|------|---------|------|
| ProjectSession | `sess_` | `sess_repo_001` |
| EpicTask | `epic_` | `epic_login_flow_001` |
| WorkTask | `work_` | `work_api_schema_003` |
| ApprovalGate | `gate_` | `gate_design_014` |
| ProviderRun | `run_` | `run_codex_092` |
| WorktreeLease | `wt_` | `wt_work_api_schema_003` |
| Artifact | `art_` | `art_design_review_021` |
| EventRecord | `evt_` | `evt_phase_transition_5001` |
| 节点 | `Nxx` / `Xxx` | `N07`、`X04` |

## 4. 状态语义

### 4.1 Task 状态

- `pending`：已注册，未执行
- `running`：当前节点执行中
- `blocked`：被审批、依赖、恢复或冲突阻塞
- `recovering`：daemon 重启或异常恢复中
- `failed`：当前节点失败且未完成回流
- `completed`：已完成并通过交付校验
- `terminated`：人工终止

### 4.2 Node 结果

- `success`
- `retryable_failure`
- `gate_required`
- `manual_intervention_required`
- `rework_required`
- `terminal_failure`

## 5. 交接包规则

任意节点交接给后继节点时，Aria 必须生成统一交接包，至少包含：

- `sessionId`
- `taskId`
- `nodeId`
- `phase`
- `effectivePolicy`
- `artifactRefs`
- `providerRunRefs`
- `worktreeRef`
- `latestCheckpointRef`
- `handoffSummary`

没有完整交接包，不允许进入后继节点。

## 6. 产物分类

- `managed_doc`
  用户可审阅文档，存放在 `cadence/` 下
- `runtime_snapshot`
  运行快照，供 Aria 内部恢复和判定使用
- `validation_record`
  校验结果记录
- `handoff_package`
  节点交接载荷

## 7. 存储规则

- 用户可读文档：按文档类型写入 `cadence/prds`、`cadence/designs`、`cadence/plans`、`cadence/reports`
- 机器运行状态：写入仓库本地 `.aria/runtime/`
- worktree 元数据：写入 `.aria/runtime/worktrees/`
- event log：写入 `.aria/runtime/events/`
- checkpoint：写入 `.aria/runtime/checkpoints/`

## 8. 完成判定全局规则

任意节点判定完成必须同时满足：

1. 必需动作已执行完毕
2. 必需产物已生成
3. 必需产物通过最小结构校验
4. 若节点配置了人工闸门，则闸门已处理
5. 交接包已生成
6. checkpoint 已写入

## 9. 失败与回流全局规则

- 可自动重试的失败：最多有限重试，超过阈值后转人工闸门
- 会影响设计或代码正确性的失败：优先回流到明确上游节点
- 无法自动决策的失败：进入 `manual_intervention_required`
- 任何失败都必须保留失败上下文，不允许静默覆盖

## 10. 人工介入全局规则

- 默认策略：`conservative`
- 支持全局策略与阶段级覆写
- 任意节点在以下情况下必须挂 gate：
  - 边界信息缺失
  - 设计冲突无法裁决
  - 集成冲突需要业务判断
  - 连续失败超过阈值

## 11. Provider 接入全局规则

- 一期所有 provider 调用均通过 `spawn + CLI`
- provider adapter 只负责接入，不负责核心状态机
- 角色抽象先于 provider 绑定
- CLI 输出不是系统真相源，只有结构化运行记录与产物才是真相源

## 12. 循环边界控制全局规则

任意节点形成回流环路时，Aria 必须执行循环计数和升级，不允许无限循环。

### 12.1 WorkTask 级别计数器

- 每个 WorkTask 维护一个 `rework_counter`，初始为 0
- 每经过 N19（rework 节点）时 `rework_counter += 1`
- `rework_counter` 超过阈值（默认 3，可通过阶段级策略覆写）后：
  - 不再进入 N19，而是直接进入 X08（manual_intervention_hold）
  - intervention record 中注明 `rework_limit_exceeded` 和当前计数

### 12.2 补丁任务循环控制

- 每个 EpicTask 维护一个 `patch_round_counter`，初始为 0
- 每经过 N26（patch_followup_dispatch）时 `patch_round_counter += 1`
- `patch_round_counter` 超过阈值（默认 2）后：
  - N25 的 final_review 不再允许 `followupNeeded = true`
  - 若仍有未解决缺口，进入 manual_intervention

### 12.3 设计评审循环控制

- N08（design_review）→ N09（design_revision）→ N08 形成环路
- 每个 EpicTask 维护一个 `design_revision_counter`，初始为 0
- 每经过 N09 时 `design_revision_counter += 1`
- 超过阈值（默认 3）后：
  - 不再进入 N09，回流到 N07 重新编写设计
  - 若重新编写后仍不通过，进入 manual_intervention

### 12.4 澄清循环控制

- N04（clarification）→ N06（spec_gate_review）→ N04 形成环路
- 每个 EpicTask 维护一个 `clarification_round_counter`，初始为 0
- N06 回流到 N04 时 `clarification_round_counter += 1`
- 超过阈值（默认 3）后：
  - 不再回流到 N04，进入 X08（manual_intervention_hold）
  - intervention record 中注明 `clarification_limit_exceeded`

### 12.5 计数器存储

- 所有循环计数器存储在 runtime_snapshot 中
- 计数器值通过 checkpoint 持久化
- daemon 重启后通过恢复 checkpoint 还原计数器

## 13. 回流失效全局规则

当节点回流跨越一个或多个中间节点时，中间节点产出的产物必须标记为失效，不允许后续流程引用失效产物。

### 13.1 失效标记

- 回流发生时，Aria 将回流路径上所有中间节点的产物标记为 `superseded`
- `superseded` 标记写入 runtime_snapshot，包含：
  - `supersededArtifactRefs`：被失效的产物引用列表
  - `backtrackReason`：回流原因
  - `backtrackSourceNode`：发起回流的节点 ID
  - `backtrackTargetNode`：回流目标节点 ID
  - `supersededAt`：失效时间戳

### 13.2 失效引用规则

- 任何节点执行时，不得引用状态为 `superseded` 的产物
- Aria 在节点进入前必须校验输入契约中的所有产物引用，确保无 `superseded` 状态
- 如果输入契约中存在 `superseded` 产物，节点不得进入执行

### 13.3 重新执行规则

- 回流目标节点重新执行后，其产出的新产物自动覆盖同类型的 `superseded` 产物
- 后续节点只能引用最新产物，不允许引用历史版本（除非通过 artifact 版本机制显式引用）

### 13.4 Checkpoint 记录

- 回流事件必须写入 checkpoint
- checkpoint 中包含完整的失效范围（哪些节点、哪些产物被标记为 superseded）
- daemon 重启后恢复时，必须还原失效标记

### 13.5 常见回流路径与失效范围

| 回流路径 | 跨越节点 | 失效产物 |
|---------|---------|---------|
| N12 → N07 | N08, N09, N10, N11 | design_review, design_revision_record, readiness_check, plan |
| N11 → N07 | N08, N09, N10 | design_review, design_revision_record, readiness_check |
| N10 → N07 | N08, N09 | design_review, design_revision_record |
| N06 → N04 | N05 | spec |
| N23 → N19 | N20, N21, N22 | ready_for_integration snapshot, queue record, prepare snapshot |
| N24 → N19 | — | 无中间节点 |

## 14. 枚举值注册表

## 15. Risk Registry 全局规则

Aria 维护一个全局 Risk Registry，作为 runtime_snapshot 的一部分，用于跨产物追踪风险的创建、消除和升级。

### 15.1 风险对象

| 字段 | 类型 | 说明 |
|------|------|------|
| `riskId` | string | 唯一标识，格式 `risk_<sequence>` |
| `description` | string | 风险描述 |
| `severity` | enum | `high` / `medium` / `low` |
| `status` | enum | `open` / `mitigated` / `accepted` / `escalated` |
| `sourceArtifact` | ref | 首次发现该风险的产物引用 |
| `sourceNode` | string | 首次发现该风险的节点 ID |
| `createdAt` | timestamp | 创建时间 |
| `updatedAt` | timestamp | 最后更新时间 |
| `resolution` | string | 风险处理说明（mitigated/accepted 时必填） |

### 15.2 产物引用规则

各产物中的风险相关字段改为通过 Risk ID 引用，而非重复声明：

- `design.risk_list`：引用 `riskId` 列表，创建时自动注册到 Risk Registry
- `design_revision_record.remaining_risks`：引用 `riskId` 列表
- `coding_report.known_risks`：引用 `riskId` 列表，新增风险自动注册
- `final_summary.remaining_risks`：引用 `riskId` 列表

### 15.3 风险生命周期

1. **创建**：任意产物声明新风险时，Aria 在 Risk Registry 中创建条目
2. **消除**：后续产物明确声明该风险已解决时，状态变更为 `mitigated`
3. **接受**：人工判定该风险可接受时，状态变更为 `accepted`
4. **升级**：风险在后续阶段仍存在且严重程度上升时，状态变更为 `escalated`
5. **继承**：回流不消除已有风险；新产物只能创建新风险或修改已有风险状态

### 15.4 Checkpoint 存储

- Risk Registry 通过 checkpoint 持久化
- daemon 重启后通过恢复 checkpoint 还原 Risk Registry

## 16. Runtime Snapshot 产物规范

所有 Aria 内部节点（类型为"Aria 内部"的节点）产出的 runtime_snapshot 遵循以下统一格式。不需要为每个内部节点编写独立的产物规范文档。

### 16.1 最小字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `snapshotId` | string | 是 | 唯一标识，格式 `snap_<nodeId>_<sequence>` |
| `sessionId` | ref | 是 | 所属 session 引用 |
| `taskId` | ref | 是 | 所属 task 引用 |
| `nodeId` | string | 是 | 产出该 snapshot 的节点 ID |
| `phase` | enum | 是 | 当前阶段 |
| `timestamp` | string | 是 | ISO 8601 时间戳 |
| `effectivePolicy` | object | 是 | 当前生效策略 |
| `artifactRefs` | array | 是 | 当前已产出的产物引用列表 |
| `providerRunRefs` | array | 否 | 关联的 provider run 引用（Agent 节点填写） |
| `worktreeRef` | ref | 否 | 关联的 worktree 引用（执行阶段节点填写） |
| `reworkCounter` | integer | 是 | 当前 WorkTask 的返工计数 |
| `riskRegistry` | object | 是 | 当前 Risk Registry 快照 |
| `loopCounters` | object | 是 | 各循环计数器当前值 |
| `supersededArtifactRefs` | array | 否 | 已失效的产物引用列表 |
| `nodeSpecificFields` | object | 否 | 节点特有字段（各节点文档自行定义） |

### 16.2 校验规则

- L1：`snapshotId`、`sessionId`、`taskId`、`nodeId`、`phase`、`timestamp` 必须存在且非空
- L1：`artifactRefs` 必须是数组
- L1：`reworkCounter` 必须是 >= 0 的整数
- L2：`snapshotId` 格式符合 `snap_<nodeId>_<sequence>` 规则
- L2：`nodeId` 是合法的节点 ID（Nxx 或 Xxx）

### 16.3 存储位置

- 所有 runtime_snapshot 存储在 `.aria/runtime/checkpoints/` 目录
- 文件命名格式：`<snapshotId>.json`

### 16.4 适用节点

以下节点产出的 runtime_snapshot 遵循本规范，不需要独立产物规范文档：

| 节点 | nodeSpecificFields |
|------|-------------------|
| N00 | `bootstrapMode`, `repoRoot`, `daemonPid` |
| N02 | `epicTaskId`, `initialPhase`, `sourceIntakeBriefRef` |
| N03 | `effectivePolicy`, `phaseOverrideApplied`, `autoAdvanceAllowed` |
| N06 | `specGateDecision`（引用产物规范 ART-SPEC-GATE-DECISION） |
| N10 | `readinessCheck`（引用产物规范 ART-READINESS-CHECK） |
| N13 | `registeredWorktasks`, `dependencyGraph`, `parallelGroups` |
| N14 | `worktreeLeases`（数组，每个包含 worktreeId, branchName, worktreePath, baseRef） |
| N15 | `nextExecutionNode`, `routingReason`, `effectivePolicy` |
| N20 | `readyAt`, `validationRefs`, `worktreeRef` |
| N21 | `queuePosition`, `enqueueTime` |
| N22 | `baseRefBeforeIntegration`, `precheckStatus`, `conflictRisk` |
| N28 | `completedAt`, `finalArtifactRefs`, `releasedResources` |

## 17. 统一术语表

| 中文术语 | 英文术语 | 说明 |
|---------|---------|------|
| 产物 | artifact | 节点执行后输出的文档、记录或快照 |
| 交接包 | handoff package | 节点间传递的标准化载荷 |
| 回流 | backtrack | 流程回退到上游节点重新执行 |
| 返工 | rework | 针对失败项的修复操作（N19 节点） |
| 闸门 / 审批闸门 | approval gate | 等待人工审批的可恢复暂停点 |
| 人工介入 | manual intervention | 系统无法自动决策时的终态兜底 |
| 策略 | policy | 决定系统保守/激进程度的配置 |
| 阶段 | phase | 任务当前所在的主阶段 |
| 节点 | node | 流程中的一个执行步骤 |
| 横切能力 | cross-cutting capability | 跨多个节点的通用能力 |
| 真相源 | source of truth | 系统状态的权威数据源 |
| 检查点 | checkpoint | 运行时状态的持久化快照 |
| 工作树 | worktree | 独立的代码工作空间（git worktree） |
| 租约 | lease | 对 worktree 的持有关系 |
| 提供者 | provider | Claude Code 或 Codex 等 AI 执行者 |
| 适配器 | adapter | 屏蔽 provider 差异的接口层 |
| 集成队列 | integration queue | 串行化主线合入的队列 |
| 事件日志 | event log | append-only 的运行事件记录 |
| 恢复 | recovery | daemon 重启后重建状态的过程 |
| 失效 | superseded | 回流后被标记为不可引用的产物状态 |
