- **设计编号**：DES-2026-04-22-ARIA-DOCSET-OVERVIEW

## 1. 目标

Aria 不是单次命令行工具，而是一个本地优先的 Agent Runtime。用户通过终端 REPL 输入需求后，Aria 负责驱动 Claude Code 与 Codex 完成需求澄清、规格、设计、评审、计划、分发、并行编码、测试、代码评审、串行集成与最终总结。

## 2. 一期定位

- 形态：`REPL client + 本地 daemon runtime`
- 范围：`单用户、本机、单仓库、多任务并行`
- Provider 接入：一期只使用 `spawn + CLI`
- 角色默认映射：
  - Claude Code：orchestrator
  - Codex：executor / reviewer
- 隔离策略：`每任务独立 worktree`
- 集成策略：`并行执行、串行集成`
- 人工介入：默认保守策略，支持阶段级覆写

## 3. 目标态

- Rust core 继续承担本地 runtime、调度、恢复、worktree、事件流
- TypeScript 继续承担 provider adapter 与未来 UI/TUI/桌面壳
- provider fabric 可从 CLI adapter 升级到 Claude Agent SDK、Codex SDK、Codex app-server
- 前台形态可从 REPL 演进到 TUI 或桌面 GUI，而不改核心 runtime

## 5. 核心设计原则

1. REPL 只负责交互，daemon 才是运行时真相源。
2. 对话分叉和代码分叉分开处理，会话靠 session/thread，代码靠 worktree。
3. 节点必须有输入契约、输出契约、完成判定、失败回流。
4. 横切能力单独成文，避免在节点文档里重复漂移。
5. 产物规范单独成文，避免不同节点定义不同格式。
6. 一期做可恢复的本地 orchestrator，不做完整多租户平台。

## 7. 交付边界

本设计文档集负责定义：

- Aria 系统如何分层
- 有哪些节点与横切能力
- 节点之间如何交接
- 每类产物的固定要求

本设计文档集不负责定义：

- 具体代码任务拆分
- 代码模块实现顺序
- TDD 明细任务计划

这些内容应在后续 `cadence/plans/` 中展开。
