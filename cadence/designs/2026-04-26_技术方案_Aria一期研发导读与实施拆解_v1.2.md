## 1. 文档目标

本文件只解决三件事：

1. 帮研发团队快速理解以下两份一期真相源文档各自负责什么：
   - `cadence/designs/2026-04-23_技术方案_Aria一期MVP精简设计_v1.2.md`
   - `cadence/designs/2026-04-23_技术方案_Aria一期实现总契约_v1.0.md`
2. 指向评审后补齐的实现规格：
   - `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.4.md`
3. 把一期研发工作按模块负责人视角拆解成可执行的实施任务。

本文件是**导读与任务拆解文档**，不是新的协议真相源。

真相源仍然分三层：

| 层级 | 真相源 | 负责内容 |
|------|--------|----------|
| canonical 协议层 | 上游协议文档集 | 节点 ID、节点路由、最小字段、最小校验、统一枚举 |

| 导读与拆解层 | 本文 | 研发阅读顺序、模块分工、实施建议 |

## 2. 两份一期文档各自的目标

### 2.1 `Aria一期MVP精简设计_v1.2.md` 的目标

这份文档回答：

**Aria 一期系统要长成什么样，边界在哪里，哪些协议能力一期必须落地。**

它主要负责：

1. 定义一期系统形态  
   `REPL client + 本地 daemon runtime`
2. 定义一期范围  
   单机、单用户、单仓库、多任务并行（worktree 隔离）
3. 定义一期和上游协议的关系  
   protocol layer 不变，runtime unit 可折叠
4. 定义一期实现边界  
   哪些完整实现、哪些兼容实现、哪些二期再做
5. 定义一期成功标准  
   不仅流程能跑，还要协议不漂移

一句话理解：

**这份文档在回答“我们一期到底要造什么系统”。**

### 2.2 `Aria一期实现总契约_v1.0.md` 的目标

这份文档回答：

**研发写代码时，到底按什么对象、接口、消息、状态机、收口规则实现。**

它主要负责：

1. 定义 Aria / OpenSpec / Superpowers 的职责边界
2. 定义 canonical artifact / projection / `_aria` / runtime artifact 的承载关系
3. 定义 Aria 如何驱动 Claude Code / Codex
4. 定义 REPL <-> daemon wire protocol
5. 定义 OpenSpec 约束从绑定到失效重编译的完整生命周期
6. 定义 traceability 和阶段级收口规则

一句话理解：

**这份文档在回答“研发到底该按什么接口和数据模型写代码”。**

### 2.3 两份文档的关系

最简理解：

- `v1.2`：讲 **做什么、做到哪、边界是什么**
- `实现总契约`：讲 **具体怎么实现、对象怎么定义、消息怎么传、收口怎么判**

研发不要反过来看：

- 不要只看 `v1.2` 就开始编码，因为接口细节不够
- 也不要只看 `实现总契约` 就忽略一期范围，因为会丢掉边界约束

### 4.2 推荐实施顺序

不要先按节点平铺开发。推荐顺序如下：

1. `REPL Wire`
2. `Runtime State`
3. `Document Operation / Canonical Artifact`
4. `Projection`
5. `Phase1 Profile`
6. `OpenSpec Bundle`
7. `Traceability`
8. `Provider Contract`
9. `Prompt / Contract Registry`
10. `Planning Nodes`
11. `Execution Nodes`
12. `Final Closure`

原因很直接：

- 没有 wire protocol，REPL / daemon 无法并行联调
- 没有 projection / `_aria` / bundle，后续 routing 和 coverage 都会漂
- 没有 provider contract，节点实现会各自脑补 DTO

## 5. 最容易做错的地方

### 5.1 把 Aria 和 OpenSpec 都做成真相源

错误。

正确做法：

- **Aria 单一正式收口**
- **OpenSpec 全程约束**

### 5.2 直接解析 Markdown 原文当机器输入

错误。

正确做法：

- Markdown artifact 先产出 canonical 文档
- daemon 再编译 projection
- routing / coverage / dashboard 一律消费 projection

### 5.3 把执行模式硬塞进 canonical 最小字段

错误。

正确做法：

- `PlanProjection.work_packages[]`
- `dispatch_package._aria.worktask_routing[]`

### 5.4 让 provider 直接决定节点完成

错误。

正确做法：

- provider 只生成候选输出
- daemon 做 canonical 校验、phase1 profile 校验、OpenSpec 覆盖校验
- 只有 daemon 能推进状态

### 5.5 把 OpenSpec 作为可选输入

错误。

一期这里已经定死：

- OpenSpec 是强约束、非可选
- 每个 EpicTask 必须绑定 `change_id`

### 5.6 端到端数据流示例

研发联调时按下面链路定位数据，而不是跨层读取原始文件：

```text
1. REPL new_task
   -> N01 intake_brief
   -> N02 EpicTask runtime state
   -> N03 effective_policy + change_id
   -> runtime_snapshot/checkpoint

2. N04 clarification
   -> CanonicalNodeInput
   -> ProviderContextPackage
   -> AdapterInput
   -> AdapterOutput
   -> ProviderRunRecord
   -> daemon normalization
   -> clarification_record
   -> artifact_validate
   -> checkpoint

3. N05/N06 spec
   -> spec canonical artifact
   -> SpecProjection
   -> N06 只检查 SpecProjection 与 proposal_constraints，不以前置 requirement_constraints 阻断
   -> daemon 将通过 gate 的 spec 写回 OpenSpec spec.md
   -> OpenSpecConstraintBundle stale/recompile
   -> N07 进入前要求 requirement_constraints 非空

4. N07-N12 planning
   -> design/design_review/plan canonical artifacts
   -> DesignProjection / PlanProjection
   -> daemon 写回 OpenSpec design.md/tasks.md
   -> bundle stale/recompile
   -> dispatch_package._aria.worktask_routing[]

5. N16-N24 execution/integration
   -> worktask routing + allowed_write_scope
   -> provider reports
   -> daemon 生成 _aria.traceability_refs
   -> ArtifactTraceabilityBinding
   -> N20 candidate_commit_sha
   -> N23 integration_report

6. N25-N28 final closure
   -> final_review._aria.coverage_summary
   -> pass/followup/fail 路由
   -> followup 时 N26 先更新 OpenSpec tasks.md，再重编译 bundle，再生成新 dispatch_package
   -> final_summary
   -> session_closeout
```

这条链路的裁定含义：

1. provider stdout 只在 `AdapterOutput` / `ProviderRunRecord` 中作为候选与审计输入存在。
2. routing、coverage、traceability 只能消费 canonical artifact、projection、bundle、binding。
3. OpenSpec 是约束来源，不是 runtime truth；Aria daemon 才能推进节点和状态。

## 6. 研发完成判定

一期真正完成，不是“节点代码写了”，而是同时满足：

1. REPL 和 daemon 能基于正式 wire protocol 独立联调
2. canonical artifact 全部可以通过校验
3. projection / `_aria` / constraint bundle / traceability 都能稳定落盘
4. Claude Code / Codex 的调用链可复现、可审计
5. OpenSpec 可以从头到尾约束流程，不允许静默跑偏
6. `final_review` 可以证明 coverage 闭合
