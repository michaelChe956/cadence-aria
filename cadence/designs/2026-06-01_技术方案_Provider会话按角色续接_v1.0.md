# Provider 会话按角色续接技术方案

## 文档信息

- 文档类型：技术方案
- 日期：2026-06-01
- 版本：v1.0
- 关联问题：Story Spec author 在用户选择交互后重新调用 provider 时新开 provider 对话，导致 token 消耗增加

## 背景

在 Story Spec 生成过程中，author 可能需要向用户确认选项。例如 author 输出一个选择题后，Workspace 会显示选择窗口。用户提交选择后，系统会重新启动 author 继续生成。

当前产品层 `workspace_session_0001` 会保持不变，但 provider 层没有续接原来的 Claude Code 或 Codex 对话。具体表现是：

- `ProviderEvent::Completed` 已携带 `provider_session_id`，但 Workspace Engine 处理完成事件时忽略该字段。
- `StreamingProviderInput.session_id` 目前被填入产品层 workspace session id，例如 `workspace_session_0001`，语义不是 provider 对话 id。
- Claude Code provider 启动时没有使用 `--resume <session_id>`。
- Codex provider 每次都执行 `thread/start`，没有复用已有 thread id。
- Coding Workspace 的 provider stream 完成路径同样忽略 `provider_session_id`。

这会造成用户选择后继续 author 时，provider 重新建立对话，已有上下文无法以 provider 原生会话方式复用，并增加 token 消耗。

## 目标

1. 同一角色的后续 provider 调用应续接同一 provider 会话。
2. 不同角色之间不得混用 provider 会话，例如 author、reviewer、coder、tester 必须隔离。
3. Claude Code 和 Codex 都支持 provider 会话续接。
4. provider session 与产品层 workspace session 明确区分，避免字段语义混用。
5. provider session 需要持久化，刷新页面或 WebSocket 重连后仍可续接已完成的 provider 会话。
6. resume 失败时显式报错，不静默退化为新会话。
7. 兼容旧数据，旧 session 或 attempt 没有 provider conversation 字段时仍可打开。

## 非目标

- 不跨角色共享 provider 会话。
- 不跨 provider 共享会话，例如 Claude Code session id 不用于 Codex。
- 不跨 Coding attempt 复用 coder/tester/reviewer 会话。
- 不在本轮重构完整 provider run 审计系统。
- 不改变前端 `workspace_session_id`、WebSocket session id 或现有选择卡片 UI。

## 方案选择

### 方案 A：最小字段补丁

在 `WorkspaceEngine` 上增加 `author_provider_session_id` 和 `reviewer_provider_session_id`。

优点：实现最快。

缺点：扩展到 coder、tester、analyst、code reviewer 后字段会膨胀，且 Story/Design Workspace 与 Coding Workspace 需要重复实现。

### 方案 B：角色会话表

用统一的 provider conversation 表记录 `role + provider` 对应的 provider session id。

优点：符合角色隔离要求，适合同时覆盖 Story/Design Workspace 与 Coding Workspace，也方便后续扩展角色。

缺点：需要调整模型、持久化和测试。

### 方案 C：完整 provider run registry

新增独立 store，记录每次 provider run、角色、provider session、状态和审计信息。

优点：最完整。

缺点：超出当前问题需要，改动面和迁移成本偏大。

### 决策

采用方案 B：角色会话表。

## 核心模型

新增 provider conversation 概念：

```rust
pub struct ProviderConversationRef {
    pub role: ProviderConversationRole,
    pub provider: ProviderName,
    pub provider_session_id: String,
    pub updated_at: String,
    pub last_node_id: Option<String>,
}
```

角色枚举建议统一覆盖当前和后续角色：

```rust
pub enum ProviderConversationRole {
    Author,
    Reviewer,
    Coder,
    Tester,
    Analyst,
    CodeReviewer,
    InternalReviewer,
}
```

查找 key 为 `role + provider`。同一 role 但 provider 不同，不复用 session。provider 切换后自然查不到旧 provider 的 conversation。

## Provider 输入契约

拆分当前 `StreamingProviderInput.session_id` 的含混语义：

```rust
pub struct StreamingProviderInput {
    pub provider_type: ProviderType,
    pub role: AdapterRole,
    pub prompt: String,
    pub working_dir: PathBuf,
    pub workspace_session_id: Option<String>,
    pub resume_provider_session_id: Option<String>,
    pub permission_mode: ProviderPermissionMode,
    pub env_vars: BTreeMap<String, String>,
    pub timeout_secs: u64,
}
```

字段语义：

- `workspace_session_id`：产品层 session id，只用于审计、日志、测试控制和归属。
- `resume_provider_session_id`：provider 原生会话 id，用于续接 Claude Code session 或 Codex thread。
- `ProviderEvent::Completed.provider_session_id`：provider 本轮结束后返回的最新可续接 id。

兼容策略：

- 修改所有构造 `StreamingProviderInput` 的位置，明确填入这两个字段。
- 旧的 `session_id` 字段不继续承担双重语义。

## Workspace 数据流

### 启动 provider run

每次启动 provider run 时：

1. 根据 run 类型确定 `ProviderConversationRole`：
   - Story/Design author：`Author`
   - Story/Design reviewer：`Reviewer`
   - Story/Design revision：`Author`
2. 根据 `role + provider_name` 查询 provider conversation。
3. 构造 `StreamingProviderInput`：
   - `workspace_session_id = Some(workspace_session_0001)`
   - `resume_provider_session_id = conversation.provider_session_id`
4. 调用 provider。

### 完成 provider run

provider 返回 `ProviderEvent::Completed { provider_session_id, ... }` 后：

1. 如果 `provider_session_id` 为 `Some`，写回当前 `role + provider_name` 对应的 conversation。
2. 更新 `updated_at` 和 `last_node_id`。
3. 持久化到 session record。
4. 如果 `provider_session_id` 为 `None`，不覆盖已有 conversation。

### 用户选择后的续跑

结构化选择场景：

- provider 进程仍在等待 `ChoiceResponse` 时，继续走 active run 的 command channel。
- 不需要重新启动 provider，也不涉及 resume。

文本兜底选择题场景：

- author 已完成退出，engine 已持久化该 author provider session。
- 用户提交选择后，WebSocket handler 重新启动 author run。
- 新 author run 使用 `Author + 当前 provider` 查到的 provider session 续接。

## Claude Code 续接

Claude Code provider 行为：

- 无 `resume_provider_session_id`：保持当前 `claude -p ...`。
- 有 `resume_provider_session_id`：追加 `--resume <id>`。
- 禁止使用 `--continue`，因为它按当前目录选择最近会话，可能串到其他 Workspace 或其他角色。
- 禁止使用 `--fork-session`，因为本需求要求复用原会话。

完成时继续从 Claude `result.session_id` 解析 `provider_session_id` 并上报。

resume 失败处理：

- 如果 Claude CLI 因 resume id 无效或不可恢复而失败，向上返回 provider failed/protocol error。
- 不静默移除 `--resume` 后重试。

## Codex 续接

Codex app-server provider 行为：

- 无 `resume_provider_session_id`：
  - 调用 `thread/start`。
  - 用返回的 `thread.id` 调用 `turn/start`。
  - 完成时上报该 thread id。
- 有 `resume_provider_session_id`：
  - 不调用新的 `thread/start`。
  - 直接以该 id 作为 `turn/start.params.threadId`。
  - 完成时继续上报同一个 thread id。

resume 失败处理：

- 如果 `turn/start` 返回 thread 不存在、不可恢复或其他错误，向上返回 provider failed/protocol error。
- 不静默新建 thread。

## 持久化设计

### Story/Design Workspace

在 `WorkspaceSessionRecord` 增加字段：

```rust
#[serde(default)]
pub provider_conversations: Vec<ProviderConversationRef>,
```

旧 JSON 没有该字段时默认空表。

`WorkspaceSession` 内存模型同步增加 `provider_conversations` 字段。`WorkspaceSession::from_record` hydrate provider conversations 到 engine 内存。engine 更新 conversation 后通过 `LifecycleStore` 写回 session record。

### Coding Workspace

在 `CodingExecutionAttempt` 增加同类字段：

```rust
#[serde(default)]
pub provider_conversations: Vec<ProviderConversationRef>,
```

conversation 生命周期绑定单次 coding attempt，不跨 attempt 复用。

Coding engine 在每个 provider role run 前读取对应 conversation，在 provider 完成后写回 attempt。

## Coding 角色映射

Coding Workspace 接入同一套 provider conversation 机制：

- 代码编写 provider run：`Coder`
- 测试 provider run：`Tester`
- 代码评审 provider run：`CodeReviewer`
- 分析类 provider run：`Analyst`
- 内部评审类 provider run：`InternalReviewer`
- Git/System 节点不调用 provider，不创建 conversation。

如果当前 UI 或模型仍显示 `Author`，provider conversation 内部仍按 `Coder` 记录，避免和 Story/Design author 混淆。

## 断线与恢复

- WebSocket 断开时，当前 active provider run 仍按现有逻辑 abort。
- 已完成并保存 `provider_session_id` 的 provider conversation 可在重连后继续使用。
- 中途断开且 provider 未完成时，不保存不完整 provider session。
- 如果 session record 已保存 conversation，engine 重建后应从持久化数据恢复。

## 错误处理

- provider 没返回 `provider_session_id`：不报错，不覆盖旧值。
- provider 返回空字符串：忽略，不覆盖旧值。
- provider 切换：不复用旧 provider 的 session。
- resume 失败：显式向前端发送错误，不静默新建 provider 会话。
- 旧数据缺少 conversation 字段：按空表处理。

## 测试策略

### Workspace Engine 单元测试

覆盖：

1. author 第一次完成返回 `claude-session-a` 后被保存。
2. author 用户选择续跑时，下一次 input 的 `resume_provider_session_id` 等于 `claude-session-a`。
3. reviewer 第一次启动时不读取 author session。
4. reviewer 完成返回 `reviewer-session-b` 后，下一次 reviewer 才续接 `reviewer-session-b`。
5. provider 从 Claude Code 切换到 Codex 时，不复用 Claude Code 的 session id。
6. provider 返回 `None` 时不覆盖已有 provider session。

### Claude Code provider 测试

覆盖：

1. 无 resume id 时参数不包含 `--resume`。
2. 有 resume id 时参数包含 `--resume <id>`。
3. 参数不包含 `--continue`。
4. Claude result 中的 `session_id` 被解析为 `provider_session_id`。

### Codex provider 测试

覆盖：

1. 无 resume id 时发送 `thread/start`，然后使用返回 thread id 调 `turn/start`。
2. 有 resume id 时不发送新的 `thread/start`，直接使用该 id 调 `turn/start`。
3. 完成时返回 `provider_session_id = thread_id`。

### WebSocket 集成测试

覆盖文本选择题流程：

1. 第一次 author run 返回 provider session。
2. 收到 `choice_request`。
3. 用户提交 `choice_response`。
4. 第二次 author run 续接第一次的 provider session。
5. 最终 Story Spec 生成后才进入 reviewer。
6. reviewer 不复用 author provider session。

### Coding Workspace 测试

覆盖：

1. coder 第一次返回 `coder-thread-1` 后被保存。
2. tester 不读取 coder session。
3. code reviewer 不读取 coder/tester session。
4. coder 后续同角色 run 续接 `coder-thread-1`。

## 验收标准

- Story Spec author 遇到用户选择后，继续使用同一个 Claude Code 或 Codex provider 对话。
- author、reviewer、coder、tester 等角色的 provider 会话互不复用。
- provider 切换不会复用另一 provider 的 session。
- 页面刷新或 WebSocket 重连后，已完成并持久化的 provider session 仍能续接。
- resume 失败时显式报错，不静默退化成新 provider 会话。
- 旧 workspace session 和 coding attempt 数据仍可读取。

## 实施顺序建议

1. 新增 provider conversation 模型和 `StreamingProviderInput` 字段拆分。
2. 为 Story/Design Workspace 接入 role-based provider conversation。
3. 实现 Claude Code `--resume`。
4. 实现 Codex thread 复用。
5. 为 Coding Workspace 接入 role-based provider conversation。
6. 补齐测试并运行定向验证。
