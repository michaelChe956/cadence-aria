# PR #1 真实接入人工测试方案

## 1. 文档信息

- 目标 PR：`#1`
- PR 标题：`feat: complete real integration runtime flow and verification kit`
- 目标分支：`feature-aria-phase1-foundation`
- 基线提交：`9ac581f3d38f137a054a66cf7d99164c999fa55b`
- 测试对象：Aria 一期真实接入闭环、只读验证工具、状态恢复与人工检查链路

## 2. 测试目标

本方案用于指导测试人员对 PR #1 进行人工测试，重点确认以下内容：

1. formal flow 主链路可以从 `intake` 正常推进到 `verified`、`patching` 或 `blocked`
2. 每个关键节点都能产生正确的状态迁移和正式工件
3. `exec-result`、`review-report`、`test-report` 由 Aria 正式落盘，并与 `state.yaml` 保持一致
4. `aria:doctor` 与 `scripts/verify-real-integration.sh` 能为真实接入提供可执行的验收证据
5. 在真实 `Claude Code` / 真实 `Codex` 场景下，测试人员可以按节点检查产出物、状态和证据链
6. 受控异常场景作为附录补充，不与主线真实验收混用

## 5. 测试策略

本次人工测试分为主线 3 层，另附 1 组补充附录。

### 5.1 P0：环境与入口冒烟

验证：

- CLI 入口可用
- `aria:doctor` 能识别能力
- 构建产物可执行

### 5.2 P1：真实主线闭环

按真实用户路径验证：

1. `aria:intake`
2. `aria:start`
3. `confirm-spec`
4. `confirm-plan`
5. `aria:run`
6. `aria:status`
7. `aria:result`
8. `scripts/verify-real-integration.sh`

要求至少完成一次完整成功闭环，并核对所有正式工件。

### 5.3 P1：节点产物核对

对每个节点，都必须同时核对：

1. 命令输出
2. `state.yaml`
3. 当前节点新增或更新的工件
4. 工件中的关键字段

本次测试以 `state.yaml` 为最终状态事实源，以工件内容为正式证据源，以命令输出为辅助观察面。

### 5.4 附录 A：受控异常与恢复补充

通过环境故障或受控替身二进制，补充验证以下分支：

- capability 缺失
- codex 执行失败
- review/test 报告非法
- review/test 报告 `task_id` 不匹配
- `review verdict = needs_patch`
- `review verdict = failed`
- `aria:retry`
- `aria:cancel`

## 6. 附录 A 使用原则

### 6.1 推荐方式

优先通过环境变量覆盖 CLI 二进制路径：

- `CADENCE_CODEX_BIN`
- `CADENCE_CLAUDE_BIN`

原因：

1. 不需要修改 PR 代码
2. 能稳定复现负向路径
3. 更接近真实运行时接口

### 6.2 使用边界

受控异常注入适用于以下场景：

- 需要强制 `codex` 非零退出
- 需要强制 `claude` 输出非法 YAML
- 需要强制 `claude` 输出错误 `task_id`
- 需要强制 `review verdict = needs_patch` 或 `failed`

不建议在主线成功场景中使用替身。

## 7. 测试执行顺序

建议按以下顺序执行：

1. 环境检查与构建
2. `aria:doctor`
3. 主线成功闭环
4. 只读验证脚本
5. `status/result` 查询
6. front phase 来源证明篡改场景
7. 真实结果分流检查：`verified / patching / blocked`
8. `cancel` 查询与收尾
9. 如需完整覆盖，再执行附录 A 中的受控异常场景

原因：

- 先拿到一套成功样本，后续所有只读核验和状态对比都可以复用
- 受控异常场景会污染任务状态，不适合作为首测路径

## 8. 进入与退出准则

### 8.1 进入准则

- PR 分支已同步到 `9ac581f3d38f137a054a66cf7d99164c999fa55b`
- `pnpm build` 成功
- `aria:doctor` 可正常输出能力探测结果
- 测试人员具备目标仓库写权限

### 8.2 主线准出准则

满足以下条件，可视为 PR #1 真实主线人工测试通过：

1. P0 和 P1 全部通过
2. 主线成功闭环至少 1 次
3. `scripts/verify-real-integration.sh` 至少对成功样本通过 1 次
4. 至少覆盖 1 次真实 `aria:run` 结果分流，并核对对应产物
5. `cancel` 至少验证 1 次

### 8.3 完整覆盖准则

在主线准出之外，如需宣称“完整覆盖”而非“主线通过”，还应额外满足：

1. 附录 A 中的受控异常场景已执行
2. `verified`、`patching`、`blocked` 三类结果至少各覆盖 1 次
3. `retry` 至少验证 1 次
4. 所有失败场景都能给出明确 `block_reason_code`、`blocking_stage`、`required_action`

## 9. 结果记录要求

每条人工测试用例至少记录以下信息：

- 用例 ID
- 执行人
- 执行时间
- 执行目录
- 任务 ID
- 命令
- 实际状态
- 实际工件路径
- 结论：通过 / 失败 / 阻塞
- 差异说明

## 10. 高风险观察点

测试人员应重点关注以下风险：

1. `state.yaml` 状态与报告 verdict 不一致
2. 冻结引用与 handoff 工件不一致
3. `exec-result` 中记录的能力消费与实际 contract 不一致
4. `review-report.yaml` / `test-report.yaml` 缺字段但状态仍推进
5. `blocked` 场景未清理 `active_exec_units`
6. `patching` 场景未生成 `patch-contract-patch-01.yaml`
7. `verify-real-integration.sh` 与人工核对结论不一致
