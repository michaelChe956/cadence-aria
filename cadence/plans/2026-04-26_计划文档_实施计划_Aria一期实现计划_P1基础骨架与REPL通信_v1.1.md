# Aria Phase 1 P1 Implementation Plan

**文档信息**
- **创建日期**：2026-04-26
- **版本**：v1.1（研发可落地性 Review 修正版）
- **修正内容**：补充 `nodes.rs` / `artifacts.rs` 内容来源、`RuntimeUnit` 统一接口落位、`N00-N03` 入口函数签名示例。

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 建立 Aria 一期的 Rust 工程骨架、`N00-N03` 入口链、REPL <-> daemon wire protocol、transport、daemon session 状态与 checkpoint 基线。

**Architecture:** 先把最底层通信、进程边界和任务入口链定死，再实现最小可联调的 REPL/daemon 骨架。P1 不实现 planning/execution 业务，但必须负责 `new_task -> intake_brief -> EpicTask -> effectivePolicy/changeId` 这条进入系统的正式路径；P2 在此基础上补齐 OpenSpec 文件结构化写入和 bundle 编译。

**Tech Stack:** Rust、Cargo、serde、Unix socket / named pipe、JSON、basic fs persistence。

---

## 0. 评审后准入门槛

P1 启动前必须先落实 `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.3.md` 中与 P1 相关的裁定：

- 第 4.1-4.3、4.8：基础 Rust 类型、`CanonicalNodeInput`、`RuntimeSnapshot`
- 第 4.7.4：`RuntimeUnit` 统一 handler 契约、`RuntimeUnitResult`、`RuntimeUnitError`
- 第 7 章：daemon 启动、socket / named pipe 发现、crash recovery、event replay 窗口
- 第 8 章：`.aria/runtime/` artifact 与 snapshot 路径
- 第 14 章：approval gate 的 REPL 命令面

P1 不要求实现 Projection 或 OpenSpec compiler，但不得设计与上述补齐规格冲突的路径、字段或 handshake。

## 1. 范围与出口

P1 完成后，必须满足：

1. 仓库具备可编译的 Rust 工程骨架
2. REPL 与 daemon 可以通过正式 wire protocol 建立连接
3. 支持 `hello/attach/subscribe/get_status/detach`
4. 支持 `new_task` 形成 `intake_brief`、EpicTask runtime state、`effectivePolicy` 与稳定 `changeId`
5. daemon 能写最小 checkpoint / runtime snapshot
6. REPL 可接收 event 流并支持 reconnect 基线

不在 P1 范围内：

- provider 调用
- projection / `_aria`
- OpenSpec 文件结构化写入、bundle 编译和 coverage 校验
- planning / execution 节点业务逻辑

入口函数签名裁定：

```rust
// session_bootstrap.rs
pub fn bootstrap(workspace_root: &Path) -> Result<SessionState>;

// intake_capture.rs
pub fn capture_intake(
    request_text: &str,
    session: &SessionState,
) -> Result<IntakeBrief>;

// task_init.rs
pub fn init_task(
    intake: &IntakeBrief,
    session: &mut SessionState,
) -> Result<TaskInitResult>;

pub struct TaskInitResult {
    pub task_id: TaskId,
    pub change_id: ChangeId,
    pub effective_policy: PolicyRef,
    pub openspec_bootstrap_status: BundleStatus,
}
```

上述函数是 P1 入口链的内部 helper；daemon 状态机对外调用 runtime unit 时，仍统一走 `RuntimeUnit::execute(input, ctx)`，不得为 `N00/N01/N02/N03` 单独发明不同 handler 签名。

- [ ] **Step 1: 写失败测试，覆盖 `new_task -> N00/N01/N02/N03`**

断言：
- `new_task.request_text` 生成 `intake_brief`
- 未传 `requested_change_id` 时，daemon 生成 `chg_<taskId>`
- 传入 `requested_change_id` 时，daemon 校验并固化该值
- task runtime state 写入 `effectivePolicy`
- `N00/N01/N02/N03` 均写入 runtime snapshot 或可追踪 checkpoint 记录
- OpenSpec bootstrap 状态先记录为 `bootstrap_pending`，实际文件写入与 bundle 编译交给 P2
