# target 目录瘦身 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 通过集成测试分组合并（88→6 个二进制）、profile 调优、sccache 与清理脚本，结构性根治 cadence-aria 的 target 目录膨胀，使干净构建从 ~15G 降到 3–5G 且不再涨回 60G。

**Architecture:** 分三条主线推进。先做零风险的 profile + sccache 配置（改 `Cargo.toml` / `.cargo/config.toml`），再做核心的测试合并（`tests/*.rs` 按前缀归入 6 个入口二进制，原文件移入同名子目录作为 `mod`），最后加清理脚本。每组测试迁移后独立验证，全部完成跑完整 `cargo test` 确保 88 个测试逻辑全绿。

**Tech Stack:** Rust 1.95.0 (edition 2024)、Cargo、单 crate（lib `cadence_aria` + bin `aria`）、tokio、axum、sccache（可选）。

**对应设计文档:** `cadence/designs/2026-05-30_技术方案_target目录瘦身_v1.0.md`

## 第二阶段：集成测试分组合并（核心）

> **通用迁移机制（每组套用，不要偏离）**
>
> 以入口 `it_<group>` 为例，把属于该组的若干 `tests/<name>.rs` 文件合并为一个二进制：
>
> 1. 建子目录：`mkdir -p tests/it_<group>`
> 2. 移动文件：对该组每个 `<name>.rs`，`git mv tests/<name>.rs tests/it_<group>/<name>.rs`
> 3. 修复 `include_str!` / `include_bytes!` 等**编译期相对路径**（仅下列 6 个文件涉及，见各任务说明）：
>    - `fixtures/...` → `concat!(env!("CARGO_MANIFEST_DIR"), "/tests/fixtures/...")`
>    - `../src/...` → `concat!(env!("CARGO_MANIFEST_DIR"), "/src/...")`
>    - **运行期相对路径**（字符串字面量如 `"tests/fixtures/..."` 传给函数的，不在 `include_*!` 宏里）**保持不动**——`cargo test` 的 CWD 始终是 crate 根。
> 4. 处理 `mod support;`：若该组有文件引用 support，在子目录建桥接文件 `tests/it_<group>/support.rs`，内容仅一行：`#[path = "../support/mod.rs"] ... ` —— 见 Task 8 的 support 共享方案；引用 support 的文件把开头的 `mod support;` 删除（改由入口统一声明）。
> 5. 写入口文件 `tests/it_<group>.rs`：为该组每个文件写一行 `#[path = "it_<group>/<name>.rs"] mod <name>;`；若该组用到 support，额外加 `mod support;`（指向桥接文件）。
> 6. 验证：`cargo test --test it_<group> -- --list 2>&1 | tail` 能列出测试，再 `cargo test --test it_<group>` 全绿。
> 7. Commit。
>
> 注意：模块名（`mod <name>`）用文件名去掉 `.rs`。各 `mod` 命名空间独立，顶层 `fn` 不会冲突（已验证）。

### Task 10: 编写 target 清理脚本

**Files:**
- Create: `scripts/clean-target.sh`

- [ ] **Step 1: 创建清理脚本**

Create `scripts/clean-target.sh`:
```bash
#!/usr/bin/env bash
# target 目录清理工具。三档：--stale / --all / --dry-run
# --stale   : 清 incremental 全部缓存；deps 保留（仅 incremental 是纯累积缓存）
# --all     : 等价 cargo clean（全删 target）
# --dry-run : 只报告将释放多少空间，不删除
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"

mode="${1:---dry-run}"

size_of() { du -sh "$1" 2>/dev/null | awk '{print $1}'; }

case "$mode" in
  --all)
    echo "[clean-target] cargo clean（全删 target）..."
    cargo clean
    echo "[clean-target] 完成。"
    ;;
  --stale)
    inc="target/debug/incremental"
    if [ -d "$inc" ]; then
      echo "[clean-target] 删除增量编译缓存：$inc（当前 $(size_of "$inc")）"
      rm -rf "$inc"
    else
      echo "[clean-target] 无 incremental 缓存，跳过。"
    fi
    echo "[clean-target] 完成。deps 当前 $(size_of target/debug/deps 2>/dev/null)。"
    echo "[clean-target] 提示：deps 陈旧产物可用 'cargo clean' 整体回收后重建。"
    ;;
  --dry-run)
    echo "[clean-target] 干跑（不删除）："
    echo "  target 总计       : $(size_of target)"
    echo "  deps              : $(size_of target/debug/deps)"
    echo "  incremental(可清) : $(size_of target/debug/incremental)"
    echo "  --stale 将释放 incremental 部分；--all 释放全部 target。"
    ;;
  *)
    echo "用法: $0 [--stale|--all|--dry-run]" >&2
    exit 2
    ;;
esac
```

> 设计取舍：`--stale` 仅清 `incremental/`（纯累积缓存，删后安全）。deps 里的陈旧 hash 产物难以安全区分"最新一组 vs 旧产物"（Cargo 不暴露映射），强行删可能误伤当前产物导致全量重编。因此 deps 的回收交给 `--all`（cargo clean）。这比设计文档初稿设想的"deps 选择性删"更安全可靠。
