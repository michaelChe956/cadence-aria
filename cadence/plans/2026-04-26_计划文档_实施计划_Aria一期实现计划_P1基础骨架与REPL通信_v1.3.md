# Aria Phase 1 P1 Implementation Plan

**文档信息**
- **创建日期**：2026-04-26
- **版本**：v1.3（研发可落地性 Review 最终修正版）
- **修正内容**：修复 B-04 `openspec_bootstrap_status` 独立枚举、B-05 `N00/N01` pre-task handler 建模、B-06 `last_seen_event_id` 位置与 `created_at`->`occurred_at` 映射；补齐 H-01 扩展命令面（`list_artifacts/approve_gate/reject_gate/reply_gate`）、H-02 逐命令 payload struct、H-03 daemon discovery 任务、H-04 `intake_brief` artifact materialize 规则；补充 M-03 P1 DoD 作为 P2 准入、M-05 event registry 扩展、M-06 snake_case 全量清理。

> **自动化 agent 执行提示**：agentic worker 执行本计划时使用 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans`；人类研发按任务、测试命令和完成判定执行即可，不依赖这些 skill。

**Goal:** 建立 Aria 一期的 Rust 工程骨架、`N00-N03` 入口链、REPL <-> daemon wire protocol、transport、daemon discovery、daemon session 状态与 checkpoint 基线。

**Architecture:** 先把最底层通信、进程边界和任务入口链定死，再实现最小可联调的 REPL/daemon 骨架。P1 不实现 planning/execution 业务，但必须负责 `new_task -> intake_brief -> EpicTask -> effective_policy/change_id` 这条进入系统的正式路径；P2 在此基础上补齐 OpenSpec 文件结构化写入和 bundle 编译。P1 新增 daemon discovery / metadata / lock / auto-start 落地，确保 REPL 可稳定发现、连接或自动启动 daemon。

**Tech Stack:** Rust、Cargo、serde、Unix socket / named pipe、JSON、basic fs persistence。

## 1. 范围与出口

P1 完成后，必须满足：

1. 仓库具备可编译的 Rust 工程骨架
2. REPL 与 daemon 可以通过正式 wire protocol 建立连接
3. 支持 `hello/attach/subscribe/get_status/detach`，以及 `list_artifacts/approve_gate/reject_gate/reply_gate` 最小 handler
4. 支持 `new_task` 形成 `intake_brief`、EpicTask runtime state、`effective_policy` 与稳定 `change_id`
5. daemon 能写最小 checkpoint / runtime snapshot
6. REPL 可接收 event 流并支持 reconnect 基线
7. REPL 可通过 daemon discovery 自动发现 socket、读取 `daemon.json`、处理 stale lock、自动启动 daemon
8. `intake_brief` 按 artifact materialize 规则落盘，生成 `ArtifactRef` 与 `artifact.materialized` event

不在 P1 范围内：

- provider 调用
- projection / `_aria`
- OpenSpec 文件结构化写入、bundle 编译和 coverage 校验
- planning / execution 节点业务逻辑

### Task 2: 定义 REPL wire protocol schema

**Files:**
- Create: `src/protocol/repl_wire.rs`

- [ ] **Step 1: 写失败测试，覆盖 envelope 与命令 schema**

- [ ] **Step 1.1: 固定 envelope 序列化命名**

要求：
- request / response / event envelope Rust 类型使用 `serde(rename_all = "snake_case")`
- JSON 字段必须与 `实现总契约_v1.0` 第 7.2 章一致：`protocol_version`、`message_type`、`request_id`、`event_id`、`command`、`event_type`、`sent_at`、`occurred_at`、`ok`、`payload`、`error`
- 业务 payload 若引用第 4 章业务对象，也统一按 `snake_case` 序列化；envelope、projection、bundle、provider DTO、runtime artifact 不再维护两套字段命名规则

- [ ] **Step 2: 实现请求、响应、事件、错误对象**

必须包含命令与对应 Rust payload struct（每个 command 独立 struct，不得用裸 `serde_json::Value` 透传）：

| Command | Request Payload Struct | 成功响应 Payload 字段 |
|---------|------------------------|----------------------|
| `hello` | `HelloRequest { last_seen_event_id: Option<u64> }` | `daemon_session_id: String`、`protocol_version: String` |
| `attach` | `AttachRequest { reconnect_token: Option<String> }` | `reconnect_token: String`、`replay_cursor: Option<u64>` |
| `subscribe` | `SubscribeRequest { event_types: Option<Vec<String>> }` | `subscription_id: String` |
| `new_task` | `NewTaskRequest { request_text: String, requested_change_id: Option<String> }` | `task_id: String`、`phase: String`、`intake_ref: String`、`change_id: String` |
| `get_status` | `GetStatusRequest { task_id: Option<String> }` | `session_id: String`、`tasks: Vec<TaskSummary>`、`latest_event_id: u64` |
| `list_artifacts` | `ListArtifactsRequest { task_id: String, artifact_kind: Option<String> }` | `artifacts: Vec<ArtifactRef>` |
| `approve_gate` | `ApproveGateRequest { gate_id: String }` | `gate_id: String`、`resolution: String`、`next_route: Option<String>` |
| `reject_gate` | `RejectGateRequest { gate_id: String, reason: Option<String> }` | `gate_id: String`、`resolution: String`、`next_route: Option<String>` |
| `reply_gate` | `ReplyGateRequest { gate_id: String, reply_text: String }` | `gate_id: String`、`resolution: String`、`next_route: Option<String>` |
| `detach` | `DetachRequest {}` | `detached: bool` |

要求：
- 每个 request payload struct 必须实现 `Serialize + Deserialize`，且带 `serde(rename_all = "snake_case")`
- 每个 response success payload struct 同样必须实现 `Serialize + Deserialize`，带 `serde(rename_all = "snake_case")`
- `error` object 统一使用 `WireError { code: String, message: String, details: Option<serde_json::Value> }`
- P1 对 `list_artifacts/approve_gate/reject_gate/reply_gate` 可只做 schema 与最小 handler 骨架，不要求完整业务逻辑；但 envelope 必须能正确路由到对应 command，且响应字段必须稳定

- [ ] **Step 2.1: `last_seen_event_id` 位置裁定**

按评审后裁定（B-06）：
- `last_seen_event_id` 放在 `hello` 请求 payload 中，不在 `attach` 中重复携带
- `attach` 响应带回 `replay_cursor`（即 daemon 侧 `latest_event_id`）
- reconnect 时 REPL 先发送 `hello` 带 `last_seen_event_id`，daemon 判定是否可 replay；若窗口丢失，在后续 `attach` 响应前已通过 `hello` 响应的 error 返回 `replay_window_lost`

- [ ] **Step 2.2: event log 时间字段映射**

按评审后裁定（B-06）：
- event log 落盘记录使用 `created_at`（磁盘 JSONL 字段名）
- wire envelope event 使用 `occurred_at`（REPL 可见字段名）
- daemon 从 event log 读取后发送到 REPL 时，将 `created_at` 映射为 `occurred_at`，不得在同一 envelope 中同时出现两个字段
- `sent_at` 只用于 request/response envelope，不用于 event

- [ ] **Step 3: 加入序列化 / 反序列化边界校验**

要求：
- 非法 `message_type` 报错
- response 必须带回 `request_id`
- response 必须回显 `command`
- request envelope 的 `command` 与 `payload.command` 不一致时报 `invalid_request`
- 失败 response 必须为 `ok=false` 且带 error object
- event 必须带 `event_id`
- event 必须带 `event_type` 和 `occurred_at`

- [ ] **Step 4: 加入 event type registry 测试**

- [ ] **Step 1: 写失败测试，覆盖 daemon discovery 核心路径**

测试断言：
- 解析 workspace root 后计算 `workspace_hash`（canonical path SHA-256 前 12 位）
- Unix socket 路径：`/.aria/runtime/daemon/<workspace_hash>/daemon.sock`
- `daemon.json` 可读且包含 `daemon_session_id`、`pid`、`socket_path`、`started_at`、`protocol_version`
- `daemon.lock` 存在且 PID 存活时，REPL 判定 daemon 运行中
- `daemon.lock` 存在但 PID 不存在或 socket 不可连接时，REPL 判定 stale daemon 并允许重新启动
- 未找到 daemon 且未传 `--no-start` 时，REPL 自动执行 `aria daemon run` 并等待 ready
- `--no-start` 时未找到 daemon 返回 `daemon_not_found` 错误

- [ ] **Step 2: 实现 daemon 侧 discovery 模块**

要求：
- `src/daemon/discovery.rs` 负责：
  - daemon 启动时创建 `.aria/runtime/daemon/<workspace_hash>/` 目录
  - 写入 `daemon.lock`（含 PID）
  - 写入 `daemon.json`（最小字段见补齐规格第 7.3 章）
  - 创建 Unix socket / named pipe
  - 注册 graceful shutdown 钩子，退出时删除 lock 和 socket（crash 场景由 REPL stale lock 检测处理）
- 环境变量优先级：`ARIA_DAEMON_SOCKET` > `--socket` > 默认路径

- [ ] **Step 3: 实现 REPL 侧 discovery 模块**

要求：
- `src/repl/discovery.rs` 负责：
  - 解析 workspace root 并计算 `workspace_hash`
  - 按优先级查找 socket：`ARIA_DAEMON_SOCKET` > `--socket` > 默认路径
  - 读取 `daemon.json` 验证 `protocol_version`
  - 检查 `daemon.lock` PID 是否存活
  - 检查 socket 是否可连接
  - stale lock 处理：删除旧 lock、旧 socket、旧 `daemon.json`，触发 auto-start（除非 `--no-start`）
  - auto-start 时 spawn `aria daemon run`，轮询 `daemon.json` 和 socket 就绪，超时返回 `daemon_start_timeout`

> **B-04 裁定**：`openspec_bootstrap_status` 使用独立枚举 `OpenSpecBootstrapStatus { bootstrap_pending, bootstrapped }`，不复用 `BundleStatus`。`BundleStatus` 继续用于 `OpenSpecConstraintBundle.bundle_status`，表达 bundle readiness（bootstrap_pending / ready / stale / blocked）。

上述函数是 P1 入口链的内部 helper；daemon 状态机对外调用 runtime unit 时，仍统一走 async `RuntimeUnit::execute(input, ctx).await`，不得为 `N00/N01/N02/N03` 单独发明不同 handler 签名。

> - 替代方案（可选）：若团队认为 pre-task 条件空字段不可接受，允许 `N00/N01` 不走 `RuntimeUnit`，从 `N02` 开始统一；但必须在代码中显式注释并保持一致
