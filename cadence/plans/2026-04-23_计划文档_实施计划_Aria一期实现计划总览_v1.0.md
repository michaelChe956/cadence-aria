# Aria Phase 1 Implementation Plan Index

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 Aria 一期实现拆成研发可以分工执行、按里程碑验收的总计划与 4 个子计划。

**Architecture:** 一期实现按 4 条主线推进：基础骨架与 REPL/daemon 协议、产物与约束、provider 与规划链、执行与最终收口。总 plan 负责定义依赖和里程碑，具体执行以子 plan 为准。

**Tech Stack:** Rust、Cargo、JSON、Markdown、Markdown AST / serde 文档操作、Unix socket / named pipe、git worktree、用户本机 BYO CLI provider。

## 1. 为什么拆成多个 plan

本次 implementation plan **必须拆成多个子 plan**，原因不是文档管理，而是工程边界：

1. REPL/daemon wire protocol 是独立接口面，先定好才能并行开发。
2. canonical artifact / projection / `_aria` / OpenSpec bundle / traceability 是独立的数据面，不能散落在节点实现里。
3. provider contract 与 planning nodes 是独立执行面，可以先用 fake provider 做通路验证。
4. execution / integration / final closure 是独立收口面，依赖前面三层但本身可以作为单独里程碑验收。

如果只写一个总 plan，结果会是：

- 所有负责人都抢同一份文档
- 依赖边界变模糊
- 实施顺序和并行边界说不清

所以采用：

- **1 份总览 plan**
- **4 份子 plan**

### 3.1 基线顺序

### 3.2 允许的受控并行

若团队有足够人手，可以按以下边界受控并行：

1. **P2 可以在 P1 Task 4 完成后启动**
   - 前提：`repl_wire.rs`、基础 protocol 类型、Cargo 工程骨架、`new_task` 入口链、`changeId` 固化规则已建立，且补齐规格中的 Projection / OpenSpec / parser / fixture 规则已锁定
2. **P3 可以在 P2 Task 6 完成后启动**
   - 前提：projection、`_aria`、phase1 profile validator、OpenSpec constraint bundle、traceability binding 已可用
3. **P4 可以在 P3 Task 5 完成后启动**
   - 前提：`dispatch_package` 与 `dispatch_package._aria.worktask_routing[]` 已稳定

禁止的并行方式：

1. 在没有 wire protocol 的情况下先做 REPL 与 daemon UI 行为
2. 在没有 `new_task -> intake_brief -> EpicTask -> effectivePolicy/changeId` 入口链的情况下直接做 `N04-N12`
3. 在没有 projection / `_aria` 的情况下直接写 planning / execution 节点
4. 在没有 `dispatch_package._aria.worktask_routing[]` 的情况下直接做 execution nodes

## 4. 共享前置约束

所有子计划都必须遵守以下前置约束：

1. 以上游 canonical 协议文档为节点、路由、最小字段真相源
2. 以 `Aria一期MVP精简设计_v1.2` + `Aria一期实现总契约_v1.0` + `Aria_IO协作协议与Provider契约_v1.0` 为 phase1 implementation profile 真相源；**以 `Aria一期评审后实施规格补齐_v1.2` 为代码级实施规格补齐源**
3. 一期必须坚持：
   - Aria 单一正式收口
   - OpenSpec 全程约束、非可选
   - Superpowers 只约束 provider 内部工作纪律
4. 所有机器可消费信息必须走：
   - Document Operation
   - projection
   - `_aria`
   - runtime record
5. Markdown / JSON / YAML 文档创建、更新、归一化和校验必须走 Document Operation 层；ast-grep 只能作为可选结构搜索、lint、codemod 工具，不是 Markdown 主编辑引擎
6. Claude Code / Codex 是用户本机 BYO CLI；Aria 不内置 provider、不托管远程模型、不自动安装 CLI，只做 capability probe、compatibility matrix 和受控 spawn
7. 不允许直接靠 Markdown 原文或 provider stdout 做 routing / coverage 判定

## 5. 横切节点覆盖矩阵

4-22 节点总目录定义了 `X01-X09` 共 9 个横切节点。以下矩阵明确各横切节点在 P1-P4 中的落位与最低验收要求：

| 横切节点 | 协议语义 | 计划落位 | 最低验收 |
|----------|----------|----------|----------|
| `X01` | `approval_gate_open` | P1 / P4 | gate open event、gate registry、恢复 |
| `X02` | `approval_gate_resume` | P1 / P4 | approve / reject / reply 命令与恢复 |
| `X03` | `provider_run_dispatch` | P3 | ProviderRunRecord started event |
| `X04` | `provider_run_collect` | P3 | raw output、structured output、failed / completed event |
| `X05` | `artifact_validate` | P2 / P3 / P4 | canonical + phase1 profile validator 通过 |
| `X06` | `checkpoint_persist` | P1-P4 | 每节点 checkpoint / runtime snapshot 落盘 |
| `X07` | `runtime_recover` | P1 | reconnect / recovery smoke test |
| `X08` | `manual_intervention_hold` | P1 / P3 / P4 | reason code 与终态记录 |
| `X09` | `policy_override_apply` | P1 | effectivePolicy、降级记录、event / snapshot |

要求：

1. 即使某个横切能力在实现单元中被折叠，其 event、snapshot、checkpoint 和审计边界必须保留。
2. P1 必须覆盖 `X01/X02/X06/X07/X08/X09` 的最低验收。
3. P3 必须覆盖 `X03/X04/X05` 的最低验收。
4. P4 必须验证所有横切节点在最终闭环中可被触发和恢复。

## 6. 共享测试基线

四个子计划都应统一使用以下测试层次：

1. **unit tests**
   - protocol / wire schema
   - task intake / task init
   - document operation
   - projection compiler
   - bundle compiler
   - traceability binder
   - provider context builder
2. **integration tests**
   - REPL <-> daemon handshake
   - planning chain with fake provider
   - execution / integration / final closure with fake provider
3. **fixture-based tests**
   - fixture OpenSpec change
   - fixture canonical artifacts
   - fixture dispatch package / reports

测试文件布局规则：

1. 子计划中的 `cargo test --test <target>` 必须对应顶层 `tests/<target>.rs` 文件。
2. 共享 helper 放在 `tests/support/`，由各顶层 test target 引用。
3. 若后续改回嵌套测试文件，必须同步在 `Cargo.toml` 中声明 `[[test]]`，不能只移动文件路径。
