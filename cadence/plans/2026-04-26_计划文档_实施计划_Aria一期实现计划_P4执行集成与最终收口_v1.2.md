# Aria Phase 1 P4 Implementation Plan

**文档信息**
- **创建日期**：2026-04-26
- **版本**：v1.2（研发可落地性 Review 二次修正版）
- **修正内容**：明确 worktree scope 重叠与等待队列、N20 candidate commit daemon 执行步骤、N26 先更新 OpenSpec 再重编译 bundle 并生成新 dispatch package 的顺序。

> **自动化 agent 执行提示**：agentic worker 执行本计划时使用 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans`；人类研发按任务、测试命令和完成判定执行即可，不依赖这些 skill。

**Goal:** 建立 `N13-N28` 执行、集成与最终收口闭环，并确保 `final_review._aria.coverage_summary` 可以证明一期任务闭合。

**Architecture:** P4 先扩展 P3 已建立的 provider contract / prompt registry，使 `N16-N20/N24/N25-N27` 继续走实现总契约 §8.1 的统一执行链；再实现 worktask 注册与执行准备、coding/testing/review/rework、integration / final closure。执行面先在 fake provider 与 fixture repo 下跑通，真实 provider 接入复用同一 `ProviderContextPackage -> AdapterInput -> ProviderRunRecord` 路径。

**Tech Stack:** Rust、git worktree、fake provider、fixture repo、integration tests。

---

## 0. 评审后准入门槛

P4 启动前必须先落实 `cadence/designs/2026-04-26_技术方案_Aria一期评审后实施规格补齐_v1.4.md` 中与执行、集成和收口相关的裁定：

- 第 10 章：`_aria.traceability_refs` 生成算法与 coverage 判定
- 第 4.7 章：`NodeExecutionContract`、`WorkflowDisciplineSpec`、`NodePromptTemplateRef`、`ProviderContextPackage`、`RuntimeUnit` 统一 handler 契约
- 第 9 章：prompt render order、template manifest、provider structured output sentinel
- 第 11 章：`N13-N28` 的 `node_specific_fields`
- 第 12 章：worktree 路径、lease、并发锁
- 第 13 章：candidate commit、integration branch、N23 cherry-pick、rollback
- 第 14 章：approval gate 与 `N26` 阻断规则

P4 不允许直接把 worktree diff 合进主分支；一期固定使用 candidate commit + integration branch + rollback 记录。

## 1. 范围与出口

P4 完成后，必须满足：

1. `N16-N20/N24/N25-N27` 的 `NodeExecutionContract`、`WorkflowDisciplineSpec`、prompt template 和 context builder 映射已注册并测试通过
2. `N13-N15` 执行准备链可跑（含 `N13 worktask_register` 协议边界）
3. `N16-N19` coding/testing/review/rework 可跑
4. `M20 integration_prepare_impl` 覆盖 `N20/N21/N22`，`N20 ready` 前已生成 candidate commit，并由 `N22/N23` 消费
5. `N20-N24` 集成准备、队列、执行与验证可跑（含 `N22 integration_prepare` 与 `N23 integration_execute` 协议边界）
6. `N25/N26/N27/N28` 最终评审、补丁 followup、总结、closeout 可跑
7. `N26` followup dispatch 会更新 OpenSpec task delta、触发 bundle stale/recompile，并通过新约束覆盖校验
8. `final_review._aria.coverage_summary` 能输出 closed / uncovered / exempted

  - `N20` 判定 ready 前必须校验 worktree diff 只落在 `allowed_write_scope`

  - candidate commit 创建由 daemon 直接执行，不经过 provider；provider 不得执行 `git add` / `git commit`
  - `git add` 只能覆盖授权范围内已通过 diff 校验的文件，禁止 `git add .`
  - `N20` 必须记录 `candidate_commit_sha` 到 worktask runtime state、`N20` runtime snapshot 和后续 `N22` 输入
  - 若 candidate commit 创建失败，`N20` 不得进入 ready，必须返回稳定错误码并按 policy 进入 gate 或 manual intervention；至少覆盖 `candidate_commit_empty_diff`、`candidate_commit_author_missing`、`candidate_commit_hook_failed`、`candidate_commit_signing_failed`、`candidate_commit_submodule_dirty`、`candidate_commit_lfs_failed`
  - candidate commit 错误码测试不得依赖用户本机 git hook、GPG/SSH signing 或 LFS 配置；`hook_failed` 用 fixture repo 局部 hook 或 mock git command 覆盖，`signing_failed` 用 fixture repo 局部 git config 覆盖
  - `N21` 负责 ready worktask 入队和队列位置记录
  - 从 ready worktask 聚合输入
  - 读取 `N20` 写入的 `candidate_commit_sha` 并写入 integration record，不可绕过

  - integration branch 必须在独立 integration worktree 中创建或复用，不可绕过
  - **preflight 检查**（降低集成阶段返工成本）：
    - integration worktree 不存在时按 `<repo_parent>/.aria-worktrees/<repo_name>-<workspace_hash>/<task_id>/integration/` 创建
    - `workspace_hash` 固定为 repo root canonical path 的 SHA-256 前 12 位，必须与 daemon socket 路径使用同一算法
    - integration branch 不存在时在 integration worktree 中创建
    - candidate commit 不存在时报明确错误
    - `pre_merge_sha` 记录失败时不得进入 `N23`
    - repo root 当前 branch、index、working tree 在 `N22/N23/N24` 后必须保持不变
