## 4. 问题清单

### 4.1 P0：front phase 未真实调用 Claude Code 生成需求与计划

- 当前 `start` 阶段只是由本地代码模板化生成 `spec-artifact.md` 和 `plan-brief.md`。
- 代码证据：
  - [front-phase-orchestrator.ts](../../src/runtime/orchestrator/front-phase-orchestrator.ts)
  - [spec-artifact.ts](../../src/runtime/reports/spec-artifact.ts)
  - [plan-brief.ts](../../src/runtime/reports/plan-brief.ts)
- 这意味着当前流程并没有真实验证 `Claude Code + OpenSpec + superpowers` 在需求和计划阶段的协同产出能力。

### 4.2 P0：需求在 handoff 前被错误收窄

- 用户原始业务目标是“实现登录功能，前端+后端，无数据库，前端可登录成功”。
- 进入 spec 和 dispatch contract 后，目标被替换为“formal flow 最小闭环”。
- 这导致后续 exec 阶段处理的不是原始需求，而是一个演示性目标。

### 4.3 P0：exec prompt 阻止 Codex 执行真实实现

- 真实任务 007 的 exec prompt 明确禁止：
  - 读取仓库规则与文件
  - 执行命令
  - 修改文件
- 在这种约束下，Codex 不可能完成任何实际功能开发。
- 如果系统仍允许该任务进入 `verified`，说明当前状态推进条件不可信。

### 4.4 P0：review/test prompt 阻止 Claude Code 进行真实校验

- 真实任务 007 的 review/test prompt 同样明确禁止读取文件、执行命令、修改文件。
- 这意味着 review/test 的“通过”无法证明工件一致性、代码正确性、命令执行结果或业务功能可用性。

### 4.5 P0：`verified` 判定缺乏业务落地约束

- 当前任务在 `changed_files: []` 的情况下仍然进入了 `verified`。
- 对于“代码实现型任务”，这应当被视为直接失败，至少不能进入最终通过状态。

### 4.6 P1：缺少正式文档沉淀

- 当前流程只会在 `cadence/cache/aria/tasks/<taskId>/artifacts/` 下生成运行时工件。
- 没有任何逻辑将确认后的需求、设计、计划沉淀到项目正式目录，例如：
  - `cadence/docs/`
  - `cadence/designs/`
  - `cadence/plans/`
- 这会导致：
  - 需求不可复用
  - 设计不可评审
  - 计划不可追踪

### 4.7 P1：官方验证脚本只验证工件链路，不验证业务功能

- 当前 `verify-real-integration.sh` 可以在任务 007 上给出通过结果。
- 但它并未发现“无代码改动、无功能落地”的业务失败事实。
- 因此，该脚本当前只能证明“运行时工件闭环成立”，不能证明“真实开发闭环成立”。

## 5. 修复优先级

### 5.1 第一优先级

- 修正运行时实际 prompt 与源码期望不一致的问题，先统一真实运行产物和当前源码逻辑。
- 将 front phase 改为真实调用 Claude Code 生成 spec/plan，而不是本地模板伪造。
- 确保用户原始业务目标在 spec、plan、dispatch contract 中完整保留，不允许被固定文案替换。
- 放开 exec/review/test 所需的最小权限：
  - 允许读取工件
  - 允许读取仓库
  - 允许执行必要命令
  - 允许在白名单范围内修改文件

### 5.2 第二优先级

- 收紧状态机和最终状态判定：
  - 对“实现型任务”，`changed_files` 为空时不得进入 `verified`
  - 验证命令未实际运行时不得进入 `verified`
  - 业务验收点未满足时不得进入 `verified`
- 要求 review/test 在无法执行命令时给出 `failed`，而不是允许静默通过。

### 5.3 第三优先级

- 增加正式文档沉淀能力：
  - 需求文档输出到 `cadence/docs/`
  - 方案设计输出到 `cadence/designs/`
  - 计划文档输出到 `cadence/plans/`
- 升级 `verify-real-integration.sh`，加入业务落地校验，而不是只检查工件存在性与字段完整性。

## 6. 回归测试点

- `start` 后应能看到真实 Claude Code 生成的 spec/plan，且内容保留业务目标关键字。
- `confirm-plan` 后的 dispatch contract 中，`goal` 不得退化为固定模板文案。
- exec prompt 必须明确要求读取 spec/plan/contract，并允许在白名单范围内进行实际开发。
- review/test prompt 必须要求真实读取工件、读取代码、执行命令。
- 对实现型任务，如果 `changed_files` 为空，任务不得进入 `verified`。
- 至少保留一个真实端到端样例，要求：
  - 在目标仓库中生成前端和后端代码
  - 通过真实验证命令
  - 满足业务验收点
  - 最终再进入 `verified`
