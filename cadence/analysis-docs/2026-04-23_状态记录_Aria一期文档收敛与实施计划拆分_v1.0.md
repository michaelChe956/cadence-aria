## 2. 当前已经确定的关键结论

### 2.1 一期真相源分层

1. **canonical 协议层**：上游协议文档集
2. **phase1 implementation profile**：
   - `Aria一期MVP精简设计_v1.2`
   - `Aria一期实现总契约_v1.0`
3. **研发导读 / 实施计划层**：
   - 导读文档
   - 总 plan
   - P1-P4 子 plan

### 2.2 一期机器可消费数据承载方式

1. Markdown canonical artifact 通过 projection 提供机器索引：
   - `SpecProjection`
   - `DesignProjection`
   - `PlanProjection`
2. JSON canonical artifact 通过 `_aria` 提供实现期扩展：
   - routing
   - traceability
   - coverage
   - provider run refs
3. daemon 负责：
   - bundle 编译
   - projection 编译
   - binding 生成
   - 状态推进与最终收口

### 2.3 OpenSpec 的定位已经定死

1. 一期 **强约束、非可选**
2. 每个 EpicTask 必须绑定 `changeId`
3. 缺 `proposal/spec/design/tasks` 任一关键文件时，不允许静默降级
4. OpenSpec 不替代 Aria 正式产物，不单独维护第二套状态机

### 2.4 implementation plan 拆分方式已经定死

采用：

- 1 个总 plan
- 4 个子 plan

原因：

1. 协议面
2. 数据面
3. provider / planning 面
4. execution / closure 面

这 4 条线边界清楚、依赖清楚、可独立验收。
